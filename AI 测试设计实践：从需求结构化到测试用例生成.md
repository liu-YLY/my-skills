一、背景与目标
近期在 XXXX 项目迭代过程中，我持续探索并优化 AI Agent Skills 在测试设计领域的落地实践，逐步构建了一套基于 AI 的「需求解析 → 测试点生成 → 测试用例生成」工作流。
整体目标包括：
● 提升复杂需求场景下的测试设计效率
● 降低人工拆解测试点与编写用例的重复性工作
● 减少需求理解偏差与测试遗漏风险
● 提高业务规则、状态流转与边界场景的覆盖率
● 统一测试用例结构与 YAML 输出规范
● 提升测试用例的可评审性、可维护性与可自动化映射能力
● 为后续 AI 测试、自动化平台集成与测试体系智能化建设打基础
目前已经初步形成一套完整的 AI 测试设计工作流：
需求文档
→ Requirement Parser（需求结构化）
→ Testpoints Generator（测试点生成）
→ Testpoints To Testcases（测试用例生成）
→ YAML 标准化输出

二、早期实践中的问题
直接让 AI 从需求生成测试用例
最开始的思路其实很简单：
需求文档
→ AI
→ 测试用例
但实际效果并不好，生成的测试用例质量非常不稳定。后来发现，问题并不是模型不够强，而是任务跨度太大。
本质上，相当于让 AI 同时完成：
● 需求理解
● 规则提取
● 状态建模
● 场景拆分
● 边界分析
● 用例设计
● YAML 结构化输出
最终 AI 很容易退化成“最低满足策略”。
大量漏测
AI 更容易关注：
● 页面展示
● 按钮点击
● UI 行为
但容易遗漏：
● 状态流转
● 配置联动
● 权限控制
● 通知链路
● 业务规则
● 数据一致性
例如：
Toggle 开启后用户退订则不发送
AI 很可能只生成：
● Toggle 开启
● Toggle 关闭
却遗漏：
● 用户退订场景
● 邮件不发送逻辑
● 状态组合验证
● 配置联动验证

生成很多“看起来很多，实际没价值”的用例
例如：
title: 验证 Toggle 可以关闭
steps:
  - 刷新页面
expected_results:
  - 页面行为符合预期
这种用例的问题非常明显：
● 不可执行
● 不可评审
● 无法自动化
● 没有真实业务价值
后来发现：
AI 非常喜欢生成“低信息量内容”。
因为这是最安全、成本最低的输出方式。

AI 容易脑补需求
例如需求没有提到：
● Toast
● 接口报错
● Loading
● 重试逻辑
AI 会自己“补”，这会导致：
● 用例与真实需求不一致
● 评审困难
● 开发质疑测试设计
● 测试范围失控

三、核心转变：问题不在模型，而在输入结构
后面逐渐意识到：
AI 生成质量，80% 取决于输入结构。
真正的问题并不是：
需求 → 测试用例
而应该是：
需求
→ 结构化需求分析
→ 测试点
→ 测试用例
本质上是：
● 分阶段思考
● 分层建模
● 降低单次任务复杂度
类似于：
“先列提纲，再写文章”
覆盖率和稳定性会明显提升。

四、Requirement Parser 的价值与作用
后来开始先做 Requirement Parser。先不生成测试用例，而是先将需求结构化。
输出包括：
● 功能模块
● 状态规则
● 业务规则
● 默认值规则
● 配置联动规则
● 异常规则
● UI 规则
● 数据规则
● 风险分析
● 待确认项

实际效果提升非常明显
例如原始需求：
Toggle 开启后用户退订则不发送
AI 很容易漏掉“退订”。
但结构化后：
业务规则:
- Toggle 开启
- 用户退订
- 不发送
AI 就能稳定生成：
● Toggle 开启测试
● Toggle 关闭测试
● 用户退订测试
● 邮件发送测试
● 状态组合测试
● 配置联动测试

# Requirement Parser 协议

## 1. 目标

- 将不同来源的需求输入统一解析为以下几点的标准化需求文档。
  - 清晰
  - 结构化
  - 可评审
  - 可追溯
  - 可测试
  - 可直接生成测试点
- 生成结果必须适用于以下几点
  - 测试点生成
  - YAML 测试用例生成
  - 自动化测试设计
  - 需求评审
  - 开发联调
  - 回归分析

---

## 2. 输入来源识别（强制）

- 解析需求前，必须先识别输入来源类型。
- 包括但不限于：
| 来源类型 | 示例 |
|---|---|
| PRD | Markdown / Word / PDF |
| 钉钉文档 | 在线需求文档 |
| 飞书文档 | 在线协作文档 |
| Confluence | Wiki 文档 |
| Jira / Linear / Tapd | 需求卡片 |
| Figma | UI设计稿 |
| Dashboard 截图 | 后台页面 |
| 配置页截图 | Settings 页面 |
| 原型图 | Axure / 墨刀 |
| 流程图 | 用户流程 / 状态图 |
| Release Note | 版本说明 |
| 表格文档 | Excel / 配置表 |

- 不同来源必须使用对应解析策略。

---

## 3. 输出目标（强制）

- 输出必须统一为：# Requirement Spec Markdown
- 禁止：
  - 直接生成 YAML 测试用例
  - 直接生成 TC 编号
  - 直接生成自动化脚本

### 3.1 输出位置 (强制)

- 必须写入:releases/<版本或迭代目录>/需求/
- 禁止
  - 写入仓库根目录
  - 写入其他随机目录

---

## 4. 统一输出结构（强制）

- 输出结构必须尽量统一为：
```markdown
# 功能名称

## 功能目标

## 页面结构

## 功能模块

## 配置项

## 用户操作流程

## 状态规则

## 业务规则

## 数据规则

## 异常规则

## 权限规则

## UI规则

## 配置联动规则

## 默认值规则

## 待确认项
```
- 若某部分不适用必须明确标记：N/A或者不适用，禁止直接省略。

---

## 5. 文档结构识别（强制）

- 解析需求时必须先识别文档结构。
  - 包括但不限于：
    - 页面标题
    - 模块分组
    - Tab
    - 卡片区域
    - 配置区域
    - 表单区域
    - 按钮区域
    - 状态区域
    - Tooltip
    - 表格区域
    - 流程区域
    - 注释区域
    - Diff 区域
- 输出 Markdown 时必须尽量保留原始层级关系。
- 禁止：
  - 打散结构
  - 丢失模块归属
  - 输出无结构纯文本

---

## 6. Markdown 文档解析规则

- 若输入来源为：Markdown 需求
  - 则必须重点解析：
    - 文档层级
    - 功能模块
    - 配置项
    - 用户流程
    - 业务规则
    - Feature Flag
    - 状态流转
    - 权限规则
    - 数据逻辑
    - 异常逻辑
    - 配置联动
    - 默认值
- 必须保留：
  - 标题结构
  - 表格结构
  - 配置分组
  - 模块归属关系
- 禁止仅提取纯文本。

---

## 7. Figma / UI设计稿解析规则

### 7.1 输入来源识别 (强制)

- 若输入来源包含以下任一类型:Figma Link、Figma Dev Mode、UI设计稿
  - 则必须进入结构化需求解析模式。

### 7.2 Figma MCP / Skill 优先级规则 (强制)

- 当系统已安装:Figma MCP、Figma Skill时
  - 必须优先通过:Figma MCP或者Figma Skill获取结构化设计数据。
- 禁止:
  - 在已可访问 Figma MCP 的情况下仍完全依赖截图 OCR。

### 7.3 Figma MCP 数据获取规则

- 若检测到:Figma URL、node-id、Dev Mode Link、Frame Link
  - 则必须优先尝试通过 MCP / Skill 获取以下结构化信息:
    - 页面结构
    - UI组件
    - 配置结构
    - 页面关系
    - 状态信息

### 7.4 MCP 数据优先原则

- 需求解析优先级:
  1. Figma MCP 结构化数据
  2. Figma Skill 提取结果
  3. Dev Mode 信息
  4. 页面文字
  5. 截图 OCR
- 禁止使用低优先级数据覆盖高优先级结构信息。

### 7.5 页面结构输出要求 (强制)

- 解析结果中必须保留:
  - 页面层级 
  - 模块分组
  - 配置区域
  - 功能区域
  - Tab结构 
  - 弹窗结构
  - 页面联动关系
- 禁止将整个页面扁平化为纯文本。

---

## 8. 业务系统识别（强制）

- 解析需求时必须先识别系统类型。
  - 包括但不限于：
    - 通知系统
    - 邮件系统
    - 权限系统
    - 支付系统
    - 配置系统
    - Feature Flag 系统
    - 状态机系统
    - 搜索系统
    - 上传系统
    - 订单系统
    - Tracking 系统
    - Return 系统
    - OpenSearch 系统
    - 异步任务系统
    - 调度系统
- 识别后必须自动补充对应核心业务结构。

---

## 9. 通知 / 邮件系统专项规则

- 若需求涉及Email、Notification、Message、Subscription、Unsubscribe则必须解析：
  - 通知链路
    - 正常发送
    - 不发送
    - 重复发送
    - 延迟发送
    - 发送失败
  - 用户状态
    - 已订阅
    - 已退订
    - 黑名单
    - 无效邮箱
  - 配置影响
    - 全局开关
    - 子开关
    - 用户偏好
    - Feature Flag
  - 邮件行为
    - 收件成功
    - ESP拒收
    - 退订后行为

---

## 10. 状态机识别规则

- 若需求涉及Enable / Disable、Toggle、Status、Publish / Draft、Active / Inactive则必须解析：
  - 初始状态
  - 状态切换
  - 状态保存
  - 刷新后状态
  - 状态恢复
  - 状态互斥
  - 非法状态流转

---

## 11. 配置联动规则

- 若需求涉及全局配置、子配置、Feature Flag、多级配置、通知策略、权限配置则必须解析：
  - 上游配置影响
  - 下游行为变化
  - 配置覆盖关系
  - 配置优先级
  - 配置冲突
  - 配置恢复

---

## 12. 数据一致性规则

- 若需求涉及到保存、配置、状态切换、同步、通知、搜索、异步任务等必须覆盖以下几点之间的一致性验证
  - 页面状态
  - DB状态
  - 接口返回
  - 实际业务结果

---

## 13. UI规则识别

- 若需求涉及 UI必须解析：
  - 对齐
  - 间距
  - 字体
  - 颜色
  - 圆角
  - 布局
  - Hover状态
  - Disabled状态
  - 响应式布局
- 禁止仅提取文案。

---

## 14. 表格恢复规则（强制）

- 若图片或文档存在配置表、字段表、API表、权限矩阵、状态表、Diff表则必须：
  - 保持列顺序
  - 保持字段关系
  - 保持状态关系
- 禁止：
  - 打散表格
  - 丢失列名
  - 合并多行语义

---

## 15. 流程图解析规则

- 若输入包含流程图、状态图、用户流程、时序图则必须：
  - 保留节点顺序
  - 保留状态关系
  - 保留条件关系
- 推荐输出：
  - Markdown层级
  - Mermaid结构
- 禁止：
  - 仅提取节点文字
  - 丢失流程关系

---

## 16. OCR 不确定内容规则

- 若图片内容模糊、遮挡、分辨率低、无法识别则：
  - 使用 `[待确认]`
  - 或保留原始片段
- 禁止：
  - 脑补字段
  - 猜测业务逻辑
  - 自动补全文案

---

## 17. 测试友好型输出（强制）

- 输出结果必须有利于：
  - 测试点生成
  - YAML 用例生成
  - 自动化测试设计
  - 测试评审
- 因此必须尽量保留：
  - 功能模块
  - 配置项
  - 用户行为
  - 状态
  - 业务规则
  - 页面结构
  - 配置联动
  - 数据流转
- 禁止：
  - 输出纯 OCR 文本
  - 输出无结构内容
  - 丢失业务逻辑

---

## 18. 输出质量要求

- 生成结果必须：
  - 可直接评审
  - 可直接生成测试点
  - 可直接生成 YAML
  - 不依赖上下文猜测
  - 不重复
  - 不遗漏核心逻辑

---

## 19. 禁止事项（强制）

- 禁止：
  - 直接生成测试用例
  - 直接生成 YAML
  - 自动脑补业务规则
  - 擅自扩展产品逻辑
  - 合并多个业务目标
  - 输出低信息量内容
  - 仅输出 OCR 纯文本

---

## 20. 需求变更解析规则（强制）

若输入内容包含：
- Diff
- Release变更
- UI改版
- 配置新增
- 配置删除
- 字段修改
- 默认值变更
- Feature Flag调整
- 状态逻辑修改

则必须优先识别：
- 新增逻辑
- 删除逻辑
- 修改逻辑
- 受影响模块
- 受影响配置
- 受影响状态
- 受影响通知链路

必须输出：变更影响分析
包括：
- 影响页面
- 影响模块
- 影响配置
- 影响用户
- 影响通知
- 影响状态流转
- 回归范围建议

禁止：
- 将Diff需求按全新需求重新解析

# 测试点生成协议

## 1. 目标

- 将需求、PRD、功能说明、设计稿、流程描述拆解为测试点列表。
  - 清晰
  - 可执行验证
  - 可评审
  - 可追溯
  - 可直接转测试用例

- 测试点需具备:
  - 单一验证目标
  - 明确业务行为
  - 明确验证对象
  - 明确预期方向

- 生成结果应可直接供:
  - 人工评审
  - 自动化设计
  - testpoints-to-testcases

---

## 2. 覆盖维度 (强制)

输出中必须显式覆盖以下维度:

| 维度 | 是否必须 |
|---|---|
| 正常流程 | 必须 |
| 异常流程 | 必须 |
| 边界值 | 必须 |
| 状态流转 | 有状态时必须 |
| 权限/可见性 | 涉及权限时必须 |
| UI展示 | 涉及页面时建议 |
| 配置联动 | 涉及配置时必须 |
| 数据同步 | 涉及保存/同步时必须 |
| 响应式布局 | 涉及多端时建议 |
| 兼容性 | 涉及浏览器/App时建议 |

### 2.1 不适用项规则

- 若某维度不适用:
  - 必须明确标记:
    - `N/A`
    - 或“不适用”

- 禁止:
  - 整体省略异常场景
  - 整体省略边界场景

---

## 3. 输入要求

- 允许以下输入:
  - PRD
  - 需求文档
  - Jira / Linear / Tapd 描述
  - Markdown 文档
  - UI设计稿描述
  - 流程说明
  - 功能变更说明
  - Release Note

### 3.1 输入不足处理 (强制)

- 若需求存在明显缺失:
  - 不得脑补业务规则
  - 必须标记:
    - 待产品确认
    - 待需求补充
    - 规则未明确

### 3.2 禁止行为

- 禁止
  - 编造需求未描述的业务逻辑
  - 擅自增加产品规则
  - 将多个验证目标合并为一个测试点

---

## 4. 输出位置 (强制)

- 必须写入:releases/<版本或迭代目录>/测试点/
- 禁止
  - 写入仓库根目录
  - 写入其他随机目录

---

## 5. 输出格式 (强制)

- 默认使用 Markdown 表格格式。

### 5.1 推荐格式 (默认)

## 功能模块名

| 编号 | 测试点 | 类型 |
|------|--------|------|
| TP-01 | xxx | 正常 |
| TP-02 | xxx | 异常 |

### 5.2 字段说明

- 编号：模块缩写 + 序号
- 测试点：单条验证目标
- 类型：正常 / 异常 / 边界 / UI / 回归 等

### 5.3 编号规范

- 要求:
  - 同模块唯一
  - 顺序递增
  - 可追溯

### 5.4 描述颗粒度

- 描述测试点时，必须推测或指定 UI 交互方式（如：点击按钮、切换开关、输入文本、勾选复选框）。严禁输出只有目标、没有路径的模糊测试点。

## 6. 业务系统识别 (强制)

- 生成测试点前, 必须先识别需求所属系统类型。
  - 包括但不限于:
    - 通知系统
    - 邮件系统
    - 权限系统
    - 支付系统
    - 配置系统
    - 状态机系统
    - Feature Flag 系统
    - 任务调度系统
    - 搜索系统
    - 上传系统
    - 风控系统
    - 计费系统
  - 识别后:必须自动补充该系统对应的核心业务测试维度。

---

## 7. 通知/消息系统专项规则

- 若需求涉及: Email、Notification、Message、Subscription、Unsubscribe则必须覆盖以下几点
  - 通知发送链路
    - 正常发送
    - 不发送
    - 重复发送
    - 发送失败
    - 延迟发送
  - 用户状态
    - 已订阅
    - 已退订
    - 黑名单
    - 无效邮箱
  - 配置影响
    - 全局开关
    - 局部开关
    - Feature Flag
    - 用户偏好设置
  - 邮件行为
    - 收件成功
    - ESP 拒收
    - 退订后行为

---

## 8. 状态机规则

- 若需求存在:Enable/Disable、Active/Inactive、Toggle、Status、Publish/Draft、Start/Stop则必须覆盖以下几点
  - 初始状态
  - 状态切换
  - 状态保存
  - 刷新后状态
  - 状态互斥
  - 状态恢复
  - 非法状态流转

---

## 9. 测试点设计原则 (强制)

### 9.1 单一验证目标

- 每条测试点仅允许一个核心验证目标。
- 禁止：验证保存成功并且页面展示正确

### 9.2 禁止堆叠验证

- 禁止：
  - 一个测试点同时验证:
    - 正常 + 异常
    - 默认值 + 修改后
    - UI + 数据保存
    - 权限 + 展示

### 9.3 测试点必须可转测试用例

- 每条测试点必须:
  - 可直接生成 YAML 测试用例
  - 不依赖上下文猜测

### 9.4 要素完备性

- 每个测试点必须包含：[验证对象] + [操作/场景] + [预期结果方向]。
  - 禁止：验证 General updates 可以关闭 (太模糊)
  - 必须：验证 General updates 状态开关(Toggle)切换至关闭并保存后，配置项保持关闭状态 (包含了组件类型和保存动作)

---

## 10. 测试点内容规范 (强制)

- 测试点必须包含:
  - 验证对象
  - 操作行为 (若存在)
  - 预期方向

## 11. 异常场景生成规则 (强制)

- 涉及输入、保存、提交、上传时:必须生成异常测试点。
  - 包括但不限于:
    - 空值
    - 非法值
    - 超长值
    - 格式错误
    - 重复操作
    - 网络异常
    - 接口失败
    - 权限不足

---

## 12. 输出质量要求

- 生成的测试点必须:
  - 可直接评审
  - 可直接转 YAML
  - 不依赖额外解释
  - 不出现模板化空话
  - 不重复
  - 不遗漏核心场景

---

## 13. 禁止事项 (强制)

- 禁止：
  - 直接输出 YAML 测试用例
  - 直接生成 TC_ 编号
  - 自动补充产品未定义规则
  - 合并多个测试目标
  - 输出低信息量测试点
  - 省略异常或边界场景

禁止生成：
- 验证 XXX 可以点击
- 验证 XXX 可以关闭
- 验证 XXX 正常
- 检查 XXX 功能
- 查看 XXX 展示
这类缺少：
- 操作结果
- 状态变化
- 保存行为
- 业务影响
的低信息量测试点。

---

## 14. 核心业务链路识别 (强制)

- 生成测试点时必须优先识别以下几点
  - 核心业务链路
  - 主功能目标
  - 用户最终业务结果
- 测试点数量与覆盖应向核心业务倾斜。
- 禁止:
  - 大量生成低风险 UI 测试点
  - 页面展示类测试点明显多于核心业务测试点

- 例如通知系统的核心是:是否发送、是否正确发送、是否受退订影响
  - 而不是、Toggle UI、文案展示

## 15. 数据一致性规则

- 若需求涉及到保存、配置、状态切换、同步、通知、搜索、异步任务等必须覆盖以下几点之间的一致性验证
  - 页面状态
  - DB状态
  - 接口返回
  - 实际业务结果

---

## 16. 配置联动规则

- 若需求涉及到全局配置、子配置、Feature Flag、权限配置、多级设置等必须覆盖以下几点
  - 上游配置影响下游行为
  - 配置优先级
  - 配置覆盖关系
  - 配置冲突
  - 配置恢复

---

## 17. 测试点粒度规则

- 测试点必须保证一个测试点仅验证一个业务行为或一个状态变化或一个结果验证
- 禁止以下几点混合在同一测试点中:
  - 配置保存 + 页面展示
  - Toggle切换 + 邮件发送
  - UI展示 + DB验证
主要解决：
● 测试覆盖
● 场景拆分
● 边界分析
● 状态组合
核心价值：

● 单一验证目标
● 可评审
● 可执行
● 可直接转 YAML
的测试点。

# 测试点 → 测试用例 (YAML) 协议

## 0. 输入要求 (强约束)

输入必须是已拆解的测试点, 接受以下形式之一。

### 0.1 编号列表

```text
测试点:
1. xxx
2. xxx
```

### 0.2 Markdown 表格 (团队推荐, 与 `releases/.../测试点/*.md` 一致)

- 表头含编号, 测试点, 类型 (或同义列名)。
- 每一数据行 = 一条测试点, 不得合并单元格表达多条。

若用户只贴需求, 未给出测试点: 拒绝生成用例, 提示先使用 testpoints-generator 产出测试点。

---

## 1. 核心原则

- 一条测试点 → 至少一条测试用例 (不可遗漏)。
- 不得把多条测试点合并成一条用例。
- 不得扩展需求未给出的业务场景 (禁止脑补产品规则)。
- 允许在单条测试点含多个独立验证时, 拆分为多条 `test_cases` 。
- 所有测试用例必须:可执行、可评审、可自动化映射、可独立理解。

---

## 2. Metadata

每个测试用例文件只有一个Metadata

### 2.1 数据结构示例

```yaml
metadata:
  module: "PP Shopify"
  feature: "V26.04 Tracking page 多模板与 System notifications"
  owner: "cenyf"
  last_reviewed: "2026-05-06"
  tags:
    - "V26.04"
    - "Tracking page"
    - "System notifications"
```

### 2.2 映射关系

| metadata 字段  | test_cases 用途 |
|----------------|----------------|
| module         | 用例模块分组 |
| feature        | 用例场景定位 |
| owner          | 用例负责人 |
| tags           | 分类与检索索引 |
| last_reviewed  | 用例版本时间基线 |

### 2.3 tags 处理规则

- 去重
- 统一大小写
- 去空格
- 保留业务关键词

### 2.4 last_reviewed日期规则

- last_reviewed 必须使用当前生成日期（YYYY-MM-DD）
- 不允许使用历史日期

---

## 3. ID 与类型映射

- `id`: `TC_<功能缩写>_<三位序号>`, 按文件内递增, 全文件唯一。
- 表格类型列 → `priority` / `type` 参考 (可按风险调整, 不违背用户明示优先级):

| 测试点类型 (示例)   | 建议 type       | 建议 priority |
|-------------------|----------------|---------------|
| 正常/主流程/冒烟    | functional     | P0            |
| 异常/报错/校验      | functional     | P1            |
| 边界/极值          | functional     | P1            |
| UI/布局/文案       | ui             | P2            |
| 回归/历史遗留       | regression     | P1            |

---

## 4. Schema (强制)

- 所有字段与枚举严格符合 `schema/test-case-schema.json`。

---

## 5. YAML 输出模板

### 5.1 排版约定 (强制)

- 每一条 `test_cases` 用例之间必须保留 1 条空行, 方便人工阅读和评审。
- 字符串标量统一加双引号(含 `id`, `title`, `module`, 以及上述数组内每一项), 避免有的行带引号, 有的不带。
- `auto`: 布尔值 `true` / `false`, 不加引号。

### 5.2 数组格式 (强制)

- 以下字段必须使用块数组: preconditions、steps、expected_results、tags
- 列表项相对键名缩进 2 格, 例如 `steps:` 下一行 `      - "..."`。

---

## 6. Description 规范 (强制)

- 必须用完整自然句说明具体测试内容。
  - 需要包含测试操作场景, 目标行为和预期状态。
- description 应优先描述:
  - 核心业务行为
  - 页面变化
  - 用户感知结果
- 避免所有 description 使用完全相同句式。
- 禁止使用固定占位前缀或追溯性内容, 不得写 `"验证内容: xxx"`、`"来源测试点 TP-xxx"`、`"来源编号"`、`"测试点编号"` 等低信息量描述。

---

## 7. Tags 规范

- 禁止写测试点编号。
- 禁止包含 `TP-xxx`、`TC_` 等编号; 只保留章节、模块、类型、版本等可筛选标签。

---

## 8. Preconditions 规范 (强制)

- 若无特殊要求，默认写"无额外前置条件"。
- 若有上下文关联，需明确起始页面或者登录状态或者打开状态或者配置状态。
- Preconditions 与 Steps 不得重复。
  - 如果前置条件已写 `"已进入 xxx 页面"` 或 `"已打开 xxx 弹窗"`, steps 中不要再写 `"进入 xxx 页面"` 或 `"打开 xxx 弹窗"`。
  - steps 应从该前置状态之后的第一个实际操作开始。

---

## 9. Steps 规范 (强制)

- 使用主动语态（如“输入”、“点击”、“滑动”、“断开”）。
- 每一步仅包含一个核心动作。
  - 禁止:
    - "修改并保存"
    - "点击并跳转"
    - "输入后提交"
  - 应拆分:
    - "输入 xxx"
    - "点击 Save"
    - "查看页面展示"
- 步骤必须落到具体页面、控件、输入值、交互、配置项、用户动作。
  - 禁止仅使用刷新、查看、进入等动作替代核心操作。
- 推荐动作词：输入、点击、选择、切换、上传、删除、保存、拖拽、刷新、搜索、提交等。
- Steps 中涉及输入、选择、配置时:
  - 必须明确具体值
  - 禁止:
    - "输入测试数据"
    - "填写对应内容"
    - "选择目标配置"
- 禁止写 `"按测试点要求执行..."`、`"执行对应操作"`、`"验证相关功能"`、`"修改相关配置"`、`"调整对应设置"`、`"修改标题中描述的配置"` 等占位话术。
- 禁止以下内容作为独立步骤：`"查看 xxx 是否..."`、`"验证 xxx 是否..."`、`"检查 xxx 是否..."`、`"确认 xxx 是否..."`。
- 禁止以下内容作为唯一核心操作步骤：`"刷新当前页面"`、`"查看页面"`、`"进入页面"`。
- 若出现以上禁止的词汇，该用例判定为非法，必须根据业务逻辑重写为具体的视觉或数据描述。
- 若存在查看行为：前面必须已有明确用户操作。
- 若 Title 包含“修改/设置/开关/配置”，Steps 必须包含：1.定位组件 -> 2.交互动作 -> 3.点击保存/确定。
- 默认值验证： 若是验证默认值，Steps 必须是“进入页面 -> 查看 XX 元素的初始状态”，严禁出现修改动作。
- 如果我是测试执行人员，我看到这个 Steps，能否在不看需求文档的情况下知道点击哪个按钮、输入什么值？（如果不能，请重写 Steps）

---

## 10. Expected Results 规范 (强制)

- 基本要求：可观察、可判定、可验证。
- 必须包含明确可观察结果:页面展示、数据变化、UI变化、提示文案、跳转结果、保存状态、接口行为等。
- 禁止使用：`"结果正确"`、`"符合预期"`、`"配置正确"`、`"功能正常"`、`"应生效"`、`"与测试目标一致"`等无法直接判定的泛化描述。
- Expected Results 必须与 Steps 中的实际操作直接对应。
- 禁止生成 Steps 中未发生操作的预期结果。
- 不得扩展未执行的验证行为。
- 如果我是测试执行人员，我看到这个 Expected Results，能否直接判断测试 Pass 还是 Fail，而不需要主观理解“符合预期”的意思？（如果不能，请重写 Expected Results，描述具体的 UI 或数据表现）

---

## 11. Title 命名规范 (强制)

- 必须包含：验证对象、用户行为 (若存在)、预期结果
- 禁止例如：验证 Width、验证布局、验证 Rounded 为 16px等

---

## 12. 测试目标一致性 (强制)

- `title` / `description` / `steps` / `expected_results` 必须围绕同一个核心测试目标展开, 不得出现测试意图冲突。

- 若 `title` 或 `description` 为“默认值验证”、“默认展示”、“初始状态验证”等场景:
  - `steps` 中禁止出现 `"修改配置"`、`"编辑"`、`"保存修改"`、`"切换配置"` 等变更类操作。
  - `expected_results` 应验证默认值、默认展示、初始化状态。

- 若 `steps` 中包含修改、保存、编辑、切换等操作:
  - `title` 必须体现“修改后生效”、“配置更新”、“保存成功”等语义。
  - 不得继续使用“默认为”、“默认展示”等标题。

- 一个测试用例仅验证一个核心目标:
  - 默认值验证
  - 配置修改验证
  - 保存逻辑验证
  - 权限验证
  - 错误提示验证
  等场景应拆分为独立用例。

- 禁止在同一条用例中同时验证:
  - 默认状态 + 修改后状态
  - 首次加载 + 保存逻辑
  - 正常流程 + 异常流程

---

## 13. 用例拆分策略

以下场景必须拆分独立测试用例:

- 默认值验证
- 配置修改验证
- 保存后刷新验证
- 权限验证
- 异常输入验证
- 边界值验证

禁止为了减少用例数量而合并多个验证目标。

---

## 14. 边界测试约束 (强制)

- 若 `tags` 包含边界或者测试点类型为边界
  - 则 `Steps` 中必须体现：最小值、最大值、临界值、空值、超长值、极端布局中的至少一种。
- 禁止仅添加边界标签但缺少实际边界操作。

---

## 15. UI 类型用例约束

若 `type: "ui"`:

- Expected Results 必须包含至少一种:
  - 对齐
  - 间距
  - 字体
  - 颜色
  - 圆角
  - 布局
  - 响应式展示

- UI 用例禁止仅验证:
  - 页面正常加载
  - 功能正常

必须明确视觉结果。

---

## 16. 业务规则识别 (强制)

若需求涉及:
- 全局开关
- 状态控制
- 发送策略
- 通知逻辑
- 权限控制
- 退订逻辑
- Feature Flag
- 状态机

则必须生成对应业务规则测试用例。

禁止仅生成:
- UI展示
- Toggle切换
- 页面配置

而遗漏核心业务逻辑。

---

## 17. yaml示例

```yaml
metadata:
  module: "模块名"
  feature: "功能名"
  owner: "cenyf"
  last_reviewed: "2026-04-09"
  tags:
    - "tag1"

test_cases:
  - id: "TC_XXX_001"
    title: "验证点击 Backdrop 或 Showcase 的 Customize button 可进入对应主题的可视化编辑器"
    description: "验证在 Backdrop 或 Showcase 页面点击 Customize 按钮后, 能够正确加载并进入对应主题的可视化编辑器, 页面展示及功能正常"
    priority: "P0"
    type: "functional"
    preconditions:
      - "已登录 ParcelPanel 管理后台"
      - "已进入 Customize Showcase - Content 配置"
    steps:
      - "步骤一"
      - "步骤二"
    expected_results:
```

七、实践中的关键经验
1. 不要让 AI 直接“自由发挥”
AI 最怕：
“帮我生成测试用例”
因为它会自动走：
最低满足策略
即：
● 用最少步骤
● 最泛化描述
● 最安全话术
例如：
expected_results:
  - "功能正常"
这种内容本质上没有任何价值。

2. 约束越清晰，输出越稳定
后来开始增加大量“强约束”：
● 限制 YAML Schema
● 强制字段规范
● 禁止泛化描述
● 禁止占位话术
● 强制 Expected Results 可观察
● 强制 Steps 具体化
● 强制单一测试目标
之后生成质量提升非常明显。

八、典型问题与优化策略
坑 1：测试点质量差，会直接污染测试用例
例如：
验证 Toggle 可以关闭
AI 无法知道：
● 关闭后验证什么
● 是否保存
● 是否刷新
● 是否影响业务
● 是否影响通知发送
所以后面发现：
用例质量 ≈ 测试点质量

坑 2：必须增加“负向约束”
最开始只有：
应该怎么生成
后来发现还必须定义：
什么禁止生成
例如：
## 9. Steps 规范 (强制)
- 每一步仅包含一个核心动作。
  - 禁止:
    - "修改并保存"
    - "点击并跳转"
    - "输入后提交"
- Steps 中涉及输入、选择、配置时:
  - 必须明确具体值
  - 禁止:
    - "输入测试数据"
    - "填写对应内容"
    - "选择目标配置"
- 禁止写 `"按测试点要求执行..."`、`"执行对应操作"`、`"验证相关功能"`、`"修改相关配置"`、`"调整对应设置"`、`"修改标题中描述的配置"` 等占位话术。
- 禁止以下内容作为独立步骤：`"查看 xxx 是否..."`、`"验证 xxx 是否..."`、`"检查 xxx 是否..."`、`"确认 xxx 是否..."`。
- 禁止以下内容作为唯一核心操作步骤：`"刷新当前页面"`、`"查看页面"`、`"进入页面"`。
- - 若出现以上禁止的词汇，该用例判定为非法，必须根据业务逻辑重写为具体的视觉或数据描述。
否则 AI 会疯狂生成废话。

坑 3：AI 很喜欢“泛化 Expected Results”
例如：
expected_results:
  - "页面行为符合预期"
这种完全没有意义。
后面增加规则：
## 10. Expected Results 规范 (强制)

- 基本要求：可观察、可判定、可验证。
- 必须包含明确可观察结果:页面展示、数据变化、UI变化、提示文案、跳转结果、保存状态、接口行为等。
- 禁止使用：`"结果正确"`、`"符合预期"`、`"配置正确"`、`"功能正常"`、`"应生效"`、`"与测试目标一致"`等无法直接判定的泛化描述。
- Expected Results 必须与 Steps 中的实际操作直接对应。
- 禁止生成 Steps 中未发生操作的预期结果。
- 不得扩展未执行的验证行为。
- 如果我是测试执行人员，我看到这个 Expected Results，能否直接判断测试 Pass 还是 Fail，而不需要主观理解“符合预期”的意思？（如果不能，请重写 Expected Results，描述具体的 UI 或数据表现）
才逐渐稳定。

坑 4：AI 容易混合多个测试目标
例如同一条用例同时验证：
● 默认值
● 修改
● 保存
● 刷新
● 回显
最终导致：
● 用例特别长
● 可读性差
● 不可维护
后来增加规则：
一个用例只验证一个核心目标
效果改善非常明显。
