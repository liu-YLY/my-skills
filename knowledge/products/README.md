## 1. 概述

产品知识库用于存储各产品的业务知识，供测试工程师技能在以下阶段自动加载和使用：

- **阶段 1（理解需求）**：辅助理解需求背景、业务规则
- **阶段 2（提取测试点）**：补充高频缺陷模式、专项检查清单
- **阶段 3（编写用例）**：提供场景模板、测试数据
- **阶段 4（自检补全）**：验证覆盖度、检查遗漏

### 目录结构

```
knowledge/products/
├── README.md                    # 本文件 - 使用指南
├── products-template.md         # 产品知识模板
├── example.md                   # 示例产品知识
└── {product-id}.md              # 其他产品知识文件
```

## 2. 添加新产品知识

### 2.1 快速开始

```bash
# 1. 复制模板
cp products-template.md {product-id}.md

# 2. 编辑产品知识文件
# 填写各章节内容（可渐进式补充）

# 3. 提交到版本库
git add {product-id}.md
git commit -m "feat(knowledge): 添加 {产品名称} 产品知识"
```

### 2.2 详细步骤

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 复制模板 | 从 `products-template.md` 复制并重命名 |
| 2 | 填写元信息 | 产品ID、名称、版本、维护人 |
| 3 | 编写产品概述 | 定位、用户、核心价值 |
| 4 | 描述业务流程 | 主流程、分支流程、状态机 |
| 5 | 定义角色权限 | 角色列表、权限矩阵 |
| 6 | 记录业务规则 | 校验规则、计算规则、边界条件 |
| 7 | 整理测试场景 | 正向、异常、边界场景 |
| 8 | 积累缺陷模式 | 高频缺陷、遗留问题 |
| 9 | 完善数据字典 | 实体、字段、枚举值 |
| 10 | 更新索引 | 在「产品注册表」中添加记录 |

### 2.3 渐进式补充策略

产品知识不必一次性完成完整，可按以下优先级逐步补充：

| 优先级 | 内容 | 使用阶段 | 说明 |
|--------|------|----------|------|
| P0 | 产品概述、核心业务流程 | 阶段1 | 最基础的需求理解依据 |
| P0 | 关键业务规则 | 阶段2 | 直接影响测试点设计 |
| P1 | 高频缺陷模式 | 阶段2 | 补充防御性测试点 |
| P1 | 常见测试场景模板 | 阶段3 | 加速用例编写 |
| P2 | 数据字典 | 阶段3 | 完善测试数据设计 |
| P2 | 历史缺陷模式 | 阶段4 | 验证覆盖度 |

## 3. 知识文件命名规范

### 3.1 文件名格式

```
{product-id}.md
```

### 3.2 命名规则

| 规则 | 说明 | 示例 |
|------|------|------|
| 使用小写字母 | 全小写，避免大小写混淆 | `order-system.md` |
| 使用连字符分隔 | 多单词用 `-` 连接 | `user-center.md` |
| 使用英文 | 文件名使用英文，内容可中文 | `支付系统 → payment.md` |
| 简洁明确 | 产品ID应能清晰标识产品 | `crm.md`, `erp.md` |
| 避免特殊字符 | 仅使用字母、数字、连字符 | ❌ `order_system!.md` |

### 3.3 命名示例

| 产品名称 | 产品ID | 文件名 |
|----------|--------|--------|
| 用户中心 | `user-center` | `user-center.md` |
| 订单系统 | `order` | `order.md` |
| 支付网关 | `payment-gateway` | `payment-gateway.md` |
| CRM系统 | `crm` | `crm.md` |
| 商品管理 | `product-mgmt` | `product-mgmt.md` |

### 3.4 产品ID一致性要求

产品ID需在以下位置保持一致：

1. **知识文件名**：`{product-id}.md`
2. **文件内元信息**：`产品ID：{product-id}`
3. **产品注册表**：索引表中的 `产品ID` 列
4. **模块映射表**：代码路径与产品ID的映射

## 4. 知识内容组织结构

### 4.1 标准章节结构

每个产品知识文件应遵循以下结构（详见 `products-template.md`）：

```markdown
# 产品名称

> 最后更新：YYYY-MM-DD
> 维护人：xxx

## 1. 产品概述
   - 1.1 产品定位
   - 1.2 目标用户
   - 1.3 核心价值
   - 1.4 产品架构概览

## 2. 核心业务流程
   - 2.1 主流程
   - 2.2 分支流程
   - 2.3 流程状态机

## 3. 用户角色与权限
   - 3.1 角色定义
   - 3.2 权限矩阵
   - 3.3 角色间交互

## 4. 关键业务规则
   - 4.1 数据校验规则
   - 4.2 计算规则
   - 4.3 边界条件
   - 4.4 约束与限制

## 5. 常见测试场景
   - 5.1 正向场景
   - 5.2 异常场景
   - 5.3 边界场景
   - 5.4 兼容性/性能场景

## 6. 历史缺陷模式
   - 6.1 高频缺陷类型
   - 6.2 已知遗留问题
   - 6.3 缺陷趋势

## 7. 数据字典
   - 7.1 核心实体
   - 7.2 字段明细
   - 7.3 枚举值
   - 7.4 数据关系

## 附录
   - A. 术语表
   - B. 参考文档
   - C. 变更记录
```

### 4.2 各章节编写指南

#### 产品概述（必填）

```markdown
## 1. 产品概述

### 1.1 产品定位
<!-- 一句话说明：产品是什么、为谁、解决什么问题 -->
订单管理系统（OMS）为电商企业提供从下单到履约的全流程管理能力。

### 1.2 目标用户
<!-- 列出核心用户群体 -->
- 电商运营人员：处理订单、发货、退款
- 仓储人员：拣货、打包、出库
- 财务人员：对账、结算

### 1.3 核心价值
<!-- 3-5个关键价值点 -->
- 提升订单处理效率 50%
- 降低错发漏发率至 0.1%
- 支持日均 100 万订单处理
```

#### 核心业务流程（必填）

```markdown
## 2. 核心业务流程

### 2.1 主流程
用户下单 → 支付成功 → 商户接单 → 仓库拣货 → 打包出库 → 物流配送 → 签收完成

### 2.2 分支流程
| 触发条件 | 分支流程 | 说明 |
|---------|---------|------|
| 支付超时 | 订单自动取消 | 30分钟未支付 |
| 库存不足 | 缺货通知 | 触发补货流程 |
| 用户拒收 | 逆向退货 | 进入退款流程 |

### 2.3 流程状态机
| 当前状态 | 触发事件 | 目标状态 | 前置条件 |
|---------|---------|---------|---------|
| 待支付 | 支付成功 | 已支付 | - |
| 已支付 | 商户接单 | 待拣货 | 库存充足 |
| 待拣货 | 拣货完成 | 待打包 | - |
```

#### 关键业务规则（必填）

```markdown
## 4. 关键业务规则

### 4.1 数据校验规则
| 字段 | 规则 | 错误提示 |
|-----|------|---------|
| 手机号 | 11位数字，1开头 | 手机号格式错误 |
| 数量 | 正整数，1-9999 | 购买数量超限 |

### 4.2 计算规则
- 订单金额 = 商品单价 × 数量 - 优惠金额
- 运费 = 基础运费 + 超重附加费（重量>10kg）

### 4.3 边界条件
- 并发下单：同一SKU同时下单，库存扣减需加锁
- 幂等性：重复支付请求应返回原订单结果
```

#### 高频缺陷模式（推荐）

```markdown
## 6. 历史缺陷模式

### 6.1 高频缺陷类型
| 缺陷类型 | 典型表现 | 根因分析 | 预防措施 |
|---------|---------|---------|---------|
| 金额精度 | 计算结果出现多位小数 | 浮点数运算 | 使用整数分运算 |
| 并发冲突 | 库存超卖 | 未加锁 | 数据库行锁+乐观锁 |
| 状态混乱 | 订单状态跳变 | 缺少状态校验 | 状态机前置检查 |
```

### 4.3 编写原则

| 原则 | 说明 |
|------|------|
| **具体可测** | 规则描述应足够具体，能直接导出测试点 |
| **实例说明** | 复杂规则配示例数据，避免纯文字歧义 |
| **保持更新** | 业务变更时同步更新知识文件 |
| **避免冗余** | 已在PRD/接口文档中明确的内容可简写或引用 |

## 5. 与主流程的集成方式

### 5.1 自动加载规则

当用户的测试任务涉及特定产品时，按以下规则自动加载知识：

```
1. 识别产品
   ↓
   从用户输入、需求文档、代码模块中识别所属产品

2. 查找知识文件
   ↓
   在 knowledge/products/ 目录下查找 {product-id}.md

3. 加载并应用
   ↓
   读取产品知识，用于各阶段测试设计
```

### 5.2 产品识别方式

| 识别方式 | 优先级 | 说明 |
|----------|--------|------|
| 用户显式指定 | 高 | `测试知识库：[产品ID]` 或 `使用 [产品名称] 知识库` |
| 需求文档提取 | 中 | 从PRD标题、模块名称中识别 |
| 代码路径匹配 | 中 | 根据模块映射表自动匹配 |
| 关键词匹配 | 低 | 从功能描述中提取关键词匹配 |

### 5.3 各阶段集成详情

#### 阶段 1：理解需求

```
输入：用户需求描述
 ↓
[识别产品归属]
 ↓
[加载产品知识]
 ↓
应用：
  - 使用「产品概述」理解业务背景
  - 使用「业务规则」验证需求完整性
  - 使用「核心流程」补充隐含需求
 ↓
输出：结构化需求理解
```

#### 阶段 2：提取测试点

```
输入：结构化需求
 ↓
[加载产品知识]
 ↓
应用：
  - 使用「高频缺陷模式」补充防御性测试点
  - 使用「业务规则」设计规则验证点
  - 使用「状态机」设计状态相关测试点
 ↓
输出：测试点清单
```

#### 阶段 3：编写用例

```
输入：测试点清单
 ↓
[加载产品知识]
 ↓
应用：
  - 使用「测试场景模板」加速用例编写
  - 使用「数据字典」设计测试数据
  - 使用「边界条件」完善异常场景
 ↓
输出：完整测试用例
```

#### 阶段 4：自检补全

```
输入：测试用例初稿
 ↓
[加载产品知识]
 ↓
应用：
  - 对照「业务规则」验证覆盖度
  - 检查「高频缺陷」是否已覆盖
  - 补充「遗留问题」相关规避用例
 ↓
输出：补全后的测试用例
```

### 5.4 手动指定方式

用户可通过以下方式显式指定产品知识：

| 方式 | 示例 |
|------|------|
| 产品ID | `测试知识库：order` |
| 产品名称 | `使用订单系统知识库` |
| 场景描述 | `基于订单系统业务知识生成用例` |

## 6. 维护与更新

### 6.1 更新触发条件

| 触发事件 | 更新内容 |
|----------|----------|
| 新需求上线 | 补充业务流程、规则 |
| 发现新缺陷模式 | 更新缺陷模式库 |
| 接口变更 | 更新数据字典 |
| 角色权限调整 | 更新权限矩阵 |

### 6.2 更新流程

```
1. 识别变更内容
2. 编辑对应产品知识文件
3. 更新「变更记录」章节
4. 提交到版本库
5. 通知相关成员（可选）
```

### 6.3 质量检查清单

新增或更新产品知识时，检查以下内容：

- [ ] 产品ID与文件名一致
- [ ] 元信息完整（更新日期、维护人）
- [ ] 核心业务流程清晰可读
- [ ] 业务规则具体可测
- [ ] 状态机转换完整无遗漏
- [ ] 高频缺陷有预防措施
- [ ] 变更记录已更新
