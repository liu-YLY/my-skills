# 防御性测试点反推方法

> **何时阅读**：步骤 5 完成报告后，反推防御性测试点时加载本文件。
> **覆盖范围**：反推流程 / 5 类防御性测试点 / 每类检查清单 / 与 test-case-engineer 的转交规则。

## 1. 反推流程

根因分析完成后，按以下流程从根因反推防御性测试点：

```
根因 → 风险点 → 防御性测试点 → 优先级标注 → 清单输出
```

### 1.1 根因 → 风险点

对每个根因，回答：「若此根因再次出现（或以变体形式出现），哪些场景会出问题？」

- 列出根因涉及的所有缺陷模式（参考 [bug-patterns-index.md](bug-patterns-index.md)）
- 对每个缺陷模式，识别可能触发的同类场景
- 输出风险点清单（每条标注关联根因）

**示例**：
- 根因：订单服务缺少幂等校验
- 风险点：①支付回调可能重复处理 ②库存扣减可能重复 ③消息消费可能重复

### 1.2 风险点 → 防御性测试点

对每个风险点，按 5 类防御性测试点（见第 2 节）逐一设计测试点：

- 边界值测试点：极端输入是否触发同类问题
- 异常输入测试点：非法/异常输入是否被正确处理
- 并发测试点：并发场景下是否复现同类问题
- 资源测试点：资源耗尽/泄漏时是否复现
- 状态测试点：状态机异常转换时是否复现

### 1.3 优先级标注

| 优先级 | 判定标准 | 处理时效 |
|--------|---------|---------|
| P0 | 直接关联根因，且影响核心功能/数据安全 | 立即执行，发布前必须通过 |
| P1 | 关联根因的同类风险点，影响重要功能 | 本迭代内完成 |
| P2 | 间接关联根因，影响边缘场景 | 排入 backlog |

### 1.4 清单输出格式

```markdown
## 防御性测试点清单

| 编号 | 测试点 | 关联根因 | 优先级 | 来源 |
|------|--------|----------|--------|------|
| DT-01 | [测试点描述] | [关联根因] | P0/P1/P2 | 根因反推 |
| DT-02 | [测试点描述] | [关联根因] | P0/P1/P2 | 根因反推 |

> 完整用例生成请转交 test-case-engineer skill。
```

## 2. 5 类防御性测试点

### 2.1 边界值测试点

针对根因涉及的数值/范围/长度等边界条件设计测试点。

**适用根因类型**：
- 数值计算错误（如金额、库存、计数）
- 字符串长度/数组越界
- 时间窗口/过期判断
- 分页/批次边界

**检查清单**：
- [ ] 最小值（0、空、负数）是否被正确处理
- [ ] 最大值（Int 上限、字段长度上限）是否被正确处理
- [ ] 边界值 ±1（如 age=17/18/19，amount=0.01/0.00）是否被正确处理
- [ ] 临界条件切换（如刚好达到阈值、刚好超过阈值）是否符合预期
- [ ] 时间边界（00:00:00、23:59:59、跨年/跨月/跨日）是否被正确处理

**示例**（根因：金额计算精度丢失）：
| 编号 | 测试点 | 关联根因 | 优先级 |
|------|--------|----------|--------|
| DT-01 | 金额 0.01 元时的累加精度（100 次累加结果应为 1.00 元） | 浮点数精度丢失 | P0 |
| DT-02 | 金额 99999999.99 元时的累加精度 | 浮点数精度丢失 | P1 |

### 2.2 异常输入测试点

针对根因涉及的输入校验缺失设计测试点。

**适用根因类型**：
- 输入校验缺失/不完整
- 类型转换错误
- 空值/Null 处理
- 格式校验缺失

**检查清单**：
- [ ] Null / 空字符串 / 空数组是否被正确处理
- [ ] 类型错误（如字符串传到数值字段）是否被拒绝
- [ ] 格式错误（如非法邮箱、非法手机号）是否被拒绝
- [ ] 特殊字符（SQL 注入、XSS、Unicode 控制字符）是否被过滤
- [ ] 超大输入（远超字段长度）是否被截断或拒绝
- [ ] 依赖外部输入的场景（如第三方 API 返回异常数据）是否有兜底

**示例**（根因：用户名未做长度校验导致 DB 写入失败）：
| 编号 | 测试点 | 关联根因 | 优先级 |
|------|--------|----------|--------|
| DT-03 | 用户名为空字符串时返回 400 而非 500 | 输入校验缺失 | P0 |
| DT-04 | 用户名长度超过 255 字符时返回 400 | 输入校验缺失 | P0 |
| DT-05 | 用户名包含 SQL 注入字符时被转义 | 输入校验缺失 | P1 |

### 2.3 并发测试点

针对根因涉及的并发/竞态条件设计测试点。

**适用根因类型**：
- 竞态条件
- 幂等缺失
- 锁失效/死锁
- 缓存击穿/雪崩

**检查清单**：
- [ ] 相同请求并发 N 次（N=10/100/1000）时是否保持幂等
- [ ] 同一资源并发读改写时是否有锁保护
- [ ] 不同请求操作同一账户/订单时是否有序处理
- [ ] 缓存失效瞬间大量请求穿透到 DB 是否有保护
- [ ] 分布式锁失效/超时是否触发降级
- [ ] 死锁场景（如多资源按不同顺序加锁）是否被检测

**示例**（根因：订单创建缺少幂等校验）：
| 编号 | 测试点 | 关联根因 | 优先级 |
|------|--------|----------|--------|
| DT-06 | 同一 request_id 并发 100 次，仅 1 次成功 | 幂等缺失 | P0 |
| DT-07 | 同一用户并发提交 5 个订单，库存扣减正确 | 竞态条件 | P0 |
| DT-08 | 分布式锁超时后是否触发降级 | 锁失效 | P1 |

### 2.4 资源测试点

针对根因涉及的资源泄漏/耗尽设计测试点。

**适用根因类型**：
- 内存泄漏
- 连接泄漏（DB 连接/HTTP 连接未释放）
- 文件句柄泄漏
- 磁盘/内存耗尽
- 资源未及时释放

**检查清单**：
- [ ] 长时间运行（如 24h）后内存占用是否稳定
- [ ] 异常路径（如抛出异常）下资源是否被释放（try-finally/try-with-resources）
- [ ] 高并发下连接池是否被耗尽
- [ ] 大文件/大数据量是否触发 OOM
- [ ] 磁盘满时是否优雅降级
- [ ] 资源配额达到上限时是否拒绝新请求

**示例**（根因：异常路径下 DB 连接未释放）：
| 编号 | 测试点 | 关联根因 | 优先级 |
|------|--------|----------|--------|
| DT-09 | 连续触发 1000 次异常后，DB 连接池可用连接数仍为初始值 | 资源泄漏 | P0 |
| DT-10 | 连接池耗尽后新请求返回 503 而非阻塞 | 资源耗尽 | P1 |

### 2.5 状态测试点

针对根因涉及的状态机/流程异常设计测试点。

**适用根因类型**：
- 状态机转换错误
- 流程顺序错误（如未支付就发货）
- 状态回滚缺失
- 事务边界错误
- 中间状态可见性

**检查清单**：
- [ ] 非法状态转换（如「已取消」→「已支付」）是否被拒绝
- [ ] 重复操作（如重复支付）是否被正确处理
- [ ] 部分失败时事务是否回滚
- [ ] 中间状态对外可见性（如「支付中」时前端是否禁用提交按钮）
- [ ] 状态机并发转换（如同一订单同时被取消和支付）是否有锁保护
- [ ] 长时间停留在中间状态是否有超时处理

**示例**（根因：订单状态机允许非法转换）：
| 编号 | 测试点 | 关联根因 | 优先级 |
|------|--------|----------|--------|
| DT-11 | 已取消订单调用支付接口返回 409 而非发起支付 | 状态机错误 | P0 |
| DT-12 | 已支付订单再次调用支付接口返回幂等成功 | 状态机错误 | P0 |
| DT-13 | 订单停留在「支付中」超过 30min 后自动取消 | 状态超时 | P1 |

## 3. 完整示例

### 3.1 反推示例（基于「订单重复扣款」根因）

**根因**：订单服务缺少幂等校验 + 前端超时自动重试

**反推过程**：

1. **根因 → 风险点**：
   - 风险点 1：支付回调可能重复处理（关联根因：幂等缺失）
   - 风险点 2：库存扣减可能重复（关联根因：幂等缺失）
   - 风险点 3：前端其他接口超时重试可能触发同类问题（关联根因：前端自动重试）

2. **风险点 → 防御性测试点**（按 5 类逐一设计）：
   - 边界值：金额为 0.01 元时幂等是否生效
   - 异常输入：request_id 为空时是否被拒绝
   - 并发：100 并发同 request_id 仅 1 次成功
   - 资源：幂等校验依赖的 Redis 不可用时是否降级
   - 状态：已支付订单再次调用支付接口返回幂等成功

3. **输出清单**：

| 编号 | 测试点 | 关联根因 | 优先级 | 来源 |
|------|--------|----------|--------|------|
| DT-01 | 同一 request_id 并发 100 次，仅 1 次成功 | 幂等缺失 | P0 | 根因反推 |
| DT-02 | 前端 3s 内按钮置灰，防止重复提交 | 前端自动重试 | P1 | 根因反推 |
| DT-03 | 支付回调重复到达时仅处理一次 | 幂等缺失 | P0 | 根因反推 |
| DT-04 | 库存扣减接口幂等校验 | 幂等缺失 | P0 | 根因反推 |
| DT-05 | Redis 不可用时降级为 DB 唯一索引校验 | 幂等校验依赖 | P1 | 根因反推 |
| DT-06 | 已支付订单再次调用支付接口返回幂等成功 | 状态机错误 | P1 | 根因反推 |
| DT-07 | 金额为 0.01 元时幂等校验正常 | 边界值 | P2 | 根因反推 |
| DT-08 | request_id 为空时返回 400 | 异常输入 | P2 | 根因反推 |

## 4. 与 test-case-engineer 的转交规则

### 4.1 转交触发条件

- ✅ 防御性测试点清单已生成
- ✅ 每个测试点已标注「关联根因」字段
- ✅ 每个测试点已标注优先级（P0/P1/P2）

### 4.2 转交内容

转交给 test-case-engineer 时需提供：
1. 防御性测试点清单（含编号、描述、关联根因、优先级）
2. 根因分析报告（用于理解测试点设计意图）
3. 修复方案摘要（用于设计反向验证用例）

### 4.3 转交话术

```
bug-analyzer 已完成防御性测试点反推，输出清单如下：

| 编号 | 测试点 | 关联根因 | 优先级 |
|------|--------|----------|--------|
| DT-01 | ... | ... | P0 |
| ...  | ... | ... | ... |

请 test-case-engineer 基于以上输入：
1. 将每个 DT 展开为完整测试用例（含前置条件/操作步骤/期望结果/数据准备）
2. 优先展开 P0 级别测试点
3. 对每个修复方案设计反向验证用例（验证修复撤销时 Bug 复现）
4. 设计回归测试用例覆盖关联功能
```

### 4.4 反模式（不要做）

- ❌ 直接展开为完整用例：本 skill 不承担用例生成职责，仅输出测试点清单
- ❌ 测试点无关联根因：每个测试点必须可追溯到根因
- ❌ 测试点无优先级：修复资源无法聚焦，必须标注 P0/P1/P2
- ❌ 测试点描述模糊（如「测试并发」）：必须可执行（如「同一 request_id 并发 100 次，仅 1 次成功」）

### 4.5 边界与降级

- 若 test-case-engineer skill 未安装，在输出中显式声明「完整用例生成能力不可用」
- 若用户仅要求根因分析不要用例，仅输出测试点清单不转交
- 若 P0 测试点数量超过 10 个，提示用户聚焦核心风险，优先处理最高优先级
