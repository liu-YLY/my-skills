# Bug 根因分析报告模板

> **何时阅读**：步骤 5 生成报告时加载本文件，按字段定义逐项填写。
> **覆盖范围**：报告字段定义 / 填写规范 / 两类报告示例 / 与 test-case-engineer 的转交规则。

## 1. 报告字段定义

| 字段 | 是否必填 | 内容说明 | 来源步骤 |
|------|---------|---------|---------|
| Bug ID / 标题 | 必填 | 缺陷跟踪系统 ID 或一句话简短描述 | 用户输入 |
| 现象 | 必填 | 一句话描述 Bug 表现，禁止模糊词（如"功能不正常"） | 步骤 1 |
| 环境 | 必填 | dev/test/staging/prod + 版本号 + 构建号 | 步骤 1 |
| 复现率 | 必填 | 100% / 偶发 X% / 无法复现 | 步骤 1 |
| 附件/日志 | 选填 | trace id / 日志片段 / 截图链接 | 步骤 1 |
| 复现步骤 | 必填 | 编号步骤，含操作 + 期望 + 实际 | 步骤 1 |
| 隔离结论 | 必填 | 异常首次出现位置 + 调用链证据 + 对照缺陷模式 | 步骤 2 |
| 根因 | 必填 | 直接原因 + 深层原因 + 分析方法 + 分析过程 | 步骤 3 |
| 验证结论 | 必填 | 验证用例 + 修复方案 + 验证结果 + 验证环境约束 | 步骤 4 |
| 修复建议 | 必填 | 紧急修复 + 长期改进 + 预防措施（三档） | 步骤 5 |
| 影响范围 | 必填 | 受影响功能 / 数据 / 用户 | 步骤 5 |
| 回归测试 | 必填 | 必测场景 + 关联功能清单 | 步骤 5 |

## 2. 各字段填写规范

### 2.1 现象

- 用一句话具体描述，包含：什么操作 + 触发什么异常 + 在什么条件下
- **禁止**使用模糊词：功能不正常、有 bug、出错了、报错
- **示例**：✓「提交订单时，前端重试导致同一订单被扣款两次」 / ✗「支付有问题」

### 2.2 复现率

- 100%：每次操作必现
- 偶发 X%：明确标注复现概率（基于至少 10 次试验统计），并附触发条件（如并发数 > 50）
- 无法复现：注明已尝试的场景与未覆盖的场景

### 2.3 复现步骤

- 按编号列出，每条只描述一个操作
- 必须包含「期望结果」与「实际结果」对比
- 若为偶发 Bug，标注「需重复 N 次」或「需满足 X 条件」

### 2.4 隔离结论

- 异常首次出现位置：明确到组件/服务/模块/方法
- 调用链证据：列出每个边界的实际输入输出值（脱敏后）
- 对照缺陷模式：引用 [bug-patterns-index.md](bug-patterns-index.md) 指向的共享缺陷模式库，说明属于哪类

### 2.5 根因

- **直接原因**：触发 Bug 的具体代码/配置问题（精确到文件/行号/配置项）
- **深层原因**：导致直接原因存在的流程/规范/架构问题（如评审漏审、规范缺失、架构耦合）
- 分析方法：5 Whys / 鱼骨图 / 因果链追溯（详见 [root-cause-frameworks.md](root-cause-frameworks.md)）
- 分析过程：保留关键追问链或维度排查的关键证据
- **禁止**只写直接原因，深层原因必填；若确无深层原因，显式声明「已排查规范/评审/架构层面，未发现系统性问题」

### 2.6 验证结论

- 验证用例：最小复现用例描述（不含业务无关步骤）
- 修复方案：具体修复内容（文件/配置/代码片段摘要）
- 验证结果：必须包含三项 —— ①修复后 Bug 消失 ②回归测试通过 ③反向验证确认因果链
- **验证环境约束**（必填）：
  - 测试/预发环境完成反向验证 → 标注「实验复现确认」
  - 生产环境用替代方案（影子流量/A/B 桶/观察性验证）→ 标注「因果链由观察数据佐证而非实验复现」

### 2.7 修复建议（三档必填）

| 档位 | 时效 | 内容要求 | 示例 |
|------|------|---------|------|
| 紧急修复 | 24h 内止血 | 具体操作步骤，可立即执行 | 「回滚至 v1.2.3」「关闭特性开关 X」 |
| 长期改进 | 含涉及模块/预估工作量 | 涉及哪些模块、预估人天 | 「订单服务增加幂等校验，2 人天」 |
| 预防措施 | 含具体校验规则/监控项/测试用例编号 | 落到 CI/监控/测试用例层 | 「新增 lint 规则 R-001」「监控告警阈值: 错误率 > 1%」 |

### 2.8 影响范围

- 受影响功能：列出可能受影响的功能模块（基于调用链与数据流）
- 受影响数据：是否需要数据修复（涉及哪些表/记录数/修复脚本编号）
- 受影响用户：用户范围（全量/灰度桶/特定角色）与比例

### 2.9 回归测试

- 必测场景：修复后必须回归测试的场景清单
- 关联功能：可能受影响的关联功能清单（基于依赖关系图）

## 3. 报告示例

### 3.1 偶发复现 Bug 报告示例

```markdown
## Bug 根因分析报告

**Bug ID / 标题**：BUG-2024-0815-001 订单提交时偶发重复扣款
**现象**：用户提交订单时，前端超时重试导致同一订单被扣款两次
**环境**：prod / v2.3.1 / build 20240815.3
**复现率**：偶发 12%（基于 25 次试验，仅在并发数 > 50 时触发）
**附件/日志**：trace_id=abc123，订单服务日志片段见附件 1

### 复现步骤
1. 用户在购物车页面点击「提交订单」
2. 后端订单接口响应 > 3s（需并发 > 50）
3. 前端超时后自动重试，发起第二次请求
4. 期望：仅扣款一次；实际：扣款两次

### 隔离结论
- 异常首次出现位置：订单服务 `create_order` 方法
- 调用链证据：前端发出 2 次请求（间隔 3.1s）→ 网关透传 2 次 → 订单服务收到 2 次（每次均扣款成功）
- 对照缺陷模式：属于「状态与流程类 - 幂等缺失」（参考 ../test-case-engineer/knowledge/bug-patterns.md）

### 根因
- **直接原因**：订单服务 `create_order` 未校验请求幂等性，重复 request_id 也被当作新请求处理
- **深层原因**：①订单接口规范未要求幂等 ②代码评审 checklist 未包含「幂等校验」条目 ③上线流程未做并发压测
- 分析方法：5 Whys（详见根因分析报告附录）
- 分析过程：现象→前端重试→后端响应慢→数据库锁等待→订单表缺索引→未做压测

### 验证结论
- 验证用例：并发 100 个相同 request_id 的请求，期望仅成功 1 次
- 修复方案：①订单服务新增 request_id 幂等校验（Redis SETNX 5min 过期）②订单表新增 user_id+status 联合索引
- 验证结果：①修复后 100 并发仅 1 次成功 ②回归测试通过（23 个用例）③反向验证：撤销修复后 Bug 复现
- **验证环境约束**：实验复现确认（测试环境完成反向验证）

### 修复建议
- **紧急修复**（24h 内）：开启网关层限流（同 user_id 1s 内仅放行 1 次）
- **长期改进**：订单服务增加幂等校验中间件，涉及 order-service 模块，预估 2 人天
- **预防措施**：①代码评审 checklist 新增「幂等校验」条目 ②CI 新增并发压测任务 ③新增监控告警：重复 request_id 错误率 > 0.1%

### 影响范围
- 受影响功能：订单创建、支付回调、库存扣减
- 受影响数据：需修复 12 个重复扣款订单，修复脚本 see fix_script_20240815.sql
- 受影响用户：8 月 14 日至 15 日高并发时段下单用户，约 120 人

### 回归测试
- 必测场景：①订单创建幂等 ②支付回调幂等 ③库存扣减幂等
- 关联功能：退款流程、订单查询、对账系统
```

### 3.2 环境差异 Bug 报告示例

```markdown
## Bug 根因分析报告

**Bug ID / 标题**：BUG-2024-0820-002 预发环境登录失败但测试环境正常
**现象**：预发环境用户登录返回 500，相同代码在测试环境登录正常
**环境**：staging / v2.3.2 / build 20240820.1（对比 test / v2.3.2 / build 20240820.1）
**复现率**：100%（仅预发环境）
**附件/日志**：trace_id=def456，预发 Nginx 日志见附件 2

### 复现步骤
1. 在预发环境输入测试账号 user@example.com / pwd123
2. 点击「登录」
3. 期望：登录成功跳转首页；实际：返回 500，日志显示 JWT 签名失败

### 隔离结论
- 异常首次出现位置：auth-service 的 JWT 签名方法
- 调用链证据：前端 POST /login → 网关透传 → auth-service（500 异常首次出现）
- 对照缺陷模式：属于「环境差异类 - 配置漂移」（参考 ../test-case-engineer/knowledge/bug-patterns.md）

### 根因
- **直接原因**：预发环境 JWT_SECRET 配置为旧版本（v1.x 格式，长度 16 字节），与代码中要求的 v2.x 格式（长度 32 字节）不兼容
- **深层原因**：①配置变更走人工流程未纳入 CI 校验 ②预发环境配置基线未与代码版本对齐 ③缺少配置项 schema 校验
- 分析方法：因果链追溯（环境差异 → 配置 → JWT_SECRET → 格式不兼容）
- 分析过程：预发 vs 测试环境 diff 三项（配置/数据/依赖版本）→ 配置项 JWT_SECRET 不一致 → 长度不匹配 → 签名失败

### 验证结论
- 验证用例：在预发环境更新 JWT_SECRET 为 v2.x 格式后登录
- 修复方案：①更新预发 JWT_SECRET 配置 ②增加启动时配置 schema 校验（长度不足则启动失败）
- 验证结果：①修复后登录成功 ②回归测试通过（15 个用例）③反向验证：恢复旧配置后 Bug 复现
- **验证环境约束**：实验复现确认（预发环境完成反向验证，未涉及生产环境）

### 修复建议
- **紧急修复**（24h 内）：更新预发环境 JWT_SECRET 为 v2.x 格式
- **长期改进**：配置中心新增 schema 校验功能，涉及 config-service 模块，预估 3 人天
- **预防措施**：①CI 流水线新增配置项 schema 校验阶段 ②部署前 checklist 新增「配置基线对齐」条目 ③监控告警：服务启动失败告警

### 影响范围
- 受影响功能：仅预发环境登录功能
- 受影响数据：无数据损坏
- 受影响用户：仅预发环境测试用户（约 20 人）

### 回归测试
- 必测场景：①预发环境登录 ②JWT 签名/验签 ③配置 schema 校验启动检查
- 关联功能：SSO、权限校验、会话管理
```

## 4. 与 test-case-engineer 的转交规则

完成报告输出后，需将防御性测试点清单转交 test-case-engineer 生成完整用例。

### 4.1 转交触发条件

- ✅ 步骤 4 验证通过
- ✅ 步骤 5 报告完成且用户确认
- ✅ 防御性测试点清单已生成（详见 [defensive-test-points.md](defensive-test-points.md)）

### 4.2 转交内容

转交给 test-case-engineer 时需提供：
1. 完整根因分析报告（本文件第 3 节格式）
2. 防御性测试点清单（含编号、测试点描述、关联根因、优先级）
3. 修复方案摘要（用于 test-case-engineer 设计反向用例）

### 4.3 转交话术

```
bug-analyzer 已完成根因分析，输出：
- 根因分析报告：[报告全文]
- 防御性测试点清单：[DT-01 ... DT-N]
- 修复方案：[紧急/长期/预防三档]

请 test-case-engineer 基于以上输入：
1. 将每个防御性测试点展开为完整测试用例（含前置条件/操作步骤/期望结果）
2. 设计反向验证用例（验证修复撤销时 Bug 复现）
3. 设计回归测试用例覆盖关联功能
```

### 4.4 边界与降级

- 若用户未明确要求生成完整用例，**仅输出防御性测试点清单**，不主动展开用例
- 若 test-case-engineer skill 未安装，在输出中显式声明「完整用例生成能力不可用，建议安装 test-case-engineer skill」
