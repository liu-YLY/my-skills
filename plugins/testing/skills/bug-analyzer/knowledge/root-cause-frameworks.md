# 根因分析框架

> **何时阅读**：定位阶段需要选择合适的根因分析方法时查阅。
> **覆盖范围**：鱼骨图分类维度 / 5 Whys 分析法 / 因果链追溯 / 防御性用例反推。

## 1. 鱼骨图分类维度

根因分析时按以下五维度逐一排查，避免遗漏：

| 维度 | 含义 | 典型问题 |
|------|------|----------|
| **人** | 操作者相关 | 操作失误、权限配置错误、沟通遗漏、培训不足 |
| **机** | 基础设施 | 服务器故障、网络异常、中间件问题、环境配置错误 |
| **料** | 数据与依赖 | 数据质量差、依赖服务异常、第三方接口变更、测试数据污染 |
| **法** | 流程与逻辑 | 流程缺陷、规范缺失、代码逻辑错误、校验缺失 |
| **环** | 环境差异 | 环境不一致、版本差异、部署时序、配置漂移 |

**使用方式**：
1. 将 Bug 现象写在鱼骨图头部
2. 沿五维度逐条排查，列出可能原因
3. 对每条原因验证（可复现/有证据/符合逻辑）
4. 锁定真正贡献根因的维度与具体原因

## 2. 5 Whys 根因分析法

从问题表象逐层追问"为什么"，直到找到可落地的根因。

**规则**：
- 每层回答必须基于事实或证据，禁止猜测
- 至少追问 5 层，但不必拘泥于 5 层——找到可落地根因即可停止
- 可落地 = 可以通过修改代码/配置/流程来预防
- 若分支原因多于 1 个，分多条链追问

**示例**（重复扣款 Bug）：
```
现象：用户提交订单时偶尔重复扣款
Why 1：为什么重复扣款？→ 前端在超时后自动重试
Why 2：为什么超时？→ 后端订单接口响应慢（>3s）
Why 3：为什么后端慢？→ 高并发时数据库锁等待
Why 4：为什么有锁等待？→ 订单表缺少合适索引
Why 5：为什么缺少索引？→ 上线时未做性能压测
根因：上线流程缺少性能压测环节 + 订单表索引缺失
```

**停止条件**：
- 找到可通过具体行动修复的根因（如「缺少索引」可修，「人性贪婪」不可修）
- 已追溯到流程/制度层面（如「上线流程缺失环节」）

## 3. 因果链追溯

适用于多组件系统的 Bug 定位。

**方法**：
1. 画出数据流/调用链（如：前端 → 网关 → 服务 A → 服务 B → DB）
2. 在每个边界记录输入输出
3. 定位数据异常首次出现的组件
4. 在该组件内部继续二分定位

**示例**：
```
前端发出 {amount: 99.99}
  → 网关收到 {amount: 99.99} ✓
    → 服务 A 收到 {amount: 99.99} ✓
      → 服务 B 收到 {amount: 99.98999999} ✗  ← 异常首次出现
        → DB 存储 99.98999999
```
定位到服务 B 的浮点数转换逻辑问题。

## 4. 防御性用例反推

根因分析完成后，基于根因反推防御性测试点。

**反推流程**：
1. 列出根因涉及的所有缺陷模式
2. 为每个缺陷模式设计正向验证用例（验证修复有效）
3. 为每个缺陷模式设计反向验证用例（验证未修复时的表现）
4. 设计回归测试点（确保同类问题不再发生）
5. 输出「防御性测试点清单」，标注可由 test-case-engineer skill 生成完整用例

**输出格式**：
```markdown
## 防御性测试点清单

| 编号 | 测试点 | 关联根因 | 优先级 | 来源 |
|------|--------|----------|--------|------|
| DT-01 | 订单接口幂等性校验 - 重复请求 ID 拒绝 | 缺少幂等校验 | P0 | 根因反推 |
| DT-02 | 前端防重复提交 - 3s 内按钮置灰 | 前端自动重试 | P1 | 根因反推 |

> 完整用例生成请转交 test-case-engineer skill。
```
