# Change Impact Analyzer Skill

> 变更影响分析师技能包，v1.1.0。
> 将 git 代码变更与测试用例交叉分析，发现行为变更影响和覆盖缺口。

## 核心能力

- **四阶段工作流**：收集输入 → Diff 解析 → 交叉分析 → 生成报告
- **七种 diff 模式**：工作区 / 暂存区 / 分支对比 / 单 Commit / Commit 范围 / Revision Range / PR Diff / 外部 Patch
- **只读采集脚本**：通过结构化字段（status_short / name_status / numstat / untracked_files / unified_diff / diff_truncated）采集变更，不读未跟踪文件内容、不转二进制、超阈值截断
- **跨层影响链路追踪**：沿「页面 → 组件 → API → Controller → Service → Model/DB → 缓存/MQ/外部系统」串联上下游，找出间接影响
- **前后端契约两侧检查**：接口契约变更同时检查两侧，不只看变更的一端
- **Mode A / Mode B 双模式**：Mode A 区分代码事实/影响推断/待确认问题；Mode B 做需求 × 实现 × 测试三向对照，识别疑似超范围实现
- **风险三档清单**：高/中/低风险具体场景清单 + 「降低风险必须有证据」硬约束
- **P0/P1/P2 优先级**：明确哪些问题必须在发布前发现
- **结构化测试场景**：`[P级][类型] 前置条件 → 操作 → 可观察预期 — 证据`，禁止「验证 XX 是否正常」式描述
- **8 段报告结构**：分析范围 / 改动与需求摘要 / 影响链路 / 风险结论 / 必测内容 / 回归范围（核心+定向+可不回归）/ 需求追踪矩阵 / 待确认与限制
- **灵活用例输入**：支持 Markdown/CSV/JSON 格式，可指定文件路径或目录

## 与 testing 生态的关系

| 维度 | change-impact-analyzer | test-case-engineer | bug-analyzer |
|------|----------------------|-------------------|--------------|
| 方向 | 横向（代码改动 × 用例 → 影响） | 正向（需求 → 用例） | 逆向（Bug → 根因） |
| 输入 | git diff + 测试用例文件（可选需求文档） | 需求文档/代码变更 | Bug 现象/日志 |
| 输出 | 影响报告（行为变更 + 覆盖缺口 + 影响链路 + 风险结论 + 必测内容 + 回归范围） | 测试用例 | 根因报告 + 防御性测试点 |
| 协作 | 发现缺口后转交 test-case-engineer 补充用例 | — | — |

## 适用场景

| 场景 | 说明 |
|------|------|
| ✅ 开发提测前 | 分析当前分支相对 main 的变化，提前知道哪些模块要准备数据、是否有需求遗漏 |
| ✅ 测试分析阶段（Mode B） | 需求 + 代码变更一起输入，检查需求覆盖、实现冲突、超范围修改 |
| ✅ 代码合并前 | 对 PR Diff 做风险扫描，重点检查公共组件/接口契约/权限/DB/配置/发布顺序 |
| ✅ 发布前 | 按 P0/P1/P2 重新确认测试完成情况，把高风险遗漏优先补齐 |
| ✅ PR/MR 前检查 | 分析分支改动对现有用例的影响 |
| ✅ 回归风险评估 | 检查 commit 范围的变更是否破坏用例 |
| ✅ 覆盖缺口分析 | 找出改动功能中未被用例覆盖的部分 |

> 测试用例生成请使用 test-case-engineer，Bug 根因分析请使用 bug-analyzer。

## 不能替代的边界

代码变更无法完整还原真实产品需求，本 skill **不能替代**：需求评审、测试人员的业务判断、开发代码评审、真实环境验证、最终发布决策。它更适合作为「测试分析助手」，整理代码事实、影响链路、风险和待确认问题，减少遗漏，但不替人做最终判断。

## 敏感信息与脱敏

代码 Diff 可能包含接口地址、业务数据、密钥、用户信息等敏感信息。优先在受信任的本地环境中分析；报告对外分享前必须脱敏；涉及密钥/凭据的变更点只标注「涉及敏感字段」，不复制具体值。

## 核心工作流

```
阶段 1: 收集输入 → 阶段 2: Diff 解析 → 阶段 3: 交叉分析 → 阶段 4: 生成报告
```

### 阶段 1: 收集输入

**目标**：获取 git diff 内容和测试用例。

- 确认 diff 范围（七种模式：工作区 / 暂存区 / 分支对比 / 单 Commit / Commit 范围 / Revision Range / PR Patch）
- 推荐使用只读采集脚本采集变更上下文（保持只读、不读未跟踪内容、不转二进制、超阈值截断）
- 加载测试用例文件（Markdown/CSV/JSON）
- Mode B 下额外加载需求文档/验收标准/任务说明

🔴 **CHECKPOINT**：向用户确认分析范围、用例文件列表后再继续。

### 阶段 2: Diff 解析

**目标**：从 diff 中提取结构化变更清单。

- 提取变更文件列表（路径/类型/行数）
- 过滤无关文件（锁文件/构建产物/测试文件本身）
- 解析每个文件的变更详情（函数/接口/字段/业务逻辑）
- 跨层影响链路追踪（页面 → 组件 → API → Controller → Service → Model/DB → 缓存/MQ）
- 前后端契约两侧检查（接口契约变更同时检查对端）

🔴 **CHECKPOINT**：向用户展示变更清单 + 影响链路 + 契约检查待确认项后再继续。

### 阶段 3: 交叉分析

**目标**：将变更清单与测试用例交叉比对。

**3.0 变更固有风险参考清单**：高/中/低风险具体场景，硬约束「降低风险必须有证据」。

**3.1 第一轮 — 行为变更影响**：
- 直接引用匹配（高置信度）：用例文本包含变更的函数名/API路径/字段名
- 模块关联匹配（中置信度）：用例关键词与变更文件模块路径一致
- 依赖链匹配（低置信度）：diff 中 import 指向变更文件

**3.2 第二轮 — 覆盖缺口**：
- 检查每个变更点是否有用例覆盖
- 无覆盖的标记为缺口，按风险等级排序

**3.3 Mode A / Mode B 双模式**：
- Mode A（默认）：输出三分（代码事实/影响推断/待确认问题），避免把推断当事实
- Mode B：需求 × 实现 × 测试三向对照，输出覆盖状态（已覆盖/部分覆盖/未发现实现/实现与需求冲突/需确认），识别疑似超范围实现

### 阶段 4: 生成报告

**目标**：输出 8 段结构化 Markdown 报告。

报告章节：① 分析范围 → ② 改动与需求摘要 → ③ 影响链路 → ④ 风险结论 → ⑤ 必测内容（P0/P1/P2 + 结构化场景模板）→ ⑥ 回归范围（核心+定向+可不回归）→ ⑦ 需求追踪矩阵（Mode B）→ ⑧ 待确认与限制。

## 使用方式

### 快速开始

```
用户：帮我分析一下当前改动对测试用例的影响，用例在 docs/test-cases.md
```

### 分支对比

```
用户：对比 main 和 feat/order-cancel 分支的改动，检查对 order 相关用例的影响
```

### Commit 范围

```
用户：检查 commit abc1234 的改动是否影响了 regression 用例
```

### Mode B 需求对照

```
用户：对照 docs/requirement.md，分析 main...当前分支的代码改动，输出需求覆盖情况和测试范围
```

### 外部 Patch

```
用户：分析这个 patch 对现有用例的影响，patch 内容如下：
diff --git a/src/api/user.ts ...
```

## 反例与黑名单

| # | 反模式 | 替代做法 |
|---|--------|---------|
| 1 | 跳过 CHECKPOINT 直接分析 | 必须让用户确认 diff 范围和用例文件 |
| 2 | 将测试文件变更计入影响分析 | 阶段 2 过滤 `*.test.*`/`*.spec.*` |
| 3 | 所有变更都标记"高风险" | 严格按风险参考清单打标，低风险也要标注 |
| 4 | 降低风险没有证据就下调等级 | 必须给出「调用方数量/契约一致性/自动化覆盖」等具体证据 |
| 5 | Mode A 把推断当事实陈述 | 必须区分代码事实/影响推断/待确认问题，推断需标注置信度 |
| 6 | 验收标准凭空创造 | 测试范围可推断，但验收标准不能臆测，必须列入待确认问题 |
| 7 | 测试场景写"验证 XX 是否正常" | 必须用 `[P级][类型] 前置→操作→预期—证据` 结构 |
| 8 | 只看变更一侧不查契约对端 | 接口契约变更必须同时检查前后端两侧 |
| 9 | 报告只列问题不给建议 | 每个问题附带建议操作 |
| 10 | 强行解析 Excel 格式 | 提示转换为 CSV/Markdown |
| 11 | 把整个仓库塞给模型 | 用只读采集脚本采集变更上下文，定向读取相关文件 |
| 12 | 报告对外分享前未脱敏 | 替换接口路径/用户 ID/密钥等敏感信息 |
