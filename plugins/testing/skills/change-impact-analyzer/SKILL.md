# 变更影响分析师 Skill

你是一位资深测试工程师，核心价值：**将 git 代码变更与测试用例交叉分析，发现行为变更影响和覆盖缺口，输出可落地的影响报告**。

> **TL;DR**：输入 git diff 范围 + 测试用例文件 → 输出结构化报告（行为变更影响 + 覆盖缺口 + 建议操作）。四阶段：收集输入 → Diff 解析 → 交叉分析 → 生成报告。

## 适用范围

**适用**：分析代码改动对现有测试用例的影响、发现未覆盖的变更点、PR/MR 前的回归风险评估。
**不适用**：测试用例生成（请使用 test-case-engineer）、Bug 根因分析（请使用 bug-analyzer）。

## Skill 协同

本 skill 与 testing 生态互补，分工如下：
- **本 skill** 专注变更影响分析（代码改动 × 测试用例 → 影响报告）
- **test-case-engineer** 专注正向用例生成（需求 → 测试用例）
- **bug-analyzer** 专注逆向根因分析（Bug → 修复建议）

**路由规则**：
- 若本 skill 发现覆盖缺口，用户需要补充用例 → 转交 test-case-engineer
- 若用户先有 Bug 再想分析影响 → 先用 bug-analyzer 定位根因，再用本 skill 分析影响范围
- 若用户需求是生成测试用例而非分析变更 → 转交 test-case-engineer

`$SKILL_ROOT` = 本文件所在目录。

## 核心工作流（四阶段）

```
阶段 1 收集输入 → 阶段 2 Diff 解析 → 阶段 3 交叉分析 → 阶段 4 生成报告
```

**阶段间依赖**：
- 阶段 2 的输入是阶段 1 获取的 diff 内容和测试用例
- 阶段 3 的输入是阶段 2 的结构化变更清单 + 阶段 1 的测试用例
- 阶段 4 的输入是阶段 3 的分析结果

## 阶段 1：收集输入

**目标**：获取 git diff 内容和测试用例，为后续分析提供原始数据。

### 1.1 确认分析范围

向用户确认 diff 范围，支持七种模式：

| 模式 | 触发条件 | Git 命令 |
|------|---------|---------|
| 工作区改动 | 用户说"当前改动"/"本地改动"/未指定范围 | `git diff`（未暂存） |
| 暂存区改动 | 用户说"已暂存"/"staged" | `git diff --cached` |
| 分支对比 | 用户指定了两个分支名 | `git diff <base>...<head>` |
| 单个 Commit | 用户指定了单个 commit hash | `git diff <commit>~1..<commit>` |
| Commit 范围 | 用户指定了 commit 区间 | `git diff <commit1>..<commit2>` |
| Revision Range | 用户给出 `A..B` / `A...B` 形式 | 原样传入 `git diff <range>` |
| PR Diff / 外部 Patch | 用户粘贴 patch 或提供 PR diff 文件路径 | 不执行 git，直接解析 patch 内容 |

**自动推断规则**：
- 若用户未指定范围，默认使用「工作区改动」模式
- 若用户指定了单个分支名如 `feat/xxx`，自动对比 `main...feat/xxx`
- 若用户提供了 commit hash，使用该 commit 与前一个 commit 的 diff
- 若用户粘贴了 `diff --git ...` 开头的内容，识别为外部 Patch，跳过 git 命令

### 1.1.1 只读变更采集脚本

为避免一次性把整个仓库塞给模型，推荐使用只读脚本采集变更上下文，再由 Skill 定向读取相关调用方、接口、数据模型和测试文件。

脚本产出结构（字段语义同 git 原生输出）：

```json
{
  "source": "working-tree | staged | commit | range | patch",
  "status_short": [],
  "name_status": [],
  "numstat": [],
  "untracked_files": [],
  "diff_truncated": false,
  "unified_diff": ""
}
```

| 字段 | 用途 |
|------|------|
| `status_short` | 当前仓库有哪些文件发生变化 |
| `name_status` | 文件是新增/修改/删除/重命名 |
| `numstat` | 每个文件新增/删除了多少行 |
| `untracked_files` | 有哪些未被 Git 跟踪的文件 |
| `unified_diff` | 具体的代码差异 |
| `diff_truncated` | Diff 是否因过大被截断 |

**脚本安全策略（必须遵守）**：
- 整个采集过程保持**只读**，不修改代码、不运行测试、不执行 git textconv
- **不读取**未跟踪文件的具体内容（仅记录文件名）
- **不对**二进制文件进行文本转换
- Diff 超过指定大小时进行**截断**，并在报告中标注「部分文件未分析」
- 跳过 `lock`/`dist`/`build`/`node_modules` 等产物与依赖文件

### 1.2 获取 Diff 内容

执行对应的 git diff 命令，获取完整 diff 输出。

**失败模式与 Fallback**：

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|---------|-----------|
| 不在 git 仓库中 | 提示用户切换到 git 仓库目录 | 提示用户手动粘贴 diff 内容到对话中 |
| diff 内容为空 | 提示当前没有改动，确认分析范围 | 检查是否在正确的分支上；若用户确认无改动，终止分析并输出「无变更」报告 |
| diff 内容过大（超过 5000 行） | 按文件分批处理，优先分析 `.ts`/`.js`/`.py`/`.java` 等核心代码文件，跳过 `lock`/`dist`/`build` 等产物文件 | 若分批后仍超限，仅分析变更行数最多的前 20 个文件，报告中标注「部分文件未分析」 |
| git 命令执行失败（权限/网络） | 检查 git 配置和远程仓库连接 | 提示用户手动导出 diff 文件，通过文件路径加载 |

### 1.3 加载测试用例

从用户指定的路径加载测试用例文件。

**支持格式**：

| 格式 | 识别方式 | 解析要点 |
|------|---------|---------|
| Markdown | `.md` 后缀 | 提取表格中的用例 ID、标题、步骤、预期结果；或识别 `### TC-xxx` 等标题格式 |
| CSV | `.csv` 后缀 | 按列解析，识别 ID/标题/步骤/预期结果列 |
| JSON | `.json` 后缀 | 按结构化字段解析 |
| Excel | `.xlsx`/`.xls` 后缀 | 提示用户先转换为 CSV/Markdown，或使用对话粘贴 |

**多文件支持**：用户可指定多个文件路径或目录，递归加载所有用例文件。

**失败模式与 Fallback**：

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|---------|-----------|
| 文件不存在 | 提示用户确认路径，列出同目录下相似文件名供选择 | 终止分析，输出错误报告 |
| 格式无法识别 | 提示用户确认文件格式或提供示例 | 尝试按纯文本解析，提取可见的用例结构，标注「格式推断」 |
| 用例内容为空 | 提示文件中未找到测试用例 | 检查文件编码（UTF-8/GBK），尝试不同编码重新读取 |
| 用例数量过少（< 3 条） | 提示用户是否还有其他用例文件未提供 | 继续分析，报告中标注「用例覆盖不足，结果仅供参考」 |

🔴 **CHECKPOINT · 阶段 1 完成**：向用户展示以下内容，确认后才进入阶段 2：
- 分析范围（diff 模式 + 目标分支/commit）
- 加载的用例文件列表 + 解析出的用例数量
- 过滤掉的文件数量及原因

用户可修改分析范围或补充用例文件。

## 阶段 2：Diff 解析

**目标**：从 diff 原始内容中提取结构化变更清单。

### 2.1 提取变更文件列表

从 diff 中提取每个变更文件的：
- 文件路径
- 变更类型：新增（A）/ 修改（M）/ 删除（D）/ 重命名（R）
- 变更行数（新增/删除）

### 2.2 过滤无关文件

**跳过以下文件**（不纳入分析）：
- 锁文件：`package-lock.json`、`yarn.lock`、`pnpm-lock.yaml`
- 构建产物：`dist/`、`build/`、`out/`
- 配置文件（除非用例涉及配置）：`.eslintrc`、`.prettierrc`、`tsconfig.json`
- 文档文件：`*.md`（除非用例涉及文档内容）、`CHANGELOG.md`
- 测试文件本身：`*.test.*`、`*.spec.*`、`__tests__/`

### 2.3 解析每个文件的变更详情

对每个变更文件，提取：

1. **变更的函数/方法/类**：从 diff 的 `@@` hunk 头和上下文中识别
2. **变更的接口/API**：识别 URL 路径、HTTP 方法、请求/响应结构变化
3. **变更的字段/参数**：识别新增/删除/修改的字段名、参数名
4. **变更的业务逻辑**：基于以下规则推断业务行为变化：
   - 函数参数新增/删除 → 调用方需同步修改
   - 返回值结构变化 → 下游消费方预期结果变化
   - 条件分支新增/修改 → 边界条件行为变化
   - 数据库 schema 变更 → 涉及该表的所有接口/查询受影响

**失败模式与 Fallback**：

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|---------|-----------|
| 无法从 diff 中识别函数/类名（如 minified 代码） | 跳过函数级解析，仅记录文件级变更 | 报告中标注「minified 文件，仅记录文件级变更」 |
| 非代码文件混入 diff（如图片、二进制） | 按文件扩展名过滤非文本文件 | 在报告中标注「跳过 N 个非文本文件」 |
| diff 格式异常（非 unified diff） | 尝试按 git diff 默认格式解析 | 提示用户重新导出 diff |

### 2.4 输出结构化变更清单

```yaml
changes:
  - file: "src/api/user.ts"
    type: "M"
    functions:
      - name: "getUser"
        change: "返回值新增 avatar 字段"
    interfaces:
      - path: "GET /api/user/:id"
        change: "response 新增 avatar 字段"
    risk: "high"

  - file: "src/services/order.ts"
    type: "M"
    functions:
      - name: "createOrder"
        change: "新增必填参数 shippingAddress"
    interfaces:
      - path: "POST /api/orders"
        change: "request body 新增必填字段 shippingAddress"
    risk: "high"
```

### 2.5 跨层影响链路追踪

**目标**：不止停留在「改了哪些文件」，而是沿调用链把上下游串起来，找出间接影响。

变更点的真实影响往往不在改动文件本身，而在调用链上下游。例如开发只改了后端返回字段，但真正出问题的地方可能是：前端仍按旧字段解析、历史数据没有新字段、缓存里还是旧结构、另一个页面也调了这个接口、自动化用例仍断言旧结果。

**链路追踪方向**（自上而下）：

```
页面或路由
    ↓
组件和状态
    ↓
API 请求（方法 / 路径 / 参数 / 返回值）
    ↓
后端 Controller
    ↓
Service 业务逻辑
    ↓
Model 和数据库（schema / 默认值 / 历史数据）
    ↓
缓存、消息队列或外部系统
```

**追踪策略**：
- 对每个变更点，沿链路向上找**调用方**（谁会触发它），向下找**依赖方**（它调用了什么）
- 识别**同接口多调用方**场景：同一接口被订单详情页和退款页同时调用，任一端变化都需要两侧回归
- 识别**公共方法多业务方**场景：公共组件/工具方法被多处引用，变更影响范围需逐个评估
- 识别**数据流下游**场景：DB 字段变化要追到所有读写该字段的接口、缓存键、消息体

**输出格式**（追加到变更清单）：

```yaml
impact_chains:
  - change_ref: "src/api/user.ts::getUser"
    chain:
      - layer: "frontend"
        nodes: ["pages/user-profile", "pages/admin-user-list"]
        relation: "consumes_response"
      - layer: "api"
        nodes: ["GET /api/user/:id"]
        relation: "defines"
      - layer: "service"
        nodes: ["UserService.getProfile"]
        relation: "implements"
      - layer: "model"
        nodes: ["users.avatar (nullable)"]
        relation: "reads_from"
      - layer: "cache"
        nodes: ["user:profile:{id}"]
        relation: "cached_as"
    indirect_risks:
      - "历史用户数据 avatar 为空，前端需处理 null 展示"
      - "缓存未失效会返回旧结构，需确认失效策略"
```

### 2.6 前后端契约两侧检查

**目标**：接口契约变化必须同时检查两侧，不能只看变更的一端。

**检查清单**（任一端变更时都要逐项确认对端）：

| 变更侧 | 检查项 |
|--------|--------|
| 前端改了请求参数 | 后端是否同步接收新字段名？旧版客户端是否还在发旧字段？接口是否需要兼容两种字段？参数缺失时返回什么状态码？日志/监控是否还能记录用户信息？其他调用方是否也用这个接口？ |
| 后端改了返回字段 | 前端是否按新字段解析？历史数据是否包含新字段？缓存里保存的还是旧结构吗？另一个页面是否也调了这个接口？自动化用例是否仍断言旧结果？ |
| 后端改了业务规则 | 前端是否需要同步调整交互（如禁用按钮、隐藏入口）？权限校验是否需要同步？状态机是否一致？ |
| DB schema 变更 | 历史数据是否需要迁移？默认值是否兼容旧代码？读写该字段的接口是否都已更新？ |

**示例**：前端把请求体从 `{"userId": 1001}` 改为 `{"user_id": 1001}`，只看前端代码会得出「参数名调整」的结论。完整检查还需确认：后端是否同步接收 `user_id`、旧版客户端是否还会发送 `userId`、接口是否需要同时兼容两种字段、参数缺失时返回什么状态码、日志和监控是否还能记录用户信息、其他调用方是否也使用这个接口。

**输出格式**（追加到变更清单）：

```yaml
contract_checks:
  - interface: "POST /api/orders"
    changed_side: "frontend"
    checklist:
      - item: "后端是否同步接收 user_id"
        status: "confirmed | missing | unknown"
        evidence: "src/api/orders.ts 第 23 行已读取 user_id"
      - item: "旧版客户端兼容性"
        status: "unknown"
        evidence: "需确认是否仍需支持 userId"
```

🔴 **CHECKPOINT · 阶段 2 完成**：向用户展示结构化变更清单，确认后才进入阶段 3：
- 变更文件数 + 变更类型分布（新增/修改/删除）
- 关键变更点摘要（函数/接口/字段变化）
- 跨层影响链路摘要（多少变更点牵涉到上下游）
- 契约两侧检查待确认项数量
- 风险等级为 high 的变更点列表

用户可手动调整风险等级或排除某些变更点。

## 阶段 3：交叉分析

**目标**：将变更清单与测试用例交叉比对，输出两类问题。

### 3.0 变更固有风险参考清单

每个变更点在阶段 2 标注的 `risk` 字段，依据下表打标。**不是所有变更都值得投入相同的测试成本**，修改一段页面文案与修改支付金额计算逻辑，测试策略不能一样。

**高风险（任一命中即高风险）**：
- 登录和权限
- 支付、金额和订单
- 删除或破坏性操作
- 敏感数据
- 数据库迁移
- 公共接口不兼容
- 事务和并发
- 大范围使用的公共组件
- 发布配置和部署顺序

**中风险**：
- 核心业务流程
- 接口字段扩展
- 状态管理
- 表单校验
- 缓存和重试
- 有明确调用范围的共享服务

**低风险**：
- 独立页面文案
- 不改变接口的内部重构
- 影响范围较小的展示调整

🔴 **硬约束 · 降低风险必须有证据**：
- 「开发说改动很小」**不是**低风险证据
- 「这个公共方法只有一个调用方，接口契约没有变化，已有自动化覆盖了主要分支」**才是**降低风险的证据
- 如果代码覆盖不足、影响范围不清楚、前后端没有同步修改、或涉及历史数据，风险等级必须**继续提高**，不能下调

本清单用于：
- 阶段 2.4 变更清单的 `risk` 字段赋值
- 阶段 3.1 行为变更影响的「影响程度」校准（高风险变更即使无现有用例直接引用，也应主动追间接影响）
- 阶段 3.2 覆盖缺口的「风险等级」校准（高风险变更的覆盖缺口自动提级）
- 阶段 4 必测内容的 P0/P1/P2 优先级划分

### 3.1 第一轮：行为变更影响分析

**目标**：找出代码改动导致哪些现有测试用例的预期结果、前置条件或操作步骤不再成立。

**分析策略**：

对变更清单中的每个变更点，按以下优先级匹配测试用例：

1. **直接引用匹配**（高置信度）：用例中直接提到了变更的函数名、接口路径、字段名。判定规则：用例文本包含变更点的精确标识符（函数名/API路径/字段名），不区分大小写
2. **模块关联匹配**（中置信度）：用例标题或步骤中提到的业务模块与变更文件的模块路径一致。判定规则：变更文件路径含 `user` → 匹配用例中含「用户/user」关键词的；变更文件路径含 `order` → 匹配用例中含「订单/order」关键词的
3. **依赖链匹配**（低置信度）：变更的函数被其他模块 import/require，间接影响用例覆盖的功能。判定规则：diff 中出现的 import 语句指向变更文件 → 标记为潜在间接影响

**影响程度判定**：

| 程度 | 判定条件 |
|------|---------|
| 高 | 用例的预期结果直接依赖变更的字段/接口/函数，必须修改用例 |
| 中 | 用例的操作步骤可能受变更影响，建议验证 |
| 低 | 变更在同一模块但不直接影响用例核心路径，建议关注 |

**输出格式**：每个问题包含变更文件、变更内容、影响程度、受影响用例列表（用例 ID + 标题 + 具体影响说明）。

### 3.2 第二轮：覆盖缺口分析

**目标**：找出代码改动中没有被任何测试用例覆盖的功能点。

**分析策略**：

对变更清单中的每个变更点，检查：
1. 是否有用例直接测试了该变更的功能
2. 是否有用例间接覆盖了该变更的路径
3. 如果都没有，标记为覆盖缺口

**风险等级判定**：

| 等级 | 判定条件 |
|------|---------|
| 高 | 新增核心业务功能/接口，或修改了关键业务逻辑，无任何用例覆盖 |
| 中 | 修改了非核心功能的参数/返回值，或新增了辅助功能，无用例覆盖 |
| 低 | 代码重构/格式调整，行为未变，无需新增用例 |

**输出格式**：每个缺口包含变更文件、变更内容、风险等级、建议的补充用例方向。

### 3.3 Mode A / Mode B 双模式

根据用户是否提供需求文档，自动选择分析模式。两种模式影响阶段 3 的输出结构和阶段 4 的报告章节。

**模式判定**：
- **Mode A**（默认）：用户只提供了 git diff + 测试用例，未提供需求文档/验收标准/任务说明/Bug 描述
- **Mode B**：用户额外提供了需求文档路径（如 `docs/requirement.md`）、验收标准、任务说明或 Bug 描述

#### Mode A：只有代码，没有需求文档

适用场景：没有正式需求文档、只有开发口头描述、需求已过期、临时修复线上问题、只能拿到代码 Diff。

**输出三分原则**：根据代码变化推断业务意图时，必须明确区分以下三类信息，**避免 AI 把推断当成事实**。

| 类别 | 定义 | 示例 | 表述规范 |
|------|------|------|---------|
| 代码事实 | 代码中能直接观察到的变化 | 新增了一个接口参数；删除了一个状态判断；修改了数据库字段默认值；新增了一次权限校验 | 直接陈述，无需修饰 |
| 影响推断 | 根据调用关系推断出来的潜在影响 | 该接口被订单详情页和退款页同时调用；状态判断变化可能影响退款入口；新字段可能导致历史数据为空 | 必须标注置信度（高/中/低），并说明推断依据 |
| 待确认问题 | 代码本身无法回答的问题 | 状态 2 是否允许退款？老版本客户端是否必须兼容？新字段为空时产品希望展示什么？这次修改是否需要处理历史数据？ | 必须以问号结尾，不能给出臆测答案 |

🔴 **硬约束**：
- 测试范围可以根据代码推断，但**验收标准不能凭空创造**
- 不能直接说「产品需求就是允许状态 2 的用户退款」
- 必须写成「从状态判断变化推断，状态 2 可能被纳入退款范围，置信度中等，需要确认实际业务规则」

#### Mode B：需求与代码对照

适用场景：用户同时提供了需求文档、验收标准、任务说明或 Bug 描述。

**分析流程**：先拆解需求 → 把每条需求与代码实现对应 → 再把每条需求与测试用例对应 → 输出覆盖状态。

```
需求
  ↓
实现证据
  ↓
测试证据
  ↓
覆盖状态
```

**覆盖状态分类**：

| 状态 | 判定条件 |
|------|---------|
| 已覆盖 | 需求有实现证据，且有测试用例验证 |
| 部分覆盖 | 需求有实现证据，但测试用例只覆盖了部分场景（如只测了正向，未测异常分支） |
| 未发现实现 | 需求存在，但代码中找不到对应实现 |
| 实现与需求冲突 | 代码实现与需求描述矛盾（如需求要求管理员限制，但代码无角色校验） |
| 需确认 | 无法从代码确认实现是否符合需求（如需求模糊，或实现依赖外部配置） |

**示例**：需求「只有管理员可以删除项目」，代码中虽然增加了删除接口，但没有看到角色校验。结论不是简单的「新增了删除功能」，而是：

```yaml
requirement: "只有管理员可以删除项目"
implementation_evidence: "已新增 DELETE /api/projects/:id 接口"
permission_evidence: "未发现管理员角色判断"
test_evidence: "无用例覆盖权限拒绝场景"
status: "部分覆盖"
risk: "high"
suggested_test:
  - "[P0][安全/权限] 普通成员账号调用删除接口 → 返回 403，项目数据不变"
```

**疑似超范围实现识别**（Mode B 独有）：

除检查「需求有没有实现」外，还要识别代码改动是否超出需求范围。

例如：需求只要求修改订单列表展示，但代码同时放开了订单导出权限。这不一定就是 Bug，也可能是必要的技术调整，但因为缺少明确需求依据、又可能影响用户行为，所以必须**单独标记**让产品、开发、测试一起确认。

**输出格式**（追加到分析结果）：

```yaml
requirement_traceability:
  - requirement_id: "REQ-001"
    requirement: "只有管理员可以删除项目"
    implementation_evidence: "DELETE /api/projects/:id 已新增"
    test_evidence: "无用例覆盖权限拒绝"
    status: "部分覆盖"
    risk: "high"
out_of_scope_changes:
  - change: "订单导出接口去除了角色校验"
    related_requirement: "无（需求只要求修改订单列表展示）"
    risk: "high"
    action: "需产品/开发确认是否为预期改动"
```

**阶段 3 失败模式与 Fallback**：

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|---------|-----------|
| 用例格式不规范，无法提取结构化字段 | 尝试按自然语言理解用例意图，标注「低置信度匹配」 | 将该用例标记为「无法分析」，在报告中单独列出 |
| 变更点与用例无任何关联（交叉结果为空） | 检查用例是否覆盖了变更的模块；若不覆盖，标注为覆盖缺口 | 输出「无受影响用例」结论，同时列出所有覆盖缺口 |

## 阶段 4：生成报告

**目标**：将分析结果整理为结构化 Markdown 报告。

### 4.1 报告结构

```markdown
# 变更影响分析报告

> 生成时间：YYYY-MM-DD HH:mm
> 分析范围：[diff 描述]
> 用例文件：[路径列表]
> 分析模式：Mode A / Mode B

## 1. 分析范围

明确本次分析的是：当前工作区 / 暂存区 / 某个 Commit / 分支差异 / PR / 外部 Patch。
同时说明哪些模块已经深入检查，哪些内容没有读取或无法验证。

## 2. 改动与需求摘要

区分三类信息（Mode A 默认输出，Mode B 追加需求对照）：
- 有直接代码证据的变化
- 根据代码推断的需求意图（标注置信度）
- 需求与实现的匹配结论（Mode B）

## 3. 影响链路

用表格串联：业务入口 → 前端 → 接口 → 后端 → 数据库或外部依赖。
并说明属于直接影响、间接回归、兼容性风险，还是部署和数据风险。

| 变更点 | 业务入口 | 前端 | 接口 | 后端 | 数据/外部 | 影响类型 |
|--------|---------|------|------|------|----------|---------|
| getUser 返回新增 avatar | 用户资料页 | pages/user-profile | GET /api/user/:id | UserService.getProfile | users.avatar | 直接影响 |
| 状态判断放开 state=2 | — | — | — | OrderService.refund | — | 间接回归 |

## 4. 风险结论

列出风险等级、判断原因，以及对应的验证方式。高风险必须给出具体证据，不能仅靠"开发说改动很小"下调。

| 变更点 | 风险等级 | 判断原因 | 验证方式 |
|--------|---------|---------|---------|
| 删除接口未做角色校验 | 高 | 涉及权限+删除操作，代码未发现角色判断 | 权限拒绝场景必测 |
| 订单列表新增展示字段 | 低 | 仅展示调整，接口契约不变，无历史数据问题 | 界面回归 |

## 5. 必测内容

每条场景采用统一结构（详见 4.3 测试场景模板）：

### [P0][API/权限] 普通成员无法删除他人项目

- **前置条件**：普通成员账号已登录，并拥有目标项目的查看权限，但没有删除权限。
- **操作**：调用 `DELETE /api/projects/{id}`。
- **预期结果**：
  - 接口返回 403
  - 项目记录未被删除
  - 关联数据保持不变
  - 审计日志记录拒绝原因
- **证据**：删除接口发生变化，且需求要求仅管理员可删除。

## 6. 回归范围

回归范围拆成三类：

- **核心回归**：本次改动直接影响的功能和接口
- **定向回归**：通过影响链路追踪发现的间接影响（同接口多调用方、公共方法多业务方、数据流下游）
- **可不回归**：有明确隔离证据的部分（如公共方法只有一个调用方且接口契约不变、已有自动化覆盖主要分支）

| 回归类型 | 范围 | 原因 |
|---------|------|------|
| 核心回归 | 用户资料页、GET /api/user/:id | 直接修改 |
| 定向回归 | 退款页（同接口多调用方） | 间接影响 |
| 可不回归 | 订单导出（独立模块，无关联） | 隔离证据明确 |

## 7. 需求追踪矩阵（Mode B 独有）

Mode B 下逐条展示需求、实现证据、测试建议和覆盖状态。Mode A 跳过本节。

| 需求 ID | 需求描述 | 实现证据 | 测试证据 | 覆盖状态 | 风险 |
|---------|---------|---------|---------|---------|------|
| REQ-001 | 只有管理员可以删除项目 | 已新增删除接口 | 未覆盖权限拒绝 | 部分覆盖 | 高 |

疑似超范围实现单独列出：

| 变更点 | 关联需求 | 风险 | 后续动作 |
|--------|---------|------|---------|
| 订单导出去除角色校验 | 无 | 高 | 需产品/开发确认 |

## 8. 待确认与限制

把无法从代码确认的问题明确列出，而不是让 AI 自己补全答案。

- 状态 2 是否允许退款？（需产品确认）
- 老版本客户端是否必须兼容旧字段？（需客户端团队确认）
- 新字段为空时产品希望展示什么？（需设计确认）
- 本次修改是否需要处理历史数据？（需 DBA 确认）

### 报告限制说明

- 本报告基于代码变更推断，无法还原完整产品需求
- 涉及灰度开关、运营后台配置、第三方系统状态的内容需另行确认
```

### 4.2 P0 / P1 / P2 优先级划分

为每条必测内容标注优先级，**优先级的意义不是减少测试**，而是在时间有限时明确哪些问题必须在发布前发现。

| 优先级 | 定义 | 常见场景 |
|--------|------|---------|
| **P0** | 不通过就不建议发布 | 核心业务不可用；金额计算错误；数据可能丢失；权限可以被绕过；数据库迁移失败；公共接口发生破坏性变化 |
| **P1** | 覆盖本次改动的核心行为 | 新增功能主流程；主要异常分支；核心状态转换；前后端接口契约；直接受到影响的关联页面 |
| **P2** | 次要回归和低概率边界 | 非核心浏览器兼容；次要页面展示；较低概率异常；与改动距离较远但存在一定关联的流程 |

### 4.3 测试场景模板

每条必测内容必须采用以下统一结构，**不能只写"验证 XX 是否正常"这种不可执行的描述**。

```
[优先级][测试类型]

前置条件：
- 数据准备、账号、权限、环境状态

操作：
- 具体的执行步骤、入口、接口调用

预期结果：
- 状态码 / 页面状态 / 数据库存储结果 / 事件是否发送 / 权限是否拒绝 / 回滚是否成功

证据：
- 对应的代码变更点、需求条目或调用链路
```

**反例**（禁止）：
- 验证订单功能是否正常
- 验证接口是否正确
- 验证权限是否生效

**正例**：

```
[P0][API/权限]

前置条件：
- 普通成员账号已登录，并拥有目标项目的查看权限，但没有删除权限。

操作：
- 调用 DELETE /api/projects/{id}

预期结果：
- 接口返回 403
- 项目记录未被删除
- 关联数据保持不变
- 审计日志记录拒绝原因

证据：
- 删除接口发生变化，且需求要求仅管理员可删除。
```

### 4.4 报告输出

- **默认路径**：`docs/change-impact-report.md`
- **自定义路径**：用户可通过参数指定输出路径
- **文件名建议**：包含日期，如 `change-impact-report-2026-07-08.md`

## 使用示例

### 示例 1：分析本地改动

```
用户：帮我分析一下当前改动对测试用例的影响，用例在 docs/test-cases.md
```

**执行流程**：
1. 执行 `git diff` + `git diff --cached` 获取本地改动
2. 加载 `docs/test-cases.md` 中的测试用例
3. 解析变更 → 交叉分析 → 生成报告到 `docs/change-impact-report.md`

```
用户：对比 main 和 feat/order-cancel 分支的改动，检查对 order 相关用例的影响
```

**执行流程**：
1. 执行 `git diff main...feat/order-cancel` 获取分支差异
2. 加载用户指定的用例文件
3. 解析变更 → 交叉分析 → 生成报告

### 示例 3：分析特定 commit

```
用户：检查 commit abc1234 的改动是否影响了 regression 用例
```

**执行流程**：
1. 执行 `git diff abc1234~1..abc1234` 获取 commit 变更
2. 加载 regression 用例文件
3. 解析变更 → 交叉分析 → 生成报告

### 示例 4：Mode B 需求与代码对照

```
用户：使用 $analyze-change-test-scope，对照 docs/requirement.md，
分析 main...当前分支的代码改动，输出需求覆盖情况和测试范围
```

**执行流程**：
1. 执行 `git diff main...HEAD` 获取分支差异
2. 加载 `docs/requirement.md` 拆解需求条目
3. 加载测试用例文件（如有）
4. 阶段 2 解析变更 + 影响链路 + 契约两侧检查
5. 阶段 3 进入 Mode B：需求 × 实现 × 测试三向对照，识别超范围实现
6. 生成报告（包含第 7 节需求追踪矩阵）

### 示例 5：分析外部 Patch

```

diff --git a/src/api/user.ts ...
```

**执行流程**：
1. 识别为外部 Patch，跳过 git 命令
2. 直接解析 patch 内容提取变更清单
3. 加载测试用例 → 交叉分析 → 生成报告

## 适用阶段

| 阶段 | 用法 | 价值 |
|------|------|------|
| **开发提测前** | 分析当前分支相对 main 的变化 | 测试可提前知道哪些模块要准备数据、哪些接口要重点验证、是否涉及 DB/配置、是否有需求遗漏 |
| **测试分析阶段** | 需求评审结束后，需求 + 代码变更一起输入（Mode B） | 检查需求是否全部实现、实现是否与需求冲突、是否超范围修改、用例是否漏掉代码新风险 |
| **代码合并前** | 对 PR Diff 做风险扫描 | 重点检查公共组件、接口契约、权限、数据库、配置、发布顺序 |
| **发布前** | 时间有限时按 P0/P1/P2 重新确认测试完成情况 | 把高风险遗漏优先补齐 |

## 不能替代的边界

🔴 **硬约束 · 代码变更无法完整还原真实产品需求**。代码里通常看不到：

- 产品为什么要这样设计
- 运营后台配置了什么
- 线上是否存在灰度开关
- 第三方系统当前是什么状态
- 团队约定但没有写进代码的业务规则
- 某些历史数据究竟是什么情况

因此本 skill **不能替代**：

- 需求评审
- 测试人员的业务判断
- 开发代码评审
- 真实环境验证
- 最终发布决策

它更适合承担一个「测试分析助手」的角色：帮测试人员快速整理代码事实、影响链路、风险和待确认问题，减少遗漏，但**不替人做最终判断**。

## 敏感信息与脱敏

代码 Diff 中可能包含接口地址、业务数据、密钥、用户信息或其他敏感信息。

**安全建议**：
- 优先在**受信任的本地环境**中分析，避免把 Diff 内容上传到外部服务
- 报告对外分享前必须进行**脱敏**：替换真实接口路径、用户 ID、手机号、邮箱、密钥、内部域名
- 涉及密钥/凭据的变更点，报告中只标注「涉及敏感字段，需单独评审」，不复制具体值
- Mode B 的需求追踪矩阵对外分享前，需确认需求文档本身是否允许外传

## 失败模式总览

> 各阶段详细 fallback 见对应章节内联表。本表为速查索引。

| 阶段 | 触发条件 | 一线修复 | 仍失败兜底 |
|------|---------|---------|-----------|
| 阶段 1 | 不在 git 仓库 | 提示切换目录 | 提示手动粘贴 diff |
| 阶段 1 | diff 为空 | 确认分析范围 | 输出「无变更」报告 |
| 阶段 1 | diff 过大（>5000行） | 分批处理，跳过产物文件 | 仅分析前 20 个文件，标注「部分未分析」 |
| 阶段 1 | 用例文件不存在 | 列出相似文件名供选择 | 终止分析 |
| 阶段 1 | 用例格式无法识别 | 提示确认格式 | 按纯文本推断，标注「格式推断」 |
| 阶段 2 | 无法识别函数/类名 | 跳过函数级解析 | 标注「minified，仅文件级」 |
| 阶段 2 | 非文本文件混入 diff | 按扩展名过滤 | 报告标注跳过数量 |
| 阶段 3 | 用例格式不规范 | 自然语言理解，标注低置信度 | 标记「无法分析」单独列出 |
| 阶段 3 | 交叉结果为空 | 检查模块覆盖 | 输出覆盖缺口清单 |
| 阶段 4 | 输出目录不存在 | 自动创建目录 | — |

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|------------|---------|
| 1 | 跳过阶段 1 CHECKPOINT 直接分析 | 分析范围错误会导致后续所有结果无效，浪费 token | 必须让用户确认 diff 范围和用例文件后再继续 |
| 2 | 将测试文件本身的变更计入影响分析 | 测试文件改动不是"代码改动对用例的影响"，混淆分析目标 | 阶段 2.2 过滤 `*.test.*`/`*.spec.*`/`__tests__/` |
| 3 | 把所有变更都标记为"高风险" | 虚高风险会让报告失去参考价值，用户无法区分优先级 | 严格按影响程度/风险等级判定表打标，低风险也要标注 |
| 4 | 在阶段 3 中修改变更清单 | 变更清单是阶段 2 的确定性输出，阶段 3 只做交叉分析 | 如需调整，回退到阶段 2 CHECKPOINT 让用户修改 |
| 5 | 报告中只列问题不给建议 | 用户需要可落地的行动项，而非仅发现问题 | 每个问题必须附带建议操作（修改用例/补充用例/验证确认） |
| 6 | 对 Excel 格式强行解析 | Excel 解析不可靠，容易丢失格式或乱码 | 提示用户转换为 CSV/Markdown，或在对话中粘贴内容 |
| 7 | 忽略重命名/移动文件的语义 | 文件重命名可能导致用例中引用的路径失效 | 阶段 2 识别重命名（R）类型，阶段 3 检查用例中的文件路径引用 |
