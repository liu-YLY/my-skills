# 反例与黑名单

> **何时阅读**：贯穿阶段 1-4 执行时查阅，避免常见反模式导致分析失真或报告失效。
> **覆盖范围**：7 条核心反例（取自 SKILL.md）+ 7 条扩展项（README.md 中标注但 SKILL.md 主流程未列入核心）+ 触发检查时机。
> **设计依据**：基于 SkillLens 论文（arXiv 2605.23899）实证——只写"应该做 X"没有"不要做 Y"会导致 LLM judge 准确率下降。

## 1. 核心反模式（7 条，取自 SKILL.md）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|------------|---------|
| 1 | 跳过阶段 1 CHECKPOINT 直接分析 | 分析范围错误会导致后续所有结果无效，浪费 token | 必须让用户确认 diff 范围和用例文件后再继续 |
| 2 | 将测试文件本身的变更计入影响分析 | 测试文件改动不是"代码改动对用例的影响"，混淆分析目标 | 阶段 2.2 过滤 `*.test.*`/`*.spec.*`/`__tests__/` |
| 3 | 把所有变更都标记为"高风险" | 虚高风险会让报告失去参考价值，用户无法区分优先级 | 严格按影响程度/风险等级判定表打标，低风险也要标注 |
| 4 | 在阶段 3 中修改变更清单 | 变更清单是阶段 2 的确定性输出，阶段 3 只做交叉分析 | 如需调整，回退到阶段 2 CHECKPOINT 让用户修改 |
| 5 | 报告中只列问题不给建议 | 用户需要可落地的行动项，而非仅发现问题 | 每个问题必须附带建议操作（修改用例/补充用例/验证确认） |
| 6 | 对 Excel 格式强行解析 | Excel 解析不可靠，容易丢失格式或乱码 | 提示用户转换为 CSV/Markdown，或在对话中粘贴内容 |
| 7 | 忽略重命名/移动文件的语义 | 文件重命名可能导致用例中引用的路径失效 | 阶段 2 识别重命名（R）类型，阶段 3 检查用例中的文件路径引用 |

## 2. 核心反模式详解

### 反模式 1：跳过阶段 1 CHECKPOINT 直接分析

**典型表现**：用户说"分析当前改动"后，未确认 diff 模式与用例文件就直接进入阶段 2。

**问题**：
- diff 范围可能错误（如本应分析暂存区，实际分析了工作区）
- 用例文件路径可能不存在或格式不识别
- 后续所有结果都基于错误前提，浪费 token 与时间

**替代做法**：阶段 1 完成后必须展示 🔴 CHECKPOINT，明确分析范围、用例文件列表、解析出的用例数量、过滤掉的文件，由用户确认后进入阶段 2。

### 反模式 2：将测试文件本身的变更计入影响分析

**典型表现**：diff 中包含 `user.test.ts` 的改动，将其作为"代码变更"参与交叉分析。

**问题**：
- 测试文件改动是"测试本身的更新"，不是"被测代码的变更"
- 混淆分析目标，导致报告出现"测试改动影响了测试"的循环结论
- 浪费分析预算

**替代做法**：阶段 2.2 主动过滤 `*.test.*`/`*.spec.*`/`__tests__/`，并在 CHECKPOINT 中说明过滤掉了多少测试文件。

### 反模式 3：把所有变更都标记为"高风险"

**典型表现**：对所有变更点都打 `risk: high`，认为"宁可信其高"。

**问题**：
- 虚高风险让报告失去参考价值，用户无法区分真正需要关注的点
- P0/P1/P2 优先级失去校准基准
- 测试团队无法据此分配资源

**替代做法**：严格按 [cross-impact-analysis.md](cross-impact-analysis.md) 第 3 节风险三档清单打标。低风险也要明确标注（如"独立页面文案调整"），让用户看到完整风险分布。

### 反模式 4：在阶段 3 中修改变更清单

**典型表现**：阶段 3 发现某变更点风险很高，直接在变更清单中调整其 `risk` 字段或删除某些条目。

**问题**：
- 变更清单是阶段 2 的确定性输出，是阶段 3 的输入
- 在阶段 3 修改输入会破坏数据流的可追溯性
- 用户难以察觉清单已被改动

**替代做法**：如需调整变更清单（如修正风险等级、排除误识别条目），回退到阶段 2 CHECKPOINT 让用户修改，再重新进入阶段 3。

### 反模式 5：报告中只列问题不给建议

**典型表现**：报告仅写"DELETE 接口缺少权限校验"，不说明后续应做什么。

**问题**：
- 用户需要可落地的行动项，而非问题清单
- 没有建议的问题需要用户二次思考，增加沟通成本
- 报告价值未被充分释放

**替代做法**：每个问题必须附带建议操作，至少包括：
- 修改用例 / 补充用例 / 验证确认 三选一
- 涉及的用例 ID 或建议新增的用例方向
- 优先级标注（P0/P1/P2）

### 反模式 6：对 Excel 格式强行解析

**典型表现**：用户提供 `.xlsx` 用例文件，直接尝试按单元格解析。

**问题**：
- Excel 解析不可靠，合并单元格、公式、数据验证等都会丢失信息
- 不同 Excel 版本可能乱码
- 表头识别失败导致字段映射错误

**替代做法**：提示用户先转换为 CSV/Markdown，或在对话中直接粘贴用例内容。若用户坚持使用 Excel，标注"格式推断，结果仅供参考"。

### 反模式 7：忽略重命名/移动文件的语义

**典型表现**：diff 中出现 `R old/path.ts → new/path.ts`，仅当作"删除+新增"两个独立事件处理。

**问题**：
- 测试用例中可能仍引用旧路径（如 `require('../../../old/path')`）
- 重命名可能伴随模块职责变化，影响链路追踪
- 漏报"用例中路径引用失效"的高风险问题

**替代做法**：
- 阶段 2 识别 `R`（rename）类型，保留 old/new 两个路径
- 阶段 3 检查用例文本是否引用了旧路径
- 在变更清单中标注重命名，提示用户检查上下游引用

## 3. 扩展项（README.md 中标注但 SKILL.md 主流程未列入核心）

> README.md 共 12 条反例，本文件核心 7 条中 5 条与 README.md 重叠、2 条为 SKILL.md 独有。以下 7 条为 README.md 中标注但未列入 SKILL.md 主流程核心的反例，作为扩展项一并遵守。

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|------------|---------|
| E1 | 降低风险没有证据就下调等级 | "开发说改动很小"不构成低风险证据，凭印象下调会漏掉高风险 | 必须给出「调用方数量/契约一致性/自动化覆盖」等具体证据才能下调 |
| E2 | Mode A 把推断当事实陈述 | 推断与事实混淆会让用户误以为代码已经验证，导致决策失误 | 必须区分代码事实/影响推断/待确认问题，推断需标注置信度 |
| E3 | 验收标准凭空创造 | 测试范围可推断，但验收标准不能臆测，否则会以假需求指导测试 | 验收标准不明确时列入「待确认问题」，不能给出臆测答案 |
| E4 | 测试场景写"验证 XX 是否正常" | 不可执行，依赖主观判断，无法判定 pass/fail | 必须用 `[P级][类型] 前置→操作→预期—证据` 结构（详见 [report-template.md](report-template.md) 第 3 节） |
| E5 | 只看变更一侧不查契约对端 | 接口契约变更只看一端会漏掉对端未同步修改的问题 | 接口契约变更必须同时检查前后端两侧（详见 [cross-impact-analysis.md](cross-impact-analysis.md) 第 2 节） |
| E6 | 把整个仓库塞给模型 | 上下文超限导致模型注意力分散，关键变更点被淹没 | 用只读采集脚本采集变更上下文，定向读取相关文件（详见 [diff-modes.md](diff-modes.md) 第 4 节） |
| E7 | 报告对外分享前未脱敏 | Diff 中可能包含接口地址、密钥、用户信息，直接分享会泄露敏感数据 | 替换接口路径/用户 ID/密钥等敏感信息，涉及密钥只标注"涉及敏感字段" |

## 5. 自检清单

阶段 4 输出前，对照以下清单自检：

- [ ] 阶段 1 CHECKPOINT 已由用户确认？（核心 1）
- [ ] 测试文件改动已过滤？（核心 2）
- [ ] 风险等级有证据支撑，不是全部"高风险"？（核心 3、扩展 E1）
- [ ] 阶段 3 未修改变更清单？（核心 4）
- [ ] 每个问题都附带了建议操作？（核心 5）
- [ ] Excel 用例未强行解析？（核心 6）
- [ ] 重命名/移动文件的语义已处理？（核心 7）
- [ ] Mode A 下推断与事实已区分？（扩展 E2、扩展 E3）
- [ ] 测试场景采用 `[P级][类型] 前置→操作→预期—证据` 结构？（扩展 E4）
- [ ] 接口契约两侧都已检查？（扩展 E5）
- [ ] 未把整个仓库塞给模型？（扩展 E6）
- [ ] 报告对外分享前已脱敏？（扩展 E7）

## 7. 失败模式速查（从 SKILL.md 下沉）

> 各阶段详细 fallback 见 [../SKILL.md](../SKILL.md) 对应章节内联表。本表为速查索引。

| 阶段 | 触发条件 | 一线修复 | 仍失败兜底 |
|------|---------|---------|-----------|
| 阶段 1 | 不在 git 仓库 | 提示切换目录 | 提示手动粘贴 diff |
| 阶段 1 | diff 为空 | 确认分析范围 | 输出「无变更」报告 |
| 阶段 1 | diff 过大（>5000行） | 分批处理，跳过产物文件 | 仅分析前 20 个文件，标注「部分未分析」 |
| 阶段 1 | 用例文件不存在 | 列出相似文件名供选择 | 终止分析 |
| 阶段 1 | 用例格式无法识别 | 提示确认格式 | 按纯文本推断，标注「格式推断」 |
| 阶段 2 | 无法识别函数/类名 | 跳过函数级解析 | 标注「minified，仅文件级」 |
| 阶段 2 | 非文本文件混入 diff | 按扩展名过滤 | 报告标注跳过数量 |
| 阶段 3 | 用例格式不规范 | 自然语言理解，标注低置信度 | 标记「无法分析」单独列出 |
| 阶段 3 | 交叉结果为空 | 检查模块覆盖 | 输出覆盖缺口清单 |
| 阶段 4 | 输出目录不存在 | 自动创建目录 | — |
