# 报告模板

> **何时阅读**：阶段 4「生成报告」时加载本文件，按 8 段结构组织输出。
> **覆盖范围**：8 段报告结构 / P0/P1/P2 优先级划分规则 / 测试场景模板 / 输出路径 / 完整示例。

## 1. 8 段报告结构

```markdown
# 变更影响分析报告

> 生成时间：YYYY-MM-DD HH:mm
> 分析范围：[diff 描述]
> 用例文件：[路径列表]
> 分析模式：Mode A / Mode B

## 1. 分析范围

明确本次分析的是：当前工作区 / 暂存区 / 某个 Commit / 分支差异 / PR / 外部 Patch。
同时说明哪些模块已经深入检查，哪些内容没有读取或无法验证。

## 2. 改动与需求摘要

区分三类信息（Mode A 默认输出，Mode B 追加需求对照）：
- 有直接代码证据的变化
- 根据代码推断的需求意图（标注置信度）
- 需求与实现的匹配结论（Mode B）

## 3. 影响链路

用表格串联：业务入口 → 前端 → 接口 → 后端 → 数据库或外部依赖。
并说明属于直接影响、间接回归、兼容性风险，还是部署和数据风险。

| 变更点 | 业务入口 | 前端 | 接口 | 后端 | 数据/外部 | 影响类型 |
|--------|---------|------|------|------|----------|---------|
| getUser 返回新增 avatar | 用户资料页 | pages/user-profile | GET /api/user/:id | UserService.getProfile | users.avatar | 直接影响 |
| 状态判断放开 state=2 | — | — | — | OrderService.refund | — | 间接回归 |

## 4. 风险结论

列出风险等级、判断原因，以及对应的验证方式。高风险必须给出具体证据，不能仅靠"开发说改动很小"下调。

| 变更点 | 风险等级 | 判断原因 | 验证方式 |
|--------|---------|---------|---------|
| 删除接口未做角色校验 | 高 | 涉及权限+删除操作，代码未发现角色判断 | 权限拒绝场景必测 |
| 订单列表新增展示字段 | 低 | 仅展示调整，接口契约不变，无历史数据问题 | 界面回归 |

## 5. 必测内容

每条场景采用统一结构（详见第 3 节测试场景模板）：

### [P0][API/权限] 普通成员无法删除他人项目

- **前置条件**：普通成员账号已登录，并拥有目标项目的查看权限，但没有删除权限。
- **操作**：调用 `DELETE /api/projects/{id}`。
- **预期结果**：
  - 接口返回 403
  - 项目记录未被删除
  - 关联数据保持不变
  - 审计日志记录拒绝原因
- **证据**：删除接口发生变化，且需求要求仅管理员可删除。

## 6. 回归范围

回归范围拆成三类：

- **核心回归**：本次改动直接影响的功能和接口
- **定向回归**：通过影响链路追踪发现的间接影响（同接口多调用方、公共方法多业务方、数据流下游）
- **可不回归**：有明确隔离证据的部分（如公共方法只有一个调用方且接口契约不变、已有自动化覆盖主要分支）

| 回归类型 | 范围 | 原因 |
|---------|------|------|
| 核心回归 | 用户资料页、GET /api/user/:id | 直接修改 |
| 定向回归 | 退款页（同接口多调用方） | 间接影响 |
| 可不回归 | 订单导出（独立模块，无关联） | 隔离证据明确 |

## 7. 需求追踪矩阵（Mode B 独有）

Mode B 下逐条展示需求、实现证据、测试建议和覆盖状态。Mode A 跳过本节。

| 需求 ID | 需求描述 | 实现证据 | 测试证据 | 覆盖状态 | 风险 |
|---------|---------|---------|---------|---------|------|
| REQ-001 | 只有管理员可以删除项目 | 已新增删除接口 | 未覆盖权限拒绝 | 部分覆盖 | 高 |

疑似超范围实现单独列出：

| 变更点 | 关联需求 | 风险 | 后续动作 |
|--------|---------|------|---------|
| 订单导出去除角色校验 | 无 | 高 | 需产品/开发确认 |

## 8. 待确认与限制

把无法从代码确认的问题明确列出，而不是让 AI 自己补全答案。

- 状态 2 是否允许退款？（需产品确认）
- 老版本客户端是否必须兼容旧字段？（需客户端团队确认）
- 新字段为空时产品希望展示什么？（需设计确认）
- 本次修改是否需要处理历史数据？（需 DBA 确认）

### 报告限制说明

- 本报告基于代码变更推断，无法还原完整产品需求
- 涉及灰度开关、运营后台配置、第三方系统状态的内容需另行确认
```

## 2. P0 / P1 / P2 优先级划分

为每条必测内容标注优先级，**优先级的意义不是减少测试**，而是在时间有限时明确哪些问题必须在发布前发现。

| 优先级 | 定义 | 常见场景 |
|--------|------|---------|
| **P0** | 不通过就不建议发布 | 核心业务不可用；金额计算错误；数据可能丢失；权限可以被绕过；数据库迁移失败；公共接口发生破坏性变化 |
| **P1** | 覆盖本次改动的核心行为 | 新增功能主流程；主要异常分支；核心状态转换；前后端接口契约；直接受到影响的关联页面 |
| **P2** | 次要回归和低概率边界 | 非核心浏览器兼容；次要页面展示；较低概率异常；与改动距离较远但存在一定关联的流程 |

## 3. 测试场景模板

每条必测内容必须采用以下统一结构，**不能只写"验证 XX 是否正常"这种不可执行的描述**。

```
[优先级][测试类型]

前置条件：
- 数据准备、账号、权限、环境状态

操作：
- 具体的执行步骤、入口、接口调用

预期结果：
- 状态码 / 页面状态 / 数据库存储结果 / 事件是否发送 / 权限是否拒绝 / 回滚是否成功

证据：
- 对应的代码变更点、需求条目或调用链路
```

### 3.1 反例（禁止）

- 验证订单功能是否正常
- 验证接口是否正确
- 验证权限是否生效

### 3.2 正例

```
[P0][API/权限]

前置条件：

操作：
- 调用 DELETE /api/projects/{id}

预期结果：
- 接口返回 403
- 项目记录未被删除
- 关联数据保持不变
```

## 5. 完整报告示例（Mode B 场景）

> 场景：对照 `docs/requirement.md` 分析 `main...HEAD` 分支的代码改动，需求涉及「项目删除权限控制」与「订单导出展示调整」。

```markdown
# 变更影响分析报告

> 生成时间：2026-07-08 14:32
> 分析范围：main...feat/project-delete-permission 分支差异
> 用例文件：docs/test-cases.md
> 分析模式：Mode B

## 1. 分析范围

本次分析 `feat/project-delete-permission` 分支相对 `main` 的全部改动，已深入检查：
- 项目删除接口（src/api/projects.ts）
- 订单导出接口（src/api/orders.ts）

未读取：
- 运营后台配置（无法从代码确认）
- 线上灰度开关状态（需另行确认）

## 2. 改动与需求摘要

**代码事实**：
- 新增 `DELETE /api/projects/:id` 接口
- 订单导出接口去除 `requireRole('admin')` 中间件
- 用户表新增 `avatar` 字段（nullable）

**影响推断**（置信度中）：
- 项目删除接口未发现角色校验，可能允许任意登录用户删除项目
- 订单导出去除权限校验，可能允许普通成员导出全部订单数据

**Mode B 需求对照**：
- REQ-001「只有管理员可以删除项目」→ 实现部分覆盖（缺权限校验）
- REQ-002「订单列表新增展示字段」→ 已覆盖
- 订单导出去除角色校验 → 疑似超范围实现

## 3. 影响链路

| 变更点 | 业务入口 | 前端 | 接口 | 后端 | 数据/外部 | 影响类型 |
|--------|---------|------|------|------|----------|---------|
| 新增删除接口（无权限校验） | 项目列表页 | pages/projects | DELETE /api/projects/:id | ProjectService.delete | projects 表 | 直接影响 |
| 订单导出去除 admin 校验 | 订单管理页 | pages/orders-export | GET /api/orders/export | OrderService.export | — | 间接回归 |
| users.avatar 新增 | 用户资料页 | pages/user-profile | GET /api/user/:id | UserService.getProfile | users.avatar | 兼容性风险 |

## 4. 风险结论

| 变更点 | 风险等级 | 判断原因 | 验证方式 |
|--------|---------|---------|---------|
| 删除接口未做角色校验 | 高 | 涉及权限+删除操作，代码未发现角色判断 | 权限拒绝场景必测 |
| 订单导出去除角色校验 | 高 | 涉及权限+敏感数据，且属于超范围实现 | 权限拒绝场景必测 |
| users.avatar 新增 | 中 | 涉及历史数据兼容，前端需处理 null | 历史用户访问验证 |

## 5. 必测内容

### [P0][API/权限] 普通成员无法删除他人项目

- **前置条件**：普通成员账号已登录，并拥有目标项目的查看权限，但没有删除权限。
- **操作**：调用 `DELETE /api/projects/{id}`。
- **预期结果**：
  - 接口返回 403
  - 项目记录未被删除

  - 审计日志记录拒绝原因
- **证据**：删除接口发生变化，且需求要求仅管理员可删除。

### [P0][API/权限] 普通成员无法导出全部订单

- **前置条件**：普通成员账号已登录。
- **操作**：调用 `GET /api/orders/export`。
- **预期结果**：
  - 接口返回 403
  - 不返回订单数据
- **证据**：订单导出接口去除了 admin 校验，属超范围实现，需产品确认是否预期。

### [P1][兼容性] 历史用户访问资料页不报错

- **前置条件**：数据库中存在 avatar 字段为 NULL 的历史用户。
- **操作**：登录该历史用户，访问 `/profile` 页面。
- **预期结果**：
  - 页面正常渲染
  - avatar 位置展示默认头像或占位符
  - 不报 500 错误
- **证据**：users.avatar 为新增 nullable 字段，历史数据无值。
```
