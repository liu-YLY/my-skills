# 瓶颈模式知识库

> **何时阅读**：性能测试定位阶段需要识别瓶颈类型、选择优化方向时查阅。
> **覆盖范围**：CPU / IO / 网络 / 锁 / 连接池 / GC 六类资源架构层瓶颈模式、传导链、反例。

## 层边界声明（必读）

本库**仅聚焦资源/架构层**瓶颈。**代码逻辑层**缺陷转交 `bug-analyzer` skill 处理。

| 层级 | 范围 | 典型问题 | 处理方 |
|------|------|----------|--------|
| 资源/架构层 | CPU / IO / 网络 / 锁 / 连接池 / GC 等系统级资源饱和或架构设计问题 | 慢 SQL、锁竞争、堆配置不当、带宽打满 | 本库 |
| 代码逻辑层 | 具体代码实现的逻辑错误、异常路径、数据错误 | 空指针、业务规则错误、资源未释放、死循环逻辑错误 | bug-analyzer |

**判断规则**：先看资源层指标是否饱和。若资源饱和 → 本库处理；若资源正常但行为错误 → 转 bug-analyzer。两者并存时，先定位资源层根因，再判断是否叠加代码逻辑缺陷。

## 1. CPU 飙升

| 现象 | 定位指标 | 根因候选 | 优化方向 | 是否转交 bug-analyzer |
|------|---------|---------|---------|---------------------|
| CPU 利用率 > 80%，runqueue > 核数 × 2，上下文切换激增 | CPU 利用率（总体+单核）、runqueue、上下文切换次数、火焰图热点 | 死循环、正则回溯、序列化开销、GC 频繁、计算密集业务 | 算法优化、缓存、异步化、水平扩容 | 死循环 / 正则回溯等代码逻辑缺陷 → 转交 |

**定位步骤**：
1. `top` / `mpstat` 确认 CPU 是 sys 高还是 user 高
2. sys 高 → 查上下文切换、系统调用；user 高 → 火焰图定位热点函数
3. 排除 GC 频繁导致的 CPU 假性飙升（结合 GC 指标）
4. 火焰图定位到具体方法后判断：算法/计算密集 → 本库；逻辑错误（如死循环）→ 转 bug-analyzer

## 2. IO 等待

| 现象 | 定位指标 | 根因候选 | 优化方向 | 是否转交 bug-analyzer |
|------|---------|---------|---------|---------------------|
| iowait 高、磁盘队列深度大、RT 抖动 | iowait%、IOPS、磁盘带宽、队列深度、iostat | 慢 SQL、缺索引、日志同步写、磁盘性能差、随机 IO 过多 | 加索引、读写分离、缓存、SSD、异步日志 | SQL 逻辑错误（如 N+1 查询）→ 转交 |

**定位步骤**：
1. `iostat -x 1` 确认是 IOPS 高还是带宽高、await 是否过大
2. 高 IOPS + 高 await → 查慢 SQL、缺索引；带宽打满 → 查大文件读写
3. `pt-query-digest` 或 DB 慢日志定位具体 SQL
4. SQL 本身慢（缺索引/全表扫描）→ 本库；SQL 逻辑错误（N+1、错误 join）→ 转 bug-analyzer

## 3. 网络延迟

| 现象 | 定位指标 | 根因候选 | 优化方向 | 是否转交 bug-analyzer |
|------|---------|---------|---------|---------------------|
| RT 高但 CPU/IO 低、跨机房调用慢 | 网络延迟、带宽利用率、重传率、TCP 连接数 | 跨机房调用、带宽打满、DNS 解析慢、连接复用差、防火墙 | 就近部署、连接池、CDN、压缩、批量化 | 一般不转交（网络层属架构层） |

**定位步骤**：
1. `ping` / `traceroute` / `mtr` 确认是网络层还是应用层延迟
2. 重传率高 → 带宽/网卡问题；DNS 慢 → 解析问题
3. 应用层 RT 高但网络层正常 → 查连接复用、序列化开销
4. 架构层方案（就近部署/CDN/连接池）属本库；不涉及代码逻辑转交

## 4. 锁竞争

| 现象 | 定位指标 | 根因候选 | 优化方向 | 是否转交 bug-analyzer |
|------|---------|---------|---------|---------------------|
| CPU 利用率不高但 TPS 上不去、线程等待时间长 | 线程阻塞时间、锁等待、synchronized/lock 竞争 | 粗粒度锁、锁范围过大、热点数据竞争、死锁 | 锁细化、无锁结构、分段锁、乐观锁、队列异步化 | 死锁等代码逻辑缺陷 → 转交 |

**定位步骤**：
1. `jstack` / `arthas` 抓线程 dump，统计 BLOCKED 线程数
2. 锁等待时间长但 CPU 低 → 锁竞争典型特征
3. 定位锁对象：粗粒度锁 → 细化；热点数据 → 分段锁
4. 死锁（jstack 直接报 Deadlock）→ 转 bug-analyzer；锁设计问题 → 本库

## 5. 连接池耗尽

| 现象 | 定位指标 | 根因候选 | 优化方向 | 是否转交 bug-analyzer |
|------|---------|---------|---------|---------------------|
| 请求排队、报"连接超时"、TPS 到顶后错误率飙升 | 连接池使用率、等待线程数、连接持有时间 | 连接池过小、连接泄漏、慢查询占连接、未释放 | 扩容连接池、查泄漏、慢查询优化、超时配置 | 连接泄漏（未 close）等代码逻辑缺陷 → 转交 |

**定位步骤**：
1. 监控连接池 active/idle/wait 状态，wait 数持续增长 → 耗尽
2. 查连接持有时间分布：长持有 → 慢查询/未释放；短持有但池小 → 扩容
3. 连接数不归还 → 排查代码未 close 路径 → 转 bug-analyzer
4. 慢查询占连接 → 走 IO 瓶颈定位链

## 6. GC 停顿

| 现象 | 定位指标 | 根因候选 | 优化方向 | 是否转交 bug-analyzer |
|------|---------|---------|---------|---------------------|
| 周期性 RT 陡增、TPS 周期性下跌、应用日志无异常 | GC 频率、GC 耗时、堆内存占用、Full GC 次数 | 堆太小、对象创建过快、内存泄漏、大对象、Full GC 频繁 | 扩堆、调 GC 参数、减少对象创建、对象池 | 内存泄漏等代码逻辑缺陷 → 转交 |

**定位步骤**：
1. `jstat -gc` / GC 日志看 YGC / FGC 频率与耗时
2. FGC 频繁且耗时大 → 周期性 RT 抖动主因
3. 堆占用持续上涨不下降 → 内存泄漏 → 转 bug-analyzer
4. 短期对象过多 → 减少创建 / 对象池；堆配置问题 → 调参

## 瓶颈传导链

一个瓶颈未及时处理会引发连锁反应，**必须沿链追溯根因**，否则治标不治本。

| 传导链 | 起点现象 | 中间环节 | 终点现象 | 治标风险 |
|--------|----------|----------|----------|----------|
| 传导链 1 | GC 停顿 → | 请求堆积 → 连接池耗尽 → | 错误率飙升 | 只扩连接池不解决 GC，延迟复发 |
| 传导链 2 | 慢 SQL → | 连接占用久 → 连接池耗尽 → | 上游超时级联 | 只调超时阈值不优化 SQL，问题转移 |
| 传导链 3 | 锁竞争 → | 线程阻塞 → CPU 利用率看似低 → | 误判为容量足继续加压 | 加压反而加剧锁竞争，TPS 反降 |
| 传导链 4 | 网络抖动 → | 重试风暴 → 下游请求翻倍 → | 下游 CPU 飙升 | 只扩下游不治理重试，越扩越崩 |
| 传导链 5 | 日志同步写 → | IO 队列深度大 → 请求线程阻塞 → | RT 整体抬升 | 只扩磁盘不改异步日志，IO 仍饱和 |

**传导链分析方法**：
1. 从最终错误现象倒推，每一步问"为什么会这样"
2. 每个环节都要有指标证据，禁止跳跃
3. 锁定传导链起点 = 真根因，对起点优化才能根治

## 反例（不要这样做）

| 反例 | 错误做法 | 正确做法 |
|------|----------|----------|
| 把代码逻辑缺陷当性能瓶颈 | 把 NullPointerException 导致的错误率上升当性能问题分析 | 转交 bug-analyzer，按代码缺陷处理 |
| 只看单一指标下结论 | CPU 高就断定 CPU 瓶颈 | 排除 GC 频繁导致的 CPU 假性飙升，结合多指标交叉验证 |
| 忽略瓶颈传导 | 修了连接池但根因是慢 SQL，问题反复 | 沿传导链追溯到起点，对起点优化 |
| 跳过资源层直接猜架构 | 没有指标数据就断定"需要加机器/拆服务" | 先用资源层指标证明饱和，再做架构判断 |
| 凭经验套方案 | 看到 RT 高就加缓存 | 先定位 RT 高的根因（网络/锁/IO/GC），再选方案 |
| 用压测替代定位 | 发现瓶颈就停止分析，直接说"扩容解决" | 扩容是缓解不是定位，必须给出根因和根治方案 |

## 与其他知识库的协作

| 场景 | 协作方 | 协作方式 |
|------|--------|----------|
| 瓶颈定位完成，需生成压测用例验证优化效果 | test-case-engineer | 输出性能测试点清单，转交生成完整用例 |
| 瓶颈根因涉及代码逻辑缺陷 | bug-analyzer | 转交，附上已定位的资源层证据 |
| 瓶颈需深入根因分析 | bug-analyzer 的 root-cause-frameworks | 引用 5 Whys / 因果链方法继续分析 |
