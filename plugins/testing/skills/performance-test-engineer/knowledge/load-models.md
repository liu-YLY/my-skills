# 负载模型

> **何时阅读**：性能测试设计阶段需要选择负载施加模式时查阅。
> **覆盖范围**：阶梯递增 / 峰值 / 疲劳 / 容量 四种负载模型的定义、参数、适用场景与风险。

## 1. 模型总览

| 模型 | 核心目标 | 典型时长 | 关键产出 |
|------|----------|----------|----------|
| 阶梯递增 | 找性能拐点 | 30-90 分钟 | 拐点并发数、拐点 TPS |
| 峰值 | 验证瞬时冲击承受力 | 5-15 分钟 | 崩溃阈值、恢复时长 |
| 疲劳 | 暴露泄漏与慢查询累积 | 8-24 小时 | 内存/连接增长曲线 |
| 容量 | 量化最大处理能力 | 1-4 小时 | 最大 TPS、最大并发用户 |

## 2. 阶梯递增（Ramp-up / Step Load）

**定义**：逐步增加并发用户数，观察系统响应随负载变化的拐点。

| 项 | 内容 |
|----|------|
| 适用场景 | 找出系统性能拐点、容量规划、回归测试 |
| 不适用场景 | 验证峰值承受能力（用峰值模型）；长期稳定性（用疲劳模型） |
| 参数示例 | 起始 10 用户，每 2 分钟增加 20 用户，持续到 TPS 下降或 RT 超阈值 |
| 典型时长 | 30-90 分钟 |
| 风险点 | 阶梯步长过大可能跳过拐点 |

**执行要点**：
1. 每阶梯保持至少 2 分钟，让系统进入稳态再采数
2. 终止条件必须量化：RT > 500ms 或 错误率 > 0.1% 或 TPS 连续两阶梯下降
3. 步长不超过预估拐点的 10%（预估拐点 200 并发，步长 ≤ 20）

## 3. 峰值（Spike）

**定义**：短时间内施加远超日常的瞬时负载，观察系统是否崩溃及恢复能力。

| 项 | 内容 |
|----|------|
| 适用场景 | 秒杀活动、突发流量、容灾验证 |
| 不适用场景 | 测容量上限（用容量模型）；长期稳定性（用疲劳模型） |
| 参数示例 | 日常 500 QPS，峰值 5000 QPS，持续 30 秒后回落到日常 |
| 典型时长 | 5-15 分钟（含恢复观察期） |
| 风险点 | 忽略恢复期观察，漏测雪崩 |

**执行要点**：
1. 峰值持续时间 ≥ 30 秒，覆盖连接池打满、缓存击穿的时间窗
2. 回落后必须观察 ≥ 5 分钟，记录 RT、错误率、线程池恢复时长
3. 峰值倍数取日常的 5-10 倍，秒杀类场景取 10 倍以上

## 4. 疲劳（Soak / Endurance）

**定义**：在预期生产负载下长时间运行，检测内存泄漏、连接泄漏、慢查询累积。

| 项 | 内容 |
|----|------|
| 适用场景 | 上线前稳定性验证、内存泄漏检测、慢查询暴露 |
| 不适用场景 | 找拐点（用阶梯）；测峰值（用峰值） |
| 参数示例 | 日常 500 QPS，持续 8-24 小时 |
| 典型时长 | 8-24 小时 |
| 风险点 | 时长不足暴露不了泄漏；环境不一致导致结果失真 |

**执行要点**：
1. 时长 ≥ 8 小时，内存泄漏通常在 4 小时后显现
2. 负载取日常峰值的 70-80%，模拟真实生产水位
3. 每 30 分钟采样一次 JVM 堆、GC 次数、DB 连接数、慢查询数
4. 通过标准：RT 无上升趋势、错误率 < 0.01%、堆内存无持续增长

## 5. 容量（Capacity）

**定义**：持续施加负载找到系统最大处理能力（最大 TPS / 最大并发用户）。

| 项 | 内容 |
|----|------|
| 适用场景 | 容量规划、扩容决策、SLA 谈判 |
| 不适用场景 | 稳定性验证（用疲劳）；瞬时峰值（用峰值） |
| 参数示例 | 从 100 并发起，每 5 分钟增加 50，直到错误率 > 1% 或 RT 超阈值 |
| 典型时长 | 1-4 小时 |
| 风险点 | 单接口容量不等于系统容量；忽略依赖方瓶颈 |

**执行要点**：
1. 每阶梯保持 ≥ 5 分钟，确保资源池打满后再采数
2. 终止条件量化：错误率 > 1% 或 RT > 阈值（如 P95 > 800ms）
3. 必须跑混合场景（核心接口按线上比例配比），单接口容量仅作参考
4. 同步监控依赖方（DB、缓存、下游服务），记录首个达到瓶颈的组件

## 6. 决策表（业务场景 → 推荐负载模型）

| 业务场景 | 推荐模型 | 理由 |
|---------|---------|------|
| 秒杀/大促 | 峰值 + 容量 | 峰值验证承受力，容量定上限 |
| 日常上线回归 | 阶梯递增 | 快速找拐点，时长可控 |
| 上线前稳定性 | 疲劳 | 暴泄漏与慢查询 |
| 扩容决策 | 容量 | 量化最大处理能力 |
| 容灾演练 | 峰值 | 验证突发流量下的恢复 |

## 7. 反例（不要这样做）

| 反例 | 问题 | 正确做法 |
|------|------|----------|
| 用峰值模型测容量上限 | 峰值是瞬时，容量是持续，结果不可比 | 用容量模型，阶梯持续加载 |
| 阶梯模型步长过大（如一次加 200 并发） | 跳过拐点，得到错误的容量结论 | 步长 ≤ 预估拐点的 10% |
| 疲劳测试只跑 1 小时 | 内存泄漏往往在 4 小时后才显现 | 时长 ≥ 8 小时 |
| 只跑单接口容量 | 忽略混合场景下的资源争用 | 跑混合场景，单接口仅作参考 |
| 峰值测试不观察恢复期 | 漏测雪崩与服务降级未恢复 | 峰值后观察 ≥ 5 分钟 |
| 容量测试忽略依赖方监控 | 把下游瓶颈误判为自身容量上限 | 同步监控 DB/缓存/下游 |
