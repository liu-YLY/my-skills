# 指标体系

> **何时阅读**：设计压测方案、采集性能数据、判定瓶颈位置时查阅。
> **覆盖范围**：吞吐量 / 响应时间 / 错误率 / 资源利用率 四项核心指标的定义、采集、阈值与相互关系。

## 1. 四项核心指标

### 1.1 TPS / QPS（吞吐量）

| 属性 | 说明 |
|------|------|
| **定义** | 单位时间内系统处理的请求数（事务/秒 或 查询/秒） |
| **采集方式** | 压测工具原生统计（JMeter Summary Report / Wrk / Locust Stats）；APM 全量采集 |
| **关键观察点** | 随并发上升的拐点；稳态值（持续 N 分钟不下降）；加压后不再增长的顶点 |
| **单位选择** | TPS 面向事务（含多步），QPS 面向单次查询；同一报告内禁止混用 |

### 1.2 响应时间 RT（延迟）

| 分位数 | 含义 | 用途 |
|--------|------|------|
| **P50** | 中位数 | 反映典型用户体验 |
| **P95** | 95 分位 | 反映大多数用户体验的下限 |
| **P99** | 99 分位 | 反映长尾用户感受，是 SLA 核心指标 |
| **Max** | 最大值 | 反映最差体验，但易受噪声干扰 |

| 采集位置 | 含义 | 适用场景 |
|----------|------|----------|
| **客户端采集** | 包含网络往返、序列化、压测机调度开销 | 评估真实用户体验 |
| **服务端采集** | 纯处理时间（从接收请求到发出响应） | 定位服务自身瓶颈 |

| 关键观察点 | 说明 |
|------------|------|
| **P99 与 P50 差距** | 差距 > 3 倍即存在长尾问题，需追查 GC / 锁竞争 / 慢查询 |
| **随并发的陡增点** | RT 在某并发值后突然陡增，该值即系统承载上限 |
| **分位数发散** | 高并发下 P99 远离 P50，是排队效应开始的信号 |

### 1.3 错误率

| 属性 | 说明 |
|------|------|
| **定义** | 失败请求数 / 总请求数 |
| **采集方式** | 压测工具断言统计 + 服务端日志 + APM 错误追踪 |

**错误分类**：

| 类型 | 表现 | 责任方 |
|------|------|--------|
| 业务错误 | HTTP 4xx（参数错、鉴权失败、业务规则拒绝） | 调用方 / 业务逻辑 |
| 系统错误 | HTTP 5xx（空指针、依赖崩溃、OOM） | 服务提供方 |
| 超时 | 客户端超时 / 网关超时 / 数据库超时 | 多方定位 |
| 断言失败 | 压测脚本断言未通过（响应体字段缺失/值错） | 接口契约问题 |

| 关键观察点 | 说明 |
|------------|------|
| **突增并发阈值** | 错误率从 0 跃升到 >0 的并发值，是容量规划硬上限 |
| **错误类型切换** | 先出现超时、后出现 5xx，是资源耗尽的典型演变 |
| **错误分布** | 单机错误集中 → 负载不均；全机错误同步 → 容量不足 |

### 1.4 资源利用率

| 资源 | 指标 | 关键观察点 |
|------|------|------------|
| **CPU** | 总体利用率 + 单核利用率 | 单核 100% 而总体 30% → 单线程瓶颈；总体 > 80% 触发告警 |
| **内存** | 已用 / 可用 + GC 频率与耗时 | Full GC 频次 > 1 次/分钟或 STW > 200ms 即为瓶颈信号 |
| **磁盘 IO** | IOPS + 吞吐带宽 + await | await > 20ms 或 util > 80% 视为饱和 |
| **网络** | 入站带宽 + 出站带宽 + 重传率 | 重传率 > 1% 或带宽打满 > 90% 视为饱和 |

| 采集工具 | 适用场景 |
|----------|----------|
| node_exporter + Prometheus | Kubernetes / 容器化环境，持续采集 |
| top / htop | 快速查看 CPU 与内存，单机定位 |
| iostat -x 1 | 磁盘 IO 详细指标（await、util、svctm） |
| sar -n DEV 1 | 网络流量与重传，历史回溯 |
| pidstat 1 | 进程级 CPU / 内存 / IO 归因 |

| 关键观察点 | 说明 |
|------------|------|
| **利用率阈值** | 任一资源利用率 > 80% 触发告警，> 90% 视为饱和 |
| **饱和度信号** | CPU runqueue > 核数 × 2、iowait > 20%、GC STW 持续上升 |
| **单核打满** | 总体 CPU 不高但单核 100% → 串行化瓶颈（锁、单线程模型） |

## 2. 指标阈值参考表

按业务类型分档，实际 SLA 以业务合同为准：

| 业务类型 | P99 RT | 错误率 | CPU 利用率 | 备注 |
|---------|--------|--------|-----------|------|
| 互联网应用（C 端） | ≤ 500ms | ≤ 0.1% | ≤ 70% | 用户体验优先 |
| 互联网应用（B 端） | ≤ 1s | ≤ 0.5% | ≤ 75% | 容忍度略高 |
| 金融交易 | ≤ 200ms | ≤ 0.01% | ≤ 60% | 强一致 + 低延迟 |
| 物联网/设备上报 | ≤ 2s | ≤ 1% | ≤ 80% | 海量低频 |
| 内部管理后台 | ≤ 2s | ≤ 1% | ≤ 80% | 低优先级 |

## 3. 指标间关系

### 3.1 Little's Law

公式：**L = λW**

| 符号 | 含义 | 单位 |
|------|------|------|
| L | 系统内平均并发数 | 个 |
| λ | 到达率（吞吐量） | 请求/秒 |
| W | 平均响应时间 | 秒 |

**应用场景**：已知目标 TPS(λ) 和可接受 RT(W)，推算所需并发数(L)。

**示例**：
```
目标：TPS = 1000，RT = 0.2s
推算：L = 1000 × 0.2 = 200
结论：需要 200 并发才能压出 1000 TPS
```

### 3.2 RT 陡增与 TPS 上限

TPS 到顶后，继续加压只会让 RT 线性上升，TPS 不再增长。该顶点即系统最大吞吐能力，超过该点的并发都是排队等待。

```
并发 100  → TPS 500, RT 200ms  （正常）
并发 200  → TPS 1000, RT 200ms （线性扩展）
并发 400  → TPS 1000, RT 400ms （顶点，开始排队）
并发 800  → TPS 1000, RT 800ms （纯排队，TPS 已封顶）
```

### 3.3 错误率与资源饱和

资源饱和后错误率突增是典型过载信号。常见演变顺序：

1. 资源利用率上升至 > 80%
2. RT 开始陡增（排队）
3. 出现超时（客户端或网关）
4. 出现 5xx（连接池耗尽、OOM、线程打满）

## 4. 采集建议

| 维度 | 要求 |
|------|------|
| **采样频率** | 资源指标 ≥ 1Hz；压测期间请求指标全量采集，禁止抽样 |
| **采样窗口** | 取稳态段，去掉冷启动前 10% 和收尾后 10% |
| **冷启动排除** | JIT 编译、连接池预热、缓存填充导致前几分钟数据失真，必须剔除 |
| **多机采样** | 分布式系统分机器采样 + 汇总，禁止只看一台 |
| **时间对齐** | 压测工具时间戳与服务端日志时间戳必须 NTP 同步，偏差 < 1s |

## 5. 反例（不要这样做）

| 反例 | 后果 | 正确做法 |
|------|------|----------|
| 只看平均值不看分位数 | 平均 100ms 可能掩盖 P99 = 2s 的长尾问题 | 同时报告 P50 / P95 / P99 / Max |
| 忽略冷启动数据 | 把预热期的高 RT 当成系统性能差 | 剔除前 10% 数据，或显式预热后再开始统计 |
| 只看一台机器的资源 | 漏掉负载不均导致的单点瓶颈 | 分机器采样，单机利用率单独报告 |
| TPS 到顶后继续加压看 RT | 误以为系统"变慢"，实际是过载排队 | 在 TPS 不再增长时停止加压，记录该顶点为容量上限 |
| 混淆客户端 RT 与服务端 RT | 网络开销被算进处理时间，误判服务慢 | 两端分别采集，定位瓶颈时对照差异 |
| 用压测机 CPU 占用判断服务负载 | 压测机自身瓶颈被当成服务瓶颈 | 监控压测机资源，CPU > 70% 即扩容压测机或分布式发压 |
| 错误率只看总数不分类型 | 4xx 业务错误与 5xx 系统错误混为一谈，无法定位 | 按状态码 + 超时 + 断言失败分类统计 |
