# 反模式黑名单

> **何时阅读**：贯穿阶段 1-4 执行时查阅。汇总所有会导致状态机测试质量下降的反模式。
> **设计依据**：基于 SkillLens 论文（arXiv 2605.23899）实证——只写"应该做 X"没有"不要做 Y"会导致 LLM judge 准确率下降。
> **覆盖范围**：8 条核心反模式 + 工作流反模式 + 输出格式反模式 + 触发检查时机。

## 1. 核心反模式（8 条）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|---|---|---|
| 1 | 把"页面提示"等同于"业务状态" | 页面提示 ≠ 后台真实状态，混淆会导致漏测真实状态变化 | 状态/事件/结果必须分层描述，区分"接口调用成功""申请已受理""业务最终成功" |
| 2 | 把"请求受理"等同于"处理完成" | 受理只是提交结果，不代表业务完成（如退款申请成功 ≠ 退款成功） | 转换必须明确 FROM → TO，不能跳过中间态 |
| 3 | PRD 缺权限/异常路径时自行脑补 | LLM 脑补管理员逻辑会引入幻觉，与真实需求不符 | 必须输出"待确认"节点，暴露给用户裁定 |
| 4 | 不标注依据类型 | 无法区分哪些是 PRD 明确、哪些是推理，评审无法聚焦 | 每条 transition 和场景必须标 `需求明确/合理推理/待确认` |
| 5 | 把状态用动词描述（如"支付中"） | 动词状态边界模糊，易与事件混淆 | 状态用名词（如"待支付"/"已支付"），事件用动词 |
| 6 | 不定义禁止转换 | 只测合法路径会漏掉非法跳转的高风险场景 | 每个状态必须明确"禁止进入哪些状态"及理由 |
| 7 | 把"申请成功"作为终态 | 申请成功只是提交结果，业务可能仍在中态（如退款中） | 终态必须是业务最终态（退款成功/退款失败），不是受理态 |
| 8 | 直接输出用例步骤 | 与 test-case-engineer 职责重叠，导致用例重复 | 只输出场景清单，用例步骤交给 test-case-engineer（链 5 协同） |

## 2. 各反模式详解

### 反模式 1: 把"页面提示"等同于"业务状态"

**典型表现**：

```
❌ 错误建模：
状态：支付成功提示已显示
事件：用户点击支付
```

**问题**：
- "支付成功提示已显示"是 UI 状态，不是业务状态
- UI 状态可能在网络抖动时与后台不一致
- 测试 UI 状态会漏测后台真实状态变化

**正确做法**：

```
✅ 正确建模：
状态：已支付（业务状态）
事件：支付成功回调（外部回调）
副作用：生成支付记录、触发履约（后台动作）
```

UI 提示只作为副作用的一部分，不作为状态本身。

### 反模式 2: 把"请求受理"等同于"处理完成"

**典型表现**：

```
❌ 错误转换：
待支付 →（用户申请退款）→ 退款成功
```

**问题**：
- "用户申请退款"只是受理，退款尚未真正完成
- 漏掉"退款中"中间态，无法测试退款失败/超时场景
- 把受理态当终态，违反反模式 7

**正确做法**：

```
✅ 正确转换：
待支付 →（用户申请退款）→ 退款中
退款中 →（退款成功回调）→ 退款成功
退款中 →（退款失败回调）→ 退款失败
```

受理态（退款中）是中间态，不是终态。

### 反模式 3: PRD 缺权限/异常路径时自行脑补

**典型表现**：

```
❌ 错误脑补：
PRD 未说明退款权限，LLM 自行假设"仅管理员可发起退款"
→ 生成 access_control 场景全部基于"管理员"假设
→ 真实业务可能是"用户也可发起"，测试用例全错
```

**问题**：
- LLM 脑补基于训练数据分布，与具体业务规则无关
- 脑补的内容会被当作"需求明确"对待，评审时无法识别
- 真实业务规则可能与脑补相反，导致测试用例无效

**正确做法**：

```
✅ 正确暴露：
PRD 未说明退款权限 → 标 evidence_type: 待确认
→ 输出 access_control 场景时，expected_target_state 写"待确认"
→ 在 ambiguities 中列出"退款操作权限矩阵未明确"
→ CHECKPOINT 时要求用户澄清
```

### 反模式 4: 不标注依据类型

**典型表现**：

```
❌ 错误输出：
transitions:
  - from: 待支付
    to: 已支付
    event: 支付成功回调
    # 缺 evidence_type
```

**问题**：
- 评审无法区分哪些是 PRD 明确、哪些是 LLM 推理
- 推理内容被当作"需求明确"对待，无法聚焦评审
- 违反 pydantic Schema（evidence_type 必填）

**正确做法**：

```
✅ 正确标注：
transitions:
  - from: 待支付
    to: 已支付
    event: 支付成功回调
    evidence_type: 需求明确
    source: PRD §3.2
```

### 反模式 5: 把状态用动词描述

**典型表现**：

```
❌ 错误状态名：
- 支付中
- 退款中
- 审批中
- 取消了
```

**问题**：
- "支付中"边界模糊：是"等待支付"还是"支付处理中"？
- "退款中"与事件"退款"混淆
- 动词状态难以定义不变量（"支付中的不变量是什么？"）

**正确做法**：

```
✅ 正确状态名（名词）：
- 待支付
- 已支付
- 退款处理中（名词短语）
- 已取消
```

事件保持动词：支付成功回调、用户取消、退款成功回调。

### 反模式 6: 不定义禁止转换

**典型表现**：

```
❌ 错误：只写 transitions，不写 forbidden
→ illegal_transition 场景无法生成（无 forbidden 可参照）
→ 高危的"已取消订单尝试支付"场景漏测
```

**问题**：
- 只测合法路径会漏掉所有非法跳转
- 非法跳转是状态机测试的核心价值之一
- 终态吸收规则无法表达

**正确做法**：

```
✅ 正确：transitions + forbidden 都写
forbidden:
  - from: 已取消
    to: 已支付
    reason: 已取消订单不可支付
  - from: 退款成功
    to: "*"
    reason: 终态吸收
```

### 反模式 7: 把"申请成功"作为终态

**典型表现**：

```
❌ 错误终态：
- 退款申请成功（is_terminal: true）
- 审批已提交（is_terminal: true）
```

**问题**：
- 申请成功只是提交结果，业务仍在中态
- 把受理态当终态，会导致"终态吸收"规则错误应用
- 漏测"退款中 → 退款成功/退款失败"的后续转换

**正确做法**：

```
✅ 正确终态（业务最终态）：
- 退款成功（is_terminal: true）
- 退款失败（is_terminal: true）
- 已完成（is_terminal: true）

中间态（非终态）：
- 退款中（is_terminal: false）
- 退款申请成功（is_terminal: false）  ← 这是受理态，不是终态
```

**判定规则**：问"这个状态之后还会变吗？"会变 → 非终态；不会变 → 终态；不确定 → 标"待确认"。

### 反模式 8: 直接输出用例步骤

**典型表现**：

```
❌ 错误输出（state-machine 直接出用例步骤）：
场景 SM-001:
  步骤 1: 登录系统
  步骤 2: 进入订单详情页
  步骤 3: 点击"取消订单"按钮
  步骤 4: 确认取消
  预期: 订单状态变为"已取消"
```

**问题**：
- 与 test-case-engineer 职责重叠，导致用例重复
- state-machine 的价值是"状态网络穷举"，不是"用例步骤编写"
- 用例步骤需要考虑 UI/接口/数据三层，state-machine 不擅长

**正确做法**：

```
✅ 正确输出（只出场景清单）：
场景 SM-001:
  current_state: 待支付
  trigger_event: 用户取消
  expected_target_state: 已取消
  risk_type: legal_transition
  evidence_type: 需求明确

→ 转交 test-case-engineer 落地为完整用例步骤
```

## 3. 工作流反模式

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|---|---|---|
| W1 | 跳过阶段 1 直接建模 | 需求理解不完整导致状态机偏离业务意图 | 必须完成阶段 1 输出业务对象清单 + 歧义清单 |
| W2 | 跳过 CHECKPOINT 直接进阶段 3 | 用户无法修正建模错误，后续工作可能白做 | 阶段 2 后必须展示模型给用户确认 |
| W3 | 完整性检查 fail 仍进入阶段 4 | 场景穷举基于有缺陷的模型，浪费工作 | fail 必须触发 CHECKPOINT，修正后才进阶段 4 |
| W4 | MCP 失败时停止输出 | skill 是主，MCP 是辅，MCP 失败不应阻塞 | 自动降级到独立模式，首行标记降级原因 |
| W5 | MCP 与 skill 结果冲突时自动取舍 | LLM 自动取舍可能选错，掩盖真实问题 | 标"待确认"交给用户裁定 |

## 4. 输出格式反模式

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|---|---|---|
| O1 | 状态名中英文混用 | 评审困难，下游消费时歧义 | 统一中文或统一英文，PRD 原文为准 |
| O2 | transition 缺 source 字段 | 无法追溯依据，评审困难 | source 必填，PRD 章节或推理依据 |
| O3 | 场景 id 不规范 | 无法唯一标识，追溯困难 | 用 `SM-{三位序号}` 格式 |
| O4 | risk_type 用中文 | 与 pydantic Schema 不一致，MCP 校验失败 | 用英文枚举（legal_transition 等） |
| O5 | evidence_type 用英文 | 与 pydantic Schema 不一致 | 用中文枚举（需求明确/合理推理/待确认） |

**注意**：risk_type 用英文（与 pydantic Schema 对齐），evidence_type 用中文（与 pydantic Schema 的枚举值对齐）。这是设计文档明确规定的，不要混用。

## 5. 触发检查时机

| 阶段 | 检查的反模式 |
|---|---|
| 阶段 1（需求识别） | W1、反模式 1、反模式 3 |
| 阶段 2（建模） | 反模式 2、反模式 4、反模式 5、反模式 6、反模式 7、O1、O2、O4、O5 |
| 阶段 2 后 CHECKPOINT | W2 |
| 阶段 3（完整性检查） | W3 |
| 阶段 4（场景穷举） | 反模式 8、O3、O4、O5 |
| 阶段 5（MCP 增强） | W4、W5 |

## 6. 自检清单

阶段 4 输出前，对照以下清单自检：

- [ ] 所有状态用名词，所有事件用动词？（反模式 5）
- [ ] 所有 transition 和 forbidden 都有 evidence_type？（反模式 4）
- [ ] PRD 未说明的权限/异常路径都标"待确认"？（反模式 3）
- [ ] 终态都是业务最终态，不是受理态？（反模式 7）
- [ ] forbidden 包含终态吸收规则？（反模式 6）
- [ ] 输出只有场景清单，无用例步骤？（反模式 8）
- [ ] risk_type 用英文，evidence_type 用中文？（O4、O5）
- [ ] 场景 id 用 SM-{三位序号}？（O3）
