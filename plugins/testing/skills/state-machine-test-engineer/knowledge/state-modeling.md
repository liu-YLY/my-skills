# 状态机建模方法论

> **何时阅读**：阶段 2（状态机建模）执行时查阅。定义状态机的 6 要素、MAE 流程建模、不变量与终态吸收规则。
> **覆盖范围**：6 要素定义 / MAE 三层流程 / 不变量写法 / 终态吸收规则 / 状态命名规范 / 转换定义规范。

## 1. 状态机 6 要素

每个状态机模型必须包含 6 个要素，缺一不可：

| 要素 | 字段 | 说明 | 示例 |
|---|---|---|---|
| 状态（State） | `states[]` | 业务对象可能处于的稳定形态 | 待支付/已支付/已取消 |
| 事件（Event） | `transitions[].event` | 触发状态变化的外部输入 | 支付成功回调/用户取消 |
| 转换（Transition） | `transitions[]` | 从一个状态到另一个状态的有向边 | 待支付 → 已支付 |
| 守卫（Guard） | `transitions[].guards` | 转换前置条件，不满足则拒绝 | 订单有效/金额一致 |
| 副作用（Side Effect） | `transitions[].side_effects` | 转换后必须执行的动作 | 生成支付记录/触发履约 |
| 不变量（Invariant） | `states[].invariants` | 状态内不可破坏的约束 | 待支付状态订单金额不可修改 |

**禁止转换**（Forbidden）是第 7 个可选要素，用于显式声明业务上不允许的跳转（如"已取消 → 已支付"）。

## 2. MAE 三层流程建模

MAE = Main flow / Alternative flow / Exception flow，对每条业务流程做三层建模：

### 2.1 Main flow（主流程）

PRD 直接描述的"正常路径"，通常是用户视角的"成功流程"。

```
订单创建 → 待支付 →（支付成功回调）→ 已支付 →（履约完成）→ 已完成
```

主流程的 transition 标 `evidence_type: 需求明确`。

### 2.2 Alternative flow（替代流程）

PRD 描述的"非主路径但合理"的流程，如取消/退款/撤回。

```
待支付 →（用户取消）→ 已取消
已支付 →（用户申请退款）→ 退款中 →（退款成功回调）→ 退款成功
```

替代流程的 transition 通常标 `evidence_type: 需求明确`（PRD 有描述）或 `合理推理`（PRD 未明确但业务常识）。

### 2.3 Exception flow（异常流程）

PRD 未完整描述但业务必须处理的流程，如超时/失败/重试。

```
待支付 →（支付超时 30 分钟）→ 已取消  ← 合理推理（PRD 可能未提超时时长）
退款中 →（退款失败回调）→ 退款失败  ← 需求明确（若 PRD 有写）/ 待确认（若 PRD 未写）
退款失败 →（?）→ ???  ← 待确认（PRD 未说明失败后能否重试）
```

异常流程的 transition 大量标 `evidence_type: 待确认`，这正是 LLM 必须暴露而非脑补的部分。

### 2.4 MAE 应用规则

1. **先建主流程**：从 PRD 提取所有"成功路径"，作为状态机骨架
2. **再加替代流程**：补充取消/退款/撤回等"用户主动偏离主路径"的转换
3. **最后补异常流程**：识别 PRD 未覆盖的异常情况，**全部标"待确认"**，禁止脑补
4. **三层标注依据类型**：主流程=需求明确、替代流程=需求明确或合理推理、异常流程=待确认（除非 PRD 明确）

## 3. 不变量（Invariant）

不变量是状态内不可破坏的约束，违反不变量意味着状态机模型有缺陷。

### 3.1 不变量写法

每个状态可列 0-N 条不变量，写法为"主语 + 不可/必须 + 谓语"：

```yaml
states:
  - name: 待支付
    invariants:
      - 订单金额不可修改
      - 订单项不可删除
      - 必须保留创建时间
  - name: 已支付
    invariants:
      - 支付金额不可修改
      - 支付时间不可篡改
      - 必须保留支付渠道
  - name: 退款成功
    invariants:
      - 退款金额不可再修改
      - 不可再次发起退款
```

### 3.2 不变量检查时机

不变量主要在阶段 3 完整性检查的"状态变化有副作用定义"项中使用：
- 如果 transition 的副作用会破坏目标状态的不变量 → 标 contradiction
- 如果 transition 没有副作用但目标状态有不变量 → 标 warning

## 4. 终态与终态吸收

### 4.1 终态定义

终态是业务最终态，**不可再变化**。终态必须满足：
- `is_terminal: true`
- 不出现在任何 transition 的 `from`
- 不变量包含"不可再修改"类约束

### 4.2 终态吸收规则

终态吸收 = 终态到任意状态都禁止，写法：

```yaml
forbidden:
  - from: 退款成功
    to: "*"
    reason: 终态吸收
    evidence_type: 需求明确
  - from: 退款失败
    to: "*"
    reason: 终态吸收
    evidence_type: 待确认  # 若 PRD 未说明退款失败后能否重新发起
```

### 4.3 终态 vs 受理态（反模式 #7）

| 类型 | 示例 | 是否终态 |
|---|---|---|
| 业务最终态 | 退款成功/退款失败/已完成 | ✅ 是终态 |
| 受理态 | 退款申请成功/审批已提交 | ❌ 不是终态（只是提交结果，业务仍在中态） |
| 处理态 | 退款中/审批中 | ❌ 不是终态（业务仍在处理） |

**判定规则**：问"这个状态之后还会变吗？"
- 会变 → 不是终态
- 不会变 → 是终态
- 不确定 → 标"待确认"，不强行定为终态

## 5. 状态命名规范

### 5.1 状态用名词

| ✅ 正确（名词） | ❌ 错误（动词） | 原因 |
|---|---|---|
| 待支付 | 支付中 | "支付中"边界模糊，无法判断是"等待支付"还是"支付处理中" |
| 已支付 | 支付完成 | "完成"是动作结果，不是稳定状态 |
| 退款处理中 | 退款中 | "退款中"是动作，应改为名词短语 |
| 已取消 | 取消了 | "取消了"是过去式动词 |

### 5.2 事件用动词

| ✅ 正确（动词） | ❌ 错误（名词） | 原因 |
|---|---|---|
| 支付成功回调 | 支付 | "支付"过于笼统，无法区分"用户发起"和"渠道回调" |
| 用户取消 | 取消 | 同上 |
| 退款成功回调 | 退款 | 同上 |

### 5.3 命名冲突检查

- 状态名与事件名不可重复（避免"退款成功"既是状态又是事件）
- 若不可避免（PRD 原文如此），状态加"态"后缀（如"退款成功态"），事件加"回调"后缀（如"退款成功回调"）

## 6. 转换定义规范

### 6.1 完整转换示例

```yaml
transitions:
  - from: 待支付
    to: 已支付
    event: 支付成功回调
    guards:
      - 订单有效
      - 金额一致
      - 回调可信
    side_effects:
      - 生成支付记录
      - 触发履约流程
      - 通知用户
    evidence_type: 需求明确
    source: PRD §3.2
```

### 6.2 字段填写规则

| 字段 | 必填 | 规则 |
|---|---|---|
| `from` | ✅ | 必须是 `states[]` 中已定义的状态名 |
| `to` | ✅ | 必须是 `states[]` 中已定义的状态名 |
| `event` | ✅ | 动词短语，描述触发原因 |
| `guards` | 可选 | 前置条件，不满足则转换被拒绝 |
| `side_effects` | 可选 | 后置动作，转换成功后必须执行 |
| `evidence_type` | ✅ | 必填，枚举：需求明确/合理推理/待确认 |
| `source` | ✅ | PRD 章节引用或推理依据 |

### 6.3 同向多事件处理

若多个事件触发同一转换（如"用户取消"和"超时取消"都触发"待支付 → 已取消"），写两条 transition：

```yaml
transitions:
  - from: 待支付
    to: 已取消
    event: 用户取消
    evidence_type: 需求明确
    source: PRD §3.3
  - from: 待支付
    to: 已取消
    event: 超时取消（30 分钟未支付）
    guards: [创建后 30 分钟未支付]
    evidence_type: 合理推理
    source: 业务常识（PRD 未提超时时长）
```

不要合并为一条 transition 用 `event: 用户取消或超时取消`，会丢失场景穷举的精度。

## 7. 禁止转换定义规范

### 7.1 何时定义禁止转换

只要业务上"不允许 A → B"，就必须在 `forbidden[]` 中显式声明：

```yaml
forbidden:
  - from: 已取消
    to: 已支付
    reason: 已取消订单不可支付
    evidence_type: 需求明确
    source: PRD §3.4
  - from: 退款成功
    to: "*"
    reason: 终态吸收
    evidence_type: 需求明确
    source: PRD §3.5
```

### 7.2 通配符 `*` 的使用

`to: "*"` 表示"任意状态"，仅用于终态吸收场景。其他场景必须明确写出目标状态。

### 7.3 禁止转换与 transition 的关系

- 一对状态可以既出现在 `transitions[]`（合法）又出现在 `forbidden[]`（非法）——这是**矛盾**，阶段 3 完整性检查会标记
- 一对状态只出现在 `forbidden[]` 不出现在 `transitions[]`——正常，表示业务明确禁止
- 一对状态只出现在 `transitions[]` 不出现在 `forbidden[]`——正常，表示业务允许
