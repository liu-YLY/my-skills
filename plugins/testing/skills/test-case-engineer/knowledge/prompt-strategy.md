# AI 用例生成 Prompt 策略

> **何时阅读**:阶段 3 进入 AI 生成模式时必读。
> **覆盖范围**:结构化提示词模板 / Prompt 调优要点。
> **跳过条件**:用户明确要求人工编写用例时必须跳过本文件。

## 核心理念

AI 负责生成 70%-90% 的基础用例，测试人员负责审核、修正和补充业务特有的复杂逻辑。
将"编写"转变为"评审"，实现从"劳动密集型"向"智能审核型"的转变。

## 结构化提示词模板

### 基础模板（通用）

将阶段 1 的需求理解 + 阶段 2 的测试点清单作为输入，使用以下 Prompt：

```
你是一名资深测试工程师。请根据以下需求描述和测试点清单，生成测试用例。

【需求描述】
{阶段1输出的需求理解}

【测试点清单】
{阶段2输出的测试点清单}

【输出要求】
1. 每条用例包含：前置条件(preconditions)、测试步骤(steps)、预期结果(expected_results)
2. 覆盖以下场景类型：
   - 正常场景（Happy Path）
   - 边界值场景（边界、极限、零值）
   - 异常场景（非法输入、网络异常、权限不足）
3. 测试步骤使用祈使句，每步一个操作
4. 预期结果必须具体可验证，禁用模糊词
5. 每条用例只验证一个测试逻辑
6. steps 和 expected_results 数量一一对应
```

### 增强模板（含缺陷模式）

在基础模板上追加：

```
【历史缺陷模式】
{从 bug-patterns.md 中匹配当前功能的缺陷模式}

请额外补充针对上述缺陷模式的防御性测试用例。
```

## Prompt 调优要点

### 输入质量优化

| 策略 | 做法 | 效果 |
|------|------|------|
| 结构化需求 | 需求描述按"功能概述/用户角色/业务规则/UI交互/数据流向"分段 | AI 理解更准确 |
| 测试点带编号 | 测试点清单包含编号、维度、优先级 | 生成用例可追溯 |
| Few-shot 参考 | 提供 2-3 条已有用例作为风格参考 | 输出格式更一致 |
| 缺陷模式匹配 | 按功能特征从 bug-patterns.md 筛选相关模式 | 防御性用例更精准 |

### 输出质量控制

| 策略 | 做法 | 效果 |
|------|------|------|
| 格式约束 | 要求 AI 按通用用例格式输出（Markdown 表格 + 列表） | 便于阅读和转换 |
| 追溯标注 | 要求 AI 标注每条用例对应的测试点编号 (trace) | 可追溯性 |
| 分批生成 | 单次生成 20-30 条用例 | 避免上下文溢出，质量更可控 |
| 优先级标注 | 要求 AI 按 P0-P3 标注优先级 | 减少人工调整 |

### 迭代生成策略

```
第一轮：基础用例生成
  → 输入：需求理解 + 测试点清单
  → 产出：覆盖正向功能 + 基本反向校验的用例

第二轮：补充遗漏
  → 输入：第一轮产出 + 未覆盖的测试点
  → 产出：补充遗漏场景的用例

第三轮：防御性用例
  → 输入：前两轮产出 + 匹配的缺陷模式
  → 产出：针对历史高频缺陷的防御性用例
```

## AI 生成质量评估标准

| 维度 | 合格标准 | 评估方法 |
|------|----------|----------|
| 覆盖率 | AI 生成用例覆盖 >= 80% 的测试点 | 测试点编号比对 |
| 格式合规 | 100% 符合通用用例格式 | 人工检查 |
| 步骤可执行 | >= 90% 的步骤可直接执行无需补充 | 人工审核 |
| 预期结果明确 | >= 90% 的预期结果无模糊词 | 模糊词扫描 |
| 优先级合理 | P0/P1/P2/P3 比例在 10-15%/30-40%/30-40%/10-15% | 比例统计 |

**若 AI 生成质量不达标**：
1. 检查输入的结构化程度（需求描述是否清晰、测试点是否完整）
2. 调整 Prompt（增加约束条件、补充 Few-shot 示例）
3. 分批生成（缩小单次范围，提高精准度）
