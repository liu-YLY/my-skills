# 评审模式

> 本文件从 test-case-engineer-core.md 拆分，定义评审模式完整工作流。

> **与阶段 4 自检的边界**：阶段 4 自检是 AI 对**自己生成**的用例做覆盖度自检；评审模式是对**他人/已有**的用例做全面质量评审。两者独立，不互相替代。

## 触发条件

满足以下任一条件**必须**进入评审模式，不走四阶段生成流程：
- 用户输入包含已有用例（文件路径 / 直接粘贴 / 目录引用），且使用「评审/review/检查质量」等触发词
- 用户明确要求「评审这些用例」「检查用例质量」
- 输入仅有用例无需求文档 → 进入评审模式（溯源维度降级，见失败模式表）

## 评审工作流（3 步）

```
步骤 R1 解析输入 → 步骤 R2 6 维度评审 → 🔴 CHECKPOINT 步骤 R3 输出意见表+建议清单
```

**步骤 R1：解析输入**
- 解析已有用例的结构（ID/标题/优先级/类型/步骤/预期结果/前置条件）
- **支持格式**：Markdown（`.md`，提取表格或 `### TC-xxx` 标题）/ CSV（`.csv`，按列解析 ID/标题/步骤/预期结果）/ JSON（`.json`，按结构化字段解析）/ Excel（`.xlsx`/`.xls`，提示用户转为 CSV/Markdown 或对话粘贴）
- 若提供需求文档 → 解析需求要点，用于溯源维度
- 若无需求文档 → 标注「溯源维度降级，仅检查用例内部一致性」
- 若用户指定「增量评审」→ 仅解析 git diff 中变更的用例（详见「增量评审子模式」）

**步骤 R2：9 维度评审**

> **维度扩展说明**：v8.2.0 在原 6 维度基础上新增 3 维度（可维护性/可自动化/测试数据依赖），覆盖用例的全生命周期质量。新增维度均为 P2 级（建议改进，非阻塞）。

| 维度 | 检查点 | 判定规则（可机器校验） | 依据 |
|------|--------|----------------------|------|
| 覆盖度 | 正向/逆向/边界/异常是否齐全；对照防御性场景 | 缺失类型判定：按用例 `scenario` 字段（正向/逆向/边界/异常）统计，4 类任一为 0 → P0；若用例无 `scenario` 字段，按 `title` 关键词推断（含「异常/错误/失败/超时」→逆向/异常，含「边界/极限/空」→边界，其余→正向） | bug-patterns.md |
| 优先级合理性 | P0/P1/P2/P3 分配符合业务风险与比例指导 | 量化阈值（与 [test-standards.md](test-standards.md) 比例指导对齐）：P0 占比偏离 10%~15% 区间 → P1；P1 占比偏离 30%~40% 区间 → P1；P2 占比偏离 30%~40% 区间 → P1；P3 占比偏离 10%~15% 区间 → P1（若目标系统不支持 P3，按 test-standards.md 降级规则处理，不触发本判定）；P0+P1 总和 <50% → P1 | test-standards.md |
| 字段规范 | 字段完整；无模糊词 | 字段缺失判定：必填字段（ID/title/priority/type/steps/expected_results）任一为空 → P0；模糊词命中（"等"/"之类"/"正确"/"正常"/"类似"）→ P1；标题长度为软指导（见 [test-standards.md](test-standards.md) 超长标题例外判定），仅当标题过长且未记录例外时 → P2 | test-standards.md |
| 可执行性 | 步骤可复现；预期可验证；数据具体 | 占位符正则：steps 匹配 `/(xxx\|某[个条]?\|一些\|对应\|相应\|相关\|按.{0,4}要求)/` → P0；模糊预期正则：expected_results 匹配 `/(正确\|正常\|符合预期\|功能正常\|应生效\|与测试目标一致\|结果正确\|配置正确\|对应\|相应)/` → P1；步骤数 >7 → P2 | - |
| 冗余 | 重复用例；可合并；过度测试 | 重复判定：两条用例 `title` 相同 OR steps 前 3 步一致 → P1；同测试点用例数 >3 → P2 | - |
| 溯源 | 每条用例可追溯到需求/测试点；无孤儿用例 | 孤儿判定：用例 `test_point_id` 字段为空 OR 不在测试点清单中 → P0 | 需求文档（若有） |
| 可维护性 | 用例是否易于维护（步骤耦合度/依赖稳定性） | 步骤耦合判定：steps 引用其他用例的执行结果（如「执行 TC_001 后继续」）且未在 preconditions 中声明依赖 → P2；UI 元素引用判定：steps 引用绝对坐标/动态生成的 selector → P2 | - |
| 可自动化 | 用例是否易于自动化（断言明确度/数据依赖） | 断言模糊判定：expected_results 含「页面显示正确」「布局正常」等无法自动化断验的描述 → P2；强数据依赖判定：preconditions 要求特定数据库状态但未提供造数方式 → P2 | - |
| 测试数据依赖 | 造数成本是否过高/是否有替代方案 | 高成本造数判定：preconditions 要求生产环境数据/特定用户权限/第三方系统状态，但未提供 mock 或替代方案 → P2 | - |

**步骤 R3：输出评审意见表 + 改进建议清单 + 评审度量报告**

🔴 **CHECKPOINT · 评审输出**：必须展示评审意见表 + 改进建议清单 + 评审度量报告给用户确认。用户确认后再决定是否进入修订（修订走「评审修订闭环」，详见下方小节）。

## 评审意见表模板

```markdown
## 用例评审意见表

**评审范围**：[用例文件/模块]
**评审日期**：[YYYY-MM-DD]
**用例总数**：N 条
**整体评级**：[A 优秀 / B 合格 / C 需改进 / D 不合格]

| 维度 | 问题数 | 严重等级 | 典型问题 | 建议改进 |
|------|--------|---------|---------|---------|
| 覆盖度 | 2 | P0 | TC_005-TC_008 缺少异常类型（type=异常 的用例为 0 条） | 新增异常场景用例：网络断开/超时/依赖服务不可用 |
| 优先级合理性 | 1 | P1 | P0 占比 45%（9/20），超过 30% 阈值 | 将 TC_003/TC_007 降级为 P1（非核心路径） |
| 字段规范 | 3 | P0 | TC_002 expected_results 为空；TC_006 title 含"等"字 | TC_002 补充预期结果；TC_006 title 改为「登录-手机号错误密码-提示错误」 |
| 可执行性 | 1 | P1 | TC_004 expected_results 含「正常」模糊词（命中正则 `/(正常)/`） | 改为「页面跳转至首页，顶部显示用户名」 |
| 冗余 | 1 | P2 | TC_009 与 TC_010 steps 前 3 步一致（同测试点） | 合并为一条用例，用参数化覆盖两种数据 |
| 溯源 | 1 | P0 | TC_011 的 test_point_id 不在测试点清单中 | 核对需求，补充对应测试点或删除该孤儿用例 |
```

## 评审度量报告模板

> **与生成模式效率度量对称**：生成模式在阶段 4 输出「效率度量报告」（AI 生成占比/工时节省率），评审模式在 R3 输出「评审度量报告」，使评审效果可量化、可对比、可追踪。

```markdown
## 评审度量报告

| 指标 | 本次值 | 说明 |
|------|--------|------|
| 用例总数 | N 条 | 评审范围内的用例总数 |
| 问题用例数 | M 条 | 至少有 1 个问题的用例数 |
| 通过率 | (N-M)/N * 100% | 通过率 = 无问题用例 / 总用例 |
| 问题总数 | K 个 | 跨 9 维度累加的问题条数 |
| 问题密度 | K/N | 平均每条用例的问题数（基准：< 0.5 优秀 / 0.5-1.0 合格 / > 1.0 需改进） |
| 整体评级 | [A/B/C/D] | A:通过率≥95% 且 问题密度<0.5；B:通过率≥80%；C:通过率≥60%；D:通过率<60% |

**维度分布**：

| 维度 | 问题数 | 占比 | 主要严重等级 |
|------|--------|------|-------------|
| 覆盖度 | x | x% | P0 |
| 优先级合理性 | x | x% | P1 |
| 字段规范 | x | x% | P0 |
| 可执行性 | x | x% | P1 |
| 冗余 | x | x% | P2 |
| 溯源 | x | x% | P0 |
| 可维护性 | x | x% | P2 |
| 可自动化 | x | x% | P2 |
| 测试数据依赖 | x | x% | P2 |

**严重等级分布**：

| 严重等级 | 问题数 | 占比 | 处置建议 |
|---------|--------|------|---------|
| P0 | x | x% | 必须修订后方可执行 |
| P1 | x | x% | 建议本次迭代修订 |
| P2 | x | x% | 可纳入下个迭代修订 |

**评审结论**：[一段话总结评审发现，指出最严重的 1-2 个系统性问题及改进方向]
```

**度量报告使用场景**：
- 同模块多次评审的趋势对比（通过率是否提升、问题密度是否下降）
- 跨模块横向对比（识别问题集中的模块，作为后续测试设计重点）
- 评审改进措施的成效验证（如培训后某维度问题占比是否下降）

## 评审修订闭环

> **目的**：定义从评审结论到修订完成的完整链路，确保修订有据可查、修订后必须复审、上下文不丢失。

### 修订流程（4 步）

```
步骤 V1 用户确认修订范围 → 步骤 V2 转交生成模式修订 → 步骤 V3 复审受影响维度 → 🔴 CHECKPOINT 步骤 V4 输出修订报告
```

**步骤 V1：用户确认修订范围**
- 用户基于评审意见表 + 改进建议清单，圈定本次需修订的用例 ID 列表与对应问题
- 用户可选择性修订（如仅修订 P0 问题，P1/P2 留待下迭代）
- 明确「不修订」的用例须在修订报告中标注原因

**步骤 V2：转交生成模式修订**
- 调用默认生成模式（四阶段流程），按修订范围逐条修订
- 修订时必须携带评审上下文（见下方 schema），不得脱离评审意见重新生成

**步骤 V3：复审受影响维度**
- **强制复审**：修订后的用例必须重新走 R2 评审，但仅复审「受影响维度」
- 受影响维度判定：
  - 修订了 title → 复审「字段规范」「冗余」「可维护性」
  - 修订了 steps → 复审「可执行性」「冗余」「可维护性」「可自动化」
  - 修订了 expected_results → 复审「字段规范」「可执行性」「可自动化」
  - 修订了 priority → 复审「优先级合理性」
  - 修订了 preconditions → 复审「可维护性」「测试数据依赖」
  - 新增用例 → 复审「覆盖度」「优先级合理性」「溯源」「可维护性」「可自动化」「测试数据依赖」
  - 删除用例 → 复审「覆盖度」「溯源」
- 若复审发现新问题 → 回到 V1 重新确认修订范围

**步骤 V4：输出修订报告**
- 输出修订前后对比 + 复审结果 + 修订度量（修订条数 / 复审通过率 / 残留问题数）

🔴 **CHECKPOINT · 修订完成**：必须展示修订报告给用户确认，用户确认后评审流程才视为完成。

## 补充用例 ID 生成规则（强制）

评审模式下发现缺失用例需要补充时，**必须**按以下规则生成 ID：

**ID 格式**：`TC_{模块}_{功能}_{三位序号}`

**生成步骤**：
1. **确定模块缩写**：从已有用例 ID 中提取模块前缀（如 `TC_WEBHOOK_ADD_001` → `WEBHOOK`）
2. **确定功能缩写**：从已有用例 ID 中提取功能前缀（如 `TC_WEBHOOK_ADD_001` → `ADD`）
3. **确定序号**：扫描已有用例 ID，找到相同模块+功能的最大序号，新用例序号 = 最大序号 + 1

**示例**：
- 已有用例：`TC_WEBHOOK_ADD_001`, `TC_WEBHOOK_ADD_002`, `TC_WEBHOOK_EDIT_001`
- 补充新增用例：`TC_WEBHOOK_ADD_003`（最大序号 2 + 1）
- 补充编辑用例：`TC_WEBHOOK_EDIT_002`（最大序号 1 + 1）

**禁止**：
- 使用不规范格式：`TC_001`、`TC_NEW_001`、`TC补充_001`
- 序号跳跃：已有 001、002，新增用 005
- 模块/功能缩写不一致：已有 `WEBHOOK`，新增用 `WH` 或 `webhook`

## 评审模式失败模式与 Fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| 用例格式不规范（缺字段/结构混乱）无法解析 | 列出已识别字段，提示用户补充缺失字段后重新评审 | 仅评审可解析部分，标注「格式不规范，评审不完整」 |
| 无需求文档（溯源维度无法执行） | 仅检查用例内部一致性（步骤与预期是否对应），标注「溯源维度降级」 | 跳过溯源维度，在意见表中标注「未评审：溯源（缺需求文档）」 |
| 用例数量过大（>100 条） | 按模块/优先级分批评审，每批 ≤50 条 | 分层抽样：按优先级（P0/P1/P2/P3）× 模块交叉分层，每层抽 20%（最少 1 条），记录抽样种子（如 `seed=20260722`）确保可复现，标注「抽样评审」 |

## 评审模式反例（不要做什么）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|----------|
| 1 | **直接修订用例而不输出意见表** | 用户无法理解评审结论，失去评审透明度 | 必须先输出意见表+建议清单，修订走用户确认后的默认生成模式 |
| 2 | **评审意见使用模糊词** | 用户无法定位具体问题 | 引用具体用例 ID + 具体字段 + 具体问题描述 |
| 3 | **跳过 CHECKPOINT 直接修订** | 违反用户确认原则，用户无法控制修订方向 | 必须展示意见表，用户确认后才进入修订 |
| 4 | **评审维度全打 P0 或全打 P2** | 失去优先级指导价值，用户无法聚焦 | 按问题严重程度分级，P0 占比应 < 30% |
| 5 | **评审越界：擅自改业务规则** | 评审的职责是检查用例质量，不是改业务规则；擅自改规则会让用例脱离需求意图 | 评审意见只引用规则、指出问题，业务规则变更须回溯需求文档并由产品确认后，再通过修订闭环走生成模式修订 |
| 6 | **评审主观化：不引用判定规则** | 「感觉这个用例不好」「这个步骤写得不清楚」等主观判断无法复核，也无法转化为机器可校验的规则 | 每条评审意见必须引用 R2 表中对应的判定规则（如「命中模糊词正则」「P0 占比偏离 10%~15%」），并给出具体用例 ID + 字段 + 命中证据 |
| 7 | **评审遗漏维度** | 只评覆盖度和字段规范、跳过优先级合理性/可执行性/冗余/溯源/可维护性/可自动化/测试数据依赖，导致系统性问题被掩盖 | R2 必须覆盖全部 9 维度；溯源维度在无需求文档时按失败模式 #2 降级，但不得直接跳过；度量报告的维度分布表用于自检 9 维度是否均有产出 |

## 增量评审子模式

> **目的**：迭代场景下仅评审本次新增/修改的用例，避免全量重复评审。借鉴 change-impact-analyzer 的 diff 模式。

### 触发条件

满足以下任一条件进入增量评审：
- 用户明确要求「增量评审」「只评审改动的用例」「评审本次新增」
- 用户提供 git diff 范围 + 用例文件路径（如「评审 feat/order 分支相对 main 的用例改动」）

### 工作流

增量评审复用标准评审的 R1-R3 流程，但 R1 的解析范围不同：

```
R1 解析 diff 用例 → R2 6 维度评审（仅针对变更用例）→ 🔴 CHECKPOINT R3 输出增量评审意见
```

**R1 差异**：
1. 执行 `git diff <range> -- <用例文件路径>` 提取变更内容
2. 从 diff 中识别新增（`+`）和修改的用例（按用例 ID 匹配前后版本）
3. 仅将变更用例纳入 R2 评审范围；未变更用例不评审

**R2 差异**：
- 覆盖度/优先级合理性维度：基于**全量用例**统计比例（变更用例可能影响整体分布），但仅对**变更用例**输出问题
- 冗余维度：变更用例须与**全量用例**比对（变更用例可能与未变更用例重复）
- 字段规范/可执行性/溯源维度：仅检查变更用例

**R3 差异**：
- 评审意见表标注「增量评审」+ diff 范围
- 度量报告的「用例总数」= 变更用例数，非全量用例数

### 增量评审失败模式

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| 用例文件不在 git 仓库中 | 提示用户确认路径，或改用全量评审 | 降级为全量评审，标注「增量模式不可用，已转全量」 |
| diff 中无用例变更（用例文件未修改） | 提示用户本次无用例变更，确认是否评审全量 | 终止评审，输出「无用例变更」结论 |
| 变更用例无法从 diff 中提取结构（如格式损坏） | 列出 diff 中可识别的用例 ID，提示用户确认 | 仅评审可解析的变更用例，标注「部分变更用例无法解析」 |

## 评审报告文件化

> **目的**：将评审产出沉淀为文件，支持历史对比与趋势分析。默认输出到 `docs/review-reports/` 目录。

### 触发条件

满足以下任一条件触发文件化：
- 用户明确要求「保存评审报告」「输出到文件」「沉淀评审结果」
- 评审用例数 ≥20 条（规模化评审建议留存）
- 评审度量报告整体评级为 C 或 D（需改进/不合格，建议留存用于追踪改进）

### 文件命名规则

```
docs/review-reports/{YYYYMMDD}-{module}-{mode}.md
```

- `YYYYMMDD`：评审日期
- `module`：被评审模块（从用例 ID 的模块段提取，如 `TC_order_*` → `order`）
- `mode`：`full`（全量评审）/ `incremental`（增量评审）/ `sampled`（抽样评审）

**示例**：`docs/review-reports/20260722-order-incremental.md`

### 文件内容结构

```markdown
# 用例评审报告 · {模块} · {YYYY-MM-DD}

## 元信息
- 评审日期：{YYYY-MM-DD}
- 评审模式：{全量/增量/抽样}
- 评审范围：{用例文件路径 / git diff 范围}
- 用例总数：N 条
- 整体评级：{A/B/C/D}

## 评审意见表
{R3 输出的评审意见表}

## 改进建议清单
{R3 输出的改进建议清单}

## 评审度量报告
{R3 输出的评审度量报告}

## 修订记录（若有）
{若进入修订闭环，V4 输出的修订报告}

## 历史对比（若有同类报告）
- 上次评审：{日期} {评级} {通过率} {问题密度}
- 本次评审：{日期} {评级} {通过率} {问题密度}
- 趋势：{通过率变化} {问题密度变化} {评级变化}
```

### 文件化失败模式

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| `docs/review-reports/` 目录不存在 | 创建目录后写入 | 在对话中输出完整报告，提示用户手动保存 |
| 模块名无法从用例 ID 提取（如 ID 格式不规范） | 提示用户指定模块名 | 用 `unknown` 作为模块名 |
| 同日同模块同模式已存在报告文件 | 在文件名追加序号（如 `-02`） | 覆盖旧文件前提示用户确认 |
