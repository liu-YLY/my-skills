## 工作流概览

```
输入：需求文档/用户描述/代码变更
  ↓
阶段 1：理解需求 → 输出：需求理解文档（结构化）
  ↓
阶段 2：提取测试点 → 输出：测试点清单（Markdown 表格）
  ↓
阶段 3：编写用例 → 输出：测试用例（通用结构化格式）
  ↓
阶段 4：自检补全 → 输出：最终用例 + 效率报告
```

**阶段间依赖**：
- 阶段 2 的输入是阶段 1 的需求理解输出
- 阶段 3 的输入是阶段 2 的测试点清单
- 阶段 4 的输入是阶段 3 的测试用例

## 阶段 1：理解需求

**目标**：全面透彻地理解被测功能的业务背景、技术实现和产品现状。

**输入**：需求文档/用户描述/代码变更/设计稿
**输出**：需求理解文档（结构化，包含12个字段）

### 信息收集（按优先级顺序）

| 优先级 | 手段 | 具体做法 |
|--------|------|----------|
| **P0** | 扫描项目知识库 | 列举/扫描约定目录（`docs/`、`testing-tm-cases/`），读取相关文档 |
| **P0** | **加载产品知识库** | 识别被测功能所属产品，加载 `knowledge/products/{product-id}.md`（详见下方） |
| **P0** | 搜索代码库 | 搜索相关模块的代码实现、API 定义、数据模型、校验逻辑 |
| **P0** | 阅读已有用例 | 读取同模块/相近功能的已有用例，继承风格、避免重复 |
| **P1** | 公开文档检索 | 检索第三方服务文档、API 规范、竞品实现 |
| **P1** | 阅读产品代码 | 查看控制器/服务层/校验逻辑，理解实际行为 |
| **P2** | 追问用户 | 列出具体问题清单向用户确认，不做假设 |

### 产品知识库加载（阶段 1 必做）

当被测功能可识别产品归属时，**必须加载**对应的产品知识文件：

1. **识别产品**：从以下来源推断产品ID
   - 用户明确指定（如 `使用 [产品名称] 知识库`）
   - 需求文档中的产品名称/模块名
   - 代码路径（如 `src/modules/crm/` → 产品ID `crm`）
   - API 路径（如 `/api/v1/payment/` → 产品ID `payment`）
   - 配置文件或数据库表名中的产品标识

2. **查找知识文件**：在 `knowledge/products/` 目录下查找 `{product-id}.md`
   - 精确匹配：`{product-id}.md`
   - 模糊匹配：若无精确匹配，搜索文件名包含产品关键词的文件
   - 索引文件：参考 `knowledge/products/README.md` 中的产品列表

3. **加载并应用**：
   - 读取「业务规则与约束」→ 用于阶段 1 需求理解，提取产品特定的业务逻辑
   - 读取「高频缺陷模式」→ 用于阶段 2 补充防御性测试点，避免已知缺陷
   - 读取「专项测试检查清单」→ 用于阶段 2 确保重点领域覆盖
   - 读取「状态流转规则」→ 用于阶段 2 设计状态相关测试点
   - 读取「常见测试场景模板」→ 用于阶段 3 加速用例编写
   - 读取「数据字典与校验规则」→ 用于阶段 2/3 确保输入校验测试点准确

4. **加载后处理**：
   - 将产品知识中的业务规则与需求文档进行交叉验证
   - 识别产品特定的术语和概念，在后续阶段使用准确的业务语言
   - 标记产品知识中与当前需求相关的章节，便于阶段 2 快速定位

5. **若无产品知识文件**：跳过此步骤，按通用流程执行，但在阶段 1 输出中标注「未找到产品知识文件，按通用流程执行」

> **产品知识索引**：[knowledge/products/README.md](knowledge/products/README.md)
> **知识模板**：[knowledge/products/products-template.md](knowledge/products/products-template.md)

**信息来源追踪规则**：执行期间每读取/搜索一个文件或文档，立即追加到「信息来源」列表。

🔴 **CHECKPOINT · 阶段 1 完成**：需求理解输出后，必须展示给用户确认再进入阶段 2。用户确认前不得开始提取测试点。

### 阶段 1 失败模式与 Fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| 项目文档缺失/不完整 | 扫描代码库提取业务逻辑，搜索注释和 README | 列出「存疑点」清单，要求用户补充确认 |
| 代码库无权限或不可达 | 请求用户提供关键代码片段或 API 文档 | 基于需求文档推断，标注「待验证」 |
| 已有用例风格不一致 | 取最近 3 个模块的用例作为参考基线 | 采用通用用例格式，阶段 4 统一校验 |
| 用户无法确认存疑点 | 按最保守理解设计测试点，覆盖多种可能性 | 标记为「待确认」，用例中增加条件分支 |

### 可测性与验收口径（阶段 1 必对照）

- **日志与追踪**：能否用 request id / trace id 唯一定位一次调用
- **可观测性**：指标、告警、链路是否与需求中的验收口径一致
- **造数与配置**：测试账号、造数接口、mock、特性开关是否齐备
- **验收定义**：除界面结果外，是否要求后端状态、对账结果、消息投递等可验证证据

### 业务系统识别（阶段 1 必做）

识别被测功能所属的业务系统类型，自动补充对应的核心测试维度：

| 系统类型 | 识别特征 | 核心测试维度 |
|----------|----------|--------------|
| **通知/邮件系统** | Email、Notification、Message、Subscription、Unsubscribe | 通知链路（发送/不发送/重复/失败）、用户状态（已订阅/已退订/黑名单）、配置影响（全局开关/子开关/用户偏好） |
| **状态机系统** | Enable/Disable、Toggle、Status、Publish/Draft、Active/Inactive | 初始状态、状态切换、状态保存、刷新后状态、状态互斥、非法状态流转 |
| **配置系统** | 全局配置、子配置、Feature Flag、多级配置 | 上游配置影响、下游行为变化、配置覆盖关系、配置优先级、配置冲突 |
| **支付/订单系统** | Payment、Order、Refund、Invoice | 金额精度、重复支付、部分退款、支付超时、订单状态流转 |
| **权限系统** | Role、Permission、Auth、Access Control | 角色权限矩阵、越权访问、Token 过期、敏感数据保护 |
| **搜索/筛选系统** | Search、Filter、Sort、Pagination | 特殊字符、空结果、大结果集、排序稳定性、筛选组合 |
| **异步任务系统** | Job、Queue、Scheduler、Cron、Async | 任务触发、重试逻辑、死信处理、延迟执行、并发控制 |

**识别后处理**：
1. 在需求理解中标注识别出的系统类型
2. 自动补充对应的核心测试维度到测试点清单
3. 若涉及多个系统，逐一覆盖各系统的核心维度

## 阶段 2：精准提取测试点

**目标**：从需求理解中系统性提取所有可测试点，做到不遗漏。

**输入**：阶段 1 的需求理解文档
**输出**：测试点清单（Markdown 表格，包含编号/测试点/维度/设计方法/优先级/用例数/需求引用/来源）

### 风险与优先级（粗分级）

在展开 7 维度扫描前，先对本次功能做粗粒度风险评估：变更面（影响模块多少）、业务影响（资金/合规/核心路径）、技术复杂度（新依赖、并发、分布式）。据此辅助映射 P0–P3。

### 7 维度扫描框架

对每个功能/页面，按以下 7 个维度逐一扫描：

| 维度 | 检查内容 | 最低覆盖深度 |
|------|----------|-------------|
| **正向功能** | 核心业务流程的正常路径 | 每个用户角色至少 1 条 Happy Path |
| **反向校验** | 每个输入字段的异常/非法/边界输入 | 每个字段至少 4 类：有效值、空值、格式错误、边界值 |
| **权限/角色** | 不同用户类型看到的不同内容和可用操作 | 每个角色 × 每个功能点 = 1 条 |
| **状态联动** | 选项 A 变化时选项 B 的联动响应 | 每对联动关系 × 源状态数 |
| **边界与限制** | 数量上限、长度限制、重复校验 | 上限值、上限+1、重复值、零值 |
| **UI 交互** | 按钮状态、提示文案、页面跳转、加载状态 | 每个按钮/状态变化点 = 1 条 |
| **异常与容错** | 网络异常、并发操作、服务端错误响应 | 网络断开、服务端 5xx、并发冲突 |

### 测试点设计原则（强制）

**单一验证目标**：
- 每条测试点仅允许一个核心验证目标
- 禁止：`验证保存成功并且页面展示正确`
- 必须：拆分为 `验证保存成功` + `验证页面展示正确` 两条测试点

**禁止堆叠验证**：
- 禁止在一个测试点中同时验证：
  - 正常 + 异常
  - 默认值 + 修改后
  - UI + 数据保存
  - 权限 + 展示

**要素完备性**：
- 每个测试点必须包含：[验证对象] + [操作/场景] + [预期结果方向]
- 禁止：`验证 General updates 可以关闭`（太模糊）
- 必须：`验证 General updates 状态开关(Toggle)切换至关闭并保存后，配置项保持关闭状态`

**核心业务优先**：
- 测试点数量与覆盖应向核心业务倾斜
- 禁止大量生成低风险 UI 测试点
- 例如通知系统的核心是：是否发送、是否正确发送、是否受退订影响
- 而不是：Toggle UI、文案展示

### 缺陷模式对照补充（必须执行）

7 维度扫描完成后，执行两轮缺陷模式对照：

**第一轮：通用缺陷模式**

读取 [knowledge/bug-patterns.md](knowledge/bug-patterns.md) 中的「缺陷模式→测试点映射」表，根据当前功能特征补充防御性测试点：

- 有输入框？→ 检查是否补充了 XSS、超长截断、编码问题
- 有提交按钮？→ 检查是否补充了重复提交
- 有列表页？→ 检查是否补充了空数据、分页
- 涉及权限？→ 检查是否补充了越权访问
- 有状态流转？→ 检查是否补充了状态回退、幽灵状态
- 有弹窗？→ 检查是否补充了弹窗残留

**第二轮：产品专项缺陷模式**

若阶段 1 已加载产品知识文件，读取其中的「测试重点领域」章节：

- 「历史高频缺陷」→ 对照当前功能是否涉及，补充对应测试点
- 「专项测试检查清单」→ 逐项检查是否已覆盖
- 「性能基线」→ 若涉及性能测试，参考基线值设计测试点
- 「状态流转规则」→ 确保状态相关测试点覆盖所有流转路径

### 产品特定测试点提取（阶段 2 必做）

若阶段 1 已加载产品知识文件，**必须**基于产品知识提取产品特定测试点：

1. **业务规则测试点**：
   - 从产品知识的「业务规则与约束」章节提取所有业务规则
   - 每条业务规则至少对应 1 个正向验证测试点（规则满足时的行为）
   - 每条业务规则至少对应 1 个反向验证测试点（规则违反时的错误处理）
   - 标注来源为「产品知识-业务规则」

2. **状态流转测试点**：
   - 从产品知识的「状态流转规则」章节提取所有状态流转路径
   - 每条合法流转路径对应 1 个正向测试点
   - 每条非法流转路径（不允许的状态变更）对应 1 个反向测试点
   - 覆盖所有状态组合，包括并发状态变更场景
   - 标注来源为「产品知识-状态流转」

3. **高频缺陷测试点**：
   - 从产品知识的「高频缺陷模式」章节提取历史缺陷模式
   - 若当前功能涉及相关模块/场景，补充对应的防御性测试点
   - 标注来源为「产品知识-高频缺陷」

4. **专项检查测试点**：
   - 从产品知识的「专项测试检查清单」章节提取检查项
   - 逐项检查当前功能是否涉及，涉及则补充对应测试点
   - 标注来源为「产品知识-专项检查」

5. **数据校验测试点**：
   - 从产品知识的「数据字典与校验规则」章节提取字段校验规则
   - 每个字段的校验规则对应多个测试点（有效值、无效值、边界值）
   - 标注来源为「产品知识-数据校验」

**产品特定测试点标注规则**：
- 在测试点清单的「来源」列标注「产品知识-{章节名}」
- 便于追溯测试点的来源，确保产品知识的完整应用

### 设计方法选择决策树

- 输入域有明确范围？→ 等价类划分 + 边界值分析
- 存在多个条件组合？→ 判定表/因果图
- 系统有明确状态流转？→ 状态迁移法
- 业务流程测试？→ 场景法（基本流 + 备选流 + 异常流）

```markdown
### 阶段 2 失败模式与 Fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| 测试点遗漏（阶段 4 发现） | 回溯 7 维度逐项核对，补充遗漏维度的测试点 | 标记遗漏维度，用例中标注「待补充」 |
| 缺陷模式库无匹配 | 使用通用缺陷模式（输入校验/状态/权限/数据/UI） | 仅输出 7 维度扫描结果，不强制补充 |
| 设计方法选择困难 | 优先使用场景法（基本流+备选流+异常流） | 多种方法并用，阶段 4 去重 |
```

## 阶段 3：编写测试用例

**输入**：阶段 1 的需求理解 + 阶段 2 的测试点清单
**输出**：测试用例（通用结构化格式，Markdown 表格 + 列表）

### AI 生成模式（默认）

**流程**：
1. 读取 [knowledge/prompt-strategy.md](knowledge/prompt-strategy.md) 中的结构化提示词模板
2. 将需求理解 + 测试点清单填入 Prompt 模板
3. **若已加载产品知识**：将产品知识中的「业务规则」「常见测试场景模板」作为上下文注入 Prompt
4. AI 生成 70%-90% 的基础用例
5. **人工审核**：检查业务逻辑准确性、补充领域特有边界条件、验证优先级合理性
6. 人工补充 AI 遗漏的复杂业务逻辑用例

**产品知识注入要点**：
- 使用产品知识中的「数据校验规则」确保用例中的输入值符合业务约束
- 使用产品知识中的「状态流转规则」确保状态相关用例覆盖完整路径
- 使用产品知识中的「测试数据与环境」信息完善用例的前置条件
- 使用产品知识中的「常见测试场景模板」作为 Few-shot 示例

**AI 生成的用例标记** `source: ai-generated`，人工补充的标记 `source: manual`。

**审核重点**：
- 业务逻辑是否准确
- 领域特有边界条件是否覆盖
- 优先级是否合理
- steps 是否可直接执行

🛑 **STOP · 阶段 3 AI 生成后**：AI 生成的用例必须经过人工审核才能输出。未经审核的用例禁止标记 `source: ai-generated`。

### 阶段 3 失败模式与 Fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| AI 生成质量不达标（覆盖率<80%） | 调整 Prompt：增加约束、补充 Few-shot 示例 | 分批生成（单次 10-15 条），人工补充剩余 |
| AI 生成格式不合规 | 要求 AI 按通用用例格式重新输出 | 人工手动修正格式，标记 `source: manual` |
| steps 与 expected_results 数量不一致 | 检查每步操作是否都有可观察结果，补充或合并 | 重新梳理步骤，确保一步一验 |
| 用例标题超过 40 字符 | 精简为 `{被测对象} - {具体行为}` | 截断并添加省略号 |

🔴 **CHECKPOINT · 阶段 3 输出前**：在输出用例之前，必须确认以下事项：
- 所有必填字段已填写（ID、标题、优先级、类型、步骤、预期结果）
- steps 与 expected_results 数量一致
- 标题不超过 40 字符
- 无模糊词
- 用户确认前不得输出最终用例

### 分层测试策略

根据需求类型选择测试分层，确定用例的验证层级：

| 分层 | 名称 | 关注点 | 适用场景 | 目标占比 |
|------|------|--------|----------|----------|
| **L1** | 单元层 | 核心算法、复杂计算逻辑 | 核心业务逻辑型需求 | 0-10% |
| **L2-A** | 业务场景层 | 跨接口业务闭环、状态机流转 | 复杂业务流程、异步任务 | 30% |
| **L2-B** | 接口契约层 | API 参数校验、DB 读写、第三方对接 | 所有后端功能 | 30% |
| **L3** | UI/E2E 层 | 用户旅程、交互体验、端到端闭环 | 前端样式、交互相关 | 40% |

**动态调整规则**：

| 需求类型 | 典型场景 | L2 占比 | L3 占比 |
|----------|----------|---------|---------|
| 核心业务逻辑型 | 积分规则、自动化营销、VIP 判定 | 70% | 30% |
| 数据集成型 | 对接新物流商、同步订单、数据报表 | 80% | 20% |
| 纯交互/UI型 | 首页改版、新增引导弹窗、样式优化 | 20% | 80% |

**分层选择决策**：
1. 需求涉及核心算法/复杂计算？→ 必须选择 L1 单元测试
2. 需求涉及跨接口业务闭环？→ 必须选择 L2-A 业务场景测试
3. 需求涉及 API/数据/第三方？→ 必须选择 L2-B 接口测试
4. 需求涉及 UI/交互/体验？→ 必须选择 L3 E2E 测试

### 默认输出功能测试用例

除非用户明确要求或需求内容明显涉及以下类型，否则只输出功能测试用例（`type: functional`）：

| 额外类型 | 触发条件 |
|----------|----------|
| performance | 需求提到响应时间、并发量、吞吐量要求 |
| security | 涉及认证、授权、敏感数据、支付 |
| compatibility | 需求提到多浏览器、多设备、多系统版本 |

当需要额外类型时，在功能测试用例之后独立成组追加。

### 补充用例 ID 生成规则（强制）

当评审模式或阶段 4 自检发现缺失用例需要补充时，**必须**按以下规则生成 ID：

**ID 格式**：`TC_{模块}_{功能}_{三位序号}`

**生成步骤**：
1. **确定模块缩写**：从已有用例 ID 中提取模块前缀（如 `TC_WEBHOOK_ADD_001` → `WEBHOOK`）
2. **确定功能缩写**：从已有用例 ID 中提取功能前缀（如 `TC_WEBHOOK_ADD_001` → `ADD`）
3. **确定序号**：扫描已有用例 ID，找到相同模块+功能的最大序号，新用例序号 = 最大序号 + 1

**示例**：
- 已有用例：`TC_WEBHOOK_ADD_001`, `TC_WEBHOOK_ADD_002`, `TC_WEBHOOK_EDIT_001`
- 补充新增用例：`TC_WEBHOOK_ADD_003`（最大序号 2 + 1）
- 补充编辑用例：`TC_WEBHOOK_EDIT_002`（最大序号 1 + 1）

**禁止**：
- 使用不规范格式：`TC_001`、`TC_NEW_001`、`TC补充_001`
- 序号跳跃：已有 001、002，新增用 005
- 模块/功能缩写不一致：已有 `WEBHOOK`，新增用 `WH` 或 `webhook`

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| **ID** | 是 | `TC_{模块}_{功能}_{三位序号}`，作为标题 |
| **标题** | 是 | 不超过 40 字符，动宾结构 `{被测对象} - {具体行为}` |
| **优先级** | 是 | P0/P1/P2/P3，定义与比例见 [knowledge/test-standards.md](knowledge/test-standards.md) |
| **类型** | 是 | functional / ui / security / performance / compatibility / usability / accessibility / observability |
| **需求追溯** | 否 | 需求 ID（Story ID）和测试点编号（TP 编号） |
| **来源** | 否 | 用例来源：ai-generated / manual / ai-reviewed |
| **业务背景** | 否 | 补充说明测试场景的业务背景 |
| **前置条件** | 否 | 执行测试前必须满足的条件列表 |
| **测试步骤** | 是 | 操作步骤表格，每步包含操作和对应的预期结果 |
| **标签** | 否 | 场景标签，用于分类和筛选 |

### 编写与审核铁律

**title 规范**：
- 格式：`{被测对象} - {具体行为}`，不超过 40 字符，动宾结构
- 必须包含：验证对象、用户行为（若存在）、预期结果
- 禁止：`验证 Width`、`验证布局`、`验证 Rounded 为 16px`

**行业优秀实践示例**：

| 场景 | 好的 title | 坏的 title | 原因 |
|------|------------|------------|------|
| 登录功能 | `手机号登录 - 正确凭据成功跳转首页` | `验证登录功能` | 缺少具体行为和预期结果 |
| 表单提交 | `优惠券创建 - 满减金额低于阈值提示错误` | `验证优惠券创建` | 缺少边界条件和错误场景 |
| 权限控制 | `用户管理 - 普通用户无法访问管理员页面` | `验证权限` | 缺少具体角色和行为 |
| 状态流转 | `订单状态 - 已发货订单无法修改收货地址` | `验证订单状态` | 缺少具体状态和限制 |

**steps 规范**：
- 使用主动语态（如"输入"、"点击"、"滑动"、"断开"）
- 每一步仅包含一个核心动作
  - 禁止：`修改并保存`、`点击并跳转`、`输入后提交`
  - 应拆分：`输入 xxx` → `点击 Save` → `查看页面展示`
- 步骤必须落到具体页面、控件、输入值、交互、配置项、用户动作
- 禁止仅使用"刷新"、"查看"、"进入"等动作替代核心操作
- 推荐动作词：输入、点击、选择、切换、上传、删除、保存、拖拽、刷新、搜索、提交
- Steps 中涉及输入、选择、配置时，必须明确具体值
  - 禁止：`输入测试数据`、`填写对应内容`、`选择目标配置`
- 禁止占位话术：`按测试点要求执行...`、`执行对应操作`、`验证相关功能`
- 禁止作为独立步骤：`查看 xxx 是否...`、`验证 xxx 是否...`、`检查 xxx 是否...`
- 禁止作为唯一核心操作步骤：`刷新当前页面`、`查看页面`、`进入页面`
- **不超过 7 步**，超过 7 步必须拆分为多条用例

**行业优秀实践示例**：

| 场景 | 好的 steps | 坏的 steps | 原因 |
|------|------------|------------|------|
| 登录 | `1. 输入手机号 13800138000`<br>`2. 输入密码 Abc12345`<br>`3. 点击"登录"按钮` | `1. 输入正确的手机号和密码`<br>`2. 点击登录` | 缺少具体值，"正确"是模糊词 |
| 表单 | `1. 在"优惠券名称"输入框输入"满100减20"`<br>`2. 在"使用门槛"输入框输入"100"`<br>`3. 在"优惠金额"输入框输入"20"`<br>`4. 点击"保存"按钮` | `1. 填写优惠券信息`<br>`2. 保存` | 缺少具体字段和值 |
| 权限 | `1. 使用普通账号登录系统`<br>`2. 在地址栏输入 /admin/users`<br>`3. 按回车键访问` | `1. 尝试访问管理员页面` | 缺少具体操作路径 |

**expected_results 规范**：
- 基本要求：可观察、可判定、可验证
- 必须包含明确可观察结果：页面展示、数据变化、UI变化、提示文案、跳转结果、保存状态、接口行为
- 禁止使用：`结果正确`、`符合预期`、`配置正确`、`功能正常`、`应生效`、`与测试目标一致`
- Expected Results 必须与 Steps 中的实际操作直接对应
- 禁止生成 Steps 中未发生操作的预期结果
- 数量与 steps 一致，一一对应（一步一验）
- 自检：如果我是测试执行人员，看到这个 Expected Results，能否直接判断 Pass 还是 Fail？

**行业优秀实践示例**：

| 场景 | 好的 expected_results | 坏的 expected_results | 原因 |
|------|----------------------|----------------------|------|
| 登录成功 | `1. 密码输入框显示圆点掩码`<br>`2. 页面跳转至首页，顶部显示用户名"测试用户"` | `1. 登录成功` | 缺少具体表现 |
| 表单校验 | `1. 输入框下方显示红色提示"优惠金额不能超过使用门槛"`<br>`2. 保存按钮保持置灰状态` | `1. 提示错误信息` | 缺少具体文案和UI状态 |
| 权限拦截 | `1. 页面显示403错误页面`<br>`2. 页面标题显示"无权访问"` | `1. 无法访问` | 缺少具体错误码和页面表现 |
| 数据保存 | `1. 页面顶部显示绿色提示"保存成功"`<br>`2. 数据库中user表的status字段值变为"active"` | `1. 保存成功` | 缺少UI反馈和数据库验证 |

**steps ↔ expected_results 一一对应（强制）**：
- `steps[N]` 和 `expected_results[N]` 数量必须一致
- 每步操作后立即写出对应的预期结果
- 预备动作上移到 preconditions

**测试点→用例映射规则**：
- 同一流程不同测试数据 → 合并为 1 条用例，数据参数化
- 不同前置条件/预期结果/测试逻辑 → 拆分为独立用例
- 同一字段多类校验 → 按类拆分，每类 1 条

**用例拆分与合并策略**：

用例划分需在三个目标间寻求平衡：**单一验证目标**、**失败原因可隔离**、**执行与维护成本可控**。基础准则：原子化、互不干扰的场景是最基础的划分单元，再结合业务内聚性与复杂度决定是否合并或进一步拆分。

**以下场景必须拆分独立测试用例**：
- 默认值验证
- 配置修改验证
- 保存后刷新验证
- 权限验证
- 异常输入验证
- 边界值验证
- 验证的是不同的业务规则（如正确登录 vs 错误密码登录）
- 场景之间完全独立、无相互依赖且前置条件不同
- 某一边界条件历史上反复出现缺陷或属于关键高风险逻辑（需频繁单独回归）

**以下场景可合并为一条用例**：
- **同一业务操作触发的多个紧密结果**：如用户注册成功后同时验证"数据库新增记录 + 发送欢迎邮件 + 初始积分为0"，使用**软断言（soft assert）**一次执行所有检查，汇总全部失败。要求：检查点属于同一操作触发的结果。
- **端到端核心业务流程**：如登录→搜索→加购→下单→支付，保留为一条业务流程用例。控制步骤数在 5-8 个关键节点内，每节点设置明确检查点。若流程过长，按业务阶段拆分为多个中等粒度流程用例。
- **前置条件极其昂贵或难以重建**：如数据准备需长时间批处理或依赖难以复现的环境时，可在同一用例中沿流程验证多个关键节点以分摊成本。需在用例注释中说明合并原因，此类用例作为补充而非常态。
- **同一规则下的参数化变体**：如数值输入框边界值（-1、0、1、最大值、最大值+1），通过**数据驱动**将"相同逻辑、不同输入"组织在一起，测试方法本身仍只验证一条规则。

**基于复杂度主动拆分**：
- 用例步骤数 > 15 行 → 必须拆分
- 检查点 > 5 个且不共享同一触发条件 → 必须拆分（即使用软断言也不行）
- 用例中出现 if-else 条件分支模拟不同路径 → 按分支拆分为独立用例
- 同时验证页面展示、接口状态码、数据库字段值等不同层次的检查 → 按被测模块或验证层次拆分

**合并决策快速判断（5 步法）**：

| 步骤 | 问题 | 判断 |
|------|------|------|
| 1 | 两个场景验证的是同一个业务规则吗？ | 否 → 必须独立 |
| 2 | 合并后，一个场景失败是否会阻断其他场景的验证？ | 会（且无法用软断言解决）→ 独立 |
| 3 | 合并后是否需要软断言才能一次跑完所有检查？ | 是，且检查点属于同一操作触发的结果 → 可以合并 |
| 4 | 分开写是否会导致大量无法消除的重复准备？ | 是 → 考虑合并为流程用例，或用数据驱动方式组织 |
| 5 | 即使可合并，当前用例的步骤/检查点数量是否已显著增加维护成本？ | 是 → 按模块或层次拆分 |

**软断言说明**：当一条操作触发多个可同时验证的结果时，使用软断言在一次执行中全部验证，即使某一项失败也不会影响其他检查执行，失败输出能清晰指出哪一项不符合预期。

禁止为了减少用例数量而合并无关验证目标。

### 输出格式说明

用例输出必须采用通用结构化格式（Markdown），优势：
- 直接阅读和编辑
- 转换为任意测试管理工具格式
- 版本控制和 diff 比较

导入特定测试管理工具时，必须使用脚本或手动转换。

## 阶段 4：自检与补全

**输入**：阶段 3 的测试用例
**输出**：最终用例 + 效率报告

**行为约束**：发现遗漏直接补充到用例，不另外说明。

### 阶段 4 失败模式与 Fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| 自检发现测试点遗漏 | 回溯 7 维度逐项核对，补充遗漏维度的测试点 | 标记遗漏维度，用例中标注「待补充」 |
| 优先级比例严重偏离 | 按三步法重新评估，调整 P0-P3 分布 | 保持当前比例，在效率报告中说明偏离原因 |
| AI 生成用例覆盖率不足 | 补充 Prompt 中的测试点清单，重新生成 | 人工补充遗漏用例，标记 `source: manual` |
| 用例格式不合规 | 对照编写铁律逐项修正 | 人工手动修正，标记格式问题供后续优化 |

### 用例自检清单

```markdown
## 用例自检

**可追溯性**：
- [ ] 主要用例与测试点编号可对应
- [ ] 追溯信息（req_ref/trace 或 description 首行）完整

**覆盖度检查**：
- [ ] 所有测试点均已有对应用例
- [ ] 正向 Happy Path 全覆盖
- [ ] 每个输入字段的反向校验全覆盖
- [ ] 角色/权限矩阵全覆盖
- [ ] 联动关系全覆盖
- [ ] 边界条件全覆盖

**优先级比例检查**：
- [ ] P0 占比 10%~15%
- [ ] P1 占比 30%~40%
- [ ] P2 占比 30%~40%
- [ ] P3 占比 10%~15%

**质量检查**：
- [ ] 步骤无歧义，按流程顺序编排
- [ ] 包含具体输入数据和预期输出
- [ ] expected_results 无模糊描述
- [ ] steps 具体到可操作
- [ ] steps 与 expected_results 数量一致
- [ ] ID 无重复

**遗漏扫描**：
- [ ] 是否遗漏了某个字段/按钮/链接？
- [ ] 是否遗漏了某个用户角色？
- [ ] 是否遗漏了取消/返回/刷新等辅助操作？
- [ ] 涉及安全时：是否对照了安全测试检查清单？

**产品知识覆盖检查**（若有产品知识文件）：
- [ ] 产品知识中的「历史高频缺陷」是否已覆盖
- [ ] 产品知识中的「专项测试检查清单」是否逐项检查
- [ ] 产品知识中的「状态流转规则」是否已完整覆盖
- [ ] 产品知识中的「业务规则」是否已在用例中体现

**AI 生成质量检查**（仅 AI 生成模式）：
- [ ] AI 生成用例是否覆盖了所有测试点中的标准场景
- [ ] 人工补充的用例是否覆盖了 AI 容易遗漏的领域特有业务逻辑
- [ ] AI 生成用例的 source 标记是否准确

🔴 **CHECKPOINT · 阶段 4 完成**：自检通过后，必须展示最终用例给用户确认。用户确认前不得视为完成。
```

### 效率度量输出（AI 生成模式必做）

阶段 4 自检完成后，输出效率报告：

```markdown
## 效率度量报告

| 指标 | 本次值 |
|------|--------|
| 用例总数 | X 条 |
| AI 生成 | Y 条 (Z%) |
| 人工补充 | W 条 |
| 预估传统工时 | N 人天 |
| 实际工时 | M 人天 |
| 工时节省 | (N-M)/N * 100% |
| 效率提升 | N/M 倍 |

**质量观察**：
- AI 生成用例中需修改的比例：约 A%
- AI 补充的异常/安全场景数量：B 条
```

**传统工时估算基准**：
- 低（单表单/单页面，3-5 字段）：0.5 人天
- 中（多表单/多页面，含联动/权限）：1-2 人天
- 高（复杂业务流程，多角色多状态）：3-4 人天
- 极高（跨系统集成，核心业务链路）：5+ 人天

## 模式切换

- **默认**：不允许跳过阶段 1/2 直接写完整用例
- **快速**：压缩阶段 1/4，但必须输出测试点清单，文首声明未做完整理解与自检
- **探索式**：额外输出探索章程，与测试点清单并列
- **评审**：独立于四阶段生成流程，对他人/已有用例做 6 维度评审（详见 [knowledge/review-mode.md](knowledge/review-mode.md)）
