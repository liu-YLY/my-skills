# 准入准出标准

> **何时阅读**：制定测试策略、判定测试启动/结束条件、评估发布就绪度时查阅。
> **覆盖范围**：准入条件三类 / 准出条件四类 / 项目类型分档 / 反例。

## 1. 准入标准（Entry Criteria）

测试启动前必须满足的条件，三类全部通过方可开测。任意一项未满足即打回，禁止"边测边补"。

### 1.1 开发完成准入

- [ ] 代码评审通过（至少 1 人 approve）
- [ ] 单元测试覆盖率 ≥ 70%（核心模块 ≥ 80%）
- [ ] 单元测试全部通过
- [ ] 静态代码扫描无 P0/P1 问题
- [ ] 开发自测完成（冒烟用例全过）

### 1.2 环境就绪准入

- [ ] 测试环境部署成功
- [ ] 测试环境与生产环境规格差异 ≤ 20%（CPU/内存/DB）
- [ ] 测试数据准备完成（覆盖正常/异常/边界）
- [ ] 依赖服务可用（Mock 或真实）

### 1.3 文档就绪准入

- [ ] 需求文档已基线化
- [ ] 接口文档已更新
- [ ] 测试用例已评审通过

## 2. 准出标准（Exit Criteria）

测试完成判定条件，四类全部通过方可发布。任意一项未满足需走变更评审流程，禁止临时放宽。

### 2.1 覆盖率准出

- [ ] 测试用例执行率 ≥ 95%
- [ ] 需求覆盖率 ≥ 100%
- [ ] 风险矩阵中高风险模块用例覆盖率 ≥ 100%
- [ ] 接口覆盖率 ≥ 90%

### 2.2 缺陷收敛准出

- [ ] P0/P1 缺陷修复率 100%
- [ ] P2 缺陷修复率 ≥ 95%
- [ ] P3 缺陷修复率 ≥ 80%
- [ ] 缺陷回归通过率 100%
- [ ] 近 3 天无新增 P0/P1 缺陷

### 2.3 性能达标准出

- [ ] 核心接口 P99 RT ≤ SLA
- [ ] 系统吞吐量 ≥ 目标 TPS
- [ ] 错误率 ≤ 0.1%（支付类 ≤ 0.01%）
- [ ] 资源利用率 ≤ 70%（CPU/内存）

### 2.4 文档交付准出

- [ ] 测试报告已输出
- [ ] 已知风险清单已记录
- [ ] 遗留缺陷有明确处理计划

## 3. 准入准出标准参考表（按项目类型分档）

不同项目类型采用不同分档，项目立项时锁定档位，禁止跨档借用。

| 项目类型 | 单元覆盖率 | 用例执行率 | P0 修复率 | P99 RT |
|---------|-----------|-----------|----------|--------|
| 互联网应用 | ≥ 70% | ≥ 95% | 100% | ≤ 500ms |
| 金融系统 | ≥ 85% | ≥ 98% | 100% | ≤ 200ms |
| 内部系统 | ≥ 60% | ≥ 90% | 100% | ≤ 2s |
| 遗留系统 | ≥ 40% | ≥ 90% | 100% | 按现状 |

**档位选用规则**：
1. 金融、支付、账户类系统固定按"金融系统"档
2. 面向 C 端用户的高并发系统按"互联网应用"档
3. 仅内部员工使用的管理后台按"内部系统"档
4. 无文档、无用例的存量系统按"遗留系统"档，需同步输出重构计划

## 4. 反例（不要这样做）

| 反例 | 后果 | 正确做法 |
|------|------|---------|
| 准入条件未满足就开测 | 浪费测试资源，反复打回 | 三类准入全过方可开测，未过即打回 |
| 准出只看 P0 缺陷 | P1 缺陷堆积同样影响质量 | P0/P1 修复率均需 100% |
| 性能准出只看平均值 | 平均值掩盖长尾，P99 才是用户体验真实写照 | 以 P99 RT 为准出指标 |
| 准出标准临时放宽 | 放宽标准等于没有标准 | 走变更评审流程，留痕并定回收期限 |
