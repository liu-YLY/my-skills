> **何时阅读**：strategy skill 输出测试策略文档时按此模板填充结构与内容。
> **覆盖范围**：策略文档主结构 / 风险矩阵模板 / 分层策略模板 / 准入准出 checklist / 可选附录触发条件 / 反例。

## 2. 风险矩阵模板

引用 `risk-matrix-framework.md` 的空白模板与 5 维评分表：

```markdown
| 功能模块 | 业务复杂度(1-5) | 变更频率(1-5) | 历史缺陷(1-5) | 外部依赖(1-5) | 数据敏感性(1-5) | 总分(5-25) | 风险等级 | 测试优先级 |
|---------|-----------------|---------------|---------------|---------------|-----------------|-----------|---------|-----------|
|         |                 |               |               |               |                 |           |         |           |
|         |                 |               |               |               |                 |           |         |           |
```

**5 维评分定义**：

| 维度 | 评估问题 | 评分锚点 |
|------|---------|---------|
| 业务复杂度 | 涉及多少业务规则？状态机多复杂？ | 1=简单CRUD / 5=复杂状态机+多方交互 |
| 变更频率 | 近 3 个迭代是否频繁改动？ | 1=稳定 / 5=每迭代都改 |
| 历史缺陷 | 上线后 Bug 数量与严重度？ | 1=无Bug / 5=P0 Bug ≥1 |
| 外部依赖 | 依赖多少外部服务/接口？ | 1=无 / 5=≥3 个外部依赖 |
| 数据敏感性 | 涉及资金/隐私/合规？ | 1=无 / 5=资金/隐私核心 |

**风险等级映射**：

| 总分 | 风险等级 | 测试优先级 | 测试要求 |
|------|---------|-----------|---------|
| 18-25 | 高 | P0 | 必测 + 自动化 + 探索测试 |
| 11-17 | 中 | P1 | 必测 + 关键路径自动化 |
| 5-10 | 低 | P2 | 必测 |

**使用规则**：
- 评分必须基于事实数据（Bug 数、变更记录、依赖清单），禁止主观感觉
- 历史缺陷维度 1 个 P0 直接记 5 分
- 总分 = 5 维评分之和，禁止跳过任一维度

## 3. 分层策略模板

引用 `test-pyramid.md` 的分层比例表，按项目类型选择：

| 项目类型 | 单元测试 | 接口测试 | UI/E2E 测试 | 备注 |
|---------|---------|---------|------------|------|
| 后端服务 | 70% | 25% | 5% | E2E 仅覆盖关键 API 链路 |
| 前端应用 | 50% | 20% | 30% | 单元覆盖组件逻辑，E2E 覆盖关键用户旅程 |
| 全栈 | 60% | 25% | 15% | 前后端平衡分配 |
| 微服务 | 60% | 30% | 10% | 接口测试含契约测试 |
| 遗留系统 | 40% | 40% | 20% | 单元难补，靠接口测试 + 回归测试兜底 |

**各层职责边界**（引用 `test-pyramid.md` §2）：

| 层级 | 测试对象 | 必须做 | 禁止 |
|------|---------|--------|------|
| 单元测试 | 函数/类 | 覆盖核心逻辑分支、mock 外部依赖 | 跨模块调用、访问数据库/网络 |
| 接口测试 | 服务/接口契约 | 验证业务流程、参数校验、错误码 | UI 操作、断言样式/布局 |
| UI/E2E 测试 | 用户旅程 | 覆盖关键端到端流程 | 校验细节字段、覆盖每个按钮 |

**选择规则**：
- 项目类型必须从「项目特征」章节读取，禁止重新假设
- 比例为区间中值，可按项目实际微调 ±5%，禁止突破 ±10%
- 遗留系统无单元测试基础时，接口测试比例上调，必须说明原因

## 4. 准入准出 checklist 模板

引用 `entry-exit-criteria.md`，可直接勾选：

### 准入 checklist

```markdown
- [ ] 开发自测通过，单元测试覆盖率达标
- [ ] 提测单已提交，含变更说明与影响范围
- [ ] 测试环境就绪，数据初始化完成
- [ ] 依赖服务可用，版本对齐
- [ ] 需求文档/接口文档/设计文档齐全
- [ ] 冒烟测试用例通过
```

### 准出 checklist

```markdown
- [ ] 测试用例执行率 100%
- [ ] P0/P1 缺陷修复率 100%
- [ ] P2 缺陷修复率 ≥ 95%
- [ ] 单元测试覆盖率 ≥ 80%（或项目约定阈值）
- [ ] 接口测试覆盖率 ≥ 90%
- [ ] 性能指标达标（响应时间、吞吐量、错误率）
- [ ] 安全扫描无高危项
- [ ] 测试报告输出，遗留缺陷有应对方案
- [ ] 上线 checklist 与回滚方案就绪
```

**使用规则**：
- checklist 项必须全部勾选方可准入/准出，缺一不可
- 阈值（如覆盖率 80%）可按项目约定调整，调整必须在文档中标注理由
- 遗留缺陷必须有明确责任人 + 修复时间窗，禁止"待跟进"

## 5. 可选附录触发条件

附录章节非默认生成，仅在用户明确要求时输出。触发条件如下：

| 附录章节 | 触发关键词 | 未触发时标注 |
|---------|-----------|------------|
| 资源估算附录 | 用户提及"估算人力/环境/工具" | 未生成（用户未要求） |
| 进度排期附录 | 用户提及"排期/里程碑/时间表" | 未生成（用户未要求） |
| 风险跟踪附录 | 用户提及"风险跟踪/跟踪清单" | 未生成（用户未要求） |

**判定规则**：
- 关键词需用户主动提出，strategy skill 不主动追问、不主动生成
- 触发判定基于用户原始输入，禁止从上下文推断
- 同一附录只生成一次，禁止重复输出
- 触发后附录内容必须完整填充，禁止半成品占位
- 未触发时主模板第 7 章整体替换为单行：`附录：未生成（用户未要求）`

## 6. 反例（不要这样做）

- **模板填空式输出** → 策略文档需基于项目特征定制，不是填表。每个 `{}` 必须替换为具体决策，禁止保留占位符
- **默认生成资源/进度附录** → 附录仅在用户明确要求时生成，默认不生成以避免冗余
- **模板照搬不引用知识库** → 模板内容必须引用 `risk-matrix-framework.md` / `test-pyramid.md` / `entry-exit-criteria.md`，保持评分维度、占比、checklist 一致
- **策略文档过长（>20页）** → 策略文档聚焦决策（测什么/不测什么/何时停），执行细节交由测试计划文档
- **软化措辞** → 禁止"建议/可以考虑/灵活把握/视情况而定"，必须给出明确决策
- **跳过风险矩阵** → 风险矩阵是优先级与范围的依据，缺失则后续章节无依据
- **分层比例脱离项目类型** → 禁止所有项目统一套用 60/25/15，必须按项目类型查表
