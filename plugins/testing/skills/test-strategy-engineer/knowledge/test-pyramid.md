# 测试金字塔分层模型

> **何时阅读**：制定测试策略、分配测试资源、评估测试覆盖率合理性时查阅。
> **覆盖范围**：金字塔三层占比 / 各层职责边界 / 项目类型分档 / 工具类型选择 / 反例。

## 1. 经典金字塔模型

按以下三层分配测试用例，三层缺一不可：

| 层级 | 占比 | 执行速度 | 反馈周期 | 成本 | 目的 |
|------|------|---------|---------|------|------|
| 单元测试 | 60-70% | 毫秒级 | 秒级反馈 | 低 | 验证函数/类逻辑正确性 |
| 接口测试 | 20-30% | 秒级 | 分钟级 | 中 | 验证服务/接口契约、业务流程 |
| UI/E2E 测试 | 5-10% | 分钟级 | 小时级 | 高 | 验证用户旅程、端到端流程 |

**分配原则**：
1. 单元测试为基础层，占比最高，覆盖核心逻辑分支
2. 接口测试为中间层，验证服务间协作与契约
3. UI/E2E 测试为顶层，仅覆盖关键用户旅程，不覆盖每个按钮
4. 越往下层占比越高、执行越快、维护成本越低

## 2. 各层职责与边界

| 层级 | 测试对象 | 必须做 | 禁止做 |
|------|---------|--------|--------|
| 单元测试 | 函数/类内部逻辑 | Mock 所有外部依赖；覆盖边界条件与异常分支 | 测试框架本身；测试私有实现细节 |
| 接口测试 | API 契约 | 验证请求/响应/状态码/错误码；验证业务流程编排 | 测试 UI 渲染；依赖 UI 环境运行 |
| UI/E2E 测试 | 关键用户旅程 | 覆盖登录→下单→支付等核心链路；验证端到端流程 | 测试每个按钮；覆盖所有交互细节 |

**边界判定规则**：
- 单元测试边界：被测函数的外部依赖全部 Mock，仅验证被测函数自身输入输出
- 接口测试边界：从 API 入口发起请求，验证响应数据与状态，不启动浏览器
- UI/E2E 测试边界：通过浏览器/客户端操作完整流程，验证用户可达成业务目标

## 3. 分层比例参考（按项目类型分档）

不同项目类型的分层占比必须按下表执行：

| 项目类型 | 单元 | 接口 | UI/E2E | 备注 |
|---------|------|------|--------|------|
| 后端服务 | 70% | 25% | 5% | 重单元，UI 仅用于核心 API 验证 |
| 前端应用 | 40% | 30% | 30% | 前端单元测试较难，UI 占比相对高 |
| 全栈应用 | 50% | 35% | 15% | 平衡型 |
| 微服务 | 60% | 30% | 10% | 契约测试替代部分接口测试 |
| 遗留系统 | 30% | 50% | 20% | 单元测试难补，重接口覆盖 |

**档位选择规则**：
1. 按项目主导技术栈确定类型（后端为主→后端服务档）
2. 微服务架构优先选微服务档，强制引入契约测试
3. 遗留系统指无单元测试覆盖且代码耦合严重的存量系统
4. 选定档位后占比偏差不得超过 ±5%

## 4. 分层工具选择

本 skill 仅推荐工具类型，不编写脚本。

| 层级 | 工具类型 | 示例工具 |
|------|---------|---------|
| 单元测试 | xUnit 框架 | JUnit / pytest / Jest |
| 接口测试 | API 测试框架 | REST Assured / Postman / HttpRunner |
| UI/E2E | 浏览器自动化 | Selenium / Playwright / Cypress |
| 契约测试 | 消费者驱动 | Pact / Spring Cloud Contract |

**选择规则**：
1. 单元测试优先选用与开发语言同生态的 xUnit 框架
2. 接口测试优先选用支持参数化与断言库的框架
3. UI/E2E 测试优先选用支持录制回放与显式等待的框架
4. 微服务架构必须引入契约测试工具，消费者驱动模式

## 5. 反例（禁止做法）

| 反例 | 后果 | 正确做法 |
|------|------|---------|
| UI 测试占比 > 30% | 反馈慢、维护成本高、不稳定 | UI 占比控制在 5-10%，前端应用最高 30% |
| 单元测试测框架本身 | 测 Spring/React 框架无意义，浪费资源 | 仅测自有业务逻辑，信任框架保证 |
| 接口测试依赖 UI 环境 | 接口测试无法独立运行，耦合 UI 故障 | 接口测试直接调用 API，独立于 UI |
| 只有一层测试（全单元或全 UI） | 单一层级无法覆盖所有风险 | 三层按比例分配，缺一不可 |
| 微服务忽略契约测试 | 接口变更会破坏消费者，集成故障频发 | 微服务档强制引入契约测试 |

**自检清单**：
1. UI/E2E 测试占比是否 ≤ 30%？
2. 单元测试是否仅覆盖自有业务逻辑？
3. 接口测试是否能脱离 UI 独立运行？
4. 是否同时具备三层测试？
5. 微服务架构是否已引入契约测试？
