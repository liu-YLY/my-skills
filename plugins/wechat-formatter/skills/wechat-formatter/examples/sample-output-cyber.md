# 测试用例的正确打开方式 ⚡

> ⚡ **系统启动** | 深入测试用例编写的核心机制，预计阅读 6 分钟。

编写测试用例是测试工程师的核心工作之一。很多人觉得简单——把操作步骤罗列出来就行。但实际上，**真正高质量的测试用例背后，是深入的需求分析和系统的测试点提取**。

---

## 1. 理解需求

拿到需求之后，不要急着写用例。先问自己几个问题：

- 这个功能的**核心价值**是什么？
- 有哪些**用户角色**会用到？
- 涉及哪些**数据流转**？

以「用户注册」功能为例。表面看很简单——填表、提交、注册成功。深入分析后，你会发现需要考虑的场景非常多。

> **🔑 核心结论**：需求不明确的地方不要做假设。列出问题清单向产品经理逐条确认。

---

## 2. 七维度扫描法 🎯

我常用的方法是 **7 维度扫描法**——对每个功能模块，从七个维度逐一扫描。

| 维度 | 关注点 | 示例 |
|------|--------|------|
| 1. 正向功能 | Happy Path 正常路径 | 正确填写信息完成注册 |
| 2. 反向校验 | 空值、格式错误、边界值 | 手机号少 1 位、密码为空 |
| 3. 权限/角色 | 不同用户看到的内容差异 | 已登录用户访问注册页 |
| 4. 状态联动 | 选项之间的联动关系 | 选择国家 → 区号自动切换 |
| 5. 边界与限制 | 数量上限、长度限制、重复提交 | 连续点击注册按钮 3 次 |
| 6. UI 交互 | 按钮状态、提示文案、跳转逻辑 | 注册按钮防重复点击 |
| 7. 异常与容错 | 网络断开、服务端 5xx、超时 | 短信服务超时的表现 |

> **⚠️ 警告**：多数人只关注前两个维度。但从线上故障统计来看，**权限、状态联动、异常容错**这三个维度贡献了超过 60% 的生产事故。

---

## 3. 编写用例的规范 💻

测试用例的核心字段：

- **id**：唯一标识，推荐格式 `TC_{模块}_{功能}_{序号}`
- **title**：动宾结构，不超过 40 字符
- **priority**：P0（核心路径）/ P1（重要校验）/ P2（边界场景）/ P3（探索式）
- **steps**：祈使句，每步一个操作
- **expected_results**：与 steps **一一对应**，一步一验

标准用例示例：

```
id: TC_USER_REGISTER_001
title: 正确填写注册信息并提交
priority: P0
type: functional
steps:
  - 打开注册页面
  - 输入有效手机号
  - 输入符合规则的密码
  - 点击注册按钮
expected_results:
  - 页面显示注册表单
  - 手机号输入框显示已输入内容
  - 密码以掩码形式显示
  - 提示注册成功并跳转至登录页
```

> **🔑 核心结论**：一条用例只验证一个测试逻辑。把多个校验塞进一条用例，失败后你根本无法定位是哪个校验出了问题。

preconditions 放预备动作（打开页面、登录、准备数据），steps 放有验证价值的具体操作。不要在 steps 里塞入纯预备动作。

## 4. Python 自动化示例 🚀

对于可自动化的用例，推荐使用 `pytest` 框架。下面是一个完整的示例：

```python
import pytest


class TestUserRegister:
    """用户注册接口测试"""

    def test_register_success(self, api_client):
        """正向用例：正确填写注册信息并提交"""
        phone = "13800138000"
        password = "Test@123456"

        response = api_client.register(phone, password)

        # 精确断言到具体字段值
        assert response.status_code == 200
        assert response.json()["code"] == 0
        assert "token" in response.json()["data"]

    @pytest.mark.parametrize("phone,password,expected_msg", [
        ("", "Test@123456", "手机号不能为空"),
        ("13800138000", "", "密码不能为空"),
        ("1380013800", "Test@123456", "手机号格式错误"),
    ])
    def test_register_field_validation(
        self, api_client, phone, password, expected_msg
    ):
        """反向校验：空值和格式错误"""
        response = api_client.register(phone, password)
        assert response.json()["code"] != 0
        assert expected_msg in response.json()["message"]
```

测试数据要独立于环境——使用专门的测试账号或动态生成数据，避免依赖特定数据库记录。**断言要精确到具体字段值**，而非仅判断是否成功。

---

## 5. 总结 🔥

- **先理解需求，再提取测试点，最后写用例**——这个顺序不能乱
- **7 维度扫描法**是系统化提取测试点的有效工具，尤其不要忽视权限、状态联动、异常容错这三个「高价值维度」
- **一条用例一个测试逻辑**是保障用例质量不可妥协的铁律
- **steps 与 expected_results 一一对应**，让每条用例都可独立执行和验证
- 自动化用例的生命线是**数据独立性**和**断言精确性**
