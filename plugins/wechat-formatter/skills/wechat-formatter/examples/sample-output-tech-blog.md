# 如何编写高质量的测试用例

> **📖 导读**
>
> 本文约 **2800 字**，预计阅读 **8 分钟**。文章将带你了解一套经过项目实战验证的测试用例编写方法论，覆盖需求分析、测试点提取、用例规范及自动化实践。适合 **1-5 年经验的测试工程师** 和希望提升代码质量的**开发工程师**阅读。

---

## 一、前言

**测试用例**（Test Case）是测试工程师的核心产出物。

很多人觉得写测试用例很简单——把操作步骤罗列出来不就行了？但实际上，真正高质量的测试用例背后，是深入的需求分析和系统的测试点提取。

本文的方法论经过了多个项目的验证，核心思想只有一条：**先想清楚再动手**。

---

## 二、理解需求：不要急着写用例

拿到需求之后，第一步不是打开 Excel 开始填模板，而是回答三个问题：

- 这个功能的**核心价值**是什么？
- 有哪些**用户角色**会用到？
- 涉及哪些**数据流转**？

以「用户注册」功能为例。表面看需求只有一句话——填写手机号和密码完成注册。但深入分析，你会发现至少需要考虑：

- 手机号是否已注册？（唯一性校验）
- 密码强度规则是什么？（长度、字符类型、历史密码）
- 注册后是否需要短信验证？
- 不同渠道（App / H5 / 小程序）的流程是否一致？

这些问题，在需求理解阶段就必须明确。需求不明确的地方不要做假设，列出问题清单向产品经理逐条确认——我见过太多因为需求理解偏差导致整批用例返工的案例。

---

## 三、提取测试点：七维度扫描法

理解了需求之后，下一步是系统化地提取测试点。我常用的方法是 **7 维度扫描法**——对每个功能模块，从七个维度逐一扫描。

**维度一：正向功能**

验证核心业务流程的正常路径（Happy Path）。每个用户角色至少 1 条正向用例。正向用例通过不代表功能没问题，但不通过代表功能根本不可用。

**维度二：反向校验**

每个输入字段覆盖四类输入：有效值、空值、格式错误、边界值。

**维度三：权限与角色**

不同用户类型看到的内容和可执行的操作可能不同。按「角色 × 功能点」建立矩阵，确保每条权限规则都有对应用例。

**维度四：状态联动**

当某个选项变化时，检查关联选项是否联动变化。比如选择「中国」时区号自动变为 +86。

**维度五：边界与限制**

覆盖数量上限、长度限制、重复提交、并发操作等场景。

**维度六：UI 交互**

按钮的禁用/启用状态、提示文案的准确性、页面跳转逻辑。

**维度七：异常与容错**

网络断开、服务端返回 5xx、依赖服务超时等异常场景下的表现。

这七个维度中，多数人只关注前两个。但从线上故障统计来看，**权限、状态联动、异常容错**这三个维度贡献了超过 60% 的生产事故。

## 四、编写用例的规范

### 4.1 基本格式

测试用例的核心字段：

- **id**：唯一标识，推荐格式 `TC_{模块}_{功能}_{序号}`
- **title**：动宾结构，简洁有力，不超过 40 字符
- **priority**：P0（核心路径）/ P1（重要校验）/ P2（边界场景）/ P3（探索式）
- **steps**：祈使句，每步一个操作
- **expected_results**：与 steps **一一对应**，一步一验

标准用例示例：

```
id: TC_USER_REGISTER_001
title: 正确填写注册信息并提交
priority: P0
type: functional
preconditions:
  - 手机号未注册
  - 验证码服务正常
steps:
  - 打开注册页面
  - 输入有效的未注册手机号
  - 输入符合规则的密码
  - 点击「注册」按钮
expected_results:
  - 页面正常展示注册表单，各输入框可交互
  - 手机号输入框显示已输入的内容
  - 密码以掩码形式显示
  - 提示「注册成功」并自动跳转至登录页
```

### 4.2 一条用例只验证一个测试逻辑

这是新手最容易犯的错误——把必填校验、格式校验、边界值校验全部塞进一条用例。用例失败后你根本无法定位是哪个校验出了问题。

**正确做法**：每个测试逻辑独立成一条用例。同一逻辑下不同数据可以用参数化合并，但不同逻辑必须拆分。

### 4.3 preconditions 与 steps 的区分

`preconditions` 放预备动作（打开页面、登录、准备数据），`steps` 放有验证价值的具体操作。不要在 steps 里塞入纯预备动作。

## 五、Python 自动化示例

对于可自动化的用例，推荐使用 `pytest` 框架。下面是一个完整的示例：

```python
import pytest


class TestUserRegister:
    """用户注册接口测试"""

    def test_register_success(self, api_client):
        """正向用例：正确填写注册信息并提交"""
        phone = "13800138000"
        password = "Test@123456"

        response = api_client.register(phone, password)

        assert response.status_code == 200
        assert response.json()["code"] == 0
        assert "token" in response.json()["data"]

    @pytest.mark.parametrize("phone,password,expected_msg", [
        ("", "Test@123456", "手机号不能为空"),
        ("13800138000", "", "密码不能为空"),
        ("1380013800", "Test@123456", "手机号格式错误"),
    ])
    def test_register_field_validation(
        self, api_client, phone, password, expected_msg
    ):
        """反向校验：空值和格式错误"""
        response = api_client.register(phone, password)
        assert response.json()["code"] != 0
        assert expected_msg in response.json()["message"]
```

测试数据要独立于环境——使用专门的测试账号或动态生成数据，避免依赖特定数据库记录。断言要精确到具体字段值，而非仅判断是否成功。

## 六、总结

- **先理解需求，再提取测试点，最后写用例**——这个顺序不能乱
- **7 维度扫描法**是系统化提取测试点的有效工具，尤其不要忽视权限、状态联动、异常容错这三个「高价值维度」
- **一条用例一个测试逻辑**是保障用例质量不可妥协的铁律
- **steps 与 expected_results 一一对应**，让每条用例都可独立执行和验证
- 自动化用例的生命线是**数据独立性**和**断言精确性**

本文的方法论最适合功能测试阶段。对于性能测试、安全测试、可用性测试等专项领域，需要补充相应的测试策略。
