# 排版规则详解

> **何时阅读**：阶段 1（分析文章内容）和阶段 4（质量校验）时必读。
> **覆盖范围**：§1 文章要素识别 / §2 通用排版规则 / §3 风格适配规则 / §4 质量校验清单 / §5 混合风格处理。
> **可跳过条件**：无。阶段 1 必须分析要素，阶段 4 必须逐项校验。

---

## §1 文章要素识别（阶段 1）

拿到用户文章后，首先快速扫描并识别以下要素：

### 1.1 内容类型判定

| 要素 | 识别方法 | 标记方式 |
|------|---------|---------|
| 代码片段 | 缩进块 / 三反引号 / 行内反引号 | 统计数量，判断占比 |
| 步骤流程 | 数字编号 / "首先...然后...最后" / "Step" | 识别是否以步骤为主 |
| 理论分析 | 长段落 / 专业术语密集 / 引用多 | 判断深度和篇幅 |
| 经验总结 | 第一人称 / "我" / 时间线叙事 | 判断是否个人分享 |
| 数据图表 | 表格 / 数据罗列 / "如图" | 标记需要图表占位的位置 |
| 标题层级 | `#` / `##` / `###` 数量 | 记录原始层级结构 |

### 1.2 文章元信息提取（默认模式）

```markdown
## 文章分析

**标题**：[原始标题]
**字数**：[约 xxx 字]
**内容类型**：[编程教程 / 测试理论 / 架构分析 / 经验分享 / 其他]
**代码占比**：[约 xx%（N 个代码块）]
**主要语言**：[Python / Java / Go / ...]
**关键要素**：
- 代码块：N 个
- 列表：N 处
- 引用：N 处
- 图片引用：N 处
**最佳推荐风格**：[tech-blog / tutorial / deep-dive / casual-chat]
**推荐理由**：[一句话说明]
```

### 1.3 内容类型 → 风格推荐规则

```
if 步骤流程为主 and 代码块 ≥ 2:
    → 推荐 tutorial
elif 代码块 ≥ 5 and 理论分析占比 > 40%:
    → 推荐 tech-blog
elif 总字数 > 3000 and (理论分析 or 架构分析):
    → 推荐 deep-dive
elif 第一人称 and 经验分享:
    → 推荐 casual-chat
else:
    → 推荐 tech-blog（通用兜底）
```

## §2 通用排版规则（所有风格适用）

### 2.1 标题体系

```
# 文章标题             ← 仅一个，对应公众号标题
## 章节标题            ← 正文主要层级
### 小节标题           ← 细分内容
#### 最小标题           ← 尽量避免
```

- 禁止跳级（如 `##` 后直接 `####`）
- 同级标题风格一致（如都用动宾结构或都用名词短语）
- 标题后不跟标点符号

### 2.2 段落规则

- 段落间空一行
- 手机端每段不超过 5 行（约 200 字）
- 首段不缩进
- 中文与英文/数字之间加空格：`Python 3.11 发布了` ✅  `Python3.11发布了` ❌
- 中文与行内代码之间加空格：使用 `print()` 函数 ✅

### 2.3 列表规则

- 无序列表用 `-`，不用 `*`
- 有序列表用 `1.` `2.` `3.`
- 列表项不超过 3 行
- 嵌套列表缩进 2 空格
- 列表前后各空一行

### 2.4 强调规则

- **粗体** `**text**`：用于核心概念首次出现、关键结论
- *斜体* `*text*`：用于引用书名、次要强调（公众号中斜体显示效果差，**尽量不用**）
- `` `code` ``：用于代码中的符号、文件名、命令、配置项
- ~~删除线~~ `~~text~~`：不推荐在公众号中使用

### 2.5 链接规则

```markdown
✅ 脚注式（推荐，公众号兼容）：
详见 Python 官方文档 [1]。

---

**参考资料**

[1] Python 官方文档: https://docs.python.org/3/
[2] PEP 8 规范: https://peps.python.org/pep-0008/
```

## §3 风格适配规则

| 规则项 | tech-blog | tutorial | deep-dive | casual-chat |
|--------|-----------|----------|-----------|-------------|
| Emoji 上限 | ≤3 个 | ≤5 个 | ≤2 个 | 8-15 个 |
| 段落长度 | 2-4 句 | 2-5 句 | 3-6 句 | 1-3 句 |
| 代码块语言标注 | 必须 | 必须 | 必须+路径 | 建议 |
| 目录 | 不需要 | 不需要 | **必须** | 不需要 |
| 提示框 | 简单 `>` | TIP/WARNING/NOTE/CHECK | 核心结论框 | 无 |
| 总结章节 | 必须 | FAQ替代 | 总结+思考 | 互动引导 |
| 参考文献 | 列表 | 延伸阅读 | 带说明列表 | 可选 |
| 图表占位 | 简单 | 无 | 编号+标题 | 无 |

## §4 质量校验清单（阶段 4）

排版完成后，逐项检查：

### 4.0 自然度检查（最高优先级）

- [ ] 文章读起来像一篇「真正的文章」，而非「按模板填出来的作业」
- [ ] 提示框/结论框的使用自然，不生硬——如果去掉某个框文章反而更流畅，就去掉它
- [ ] 段落之间的过渡自然，有逻辑连接，而非突然跳转
- [ ] 编号体系（一/二/三 或 1/2/3）的使用一致且不突兀
- [ ] 整体节奏有变化（长短段落交替），而非每段都一样长

### 4.1 覆盖度检查

- [ ] 用户原文的所有内容要点均已纳入排版后文件
- [ ] 原文中的代码块全部保留并格式化
- [ ] 原文中的列表、引用、链接全部保留
- [ ] 原文的标题层级已合理转换

### 4.2 可读性检查

- [ ] 手机屏幕预览：每段不超过 5 行
- [ ] 标题层级不超过 3 级（## → ### → ####）
- [ ] 中英文/数字之间有空格
- [ ] 没有连续多个代码块（中间有说明文字）
- [ ] 没有超过 30 行的代码块（长代码考虑拆分或省略）

### 4.3 公众号兼容性检查

> 详细规则见 [wechat-markdown.md](wechat-markdown.md)

- [ ] 公众号不支持的 Markdown 语法已替换
- [ ] 代码块可在公众号中正常显示
- [ ] 表格不超过 4 列
- [ ] 图片使用了占位符 URL 且带注释说明
- [ ] 没有使用 HTML 标签（除非必要）

### 4.4 风格一致性检查

- [ ] 全篇 Emoji 数量在风格限制范围内
- [ ] 标题风格统一
- [ ] 引用块风格统一
- [ ] 全篇语气一致（正式/口语化）

## §5 混合风格处理

当用户文章同时具备多种特征时（如既有步骤教程又有深度原理分析），按以下规则处理：

### 5.1 判断规则

```
if 存在 ≥2 个明显不同的内容区块 and 各适合不同风格:
    → 标记为混合型文章
else:
    → 按主导内容类型选择单一风格
```

### 5.2 混合处理策略

| 情况 | 处理方式 | 示例 |
|------|---------|------|
| 主体教程 + 少量原理 | 全程用 `tutorial`，原理部分用 `💡 NOTE` 框融入 | Python 入门教程中解释 GIL 机制 |
| 主体分析 + 嵌入操作步骤 | 全程用 `tech-blog` 或 `deep-dive`，步骤部分用有序列表呈现 | 微服务架构分析中含部署步骤 |
| 两个独立部分各占一半 | 向用户确认后，分别用两种风格排版，生成两个独立文件 | — |

### 5.3 向用户确认

当检测到混合型文章时，告知用户：

> 您的文章包含两种内容特征：A 部分偏教程、B 部分偏深度分析。建议：
> 1. 整体用 `tutorial` 风格，B 部分用 NOTE 框补充
> 2. 整体用 `deep-dive` 风格，A 部分作为"实践验证"章节
> 3. 拆分为两篇独立文章分别排版
>
> 请选择您偏好的方式。
