# 轻松聊天风格排版规则

> **何时阅读**：用户选择 `casual-chat` 风格时必读。
> **覆盖范围**：口语化表达、Emoji 规范、对话感营造、整体结构模板。
> **可跳过条件**：未选择此风格时跳过。

---

## 风格定位

亲和、真诚、像朋友聊天。对标准一线技术博主的随笔风格——**不会太水，但也不端着**。读者读完后觉得「这哥们挺有意思」，而不是「这篇文章写得好正式」。

**视觉关键词**：短段落 / 多留白 / 对话感 / 适度 Emoji / 读完不累

**推荐配色**：橙色 `#ff6600` + 深灰 `#3f3f3f`（活力温暖，与轻松的语气呼应）

---

## 排版规则

> **⚠️ 自然度优先原则**
>
> 模板提供的是骨架，不是填空题。排版时应优先保证文章**读起来自然流畅**，其次才是符合模板结构。如果某个分隔符或 Emoji 会让文章显得刻意，宁可省略。记住：**读者不会拿着你的模板检查清单逐项比对**——他们只会判断这篇文章好不好读。

### 1. 标题规范

- 正文标题**从 `##` 起步**
- 标题尽量口语化、有趣味性，但表意清晰
- 允许标题中使用 Emoji，放在文字前面
- 标题不要太长，控制在 10 字以内

```markdown
✅ 好的标题：
## 🎯 我是怎么写测试用例的
## 那个让我通宵三天的 Bug
## 🤔 为什么你的代码总是 Bug 不断
## 一个测试工程师的日常碎碎念

❌ 太正式的标题（不符合本风格）：
## 测试用例编写方法论
## 软件开发缺陷根本原因分析报告
## 基于 TDD 的测试驱动开发实践
```

### 2. 文章头部规范

轻松聊天风格**不使用**导读框等正式结构。开场就是一两个自然段落，像朋友开口说话：

```markdown
✅ 故事类开场：
上周五下午五点，我正准备收拾东西下班。

产品经理突然冲过来："线上出 Bug 了，用户注册全挂了！"

我心想，这不是早上刚上的版本么...

✅ 提问类开场：
你有没有遇到过这种情况——测试用例写了一百多条，上线后 Bug 一个没少？

说实话，我刚入行那会儿，这个问题困扰了我整整半年。

✅ 观点类开场：
我越来越觉得，写测试用例这件事，跟写代码一样，是需要「品味」的。

不是写得越多越好，也不是覆盖越全越好。而是——每一条用例，都要有它存在的理由。
```

### 3. 段落与节奏

这是本风格**最核心**的排版要素：

- **短段落**：每段 1-3 句话，大量留白
- **一句一段也行**：重要的观点、抖包袱的句子，单独成段
- **节奏控制**：3-5 个短段落后，来一个稍长的段落（2-3 句）调节节奏
- **对话感**：多用「你」「我」「咱们」「说实话」「你懂的」
- **口语化标记**：适当使用「其实」「说白了」「后来呢」「你猜怎么着」

**反面示例（太正式，不符合本风格）**：
> 在软件测试中，测试用例的编写质量直接影响软件产品的交付质量。本文将系统性地阐述测试用例编写的最佳实践。

**正面示例（口语化）**：
> 你有没有想过一个问题：
>
> 同样在写测试用例，为什么有的人写的用例能抓到 90% 的 Bug，而有的人写了 200 条却漏了最严重的那个？
>
> 这事儿我琢磨了好久。直到有一次线上出了个大事故，我才突然想明白。

### 4. Emoji 使用规范

**允许且鼓励使用 Emoji**，但有章法：

**A. 章节标题点缀**（每 2-3 个标题用一个）：

| Emoji | 适用场景 |
|-------|---------|
| 🎯 | 核心观点、重点来了 |
| 🤔 | 引发思考、提出问题 |
| 💡 | 分享技巧、恍然大悟 |
| 🔥 | 热点话题、干货内容 |
| 🚀 | 效率提升、进步成长 |
| ⚡ | 快速技巧、踩坑预警 |

**B. 行内情绪表达**（自然穿插在段落中）：

😂 🙈 😅 💪 🤷 ✨ 👇

**C. 分隔符**：用 `—` 分隔话题切换（比 `---` 更轻、更随意）

**限制**：
- 全文 Emoji 控制在 8-15 个
- 每段不超过 2 个行内 Emoji
- 代码块附近不使用 Emoji
- 严肃的结论或数据旁不用搞笑 Emoji

### 5. 口语化转换指南

| 正式表达 | 转换为 |
|---------|--------|
| 本文将会介绍... | 今天想跟大家聊聊... |
| 首先需要配置... | 第一步，先把...装上 |
| 需要注意的是... | 这里有个坑，千万别踩！ |
| 根据实践经验... | 说实话，做了这么多年... |
| 推荐使用... | 我个人比较推荐... |
| 总结一下 | 说到底，其实就是... |
| 这种方法存在局限性 | 但它也不是万能的 |
| 建议读者... | 你可以试试... |
| 通常情况下 | 大多数时候 |

### 6. 代码块格式

代码块保持专业，但前后说明语**完全口语化**：

```markdown
说实话，刚学 Python 的时候我写出来的代码是这样的（现在看简直不忍直视 😅）：

```python
# 这种写法虽然能跑，但...
result = []
for i in range(100):
    if i % 2 == 0:
        result.append(i * 2)
```

后来有一天，一个同事 code review 的时候忍不住了，给我改成了这样：

```python
result = [i * 2 for i in range(100) if i % 2 == 0]
```

我当时就想：我去，还能这样写？？？

从那以后我就爱上了列表推导式。真的，一行代码省五行，越写越爽 🚀
```

**规则**：
- 代码块前后保留空行
- 代码块的「介绍语」要讲故事，不要像教材
- 代码块的「总结语」要有情绪，表达你的感受

### 7. 「—」分隔符用法

话题转换、节奏停顿时用 `—` 作为轻量分隔：

```markdown
聊完成功的经验，再说说我踩过的坑。

说实话，这些坑每一个都是加班加出来的 😂

—

第一个坑：以为「代码覆盖率 80%」就万事大吉了。

实际上呢？覆盖率只是一个数字。100 条没断言的用例，覆盖率 100% 又有什么用？
```

### 8. 整体结构模板

```
# [口语化标题]

[开场白：像跟朋友聊天一样自然切入，1-2 段]

—

## [第一个话题：轻松切入]

[短段落 + 观点 + 感受]

—

## [第二个话题：逐渐深入]

[可以有一点干货了，但语气保持轻松]

> **💡 说个重点**
>
> [偶尔也可以用引用块强调关键观点，但不要像 tech-blog 那么严肃，一句话就够了]

—

## 我的一些想法

[个人观点 / 经验教训 / 对行业的观察]

这部分是全文的灵魂。观点可以犀利，但表达要真诚。

—

## 写在最后

[1-2 段轻松收尾，不要突然变成严肃的总结]

**你在工作中遇到过类似的情况吗？欢迎在评论区聊聊你的故事 👇**

—

*如果觉得这篇文章有点意思，点个 **在看** 让我知道，也欢迎**转发**给需要的朋友 🙏*
```

**结构说明**：
- 用 `—` 作为话题分隔符（不是 `---`）
- 不强制要求「总结」章节，用「写在最后」或「说点心里话」替代
- **结尾必须有互动引导**（评论、点赞、在看），这是公众号的核心互动机制
- 全文保持统一的「聊天」语气，不要在开头聊天、结尾突然变成正式论文

---

## 排版检查清单

- [ ] 标题口语化，部分带 Emoji（不超过一半）
- [ ] 开场白自然、有画面感（故事/问题/观点，选一种）
- [ ] 每段 1-3 句话，有单独成段的句子
- [ ] 多次出现「你」「我」等对话感词汇
- [ ] 全文 Emoji 数量 8-15 个，分布均匀
- [ ] 代码块讲解完全口语化，有情绪表达
- [ ] 用 `—` 分隔话题（不是 `---`）
- [ ] 有至少一处引用块 `>` 用于强调关键观点
- [ ] 结尾有互动引导（评论/在看/转发）
- [ ] 全文语气一致，不生硬，不突然变正式
- [ ] 所有 Markdown 链接已转为脚注式或直接标注
