# 深度干货风格排版规则

> **何时阅读**：用户选择 `deep-dive` 风格时必读。
> **覆盖范围**：多级目录、图表占位、结论框、引用规范、整体结构模板。
> **可跳过条件**：未选择此风格时跳过。

---

## 风格定位

深入、严谨、信息密度高。对标**美团技术团队深度分析**、**阿里技术**等企业级技术复盘文章。适合 3000 字以上的长篇技术分析，强调逻辑严密性和阅读体验的平衡。

**视觉关键词**：目录导航 / 结论醒目 / 图文并茂 / 引用可溯

**推荐配色**：深蓝 `#2b5797` + 灰 `#595959`（与 tech-blog 同系，但可加深引用块边框色 `#42b983` 增强结论框视觉）

---

## 排版规则

> **⚠️ 自然度优先原则**
>
> 模板提供的是骨架，不是填空题。排版时应优先保证文章**读起来自然流畅**，其次才是符合模板结构。如果某个结论框或编号体系会让文章显得生硬，宁可省略。记住：**读者不会拿着你的模板检查清单逐项比对**——他们只会判断这篇文章好不好读。

### 1. 标题规范

- 正文标题**从 `##` 起步**
- 核心章节使用 `## N. 章节标题`（阿拉伯数字 + 英文句点，如 `## 1. 背景与问题`、`## 2. 核心概念`）
- 子主题使用 `### N.M 子标题`（如 `### 2.1 什么是装饰器`）
- 四级标题 `####` 仅在极少数需要更细粒度拆解时使用（如 `#### 2.1.1 语法糖的本质`）
- 标题不使用 Emoji
- 标题层级严格递增，不跳级

### 2. 文章头部规范

深度干货风格的头部包含**三个元素**：摘要框、阅读建议框、目录。

**A. 摘要框（TL;DR）**：

```markdown
> **📖 摘要**
>
> [3-5 句话概括全文核心内容]。本文通过 [方法/视角]，深入分析 [核心主题]，得出 [核心结论]。
>
> **关键词**：[关键词 1] · [关键词 2] · [关键词 3] · [关键词 4]
```

**B. 阅读建议框**：

```markdown
> **🕐 阅读建议**
>
> 全文约 **XXXX 字**，预计阅读 **XX 分钟**。
> - 如果你只关心结论，直接跳至「[第 N 节标题]」
> - 如果你想理解推导过程，建议按顺序通读
> - 如果你对 [某主题] 已很熟悉，可跳过「[第 M 节标题]」
```

**C. 目录**：

使用纯文本编号列表（公众号不支持锚点链接），缩进表示层级：

```markdown
## 目录

1. 背景与问题
2. 核心概念
   2.1 概念 A
   2.2 概念 B
3. 核心分析
   3.1 分析视角一
   3.2 分析视角二
4. 实践验证
5. 总结与思考
```

### 3. 核心结论框（标志性元素）

每个核心章节结束时，用结论框提炼该章节最重要的发现：

```markdown
> **🔑 核心结论**
>
> [2-4 句话，把这个章节最关键的发现讲清楚。不是复述内容，而是提炼 insight。]
>
> ```
> 一句话记忆口诀（可选）：[把复杂结论压缩成一句易懂的话]
> ```
```

**使用规则**：
- 每 600-1000 字配一个核心结论框
- 全文至少 2 个，最多 5 个
- 结论框的内容要独立可读——读者跳着看也能 get 到核心信息

### 4. 图表占位规范

深度文章依赖图表辅助理解。排版时用标准化占位符：

```markdown
**📊 图 1：[描述性标题]**

![Spring Bean 生命周期完整流程](https://your-cdn.com/bean-lifecycle.png)
<!-- ⚠️ 建议工具：Draw.io / Excalidraw / Mermaid → 导出 PNG，宽度 900px -->

> 上图展示了从 BeanDefinition 加载到 Bean 销毁的 13 个关键阶段。重点关注红色标注的三个扩展点：`postProcessBeforeInitialization`、`afterPropertiesSet`、`postProcessAfterInitialization`。
```

**图表编号规则**：全文统一编号 `图 1`、`图 2`...，编号后跟描述性标题。

### 5. 代码引用规范

代码块必须标注**来源文件和位置**：

```markdown
**入口方法：`AbstractApplicationContext.java` 第 512 行 `refresh()`**

```java
// AbstractApplicationContext.java
public void refresh() throws BeansException, IllegalStateException {
    synchronized (this.startupShutdownMonitor) {
        prepareRefresh();                                  // ← 步骤 1：准备刷新
        ConfigurableListableBeanFactory beanFactory =
            obtainFreshBeanFactory();                     // ← 步骤 2：获取 BeanFactory
        prepareBeanFactory(beanFactory);                  // ← 步骤 3：预处理
        // ... 省略中间 9 个步骤
        finishRefresh();                                  // ← 步骤 13：完成刷新
    }
}
```

> 以上展示了 `refresh()` 方法的骨架结构。13 个步骤中，步骤 5（`postProcessBeanFactory`）和步骤 8（`registerListeners`）是最常用的扩展点。

**规则**：
- 不展示完整源码，只保留关键逻辑路径
- 代码中关键行末尾用 `// ←` 标注
- 超过 30 行的代码用 `// ... 省略 N 行` 精简
- 代码块后紧跟 1-2 句解读，点明这段代码的深层含义

### 6. 数据/对比表格

对于公众号不支持的 Markdown 表格，排版时保留表格内容但添加处理建议：

```markdown
<!-- ⚠️ Markdown 表格在公众号中不支持渲染，建议用 Excel 制作后截图替换 -->
**📊 表 1：三种方案性能对比**

| 方案 | P99 延迟 (ms) | QPS | CPU 使用率 | 实现复杂度 |
|------|-------------|-----|-----------|-----------|
| 方案 A：同步阻塞 | 120 | 8,500 | 45% | 低 |
| 方案 B：线程池 | 85 | 12,000 | 62% | 中 |
| 方案 C：异步非阻塞 | 65 | 15,000 | 38% | 高 |

> **核心发现**：方案 C 在各项指标上均优于 A 和 B，但实现复杂度显著更高。如果你的业务 QPS < 10,000，方案 B 是性价比最高的选择。
```

**规则**：
- ≤ 3 列的表格，转为列表格式呈现
- > 3 列的表格，保留 Markdown 表格 + 添加 `<!-- ⚠️ -->` 注释提醒
- 每个表格后紧跟一句「核心发现」解读

### 7. 引用外部资料

深度文章常引用论文、博客、官方文档：

```markdown
> **📚 参考资料**：[N] [资料标题] [链接] — [一句话说明这份资料与本文的关联]
```

文中引用时用 `[N]` 脚注标记，文末「参考资料」章节集中列出并附简短说明。

## 排版检查清单

- [ ] 文章开头有摘要框（📖 摘要 + 关键词）
- [ ] 文章开头有阅读建议框（🕐 阅读建议：字数 + 跳读指引）
- [ ] 有纯文本目录（编号列表，无锚点）
- [ ] 核心章节使用 `## N.` 编号格式
- [ ] 全文至少 2 个 `🔑 核心结论` 框
- [ ] 图表占位符带编号和描述性标题（图 N：标题）
- [ ] 代码块首行有文件路径注释
- [ ] 关键代码行有 `// ←` 注释标注
- [ ] 每个图表/表格后有解读文字
- [ ] 表格添加了 `<!-- ⚠️ -->` 注释提醒
- [ ] 有「总结与思考」章节（核心要点 + 局限 + 实践启示）
- [ ] 文末参考资料带简短说明（说明每个资料与本文的关联）
- [ ] 全文 Emoji ≤ 5 个（摘要框等固定结构除外）
- [ ] 链接使用脚注式 `[N]` 标记 + 文末集中列出
