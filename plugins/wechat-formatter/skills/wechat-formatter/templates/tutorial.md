# 教程指南风格排版规则

> **何时阅读**：用户选择 `tutorial` 风格时必读。
> **覆盖范围**：步骤化格式、提示框体系、信息层级、整体结构模板。
> **可跳过条件**：未选择此风格时跳过。

---

## 风格定位

清晰、结构化、手把手教学。对标**阮一峰的网络日志**教程风格、**GitHub 官方 Guides**，让零基础读者也能一步步跟下来。

**视觉关键词**：步骤感强 / 提示框醒目 / 代码与说明交替 / 每步可验证

**推荐配色**：薄荷绿 `#26a69a` + 深灰 `#333333`（清新自然，引导用户使用 mdnice 等工具应用）

---

## 排版规则

> **⚠️ 自然度优先原则**
>
> 模板提供的是骨架，不是填空题。排版时应优先保证文章**读起来自然流畅**，其次才是符合模板结构。如果某个提示框会让文章显得生硬，宁可省略。记住：**读者不会拿着你的模板检查清单逐项比对**——他们只会判断这篇文章好不好读。

### 1. 标题规范

- 正文标题**从 `##` 起步**
- 主要阶段用 `## 阶段 N：阶段名称`（如 `## 阶段一：环境准备`）
- 具体步骤用 `### Step N — 步骤名称`（注意：用 `—` em dash 而非冒号，视觉上更清晰）
- 步骤编号全文连续（Step 1, Step 2, ... Step N），不随阶段重置

### 2. 文章头部规范

教程风格必须在正文开头包含**目标说明框**和**前置条件清单**：

```markdown
> **🎯 完成本教程后，你将能够：**
>
> - [能力 1]：具体可衡量的产出
> - [能力 2]：具体可衡量的产出
> - [能力 3]：具体可衡量的产出
>
> ⏱️ **预计耗时**：XX 分钟  |  🎚️ **难度**：入门 / 进阶 / 高级
```

前置条件紧接其后：

```markdown
## 前置条件

开始之前，请确保你已具备：

- [基础知识要求]
- [软件/工具要求，含版本号]
- [硬件或网络要求（如有）]

> **💡 提示**：如果尚未安装 [关键工具]，请先阅读 [相关教程或官方文档]。
```

### 3. 步骤标准结构

每个步骤遵循固定的**五段式结构**，这是本风格的核心特色：

```markdown
### Step N — [步骤名称]

**🎯 这一步的目标**：[一句话说清做完这步能收获什么]

**操作步骤**：

1. [具体操作 1 — 祈使句，动词开头]
2. [具体操作 2 — 附带关键参数或选项说明]
3. [具体操作 3]

[代码块或截图（如有）]

**✅ 预期结果**：[描述完成后的可观察状态，让读者能自我验证]

> **📌 TIP**：[效率技巧或进阶玩法]（每 2-3 步配 1 个）

> **⚠️ 注意**：[该步骤最常见的错误及避免方法]（仅在确实有坑时添加）
```

**关键要求**：
- 「目标」让读者知道自己在做什么，不想看了也能直接跳下一步
- 「操作步骤」每小步一个动词开头（打开/输入/执行/检查）
- 「预期结果」是**可观察的**——读者能看到什么变化
- TIP 是可选的锦上添花，注意是必要的避坑指南

### 4. 提示框体系（四种类型）

本风格使用 4 种提示框，通过 Emoji + 竖线分隔符 `｜` 实现视觉区分：

```markdown
> **📌 TIP ｜ 效率技巧**
>
> 用 Excel 或 Notion 建立「字段级测试矩阵」，以字段为行、校验类型为列，逐格填入测试值和预期结果，确保全覆盖无遗漏。

> **⚠️ WARNING ｜ 常见陷阱**
>
> 不要把必填校验、格式校验、边界值校验全塞进一条用例。用例失败后你根本无法定位是哪个校验出的问题。**一条用例只验证一个测试逻辑**。

> **💡 NOTE ｜ 补充说明**
>
> 以"用户注册"功能为例，表面上看只需填表提交，但实际上还要考虑手机号是否已注册、密码强度规则、不同渠道（App/H5/小程序）的流程差异。

> **✅ CHECK ｜ 阶段验证**
>
> 执行 `pytest test_register.py -v`，确认所有测试用例状态为 **PASSED**。如有失败，返回 Step 4 检查用例配置。
```

**使用规则**：
| 框类型 | 使用频率 | 使用场景 |
|---------|---------|---------|
| TIP | 每 2-3 个步骤 1 个 | 效率优化、进阶技巧、替代方案 |
| WARNING | 每个可能出错的步骤后 | 常见错误、踩坑预警、注意事项 |
| NOTE | 关键概念需要额外解释时 | 背景知识、概念补充、设计决策说明 |
| CHECK | 每个主要阶段结束时 | 可验证的完成检查点 |

### 5. 代码块格式

代码块必须配合「执行说明 → 代码 → 预期输出 → 验证」的完整呈现：

```markdown
执行以下命令安装项目依赖：

```bash
pip install flask sqlalchemy pytest
```

命令执行后，你应该看到类似输出：

```
Successfully installed flask-3.0.0 sqlalchemy-2.0.0 pytest-7.4.0
```

> **✅ CHECK**：执行 `pip list | grep flask` 确认 Flask 已出现在已安装列表中。
```

### 6. 阶段汇总

每个主要阶段（一个 `##` 标题覆盖的范围）结束时，附一个阶段验证：

```markdown
> **✅ CHECK ｜ 阶段一验证**
>
> 在进入下一阶段前，请确认：
> - [ ] 所有依赖已成功安装
> - [ ] 开发服务器能正常启动
> - [ ] 示例页面在浏览器中可访问
```
