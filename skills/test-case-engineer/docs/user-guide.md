# test-case-engineer Skill 使用指南

> 本文档提供 test-case-engineer skill 的详细使用说明，包括快速开始、使用场景、工作流程、命令参考和最佳实践。

---

## 1. 快速开始

### 1.1 触发方式

test-engineer skill 会在以下关键词出现时自动触发：
- 测试、QA、测试用例、测试点
- Bug分析、缺陷定位
- 需求分析、测试策略

### 1.2 基本使用流程

```
1. 描述需求或问题
   ↓
2. Skill 自动触发，进入阶段 1
   ↓
3. 按四阶段流程执行
   ↓
4. 输出测试用例（结构化格式）
```

### 1.3 最简示例

**用户输入**：
```
帮我为登录功能编写测试用例
```

**Skill 响应**：
1. 阶段 1：理解需求，输出需求理解文档
2. 阶段 2：提取测试点，输出测试点清单
3. 阶段 3：生成测试用例，输出结构化格式
4. 阶段 4：自检补全，输出效率报告

## 2. 使用场景详解

### 2.1 新功能测试用例编写

**场景**：为新开发的功能编写完整的测试用例

**输入示例**：
```
我们新增了一个 webhook 管理功能，支持添加、编辑、删除 webhook。
请帮我编写测试用例。
```

**Skill 执行流程**：

**阶段 1：理解需求**
- 扫描 `docs/` 目录下的需求文档
- 搜索代码库中的相关实现
- 阅读已有的测试用例
- 输出需求理解文档

**阶段 2：提取测试点**
- 7 维度扫描：
  - 正向功能：添加/编辑/删除的正常流程
  - 反向校验：URL 格式、必填字段、超长输入
  - 权限角色：不同用户类型的权限差异
  - 状态联动：Version 与 Status 的联动
  - 边界限制：webhook 数量上限、URL 重复校验
  - UI 交互：按钮状态、提示文案、页面跳转
  - 异常容错：网络异常、并发操作、服务端错误
- 缺陷模式对照：XSS、重复提交、弹窗残留等
- 输出测试点清单

**阶段 3：生成用例**
- 使用结构化 Prompt 生成 70%-90% 基础用例
- 人工审核业务逻辑、补充领域边界
- 输出结构化格式用例

**阶段 4：自检补全**
- 覆盖度检查、优先级比例检查
- 质量检查、遗漏扫描
- 输出效率报告

**输出示例**：
```markdown
### TC_WEBHOOK_ADD_001: 添加 webhook - 正常 URL 提交成功

- **优先级**: P0
- **类型**: functional
- **需求追溯**: Story-42 / TP-01
- **来源**: ai-generated
- **业务背景**: 验证用户使用正确的 URL 格式可以成功添加 webhook

**前置条件**:
- 已登录为 A 类用户
- 已打开 "Add webhook" 弹窗

**测试步骤与预期结果**:

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 在 "Endpoint URL" 输入框中输入 `https://api.example.com/webhook` | 输入框无错误提示 |
| 2 | 点击 "Add endpoint" 按钮 | 弹窗关闭，webhook 列表中显示新增记录 |

**标签**: smoke, webhook
```

### 2.2 回归测试用例补充

**场景**：需求变更后，补充遗漏的测试用例

**输入示例**：
```
登录功能新增了"记住我"选项，请帮我补充相关测试用例。
已有用例见 testing-tm-cases/releases/v1.0/login.md
```

**Skill 执行流程**：
1. 阅读已有用例，了解已覆盖场景
2. 分析"记住我"选项的影响范围
3. 提取新增测试点（勾选/不勾选、过期时间、多设备等）
4. 生成补充用例，避免与已有用例重复
5. 自检覆盖度

### 2.3 测试用例评审

**场景**：评审现有测试用例的质量和覆盖度

**输入示例**：
```
请评审 testing-tm-cases/releases/v1.0/login.md 中的测试用例
```

**Skill 执行流程**：
1. 阅读用例文件
2. 对照自检清单逐项检查：
   - 可追溯性：ID、req_ref、trace 是否完整
   - 覆盖度：是否遗漏字段、角色、联动
   - 优先级比例：P0/P1/P2/P3 分布是否合理
   - 质量：步骤是否具体、预期结果是否可验证
3. 输出评审报告，标注问题和改进建议

### 2.4 测试策略设计

**场景**：为新功能或迭代设计测试策略

**输入示例**：
```
我们即将发布 v2.0 版本，包含 3 个新功能和 5 个 Bug 修复。
请帮我设计测试策略。
```

**Skill 执行流程**：
1. 分析变更范围（新功能 + Bug 修复）
2. 评估风险等级（业务影响、技术复杂度）
3. 确定测试分层（E2E/API/单元）
4. 制定回归范围：
   - Bug 修复：修复场景 + 关联功能 P0 用例
   - 新功能：全量覆盖
   - 大版本：P0 全量 + P1 抽样
5. 输出测试策略文档

## 3. 模式切换

### 3.1 默认模式

**特点**：完整四阶段流程，不允许跳过阶段 1/2

**适用场景**：
- 新功能测试用例编写
- 需求复杂、需要深入理解
- 要求高质量、高覆盖度

**使用方式**：无需特殊指令，直接描述需求即可

### 3.2 快速模式

**特点**：压缩阶段 1/4，必须输出测试点清单

**适用场景**：
- 需求明确、简单功能
- 时间紧迫、需要快速输出
- 补充少量遗漏用例

**使用方式**：
```
快速模式：帮我为登录功能的"记住我"选项编写测试用例
```

**输出说明**：
- 文首声明"本次采用快速模式，未做完整需求理解与自检"
- 阶段 1 简化：只输出功能概述和信息来源
- 阶段 4 简化：只做质量检查和遗漏扫描

### 3.3 探索式模式

**特点**：额外输出探索章程，与测试点清单并列

**适用场景**：
- 需求模糊、不确定性高
- 需要发散探索、找深坑
- 高风险功能的深度测试

**使用方式**：
```
探索式模式：帮我为支付功能设计测试策略，重点关注潜在风险
```

**输出示例**：
```markdown
## 探索章程（Exploration Charter）

**时间盒**：60min
**范围**：支付功能全链路
**关注点**：
- 金额精度问题（浮点误差）
- 重复支付场景（网络超时重试）
- 并发支付冲突（同一订单多人支付）
**须记录证据**：
- 请求/响应日志
- 数据库状态截图
- 前端错误提示截图
**退出条件**：
- 发现阻塞级缺陷则停止并报告
```

## 4. 命令参考

### 4.1 文档转换命令

**MarkItDown（推荐）**：
```bash
# 首次使用：创建 venv 并安装
python3 -m venv .venv-tools
.venv-tools/bin/pip install -r $SKILL_ROOT/scripts/requirements.txt

# 转换单个文件
.venv-tools/bin/markitdown docs/需求文档.docx -o docs/需求文档.md

# 批量转换
for f in docs/*.docx docs/*.pptx docs/*.xlsx; do
    [ -f "$f" ] && .venv-tools/bin/markitdown "$f" -o "${f%.*}.md"
done
```

**降级方案（MarkItDown 不可用时）**：
```bash
.venv-tools/bin/python $SKILL_ROOT/scripts/convert_docs.py docs/ --recursive
```

### 4.2 SKILL_ROOT 说明

在本仓库中：`SKILL_ROOT` = `skills/test-case-engineer`

执行命令时，将 `$SKILL_ROOT` 替换为实际路径：
```bash
# 示例：转换文档
.venv-tools/bin/markitdown docs/需求文档.docx -o docs/需求文档.md
```

## 5. 最佳实践

### 5.1 需求输入最佳实践

**好的输入**：
```
我们新增了一个 webhook 管理功能，需求文档在 docs/prd/webhook.md。
功能包括：
1. 添加 webhook（URL、描述、版本选择）
2. 编辑 webhook（只能编辑描述）
3. 删除 webhook（二次确认）
4. webhook 列表（分页、搜索、筛选）

用户角色：
- A 类用户：可选择 V2-V5 版本
- C 类用户：只能选择 V5 版本

请帮我编写测试用例。
```

**不好的输入**：
```
帮我测试 webhook 功能
```

**改进建议**：
- 提供需求文档路径
- 列出功能点清单
- 说明用户角色和权限
- 指出关键业务规则

### 5.2 测试点提取最佳实践

**7 维度扫描检查清单**：
- [ ] 正向功能：每个角色的 Happy Path
- [ ] 反向校验：每个字段的有效/无效/边界/空值
- [ ] 权限角色：每个角色 × 每个功能点
- [ ] 状态联动：每对联动关系
- [ ] 边界限制：上限值、上限+1、重复值、零值
- [ ] UI 交互：每个按钮/状态变化点
- [ ] 异常容错：网络断开、5xx、并发冲突

**缺陷模式对照**：
- 有输入框？→ XSS、超长截断、编码问题
- 有提交按钮？→ 重复提交、操作中断
- 有列表页？→ 空数据、分页错误、排序不稳定
- 涉及权限？→ 越权访问、水平/垂直越权
- 有状态流转？→ 状态回退、幽灵状态
- 有弹窗？→ 弹窗残留、关闭行为

### 5.3 用例编写最佳实践

**title 规范**：
- 好：`添加 webhook - 正常 URL 提交成功`
- 坏：`测试添加功能`

**steps 规范**：
- 好：`在 "Endpoint URL" 输入框中输入 https://api.example.com/webhook`
- 坏：`输入合法 URL`

**expected_results 规范**：
- 好：`输入框下方显示红色错误提示："Wrong URL format!"`
- 坏：`提示信息正确`

**一步一验**：
```markdown
# 正确
| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 在 URL 输入框留空 | 输入框下方显示红色提示"必填" |
| 2 | 点击提交按钮 | 弹窗未关闭，提交失败 |

# 错误
| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 在 URL 输入框留空 | 提交失败，提示必填 |
| 2 | 点击提交按钮 | （无对应预期结果） |
```

### 5.4 效率提升技巧

**利用 AI 生成**：
1. 提供结构化的需求描述
2. 提供完整的测试点清单
3. 使用 Few-shot 参考已有用例
4. 分批生成（20-30 条/批）

**人工审核重点**：
1. 业务逻辑准确性
2. 领域特有边界条件
3. 优先级合理性
4. steps 可执行性

**轻量维护**：
1. 模块级批量维护
2. 需求变更时优先重新生成
3. P2/P3 用例允许过期归档
4. P0/P1 用例必须同步更新

## 6. 常见问题

### 6.1 Token 消耗过高怎么办？

**原因**：
- 需求文档过长
- 测试点清单过详细
- 生成用例数量过多

**解决方案**：
1. 使用快速模式
2. 分批生成（20-30 条/批）
3. 只加载必要的 knowledge 文件
4. 精简需求描述

### 6.2 AI 生成的用例质量不高怎么办？

**原因**：
- 需求描述不够结构化
- 测试点清单不够完整
- Prompt 策略需要优化

**解决方案**：
1. 优化需求描述（按"功能概述/用户角色/业务规则/UI交互/数据流向"分段）
2. 完善测试点清单（7 维度扫描 + 缺陷模式对照）
3. 使用增强模板（追加历史缺陷模式）
4. 提供 Few-shot 参考

### 6.3 如何处理领域特有业务逻辑？

**解决方案**：
1. 在需求理解中详细说明业务规则
2. 在测试点清单中标注领域特有测试点
3. 人工补充 AI 遗漏的领域用例
4. 将领域知识沉淀到项目知识库

### 6.4 如何与现有测试管理工具集成？

**通用集成方式**：
1. 导出为通用结构化格式（Markdown）
2. 使用脚本或手动转换为目标工具格式
3. 导入到测试管理工具

### 6.5 如何处理需求变更？

**解决方案**：
1. 识别变更影响范围
2. 评估需要补充的测试点
3. 优先使用 AI 重新生成受影响模块
4. 人工审核补充领域特有用例
5. 更新测试点清单和用例

## 7. 高级用法

### 7.1 自定义 Prompt 模板

在 `knowledge/prompt-strategy.md` 中可以添加领域专项模板：

```markdown
### 领域专项模板

**支付/金额领域**：
```
额外要求：
- 覆盖金额精度问题（浮点误差、大整数）
- 覆盖重复支付、部分退款场景
- 覆盖币种/汇率转换精度
- 覆盖支付超时后状态一致性
```
```

### 7.2 项目特定缺陷模式

在项目根目录创建 `docs/bug-patterns.md`：

```markdown
# 项目特定缺陷模式

## 模块 A

| 模式 | 表现 | 测试方法 |
|------|------|----------|
| 模式1 | 描述 | 测试方法 |
| 模式2 | 描述 | 测试方法 |
```

### 7.3 产品知识库

在 `knowledge/products/` 目录下管理产品业务知识：

**新增产品知识步骤**：
1. 复制 `products-template.md` 为 `{product-id}.md`
2. 填写各章节内容（可渐进式补充，不必一次完成）
3. 在 `README.md` 的「已注册产品列表」中添加记录
4. 在「产品-模块映射表」中添加代码路径映射
5. 提交到版本库

**知识文件结构**：
- 元信息：产品ID、名称、版本、维护人
- 产品概述：业务定位、用户角色、核心流程
- 功能模块清单：模块列表与优先级
- 业务规则与约束：核心规则、校验规则、状态流转
- 接口与集成：API 列表、第三方依赖
- 测试重点领域：高频缺陷、专项检查清单、性能基线
- 测试数据与环境：账号、环境配置、造数工具
- 常见测试场景模板：可复用的场景模板
- 参考文档：PRD、API 文档、设计稿链接

**自动加载规则**：
- 阶段 1 时，根据用户输入、需求文档、代码模块自动识别产品归属
- 若匹配到产品ID，自动加载对应知识文件
- 用户也可手动指定：`测试知识库：[产品ID]` 或 `使用 [产品名称] 知识库`

### 7.4 探索式测试

使用探索式模式进行深度测试：

```
探索式模式：帮我为支付功能设计探索测试策略，重点关注：
1. 金额精度问题
2. 并发支付冲突
3. 网络异常场景
```

**输出**：
- 测试点清单
- 探索章程（时间盒、范围、关注点、退出条件）
- 探索结论和缺陷报告

## 8. 总结

test-engineer skill 提供了一套完整的测试用例生成与管理方案，通过：

1. **系统化流程**：四阶段工作流，确保质量
2. **AI 赋能**：70%-90% 用例自动生成，提升效率
3. **质量保障**：编写铁律 + 自检清单，保证质量
4. **灵活扩展**：适配器机制 + 知识库扩展，适应不同场景

掌握本指南中的使用方法和最佳实践，可以充分发挥该 Skill 的价值，提升测试效率和质量。
