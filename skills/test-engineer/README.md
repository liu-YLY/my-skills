# Test Engineer Skill

> AI 赋能的测试工程师技能，自动生成高质量测试用例

## 简介

Test Engineer 是一个 AI 驱动的测试技能，扮演资深测试工程师角色，能够：

- **需求分析**：深入理解需求文档，提取业务规则和测试点
- **测试点提取**：系统性扫描 7 个维度，确保测试覆盖无遗漏
- **用例生成**：AI 生成 70%-90% 的基础用例，人工审核补充
- **缺陷分析**：五步定位法 + 鱼骨图 + 5 Whys 根因分析

## 适用场景

| 场景 | 说明 |
|------|------|
| ✅ 编写测试用例 | 从需求文档自动生成完整测试用例 |
| ✅ 评审测试用例 | 检查用例质量和覆盖度 |
| ✅ 需求分析 | 提取业务规则和测试点 |
| ✅ 测试策略设计 | 确定测试范围和优先级 |
| ✅ Bug 根因分析 | 定位缺陷原因并给出修复建议 |

## 核心工作流

```
阶段 1: 理解需求 → 阶段 2: 提取测试点 → 阶段 3: 编写用例 → 阶段 4: 自检补全
```

### 阶段 1: 理解需求

**目标**：全面透彻地理解被测功能的业务背景、技术实现和产品现状。

**信息收集优先级**：
- P0: 扫描项目知识库、搜索代码库、阅读已有用例
- P1: 公开文档检索、阅读产品代码
- P2: 追问用户确认

**输出**：需求理解文档（结构化，包含 12 个字段）

🔴 **CHECKPOINT**：需求理解输出后，必须展示给用户确认再进入阶段 2。

### 阶段 2: 提取测试点

**目标**：从需求理解中系统性提取所有可测试点，做到不遗漏。

**7 维度扫描框架**：
1. 正向功能 - 核心业务流程的正常路径
2. 反向校验 - 每个输入字段的异常/非法/边界输入
3. 权限/角色 - 不同用户类型的不同操作
4. 状态联动 - 选项变化时的联动响应
5. 边界与限制 - 数量上限、长度限制、重复校验
6. UI 交互 - 按钮状态、提示文案、页面跳转
7. 异常与容错 - 网络异常、并发操作、服务端错误

**输出**：测试点清单（Markdown 表格）

🔴 **CHECKPOINT**：测试点清单输出后，必须展示给用户确认再进入阶段 3。

### 阶段 3: 编写用例

**目标**：基于测试点清单生成完整可落地的测试用例。

**AI 生成模式**（默认）：
1. 读取结构化提示词模板
2. 将需求理解 + 测试点清单填入 Prompt
3. AI 生成 70%-90% 的基础用例
4. 人工审核业务逻辑准确性
5. 人工补充 AI 遗漏的复杂业务逻辑用例

**用例格式**：
```markdown
### TC_MODULE_FUNC_001: 被测对象 - 测试意图

- **优先级**: P0
- **类型**: functional
- **需求追溯**: Story-42 / TP-03

**前置条件**:
- 已登录为 A 类用户

**测试步骤与预期结果**:

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 点击"提交"按钮 | 页面显示"提交成功"提示 |
```

🛑 **STOP**：AI 生成的用例必须经过人工审核才能输出。

### 阶段 4: 自检补全

**目标**：确保用例完整性和质量。

**自检清单**：
- 可追溯性：用例与测试点编号可对应
- 覆盖度：所有测试点均已有对应用例
- 优先级比例：P0 10-15%、P1 30-40%、P2 30-40%、P3 10-15%
- 质量检查：步骤无歧义、预期结果无模糊词

**输出**：最终用例 + 效率报告

🔴 **CHECKPOINT**：自检通过后，必须展示最终用例给用户确认。

## 使用方式

### 快速开始

1. **确定模式**：
   - 默认模式：完整四阶段流程
   - 快速模式：压缩阶段 1/4，必须输出测试点清单
   - 探索式模式：额外输出探索章程

2. **提供输入**：
   - 需求文档（Word/PDF/Markdown）
   - 或用户描述
   - 或代码变更

3. **执行流程**：
   - 按阶段 1→2→3→4 顺序执行
   - 每个阶段完成后等待用户确认
   - 遇到问题查阅对应知识库文件

### 示例对话

```
用户：我有一个用户登录功能需要测试，需求如下：用户通过手机号+验证码登录...

AI（test-engineer）：
## 阶段 1: 需求理解

**功能概述**：用户通过手机号+验证码登录系统

**用户角色**：
- 普通用户
- 管理员

**业务规则**：
1. 手机号格式为 11 位数字
2. 验证码为 6 位数字，有效期 5 分钟
3. 支持记住登录状态 7 天

[展示给用户确认后继续阶段 2...]
```

## 编写铁律

### title 规范
- 格式：`{被测对象} - {具体行为}`
- 不超过 40 字符
- 动宾结构

### steps 规范
- 祈使句，每步一个操作
- 含具体输入值
- **不超过 7 步**，超过必须拆分

### expected_results 规范
- 可直接判定 pass/fail
- 引用实际 UI 文案
- **禁用模糊词**（正确、正常、符合预期等）
- 数量与 steps 一致

## 缺陷分析

五步定位法：复现 → 隔离 → 定位 → 验证 → 报告

**鱼骨图分类**：
- 人：操作失误、权限配置、沟通遗漏
- 机：服务器、网络、中间件、环境配置
- 料：数据质量、依赖服务、第三方接口
- 法：流程缺陷、规范缺失、代码逻辑
- 环：环境差异、版本差异、部署时序

## 反例与黑名单

### 工作流反模式
- ❌ 跳过阶段 1 直接写用例
- ❌ 跳过阶段 2 直接写用例
- ❌ AI 生成后不审核直接输出
- ❌ 用例中使用模糊词
- ❌ steps 与 expected_results 数量不一致

### 输出格式反模式
- ❌ 缺少必填字段
- ❌ title 超过 40 字符
- ❌ steps 超过 7 步
- ❌ ID 重复或格式不规范

## 常见问题

### Q: 如何快速生成测试用例？
A: 使用快速模式，提供需求文档或描述，AI 会自动生成测试点清单和基础用例。

### Q: AI 生成的用例质量如何保证？
A: AI 生成 70%-90% 基础用例，人工审核后补充业务特有逻辑，最终输出标记来源。

### Q: 如何处理复杂业务逻辑？
A: 阶段 1 深入理解需求，阶段 2 提取所有测试点，阶段 3 AI 生成 + 人工补充。

### Q: 支持哪些输出格式？
A: 默认输出通用 Markdown 格式，可通过适配器转换为 TEST 等特定格式。
