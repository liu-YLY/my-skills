# test-engineer Skill 分析报告

> 本文档分析 test-engineer skill 的结构、能力、优化效果及适用场景。

---

## 1. 概述

test-engineer 是一个通用的 AI 赋能测试工程师技能包，专注于测试用例生成与管理。v4.2.0 版本经过多次优化，在保持核心能力的同时显著降低了 token 消耗，并新增了产品知识库功能。

### 核心价值

- **AI 赋能用例生成**：使用结构化 Prompt 生成 70%-90% 基础用例
- **系统化测试点提取**：7 维度扫描框架 + 缺陷模式对照
- **标准化输出**：统一的 YAML 格式
- **质量保障**：编写铁律 + 自检清单 + 效率度量
- **产品专项能力**：产品知识库支持，提供领域特有测试能力

## 3. 核心能力分析

### 3.1 四阶段工作流

```
阶段 1 理解需求 → 阶段 2 提取测试点 → 阶段 3 编写用例 → 阶段 4 自检补全
```

**阶段 1：理解需求**
- 信息收集优先级表（P0/P1/P2）
- 可测性与验收口径检查
- 结构化需求理解输出模板

**阶段 2：提取测试点**
- 7 维度扫描框架（正向功能/反向校验/权限角色/状态联动/边界限制/UI交互/异常容错）
- 缺陷模式对照补充（必须执行）
- 设计方法决策树（等价类/边界值/判定表/状态迁移/场景法）

**阶段 3：编写用例**
- AI 生成模式（默认）：结构化 Prompt → 人工审核 → 补充
- 通用 YAML 格式 + 字段定义
- 编写铁律（title/steps/expected_results 规范）
- 适配器转换流程

**阶段 4：自检补全**
- 覆盖度检查、优先级比例检查、质量检查
- 遗漏扫描、AI 生成质量检查
- 效率度量输出

### 3.2 7 维度扫描框架

| 维度 | 检查内容 | 最低覆盖深度 |
|------|----------|-------------|
| 正向功能 | 核心业务流程的正常路径 | 每个角色至少 1 条 Happy Path |
| 反向校验 | 每个输入字段的异常/非法/边界输入 | 每个字段至少 4 类 |
| 权限/角色 | 不同用户类型的不同内容和操作 | 每个角色 × 每个功能点 |
| 状态联动 | 选项 A 变化时选项 B 的联动响应 | 每对联动关系 × 源状态数 |
| 边界与限制 | 数量上限、长度限制、重复校验 | 上限值、上限+1、重复值、零值 |
| UI 交互 | 按钮状态、提示文案、页面跳转 | 每个按钮/状态变化点 |
| 异常与容错 | 网络异常、并发操作、服务端错误 | 网络断开、5xx、并发冲突 |

### 3.3 缺陷模式库

**5 大类通用缺陷模式**：
- 输入校验类（必填绕过、XSS、SQL注入、超长截断、编码问题）
- 状态与流程类（状态回退、重复提交、并发冲突、幽灵状态、操作中断）
- 权限与安全类（越权访问、水平/垂直越权、Token过期、敏感数据泄漏）
- 数据与接口类（空数据、分页错误、排序不稳定、时区偏移、精度丢失）
- UI 与交互类（按钮状态不同步、弹窗残留、加载态缺失、错误提示不准确）

**领域特定核心模式**：
- 支付/金额（金额精度、重复支付、部分退款）
- 搜索/筛选（特殊字符、空结果、大结果集）
- 通知/消息（重复通知、通知丢失、模板渲染）
- 文件上传（类型绕过、大文件、并发上传）

**安全专项检查清单**：10 项检查（SQL注入、XSS、越权、鉴权、明文传输、敏感数据、密码安全、验证码、CSRF、数据健壮性）

### 3.4 AI 赋能模式

**标准作业程序**：
1. AI 生成：使用结构化 Prompt 生成 70%-90% 基础用例
2. 人工审核：检查业务逻辑、补充领域边界、验证优先级
3. 轻量维护：模块级批量维护，变更时优先重新生成

**Prompt 策略**：
- 基础模板：需求理解 + 测试点清单 → 生成用例
- 增强模板：追加历史缺陷模式 → 生成防御性用例
- 输入优化：结构化需求、测试点带编号、Few-shot 参考
- 输出控制：格式约束、追溯标注、分批生成、优先级标注

### 3.5 产品知识库

**知识沉淀机制**：
- 按产品维度组织业务知识（knowledge/products/ 目录）
- 标准化模板：产品概述、业务规则、接口清单、测试重点领域
- 自动识别：根据用户输入、需求文档、代码模块匹配产品归属

**核心价值**：
- 阶段 1：使用产品概述和业务规则辅助需求理解
- 阶段 2：使用高频缺陷模式和专项检查清单补充防御性测试点
- 阶段 3：使用常见测试场景模板加速用例编写
- 阶段 4：对照专项检查清单验证覆盖度

**维护方式**：
- 新增产品：复制模板 → 填写内容 → 注册索引 → 提交版本库
- 渐进式补充：可先填写核心章节，逐步完善
- 版本管理：变更记录追踪，支持多人协作

### 3.6 缺陷分析与定位

**五步定位法**：复现 → 隔离 → 定位 → 验证 → 报告

**分析工具**：
- 鱼骨图分类维度（人/机/料/法/环）
- 5 Whys 根因分析法

**报告模板**：
```markdown
**现象** / **环境** / **版本/构建号** / **复现率** / **附件/日志**
**根因**：[直接原因] / [深层原因]
**影响范围** / **修复建议**（紧急 + 长期） / **回归测试**
```

### 4.2 各场景 Token 消耗

| 场景 | 需加载文件 | 预估 Token |
|------|-----------|-----------|
| 完整四阶段 | SKILL.md + core + 全部 knowledge | ~8,000-10,000 |
| 快速模式 | SKILL.md + core + test-standards | ~5,000-6,000 |
| Bug 分析 | SKILL.md + bug-patterns | ~3,000-4,000 |
| 仅阶段 1-2 | SKILL.md + core + test-levels + bug-patterns | ~5,000-6,000 |
| 仅阶段 3-4 | SKILL.md + core + test-standards + prompt-strategy | ~6,000-7,000 |

### 4.3 优化策略

**合并策略**：
- 将 4 个 stage 文件合并为 1 个 core 文件
- 内联关键规则，减少文件间跳转

**精简策略**：
- 删除详细示例和正反例对比
- 删除重复的规则说明
- 删除场景模式速查（AI 已掌握）

**移除策略**：
- 删除 Python 调试相关内容（debugging.md、pep8.md）→ 按需时让 AI 自行搜索
- 删除效率度量独立文件 → 合并精简到 core
- 删除元分析文档（context-analysis.md）→ 不参与工作流

## 5. 质量保障机制

### 5.1 编写铁律

**title 规范**：
- 不超过 40 字符
- 动宾结构：`{被测对象} - {具体行为}`
- 同一模块内唯一

**steps 规范**：
- 祈使句，每步一个操作
- 包含具体输入值
- 建议 7 步以内
- 一个用例只覆盖单一测试逻辑

**expected_results 规范**：
- 可直接判定 pass/fail
- 引用实际文案
- 禁用模糊词（50+ 个禁用词）
- 与 steps 一一对应

### 5.2 自检清单

**覆盖度检查**：
- 所有测试点均有对应用例
- 正向 Happy Path 全覆盖
- 每个输入字段的反向校验全覆盖
- 角色/权限矩阵全覆盖
- 联动关系全覆盖
- 边界条件全覆盖

**优先级比例检查**：
- P0：10%-15%
- P1：30%-40%
- P2：30%-40%
- P3：10%-15%

**质量检查**：
- 步骤无歧义
- 包含具体输入数据和预期输出
- expected_results 无模糊描述
- steps 与 expected_results 数量一致
- ID 无重复

### 5.3 效率度量

**核心指标**：
- AI 生成占比（目标 >= 70%）
- 工时节省率（目标 >= 50%）
- 效率提升倍数（目标 >= 2x）
- 人工补充率（目标 <= 30%）
- 审核修改率（目标 <= 20%）

## 6. 适用场景

### 6.1 最佳适用场景

- **新功能测试用例编写**：系统化提取测试点，AI 生成基础用例
- **回归测试用例补充**：基于变更分析，补充遗漏场景
- **测试用例评审**：对照自检清单，检查覆盖度和质量
- **Bug 根因分析**：使用鱼骨图和 5 Whys 定位问题
- **测试策略设计**：基于风险评估，制定分层测试策略

### 6.2 可简化场景

- **纯重构无行为变更**：回归验证清单即可
- **快速测试点提取**：切快速模式，压缩阶段 1/4
- **探索式测试**：额外输出探索章程

### 6.3 不适用场景

- **单元测试代码编写**：本 Skill 专注于功能验收层面
- **自动化测试脚本生成**：本 Skill 输出测试用例，非自动化脚本
- **性能测试执行**：本 Skill 仅生成性能测试用例，不执行测试

## 7. 扩展性设计

### 7.1 产品知识库机制

- 支持按产品维度沉淀业务知识（knowledge/products/ 目录下）
- 自动识别产品归属：根据用户输入、需求文档、代码模块匹配产品ID
- 阶段 1 加载产品概述和业务规则，辅助需求理解
- 阶段 2 使用高频缺陷模式和专项检查清单，补充防御性测试点
- 阶段 3 使用常见测试场景模板，加速用例编写
- 新增产品知识：复制模板 → 填写内容 → 注册索引 → 提交版本库

### 7.2 知识库扩展

- 项目特定缺陷模式：项目根目录 `docs/bug-patterns.md`
- JIRA 历史缺陷查询：按模块/标签查询近期 Bug
- 领域专项模板：可按需添加新领域模板
- 产品知识文件：knowledge/products/{product-id}.md

### 7.3 模式切换

- 默认模式：完整四阶段
- 快速模式：压缩阶段 1/4
- 探索式模式：额外输出探索章程

## 8. 优化建议

### 8.1 进一步优化方向

1. **知识库动态加载**：根据场景自动判断需要加载哪些 knowledge 文件
2. **Prompt 模板缓存**：将常用 Prompt 模板缓存，减少重复生成
3. **用例模板库**：建立常见场景的用例模板库，提高生成效率
4. **集成测试管理工具**：与 TestRail、Xray 等工具集成

### 8.2 质量提升建议

1. **建立用例评审机制**：定期评审 AI 生成用例的质量
2. **积累领域知识**：将项目特定的业务规则沉淀到知识库
3. **优化 Prompt 策略**：根据实际使用效果迭代优化 Prompt
4. **建立反馈循环**：收集用例执行结果，反馈到缺陷模式库

## 9. 总结

test-engineer v3.0.0 通过精简优化，在保持核心能力的同时实现了：

- **Token 节省 45-55%**：从 ~17,000 降到 ~8,000-10,000
- **文件数减少 55%**：从 20 个降到 9 个
- **行数减少 73%**：从 ~3,210 行降到 ~850 行
- **核心能力完整保留**：四阶段流程、7 维度扫描、缺陷模式库、AI 赋能模式

该 Skill 适用于测试团队的日常测试用例生成与管理，能够显著提升测试效率，同时保证用例质量。
