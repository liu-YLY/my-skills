# 历史缺陷模式与经验沉淀

> **何时阅读**:阶段 2 测试点扫描完 7 维度后**必读**(SKILL.md 强制要求);Bug 根因分析时按需查阅。
> **覆盖范围**:5 类缺陷模式(输入校验/状态流程/权限安全/数据接口/UI 交互)+「功能特征→缺陷模式」自动映射表 + 项目特定模式管理建议(JIRA / 项目级 docs/bug-patterns.md)。
> **快速定位**:
> - 用例补漏 → 直跳「缺陷模式 → 测试点映射」表对照功能特征
> - 学习模式分类 → 阅读上方 5 个分类表
> **可跳过条件**:阶段 2 之外的场景且无 Bug 分析需求。

## 缺陷模式库

根据高频出现的缺陷类型分类。设计用例时，主动对照此库检查是否已覆盖对应风险。

### 输入校验类

| 模式 | 表现 | 测试方法 | 典型用例 |
|------|------|----------|----------|
| 必填绕过 | 前端校验但后端未校验，可通过 API 直接绕过 | 同时测前端 + 直接调 API | 前端置灰但 curl 能提交成功 |
| XSS 注入 | 输入 `<script>` 标签未转义 | 输入框注入 HTML/JS | 在名称字段输入 `<img onerror=alert(1)>` |
| SQL 注入 | 输入拼接到查询中 | 输入 `' OR 1=1 --` | 搜索框、筛选条件 |
| 超长截断 | 前端限制长度但后端未限制，或截断后语义变化 | 输入超过限制的文本 | 100 字符限制输入 200 字符 |
| 编码问题 | 中文/emoji/特殊字符导致乱码或报错 | 输入多种字符集 | 用户名包含 emoji 或日文 |

### 状态与流程类

| 模式 | 表现 | 测试方法 |
|------|------|----------|
| 状态回退 | 已到终态的资源被非法操作回退 | 对终态资源执行中间操作 |
| 重复提交 | 快速双击导致重复创建 | 快速连续点击提交按钮 |
| 并发冲突 | 两人同时编辑同一资源 | 双浏览器同时修改并保存 |
| 幽灵状态 | 页面显示状态与后端不一致 | 操作后刷新页面对比 |
| 操作中断 | 操作过程中网络断开或页面刷新 | 提交过程中断网或 F5 |

### 权限与安全类

| 模式 | 表现 | 测试方法 |
|------|------|----------|
| 越权访问 | 篡改 URL 中的 ID 可访问他人资源 | 替换 URL 中的资源 ID |
| 水平越权 | 同级别用户可操作他人数据 | A 用户操作 B 用户的数据 |
| 垂直越权 | 低权限用户通过 API 调用高权限接口 | 普通用户直接调管理员 API |
| Token 过期 | 长时间未操作后执行敏感操作 | 等待 token 过期后操作 |
| 敏感数据泄漏 | API 返回了不该返回的字段 | 检查接口返回是否包含密码/密钥 |

### 数据与接口类

| 模式 | 表现 | 测试方法 |
|------|------|----------|
| 空数据 | 列表为空时页面异常或报错 | 新账号无数据时访问列表 |
| 分页错误 | 末页/跨页操作后数据丢失或重复 | 删除当前页最后一条后的表现 |
| 排序不稳定 | 相同排序值的数据顺序不固定 | 多次刷新检查顺序一致性 |
| 时区偏移 | 日期时间显示与用户时区不匹配 | 切换浏览器时区后检查 |
| 精度丢失 | 浮点数计算或大整数传递精度丢失 | 涉及金额/ID 时用边界值检查 |

### UI 与交互类

| 模式 | 表现 | 测试方法 |
|------|------|----------|
| 按钮状态不同步 | 条件满足后按钮仍置灰 | 逐步填写必填项观察按钮变化 |
| 弹窗残留 | 关闭弹窗后再次打开仍保留上次数据 | 关闭后重新打开检查表单 |
| 加载态缺失 | 操作后无 loading，用户以为没响应 | 在慢网络下操作观察反馈 |
| 错误提示不准确 | 提示信息与实际错误不匹配 | 触发各类错误检查文案 |
| 键盘操作 | Tab 顺序混乱或无法通过键盘操作 | 纯键盘完成表单 |

## 缺陷模式 → 测试点映射

设计用例时，按以下规则自动从缺陷模式库中补充测试点：

| 功能特征 | 应检查的缺陷模式 |
|----------|------------------|
| 有输入框 | 必填绕过、超长截断、编码问题、XSS（如果面向公网） |
| 有提交按钮 | 重复提交、操作中断 |
| 有列表页 | 空数据、分页错误、排序不稳定 |
| 涉及权限 | 越权访问、水平/垂直越权 |
| 有状态流转 | 状态回退、幽灵状态 |
| 有日期时间 | 时区偏移 |
| 有弹窗/抽屉 | 弹窗残留、关闭行为 |
| 涉及金额/ID | 精度丢失 |

## 项目特定缺陷模式（与本 Skill 分离）

> 本文档中的模式为**跨项目通用抽象**，保持精简和通用性。项目特定的缺陷模式（如某模块特殊业务逻辑导致的 Bug）不在此处存放，而是通过以下方式获取：

### 方案一：JIRA 查询（推荐）

阶段 2 提取测试点时，若环境支持联网且用户授权，可**按模块/标签查询 JIRA 历史缺陷**，从中提取项目特定的高频模式作为补充测试点。示例查询思路：

- 模块 = "定价" 且 状态 = 已关闭
- 根因分类 = "边界条件" 且 发现时间近 6 个月
- 严重程度 = Critical/Major

查询到的 Bug 列表直接作为阶段 2 测试点的「来源」引用，无需在本文件重复存储。

### 方案二：项目本地文件（备选）

若团队希望离线沉淀，在项目根目录维护 `docs/bug-patterns.md`（而非本 skill 内部），格式与本文件相同。阶段 2 时若该文件存在则自动读取。

### Agent 行为规则

1. 阶段 2 提取测试点时，本文件中的通用映射表**始终对照**
2. 若识别到未被通用库覆盖的新模式：**不追加到本文件**，而是提示用户考虑记录到 JIRA 或项目级 `docs/bug-patterns.md`
3. Bug 定位/分析流程结束后，若有值得沉淀的模式，询问用户是否需要在 JIRA 上创建对应的"测试设计建议"标签或更新项目知识文件
