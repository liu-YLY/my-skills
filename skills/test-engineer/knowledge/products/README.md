# 产品知识库索引

> **何时阅读**：阶段 1 信息收集时，根据被测功能所属产品加载对应知识文件。

---

## 使用方式

### 自动加载规则

当用户的测试任务涉及特定产品时，按以下规则加载知识：

1. **识别产品**：从用户输入、需求文档、代码模块中识别所属产品
2. **查找知识文件**：在 `knowledge/products/` 目录下查找 `{product-id}.md`
3. **加载并应用**：读取产品知识，用于阶段 1 需求理解和阶段 2 测试点提取

### 手动指定

用户可通过以下方式指定产品知识：
- `测试知识库：[产品ID]`
- `使用 [产品名称] 知识库`
- `基于 [产品名称] 业务知识生成用例`

## 知识文件结构规范

每个产品知识文件应包含以下章节（详见 [products-template.md](products-template.md)）：

1. **元信息**：产品ID、名称、版本、维护人
2. **产品概述**：业务定位、用户角色、核心流程
3. **功能模块清单**：模块列表与优先级
4. **业务规则与约束**：核心规则、校验规则、状态流转
5. **接口与集成**：API 列表、第三方依赖
6. **测试重点领域**：高频缺陷、专项检查清单、性能基线
7. **测试数据与环境**：账号、环境配置、造数工具
8. **常见测试场景模板**：可复用的场景模板
9. **参考文档**：PRD、API 文档、设计稿链接

---

## 新增产品知识步骤

1. 复制 `products-template.md` 为 `{product-id}.md`
2. 填写各章节内容（可渐进式补充，不必一次完成）
3. 在本文件「已注册产品列表」中添加记录
4. 在「产品-模块映射表」中添加代码路径映射
5. 提交到版本库

---

## 与核心流程的集成

### 阶段 1（理解需求）

- 若识别到产品归属，自动加载对应知识文件
- 使用「产品概述」和「业务规则」辅助需求理解
- 使用「核心业务流程」验证需求描述的完整性

### 阶段 2（提取测试点）

- 使用「高频缺陷模式」补充防御性测试点
- 使用「专项测试检查清单」确保重点领域覆盖
- 使用「状态流转规则」设计状态相关测试点

### 阶段 3（编写用例）

- 使用「常见测试场景模板」加速用例编写
- 使用「测试数据与环境」信息完善前置条件
- 使用「业务规则」确保用例逻辑准确

### 阶段 4（自检补全）

- 对照「专项测试检查清单」验证覆盖度
- 检查「高频缺陷模式」是否已覆盖
