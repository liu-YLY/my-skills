# 测试分层与交付形态

> **何时阅读**:阶段 2/3 必读(SKILL.md 强制要求)。
> **覆盖范围**:E2E / API / 契约 / 集成 / 单元 五层各自测什么、YAML 形态差异、分层选择决策。
> **可跳过条件**:本次明确是「纯 UI 验收」且不涉及接口/契约,可只读「本 Skill 的默认重心」一节。

与 [test-standards.md](test-standards.md) 中的优先级、用例类型配合使用。本文说明**不同测试分层各自验什么**、**本 Skill 默认落点**,以及在阶段 2(测试点)、阶段 3(用例)中如何写才**可执行、不混层**。

## 为什么要分层

同一业务需求往往可同时用 **UI 端到端**、**HTTP/API**、**集成/契约**、**单元** 等多层验证；分层清晰能避免：用大量 UI 步骤掩盖接口缺陷，或用单测冒充验收。**本 Skill 不替代开发侧单测与自动化框架选型**，但应写出**与分层一致的测试点与步骤**。

## 常见分层（简图）

```text
        /\
       /  \     E2E / 验收（少而精，用户旅程）
      /----\
     /      \   API / 集成 / 契约（服务行为、错误码、兼容）
    /--------\
   / 单元/组件  \（开发主导，函数/类/模块边界）
  /--------------\
```

| 分层 | 典型目标 | 谁主导 | 本 Skill 常见产出 |
|------|----------|--------|-------------------|
| **单元 / 组件** | 函数/类边界、分支、异常 | 开发 | **测点提示**（输入边界、异常路径、mock 假设），不代替单测代码 |
| **集成** | 多模块/DB/消息队列联调 | 开发/测试 | 关键路径的**接口级或环境级**步骤与数据前提 |
| **API / 服务** | HTTP/RPC 行为与契约 | 测试（或开发） | **请求/响应/状态码/错误码** 可断言的用例 |
| **契约 / 消费者-提供者** | Schema 与版本兼容 | 视团队 | **字段、版本号、破坏性变更** 相关测试点 |
| **E2E / 功能验收** | 完整用户旅程、多端 | 测试 | **默认可执行场景用例**（本 Skill 默认重心） |

## 本 Skill 的默认重心

- **默认**：产出**可执行的场景级用例**，偏 **E2E / 功能验收**（含必要 UI 步骤），与 [SKILL.md](../SKILL.md) 中 YAML 示例一致。
- 若用户或需求明确偏后端/接口，则**重心上移到 API**，避免用纯 UI 步骤代替服务侧验证。

## 按分层：测试点应覆盖什么

### API / 后端服务（无 UI 或 UI 非重点）

在测试点中显式包含（按项目裁剪）：

- **HTTP**：方法、路径、Query/Body、Header（含鉴权）
- **响应**：HTTP 状态码、业务错误码、body 关键字段、空/异常 body
- **行为**：幂等键、分页（cursor/limit）、限流、超时、重试语义
- **安全**：鉴权失败、越权、token 过期（与 [bug-patterns.md](bug-patterns.md) 呼应）

### 契约 / 微服务

- 请求/响应 **Schema**（或 OpenAPI 版本）
- **版本兼容**：新增字段、废弃字段、破坏性变更
- 消费者与提供者的**约定**（错误模型、分页格式等）

### E2E / 功能

- 用户可见步骤、数据准备、跨页面/跨系统旅程
- 与 API 用例**可并存**：同一需求下分两组用例，避免单条用例里 UI 与接口断言混杂导致不可执行

### 单元 / 组件（本 Skill 边界）

- 输出 **给开发的测点列表** 即可（边界值、分支、异常、并发假设）
- **不要求**输出具体测试框架代码（pytest 类名等），除非用户明确要求

## 按分层：YAML 用例形态（与 SKILL 阶段 3 一致）

| 分层 | preconditions 侧重 | steps 侧重 | expected_results 侧重 |
|------|-------------------|------------|------------------------|
| **API** | base URL、环境、鉴权方式、测试数据 | 构造请求 → 调用 →（可选）轮询 | 状态码、body 字段、错误码与文案 |
| **契约** | schema 版本、兼容策略说明 | 覆盖兼容/不兼容示例 | 序列化结果、校验错误信息 |
| **E2E** | 账号、角色、页面入口、数据状态 | 用户操作链 | UI 文案、跳转、后端可核对状态（若需要） |

**禁止**：在一条「接口用例」里只写「打开页面点击」而无请求/断言；在「E2E 用例」里堆砌大量底层字段而又无操作上下文——应拆条或分组。

## 如何选择主分层（简决策）

1. 需求文档**仅描述接口/无界面** → 以 **API（+契约按需）** 为主。
2. 需求**强依赖浏览器/多端流程** → 以 **E2E** 为主，API 作为辅助或回归子集。
3. **重构无行为变更** → 回归清单即可，不必强行补全所有分层（见 SKILL「适用范围」）。
4. **不确定** → 在「存疑点」中列出，并在测试点表中标注**建议分层**列（可选）。

## 与可测性、可观测性的关系

- 分层选 API/E2E 后，**验收证据**仍须与 SKILL 阶段 1「可测性与验收口径」一致（trace、日志、对账等）。
- **可观测性**类测试点（指标、告警）可落在集成/API 层或单独非功能用例组，见 SKILL 阶段 2「可观测性」说明。
