# 测试分层与交付形态

> **何时阅读**:阶段 2/3 必读。
> **覆盖范围**:E2E / API / 契约 / 集成 / 单元 五层各自测什么、本 Skill 默认重心、禁止混层规则。

## 常见分层（简图）

```text
        /\
       /  \     E2E / 验收（少而精，用户旅程）
      /----\
     /      \   API / 集成 / 契约（服务行为、错误码、兼容）
    /--------\
   / 单元/组件  \（开发主导，函数/类/模块边界）
  /--------------\
```

| 分层 | 典型目标 | 谁主导 | 本 Skill 常见产出 |
|------|----------|--------|-------------------|
| **单元 / 组件** | 函数/类边界、分支、异常 | 开发 | **测点提示**（输入边界、异常路径） |
| **集成** | 多模块/DB/消息队列联调 | 开发/测试 | 关键路径的**接口级或环境级**步骤 |
| **API / 服务** | HTTP/RPC 行为与契约 | 测试（或开发） | **请求/响应/状态码/错误码** 可断言的用例 |
| **契约 / 消费者-提供者** | Schema 与版本兼容 | 视团队 | **字段、版本号、破坏性变更** 相关测试点 |
| **E2E / 功能验收** | 完整用户旅程、多端 | 测试 | **默认可执行场景用例**（本 Skill 默认重心） |

## 本 Skill 的默认重心

- **默认**：产出**可执行的场景级用例**，重心为 **E2E / 功能验收**（含必要 UI 步骤）
- 若用户或需求明确后端/接口，则**重心必须上移到 API**，禁止用纯 UI 步骤代替服务侧验证

## 禁止混层

- **禁止**：在一条「接口用例」里只写「打开页面点击」而无请求/断言
- **禁止**：在「E2E 用例」里堆砌大量底层字段而又无操作上下文
- 应拆条或分组，保持每条用例的分层一致性

## 分层选择简决策

1. 需求文档**仅描述接口/无界面** → 以 **API（+契约按需）** 为主
2. 需求**强依赖浏览器/多端流程** → 以 **E2E** 为主，API 作为辅助或回归子集
3. **重构无行为变更** → 回归清单即可，不必强行补全所有分层
4. **不确定** → 必须在「存疑点」中列出，并在测试点表中标注分层列
