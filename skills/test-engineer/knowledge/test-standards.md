# 团队测试规范与标准

> **何时阅读**:阶段 3 编写用例时 + 阶段 4 自检时必读。
> **覆盖范围**:优先级三步法与比例 / 字段编写核心规则 / 模糊词禁用清单。
> **快速定位**:
> - 划分 P0-P3 → 直跳「优先级划分方法(三步法)」
> - 写 title/steps/preconditions/expected_results → 直跳「字段编写标准」
> - 检查模糊词 → 直跳「模糊词禁用清单」

## 命名规范

| 元素 | 规则 | 示例 |
|------|------|------|
| 用例 ID | `TC_{模块}_{功能}_{三位序号}` | TC_WEBHOOK_ADD_001 |
| 性能用例 ID | `TC_{模块}_PERF_{三位序号}` | TC_WEBHOOK_PERF_001 |
| 安全用例 ID | `TC_{模块}_SEC_{三位序号}` | TC_AUTH_SEC_001 |
| title | `{被测对象} - {具体行为}`，**不超过 40 字符** | Endpoint URL 校验 - 为空时提示错误 |

**title 命名约束**：
- 采用动宾结构，提炼关键前置条件和观察点
- 在同一功能模块内唯一，不允许仅靠编号区分
- 避免特殊字符，可用 `-` 或 `_` 分割

## 优先级定义

| 等级 | 定义 | 回归频率 |
|------|------|----------|
| P0 | 冒烟测试：核心流程，fail 则阻塞大部分测试 | 每次构建必回归 |
| P1 | 高优先级：基本功能验证 + 重要的错误和边界测试 | 每个迭代回归 |
| P2 | 中优先级：更全面的功能验证、异常/边界/中断/UI 测试 | 版本发布前回归 |
| P3 | 低优先级：性能/压力/兼容/稳定性/安全/可用性等 | 有空余时间回归 |

> 若项目适配器不支持 P3（如 TEST），输出时由适配器自动降级为 P2。

### 优先级比例指导

一个功能模块的用例优先级分布应大致满足：

| P0 | P1 | P2 | P3 |
|----|----|----|----|
| 10%~15% | 30%~40% | 30%~40% | 10%~15% |

如果严重偏离此比例（如 P0 超过 30%），说明优先级划分粒度有问题，需要重新评估。

### 优先级划分方法（三步法）

**第一步：初定**
1. 所有功能性验证（正向路径）→ 标记为 P1
2. 所有错误/边界/UI 测试 → 标记为 P2
3. 所有非功能性测试（性能/安全/兼容）→ 标记为 P3

**第二步：升降级**
1. P1 中"不十分重要"的功能验证 → 降为 P2
2. P2 中"重要"的错误和边界测试 → 升为 P1
3. P3 中"重要"的非功能性测试 → 升为 P2

> "重要"的判断依据：Bug 高发区域、用户使用频率高、涉及核心数据/资金

**第三步：挑选冒烟用例（P0）**
- 从 P1 中挑出"最关键"的用例 → 升为 P0
- P0 用例 fail 会阻塞其他测试的执行

## 字段编写标准

以下规则参考华为 CodeArts TestPlan 文字表达 7 大规则：

**总则**：风格一致、用词一致、精准、简洁、易懂、易确认、去重。

### preconditions（前置条件）

- 必须包含：用户角色/权限、所在页面、必要的数据状态
- 格式：陈述句，明确到可直接准备环境

| 正例 | 反例 | 问题 |
|------|------|------|
| 通讯录中已添加联系人"张三" | 无前置条件 | 后续步骤依赖数据但未说明 |
| Python 版本 >= 3.4 | 相关软件已正常安装 | 未明确版本，可能不兼容 |

### steps（操作步骤）

- 每步一个操作，使用祈使句
- 包含具体的操作对象（按钮名、输入框名）和输入值
- **步骤总数不超过 7 步**，超过 7 步必须拆分为多条用例
- **一个用例只覆盖单一测试逻辑**
- 参数要显式说明具体值，禁止只写"边界值""非法值"
- 每一步操作都应产生可观察的结果（纯预备动作放入 preconditions）

| 正例 | 反例 | 问题 |
|------|------|------|
| 在 "Endpoint URL" 输入框中输入 `https://api.example.com/webhook` | 输入合法 URL | 未说明具体输入值 |
| 点击 "Add endpoint" 按钮 | 通过左上角的按钮提交 | 用位置描述代替元素名称 |

### expected_results（预期结果）

- 必须具体、可验证，可直接用于判定 pass/fail
- 引用实际 UI 文案（如 `"Wrong URL format!"`）
- **数量与 steps 一致**：`expected_results[N]` 对应 `steps[N]` 执行后的可观察结果（一步一验）

| 正例 | 反例 | 问题 |
|------|------|------|
| 输入框下方显示红色错误提示："Wrong URL format!" | 提示信息正确 | "正确"无法判断 |
| 检查 XXX 消息中参数 YYY 的取值为 ZZZ | 检查消息流程正确 | 未说明如何判断"正确" |

## 模糊词禁用清单

以下词汇**禁止**出现在 steps 和 expected_results 中：

> 很多、一些、部分、大量、少量、多次、随机、任意、一段时间、一会、一点、特殊、很长、较长、很短、较短、多个、几块、多条、数次、所有、大概、频繁、大约、某些、正常、最大、最小、超长、超小、超短、边界值、过低、过高、全部、缺省、默认、各模块、可能、有的、一定范围、适量、多于、少于、左右、上下、设备无异常、计数正常、长时间、错误信息、非法字符

遇到需要使用这些词时，**必须替换为具体数值或明确描述**：
- "多次" → "连续操作 5 次"
- "超长" → "输入 101 个字符（上限为 100）"
- "正常" → "页面显示 Webhook 列表，包含刚创建的记录"
- "默认" → "Version 下拉框选中 'Version 5.0'"

## 用例类型定义

| type | 适用场景 | 何时追加 |
|------|----------|----------|
| functional | 功能正确性验证（默认） | 始终输出 |
| performance | 响应时间、并发、吞吐 | 需求明确性能指标 或 用户要求 |
| security | 认证、授权、数据保护、注入 | 涉及权限/敏感数据 或 用户要求 |
| compatibility | 多浏览器、多设备、多版本 | 需求提到兼容要求 或 用户要求 |
| usability | 交互体验、无障碍 | 有设计稿/交互规范 或 用户要求 |
| accessibility | 无障碍专用 | 需求有无障碍规范 |
| observability | 日志、指标、告警、链路 | 需求要求可观测验收 |
| ui | UI 布局、视觉、渲染 | 有设计稿或前端改动 |

> 若项目适配器不支持某些 type，输出时按适配器规则映射。

## 测试策略标准

### 新功能测试覆盖基线

每个新功能至少覆盖：
1. **Happy Path**：至少 1 条完整正向流程
2. **字段校验**：每个输入字段的有效/无效/边界/空值
3. **权限差异**：所有涉及的用户角色
4. **联动逻辑**：所有条件联动关系
5. **辅助操作**：取消/返回/刷新的行为

### 回归测试选择标准

| 变更类型 | 回归范围 |
|----------|----------|
| Bug 修复 | 修复场景 + 关联功能的 P0 用例 |
| 小功能迭代 | 变更功能全量 + 上下游 P0 用例 |
| 大版本发布 | P0 全量 + P1 抽样 |
| 数据库/配置变更 | 受影响模块的 P0+P1 |

## 用例生命周期管理

| 状态 | 含义 | 触发事件 |
|------|------|----------|
| Draft | 新建待评审 | - |
| AI-Generated | AI 生成待审核 | AI 生成完成 |
| Active | 评审通过，可执行 | 人工审核通过 |
| Blocked | 因环境/依赖无法执行 | 执行中发现不合理 |
| Obsolete | 需求变更导致失效 | 功能下线 / 发现新缺陷 |

**状态流转**：
- AI 生成模式：Draft → AI-Generated → Active（审核通过后）→ Obsolete
- 人工编写模式：Draft → Active（评审通过后）→ Obsolete

### 轻量维护原则

| 原则 | 做法 | 适用范围 |
|------|------|----------|
| 模块级维护 | AI 生成的用例以"功能模块"为单位批量维护 | AI 生成用例 |
| 变更优先重生成 | 需求变更时，优先让 AI 重新生成受影响模块的用例 | AI 生成用例 |
| 非核心自动归档 | P2/P3 用例允许"过期自动归档" | 所有用例 |
| 核心同步更新 | P0/P1 用例必须随需求同步更新 | 所有用例 |
