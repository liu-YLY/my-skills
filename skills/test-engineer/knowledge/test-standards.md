# 团队测试规范与标准

> **何时阅读**:阶段 3 编写用例时 + 阶段 4 自检时必读。
> **覆盖范围**:8 大设计原则 / ID 与 title 命名 / 优先级三步法与比例 / 字段编写 7 大规则 / 模糊词禁用清单 / 安全检查清单 / 用例生命周期。
> **快速定位**:
> - 写 title/steps/preconditions/expected_results → 直跳「字段编写标准」
> - 划分 P0-P3 → 直跳「优先级划分方法(三步法)」
> - 检查模糊词 → 直跳「模糊词禁用清单」
> - 涉及安全 → 直跳「安全测试检查清单」

## 用例设计原则

编写和评审用例时，始终对照以下 8 项原则检查：

| 原则 | 要求 | 反面示例 |
|------|------|----------|
| **有效性** | 步骤清晰无歧义，按流程顺序编排 | 步骤含"如果...则..."的分支逻辑 |
| **清晰性** | 包含明确的输入数据和预期输出，前置条件描述完整 | "输入合法数据"未说明具体值 |
| **可复用性** | 相似场景抽象归类，数据与逻辑分离 | 每个版本重写一套几乎相同的用例 |
| **可维护性** | 需求变更时及时更新，保持与产品行为一致 | 用例描述的行为与实际产品不符 |
| **完整性** | 覆盖所有需求点，不遗漏 | 只测了正向路径，忽略异常和边界 |
| **兼容性** | 包含新老版本/数据兼容、浏览器兼容测试点 | 仅在 Chrome 上验证 |
| **易操作性** | 任何测试人员都能独立执行，不依赖口头说明 | 步骤前提是"找开发确认配置" |
| **可评估性** | 用例通过率和缺陷数可用于质量评估 | 用例粒度不一，无法统计有效通过率 |

## 用例编写规范

### 命名规范

| 元素 | 规则 | 示例 |
|------|------|------|
| 用例 ID | `TC_{模块}_{功能}_{三位序号}` | TC_WEBHOOK_ADD_001 |
| 性能用例 ID | `TC_{模块}_PERF_{三位序号}` | TC_WEBHOOK_PERF_001 |
| 安全用例 ID | `TC_{模块}_SEC_{三位序号}` | TC_AUTH_SEC_001 |
| title | `{被测对象} - {具体行为}`，**不超过 40 字符** | Endpoint URL 校验 - 为空时提示错误 |

**title 命名约束**（参考华为 CodeArts TestPlan 规范）：
- 采用动宾结构，提炼关键前置条件和观察点
- 在同一功能模块内唯一，不允许仅靠编号区分
- 避免特殊字符，可用 `-` 或 `_` 分割

| 正例 | 反例 | 问题 |
|------|------|------|
| Endpoint URL 校验 - 为空时提示错误 | 测试 URL 输入 | 缺少具体行为描述 |
| Version 选择 - A类用户可选 V2~V5 | 验证版本功能 | 过于笼统 |
| 查询流量策略 - 非法参数特殊字符 | 使用包含特殊字符构造的非法参数来查询流量策略 | 过长，未提炼关键词 |

### 优先级定义

| 等级 | 定义 | 回归频率 |
|------|------|----------|
| P0 | 冒烟测试：核心流程，fail 则阻塞大部分测试 | 每次构建必回归 |
| P1 | 高优先级：基本功能验证 + 重要的错误和边界测试 | 每个迭代回归 |
| P2 | 中优先级：更全面的功能验证、异常/边界/中断/UI 测试 | 版本发布前回归 |
| P3 | 低优先级：性能/压力/兼容/稳定性/安全/可用性等 | 有空余时间回归 |

> 若项目适配器不支持 P3（如 TEST），输出时由适配器自动降级为 P2。

### 优先级比例指导

一个功能模块的用例优先级分布应大致满足：

| P0 | P1 | P2 | P3 |
|----|----|----|----|
| 10%~15% | 30%~40% | 30%~40% | 10%~15% |

如果严重偏离此比例（如 P0 超过 30%），说明优先级划分粒度有问题，需要重新评估。

### 优先级划分方法（三步法）

**第一步：初定**
1. 所有功能性验证（正向路径）→ 标记为 P1
2. 所有错误/边界/UI 测试 → 标记为 P2
3. 所有非功能性测试（性能/安全/兼容）→ 标记为 P3

**第二步：升降级**
1. P1 中"不十分重要"的功能验证 → 降为 P2
2. P2 中"重要"的错误和边界测试 → 升为 P1
3. P3 中"重要"的非功能性测试 → 升为 P2

> "重要"的判断依据：Bug 高发区域、用户使用频率高、涉及核心数据/资金

**第三步：挑选冒烟用例（P0）**
- 从 P1 中挑出"最关键"的用例 → 升为 P0
- P0 用例 fail 会阻塞其他测试的执行

### 用例类型定义

| type | 适用场景 | 何时追加 |
|------|----------|----------|
| functional | 功能正确性验证（默认） | 始终输出 |
| performance | 响应时间、并发、吞吐 | 需求明确性能指标 或 用户要求 |
| security | 认证、授权、数据保护、注入 | 涉及权限/敏感数据 或 用户要求 |
| compatibility | 多浏览器、多设备、多版本 | 需求提到兼容要求 或 用户要求 |
| usability | 交互体验、无障碍 | 有设计稿/交互规范 或 用户要求 |
| accessibility | 无障碍专用 | 需求有无障碍规范 |
| observability | 日志、指标、告警、链路 | 需求要求可观测验收 |
| ui | UI 布局、视觉、渲染 | 有设计稿或前端改动 |

> 若项目适配器不支持某些 type（如 TEST 无 `compatibility`/`usability`/`observability`），输出时按适配器规则映射。

### 字段编写标准

以下规则参考[华为 CodeArts TestPlan 文字表达 7 大规则](https://support.huaweicloud.com/usermanual-testman/cloudtest_01_1307.html)：

**总则**：风格一致、用词一致、精准、简洁、易懂、易确认、去重。

**preconditions（前置条件）**：
- 必须包含：用户角色/权限、所在页面、必要的数据状态
- 详细简洁地说明执行测试前的准备事项，不使用无用、抽象的条件
- 格式：陈述句，明确到可直接准备环境

| 正例 | 反例 | 问题 |
|------|------|------|
| 通讯录中已添加联系人"张三" | 无前置条件 | 后续步骤依赖数据但未说明 |
| Python 版本 >= 3.4，adb 版本 >= 1.3.6 | 相关软件已正常安装 | 未明确版本，可能不兼容 |

**steps（操作步骤）**：
- 每步一个操作，使用祈使句
- 包含具体的操作对象（按钮名、输入框名）和输入值
- 不合并多个操作到一步
- **步骤总数建议 7 步以内**，超过则考虑拆分用例
- **一个用例只覆盖单一测试逻辑**，不要在一条用例中验证多个独立功能点
- 参数要显式说明具体值，禁止只写"边界值""非法值"

| 正例 | 反例 | 问题 |
|------|------|------|
| 在 "Endpoint URL" 输入框中输入 `https://api.example.com/webhook` | 输入合法 URL | 未说明具体输入值 |
| 点击 "Add endpoint" 按钮 | 通过左上角的按钮提交 | 用位置描述代替元素名称 |
| 在 "Description" 输入框中输入 101 个字符的文本 | 输入超长文本 | "超长"是模糊描述 |

**expected_results（预期结果）**：
- 必须具体、可验证，可直接用于判定 pass/fail
- 引用实际 UI 文案（如 `"Wrong URL format!"`）
- 包含所有需要验证的检查点（界面显示、数据变化、状态变更）

| 正例 | 反例 | 问题 |
|------|------|------|
| 输入框下方显示红色错误提示："Wrong URL format!" | 提示信息正确 | "正确"无法判断 |
| 检查 XXX 消息中参数 YYY 的取值为 ZZZ | 检查消息流程正确 | 未说明如何判断"正确" |

### 模糊词禁用清单

以下词汇**禁止**出现在 steps 和 expected_results 中（参考[华为规范](https://support.huaweicloud.com/usermanual-testman/cloudtest_01_1309.html)）：

> 很多、一些、部分、大量、少量、多次、随机、任意、一段时间、一会、一点、特殊、很长、较长、很短、较短、多个、几块、多条、数次、所有、大概、频繁、大约、某些、正常、最大、最小、超长、超小、超短、边界值、过低、过高、全部、缺省、默认、各模块、可能、有的、一定范围、适量、多于、少于、左右、上下、设备无异常、计数正常、长时间、错误信息、非法字符

遇到需要使用这些词时，**必须替换为具体数值或明确描述**：
- "多次" → "连续操作 5 次"
- "超长" → "输入 101 个字符（上限为 100）"
- "正常" → "页面显示 Webhook 列表，包含刚创建的记录"
- "默认" → "Version 下拉框选中 'Version 5.0'"

## 测试策略标准

### 新功能测试覆盖基线

每个新功能至少覆盖：
1. **Happy Path**：至少 1 条完整正向流程
2. **字段校验**：每个输入字段的有效/无效/边界/空值
3. **权限差异**：所有涉及的用户角色
4. **联动逻辑**：所有条件联动关系
5. **辅助操作**：取消/返回/刷新的行为

### 回归测试选择标准

| 变更类型 | 回归范围 |
|----------|----------|
| Bug 修复 | 修复场景 + 关联功能的 P0 用例 |
| 小功能迭代 | 变更功能全量 + 上下游 P0 用例 |
| 大版本发布 | P0 全量 + P1 抽样 |
| 数据库/配置变更 | 受影响模块的 P0+P1 |

## 用例生命周期管理

用例不是写完就不动了，需要持续维护：

### 状态跟踪

| 状态 | 含义 |
|------|------|
| Draft | 新建待评审 |
| Active | 评审通过，可执行 |
| Blocked | 因环境/依赖无法执行 |
| Obsolete | 需求变更导致失效，需更新或废弃 |

### 维护时机

| 触发事件 | 维护动作 |
|----------|----------|
| 需求变更 | 更新受影响用例的 steps/expected_results |
| 发现新缺陷 | 补充防御性用例 |
| 执行中发现用例不合理 | 修正 preconditions/steps |
| 功能下线 | 标记为 Obsolete |

### 用例组织

- 按功能模块组织（主要方式）
- 在模块内按优先级排序（P0 在前）
- 跨模块的端到端场景单独归集

## 安全测试检查清单

涉及安全相关功能时，对照此清单补充测试点：

| 检查项 | 测试要点 |
|--------|----------|
| SQL 注入 | 输入框、搜索框、筛选条件中注入 SQL 片段 |
| XSS 注入 | 输入 `<script>`/`<img onerror>` 等标签 |
| 越权访问 | 篡改 URL 中的资源 ID 访问他人数据 |
| 鉴权缺失 | 未登录/Token 过期直接访问受保护接口 |
| 明文传输 | 检查密码/密钥是否通过 HTTPS 传输 |
| 敏感数据泄漏 | API 响应中是否包含密码/密钥/内部 ID |
| 密码安全 | 弱密码校验、密码复杂度要求、加密存储 |
| 短信/邮箱验证 | 验证码有效期、频率限制、暴力破解防护 |
| CSRF 防护 | 关键操作是否有 Token 校验 |
| 数据健壮性 | 超大 payload、畸形 JSON、非法编码 |

## 备选：华为 5 级等级制

当项目需要更细粒度的等级划分时，可参考[华为 CodeArts TestPlan 等级规则](https://support.huaweicloud.com/usermanual-testman/cloudtest_01_1306.html)：

| 级别 | 说明 | 占比 |
|------|------|------|
| 0 级 | 最基本功能验证，每模块 10-20 个 | ~5% |
| 1 级 | 基本功能验证，继承特性/迭代验收 | ~20% |
| 2 级 | 重要特性验证，功能测试主体 | ~60% |
| 3 级 | 一般功能/异常测试 | ~10-15% |
| 4 级 | 极端特殊场景/阈值条件 | ~0-5% |

与本项目 P0-P3 的映射关系：P0 ≈ 0级, P1 ≈ 1级, P2 ≈ 2+3级, P3 ≈ 4级。

## YAML 文件组织标准

```
testing-tm-cases/
├── schema/
│   └── test-case-schema.json    # 用例 Schema 校验
└── releases/
    ├── {版本目录}/
    │   ├── {功能模块-1}.yaml
    │   ├── {功能模块-2}.yaml
    │   └── ...
    └── ...
```

- 每个 YAML 文件对应一个功能模块
- 文件名使用小写 + 连字符（如 `add-webhook.yaml`）
- 同一功能的用例放在同一文件，按操作流程排序
