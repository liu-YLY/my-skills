# 阶段 1:深入理解需求

> **何时阅读**:进入阶段 1 时必读;若用户明确切快速模式,只读 1.3「需求理解输出模板」。
> **覆盖范围**:1.1 信息收集优先级表 / 1.2 可测性与验收口径 / 1.3 需求理解输出模板。
> **关键产出**:输出结构化「需求理解」文档(含信息来源追踪)。
> **强制约束**:默认模式下不允许跳过本阶段直接进入阶段 2。

**目标**:在动手写用例之前,全面透彻地理解被测功能的业务背景、技术实现和产品现状。

**默认模式下不允许跳过此阶段**。快速模式下可压缩,但文首须声明。

## 1.1 信息收集（按优先级顺序，主动使用所有可用手段）

必须主动执行以下动作来获取上下文，而不是仅凭用户的简单描述就开始。

> **信息来源追踪规则**：执行期间**每读取/搜索一个文件或文档，立即追加到下方的「信息来源」列表**。不得事后再凭记忆补。列表在需求理解输出中呈现。

| 优先级 | 手段 | 何时使用 | 具体做法 |
|--------|------|----------|----------|
| **P0** | **扫描项目知识库** | 始终（第一步） | 列举/扫描约定目录（`docs/`、`testing-tm-cases/`），读取相关文档，理解产品全貌 |
| **P0** | **搜索代码库** | 始终 | 在仓库中搜索相关模块的代码实现、API 定义、数据模型、校验逻辑 |
| **P0** | **阅读已有用例** | 代码库中有 YAML 用例时 | 读取同模块/相近功能的已有用例，继承风格、避免重复、衔接 ID 序号 |
| **P1** | **公开文档检索** | 涉及第三方服务、行业规范、不确定的技术细节时 | 检索相关文档、API 规范、竞品实现、已知问题 |
| **P1** | **阅读产品代码** | 需求涉及具体逻辑时 | 查看控制器/服务层/校验逻辑，理解实际行为而非仅看需求描述 |
| **P2** | **追问用户** | 需求存在歧义或信息缺失时 | 列出具体问题清单向用户确认，不做假设 |

## 1.2 可测性与验收口径（阶段 1 必对照）

信息收集时主动确认或记录；若无法确认，写入「存疑点」：

- **日志与追踪**：能否用 request id / trace id / 日志唯一定位一次调用；错误码与文档是否一致。
- **可观测性**：指标、告警、链路是否与需求中的验收口径一致（与阶段 2「可观测性」测试点呼应）。
- **造数与配置**：测试账号、造数接口、mock、**特性开关**是否齐备；能否在隔离环境稳定复现。
- **验收定义**：除界面结果外，是否要求后端状态、对账结果、消息投递、DB 落库等**可验证证据**作为完成标准。

## 1.3 需求理解输出模板

必须先输出需求理解再写用例（快速模式可只保留功能概述 + 信息来源 + 测试数据与环境）：

```markdown
## 需求理解

**功能概述**：[一句话描述功能的核心目的]

**需求追溯（可选）**：
| 条目 | 需求ID/PRD章节/Story | 备注 |
|------|----------------------|------|
| [能力点1] | [ID 或 章节号] | 若无正式 ID 则填「待补充」 |

**用户角色**：[列出所有涉及的用户类型/权限角色]

**业务规则**：
1. [从需求 + 代码 + 项目文档中提取的规则 1]
2. [规则 2]
...

**UI/交互要素**：
- [页面/弹窗/表单中的关键元素及其行为]

**数据流向**：
- [前端 → API → 后端 → 数据库 的关键数据路径]

**测试数据与环境**：
- 目标环境（如 staging / 预发 / 生产只读）与前置条件
- 账号/角色矩阵（测试账号类型、权限组合）
- 数据脱敏或合成数据要求（如有）
- **数据与合规**：禁止未脱敏生产数据时须写明约束；涉及 PII/敏感字段时，覆盖**导出、日志、复制、展示脱敏**等路径；测**权限最小化**（谁能访问、是否与角色一致）
- 特性开关 / 配置项（如有）

**与现有功能的关联**：
- [该功能对其他模块的影响或依赖]

**已有用例覆盖情况**：（如项目中已有相关用例）
- [已覆盖的场景概述，本次需补充/新增的方向]

**存疑点**：（如有）
- [需求未明确的地方，附带自己的判断或向用户的提问]

**信息来源**：
- [列出本次分析实际参考了哪些文件/文档/检索结果]
```
