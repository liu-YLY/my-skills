# 阶段 2:精准提取测试点

> **何时阅读**:进入阶段 2 时必读;探索式模式下额外读「探索章程模板」。
> **覆盖范围**:2.0 风险粗分级 / 2.1 7 维度扫描框架 / 2.2 缺陷模式对照(强制) / 设计方法决策树 / 测试点清单输出模板 / 探索章程模板。
> **强制约束**:7 维度扫描完后**必须**对照 [knowledge/bug-patterns.md](../knowledge/bug-patterns.md) 补充防御性测试点;**必须**先读 [knowledge/test-levels.md](../knowledge/test-levels.md)。
> **关键产出**:测试点清单(含编号/维度/设计方法/优先级/用例数估/需求引用/来源)+ 覆盖度自检 checklist。

**目标**:从需求理解中系统性提取所有可测试点,做到不遗漏。

## 2.0 风险与优先级（粗分级）

在展开 7 维度扫描前，先对本次功能做**粗粒度风险评估**（几行即可）：变更面（影响模块多少）、业务影响（资金/合规/核心路径）、技术复杂度（新依赖、并发、分布式）。据此**辅助映射** P0–P3：高风险核心路径倾向 P0/P1；边缘与体验项倾向 P2/P3。最终仍须与 [knowledge/test-standards.md](../knowledge/test-standards.md) 中的比例自检协调。
> 若项目启用了适配器，优先级按适配器规则映射。

## 测试分层与交付形态（范围意识）

本 Skill **默认**产出可执行的 **E2E / 功能验收**向用例；若需求或用户明确偏 **接口、契约、单测测点**，分层取舍、各层测什么、YAML 形态与「禁止混层」写法见 **[knowledge/test-levels.md](../knowledge/test-levels.md)**。阶段 2 在标注测试点优先级时，可注明建议分层（可选）。

## 2.1 测试点提取框架（7 维度扫描）

对每个功能/页面，按以下 7 个维度逐一扫描：

| 维度 | 检查内容 | 最低覆盖深度 | 示例 |
|------|----------|-------------|------|
| **正向功能** | 核心业务流程的正常路径 | 每个用户角色至少 1 条 Happy Path | 填写完整表单 → 提交成功 |
| **反向校验** | 每个输入字段的异常/非法/边界输入 | 每个字段至少 4 类：有效值、空值、格式错误、边界值 | 必填为空、格式错误、超长、特殊字符 |
| **权限/角色** | 不同用户类型看到的不同内容和可用操作 | 每个角色 × 每个功能点 = 1 条 | A 类用户可选 V2~V5，C 类仅 V5 |
| **状态联动** | 选项 A 变化时选项 B 的联动响应 | 每对联动关系 × 源状态数 | Version 变更 → Status 列表变化 |
| **边界与限制** | 数量上限、长度限制、重复校验 | 上限值、上限+1、重复值、零值 | 最多 4 个 webhook、URL 不可重复 |
| **UI 交互** | 按钮状态、提示文案、页面跳转、加载状态 | 每个按钮/状态变化点 = 1 条 | 必填未满足 → 按钮置灰 |
| **异常与容错** | 网络异常、并发操作、服务端错误响应 | 网络断开、服务端 5xx、并发冲突 | 提交时网络断开 → 友好提示 |

**可观测性（按需）**：若需求涉及日志、指标、告警、链路追踪的验收，在测试点中单独标注（可与非功能用例一起输出）；与阶段 1「可测性与验收口径」对照，避免只验界面不验可证明性。

## 2.2 缺陷模式对照补充（必须执行）

7 维度扫描完成后，**必须读取** [knowledge/bug-patterns.md](../knowledge/bug-patterns.md) 中的「缺陷模式→测试点映射」表，根据当前功能特征补充防御性测试点：

- 有输入框？→ 检查是否补充了 XSS、超长截断、编码问题
- 有提交按钮？→ 检查是否补充了重复提交
- 有列表页？→ 检查是否补充了空数据、分页
- 涉及权限？→ 检查是否补充了越权访问
- 有状态流转？→ 检查是否补充了状态回退、幽灵状态
- 有弹窗？→ 检查是否补充了弹窗残留

**项目特定模式补充**（按需）：通用对照完成后，检查以下来源是否有项目特定的缺陷模式（不存入 Skill，按需读取）：
- 项目根目录是否存在 `docs/bug-patterns.md` → 如有则读取
- 是否可查询 JIRA → 如有授权且环境支持，按模块/严重程度筛选近期 Bug，提取高频模式

## 设计方法选择决策树

- 输入域有明确范围？→ 等价类划分 + 边界值分析
- 存在多个条件组合？→ 判定表/因果图
- 系统有明确状态流转？→ 状态迁移法
- 多参数需要覆盖？→ 正交实验法
- 业务流程测试？→ 场景法（基本流 + 备选流 + 异常流）

### 设计方法速查示例

| 方法 | 适用场景 | 示例 |
|------|----------|------|
| 等价类+边界值 | 输入域有明确范围 | 年龄[18,60]→测试值 17,18,19,59,60,61；无效：<0,"abc","",小数 |
| 状态迁移 | 系统有明确状态 | 识别所有状态→触发事件→迁移图→覆盖路径(0-switch/1-switch) |
| 判定表 | 多条件组合 | 列举所有条件组合及动作，不影响动作的用"-"合并 |
| 场景法 | 业务流程 | 基本流(Happy Path)→备选流(取消/修改)→异常流(错误/超时) |

### 风险与深度策略

| 风险等级 | 测试深度 | 典型场景 |
|----------|----------|----------|
| 高（核心业务/资金/合规） | 全面覆盖 + 探索性 | 支付、权限、数据一致性 |
| 中（常用功能） | 等价类 + 边界值 | 表单提交、列表筛选 |
| 低（辅助功能） | 冒烟测试 | 帮助页、关于页 |

## 测试点清单输出模板

```markdown
## 测试点清单

| 编号 | 测试点 | 维度 | 设计方法 | 优先级 | 用例数（估） | 需求引用 | 来源 |
|------|--------|------|----------|--------|-------------|----------|------|
| 1 | Endpoint URL 必填校验 | 反向校验 | 等价类+边界值 | P0 | 3 | PRD §3.2 | 需求 |
| 2 | URL 格式校验 | 反向校验 | 等价类 | P0 | 3 | PRD §3.2 | 需求 |
| 3 | Version - 不同用户类型可选范围 | 权限/角色 | 角色矩阵 | P0 | 3 | Story-42 | 需求 |
| 4 | Version → Status 联动 | 状态联动 | 状态迁移 | P0 | 4 | Story-42 | 需求 |
| 5 | 重复 URL 提交拦截 | 边界与限制 | 等价类 | P0 | 1 | PRD §3.3 | 需求 |
| 6 | XSS 防御 - URL 字段注入 | 异常与容错 | 安全模式 | P1 | 1 | - | 缺陷模式库 |
| 7 | 弹窗关闭后残留数据检查 | UI 交互 | 经验模式 | P2 | 1 | - | 缺陷模式库 |

**覆盖度自检**：
- [ ] 7 个维度均已扫描
- [ ] 缺陷模式库已对照（标注来源为「缺陷模式库」的测试点）
- [ ] 每个输入字段至少覆盖：有效值、空值、格式错误、边界值
- [ ] 每个用户角色/权限组合均已覆盖
- [ ] 所有联动关系均已识别
```

**需求引用**列可填 PRD 章节、用户故事 ID、内部需求单号等；无则填 `-`。

### 字段级测试矩阵（反向校验维度专用）

对每个表单/页面，用此矩阵确保字段覆盖无遗漏：

| 字段 | 必填 | 有效值 | 空值 | 格式错误 | 边界值 | 特殊字符 |
|------|------|--------|------|----------|--------|----------|
| Endpoint URL | 是 | `https://api.example.com` | 为空→提示必填 | `ftp://xxx`→格式错误 | 2048字符/2049字符 | `<script>alert(1)</script>` |
| Version | 是 | V5.0 | 未选择→提示必填 | - | - | - |
| Description | 否 | 正常文本 | 为空→可提交 | - | 100字符/101字符 | emoji、日文 |

填写规则：每格填入**一个代表性测试值 + 预期结果关键词**，`-` 表示该类校验不适用。

## 探索式测试模式 - 探索章程模板

当用户明确要求「发散探索」「找深坑」或需求模糊、不确定性高时，在完成最小需求理解后，除照常输出测试点清单外，**额外**输出探索章程，与测试点清单**并列**：

```markdown
## 探索章程（Exploration Charter）

**时间盒**：[如 60min / 本迭代内]
**范围**：[功能边界 / 模块]
**关注点**：[风险区域、接口联动、竞态、边界]
**须记录证据**：[截图、日志片段、trace id、请求/响应摘录]
**退出条件**：[如发现阻塞级缺陷则停止并报告]
```

探索章程**不替代** 7 维度扫描与缺陷模式对照，但应标出高风险区优先探索。探索结论中的缺陷仍用 SKILL 中「缺陷分析与定位」格式报告。
