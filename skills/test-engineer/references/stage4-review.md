# 阶段 4:自检与补全

> **何时阅读**:进入阶段 4 时必读;快速模式下可只跑「质量检查」+「遗漏扫描」两节。
> **覆盖范围**:Schema 校验确认 / 可追溯性 / 分层与合规 / 探索式 / 覆盖度 / 优先级比例 / 质量 8 原则 / 遗漏扫描。
> **行为约束**:发现遗漏直接补充到用例,不另外说明。
> **依赖**:优先级比例参考 [knowledge/test-standards.md](../knowledge/test-standards.md) 三步法。

输出用例后,**必须执行**以下检查(快速模式可压缩)。

## 前置：Schema 校验确认（如有适配器）

若阶段 3 使用了项目适配器，确认适配器指定的校验**已通过**。若未执行或失败，先回到阶段 3 修复。

## 用例自检清单

```markdown
## 用例自检

**可追溯性**：
- [ ] 主要用例与测试点编号可对应；追溯信息（req_ref/trace 或 description 首行）完整

**分层与合规（按需）**：
- [ ] 若本次为接口/契约导向：用例步骤与预期是否与分层一致（非仅 UI 套壳）
- [ ] 若涉及 PII/导出/日志/权限：相关测试点已覆盖

**探索式（若本次输出了探索章程）**：
- [ ] 已记录证据（截图、日志、trace、请求/响应）与结论或待跟进项

**覆盖度检查**：
- [ ] 所有测试点均已有对应用例
- [ ] 正向 Happy Path 全覆盖
- [ ] 每个输入字段的反向校验全覆盖
- [ ] 角色/权限矩阵全覆盖
- [ ] 联动关系全覆盖
- [ ] 边界条件全覆盖
- [ ] 场景法覆盖：基本流 + 备选流 + 异常流

**优先级比例检查**（参照 [knowledge/test-standards.md](knowledge/test-standards.md) 三步法）：
- [ ] P0 占比 10%~15%
- [ ] P1 占比 30%~40%
- [ ] P2 占比 30%~40%
- [ ] P3 占比 10%~15%
- [ ] 如偏离比例，需重新评估升降级
> 若适配器降级了优先级（如 TEST 中 P3→P2），比例以通用格式的原始划分为准。

**质量检查**（对照 8 大设计原则）：
- [ ] 有效性：步骤无歧义，按流程顺序编排
- [ ] 清晰性：包含具体输入数据和预期输出
- [ ] 易操作性：任何人可独立执行，不依赖口头说明
- [ ] expected_results 无模糊描述（禁止「正常」「正确」「符合预期」）
- [ ] steps 具体到可操作（包含实际输入值和操作对象）
- [ ] steps 与 expected_results 数量一致，一一对应（一步一验）
- [ ] ID 无重复

**遗漏扫描**（最后再过一遍 7 个维度）：
- [ ] 是否遗漏了某个字段/按钮/链接？
- [ ] 是否遗漏了某个用户角色？
- [ ] 是否遗漏了取消/返回/刷新等辅助操作？
- [ ] 涉及安全时：是否对照了安全测试检查清单？
```

**如果发现遗漏，直接补充到用例中，不另外说明。**
