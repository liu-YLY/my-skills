# 测试工程师核心流程

> 本文档合并了原四阶段指令文件的核心内容，提供完整但精简的测试用例生成流程。

---

## 阶段 1：理解需求

**目标**：全面透彻地理解被测功能的业务背景、技术实现和产品现状。

### 信息收集（按优先级顺序）

| 优先级 | 手段 | 具体做法 |
|--------|------|----------|
| **P0** | 扫描项目知识库 | 列举/扫描约定目录（`docs/`、`testing-tm-cases/`），读取相关文档 |
| **P0** | **加载产品知识库** | 识别被测功能所属产品，加载 `knowledge/products/{product-id}.md`（详见下方） |
| **P0** | 搜索代码库 | 搜索相关模块的代码实现、API 定义、数据模型、校验逻辑 |
| **P0** | 阅读已有用例 | 读取同模块/相近功能的已有用例，继承风格、避免重复 |
| **P1** | 公开文档检索 | 检索第三方服务文档、API 规范、竞品实现 |
| **P1** | 阅读产品代码 | 查看控制器/服务层/校验逻辑，理解实际行为 |
| **P2** | 追问用户 | 列出具体问题清单向用户确认，不做假设 |

### 产品知识库加载（阶段 1 必做）

当被测功能可识别产品归属时，**必须加载**对应的产品知识文件：

1. **识别产品**：从以下来源推断产品ID
   - 用户明确指定（如 `使用 [产品名称] 知识库`）
   - 需求文档中的产品名称/模块名
   - 代码路径（如 `src/modules/crm/` → 产品ID `crm`）
   - API 路径（如 `/api/v1/payment/` → 产品ID `payment`）
   - 配置文件或数据库表名中的产品标识

2. **查找知识文件**：在 `knowledge/products/` 目录下查找 `{product-id}.md`
   - 精确匹配：`{product-id}.md`
   - 模糊匹配：若无精确匹配，搜索文件名包含产品关键词的文件
   - 索引文件：参考 `knowledge/products/README.md` 中的产品列表

3. **加载并应用**：
   - 读取「业务规则与约束」→ 用于阶段 1 需求理解，提取产品特定的业务逻辑
   - 读取「高频缺陷模式」→ 用于阶段 2 补充防御性测试点，避免已知缺陷
   - 读取「专项测试检查清单」→ 用于阶段 2 确保重点领域覆盖
   - 读取「状态流转规则」→ 用于阶段 2 设计状态相关测试点
   - 读取「常见测试场景模板」→ 用于阶段 3 加速用例编写
   - 读取「数据字典与校验规则」→ 用于阶段 2/3 确保输入校验测试点准确

4. **加载后处理**：
   - 将产品知识中的业务规则与需求文档进行交叉验证
   - 识别产品特定的术语和概念，在后续阶段使用准确的业务语言
   - 标记产品知识中与当前需求相关的章节，便于阶段 2 快速定位

5. **若无产品知识文件**：跳过此步骤，按通用流程执行，但在阶段 1 输出中标注「未找到产品知识文件，按通用流程执行」

> **产品知识索引**：[knowledge/products/README.md](knowledge/products/README.md)
> **知识模板**：[knowledge/products/products-template.md](knowledge/products/products-template.md)

**信息来源追踪规则**：执行期间每读取/搜索一个文件或文档，立即追加到「信息来源」列表。

🔴 **CHECKPOINT · 阶段 1 完成**：需求理解输出后，必须展示给用户确认再进入阶段 2。用户确认前不得开始提取测试点。

### 阶段 1 失败模式与 Fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| 项目文档缺失/不完整 | 扫描代码库提取业务逻辑，搜索注释和 README | 列出「存疑点」清单，要求用户补充确认 |
| 代码库无权限或不可达 | 请求用户提供关键代码片段或 API 文档 | 基于需求文档推断，标注「待验证」 |
| 已有用例风格不一致 | 取最近 3 个模块的用例作为参考基线 | 采用通用 YAML 格式，阶段 4 统一校验 |
| 用户无法确认存疑点 | 按最保守理解设计测试点，覆盖多种可能性 | 标记为「待确认」，用例中增加条件分支 |

### 可测性与验收口径（阶段 1 必对照）

- **日志与追踪**：能否用 request id / trace id 唯一定位一次调用
- **可观测性**：指标、告警、链路是否与需求中的验收口径一致
- **造数与配置**：测试账号、造数接口、mock、特性开关是否齐备
- **验收定义**：除界面结果外，是否要求后端状态、对账结果、消息投递等可验证证据

## 阶段 2：精准提取测试点

**目标**：从需求理解中系统性提取所有可测试点，做到不遗漏。

### 风险与优先级（粗分级）

在展开 7 维度扫描前，先对本次功能做粗粒度风险评估：变更面（影响模块多少）、业务影响（资金/合规/核心路径）、技术复杂度（新依赖、并发、分布式）。据此辅助映射 P0–P3。

### 7 维度扫描框架

对每个功能/页面，按以下 7 个维度逐一扫描：

| 维度 | 检查内容 | 最低覆盖深度 |
|------|----------|-------------|
| **正向功能** | 核心业务流程的正常路径 | 每个用户角色至少 1 条 Happy Path |
| **反向校验** | 每个输入字段的异常/非法/边界输入 | 每个字段至少 4 类：有效值、空值、格式错误、边界值 |
| **权限/角色** | 不同用户类型看到的不同内容和可用操作 | 每个角色 × 每个功能点 = 1 条 |
| **状态联动** | 选项 A 变化时选项 B 的联动响应 | 每对联动关系 × 源状态数 |
| **边界与限制** | 数量上限、长度限制、重复校验 | 上限值、上限+1、重复值、零值 |
| **UI 交互** | 按钮状态、提示文案、页面跳转、加载状态 | 每个按钮/状态变化点 = 1 条 |
| **异常与容错** | 网络异常、并发操作、服务端错误响应 | 网络断开、服务端 5xx、并发冲突 |

### 缺陷模式对照补充（必须执行）

7 维度扫描完成后，执行两轮缺陷模式对照：

**第一轮：通用缺陷模式**

读取 [knowledge/bug-patterns.md](knowledge/bug-patterns.md) 中的「缺陷模式→测试点映射」表，根据当前功能特征补充防御性测试点：

- 有输入框？→ 检查是否补充了 XSS、超长截断、编码问题
- 有提交按钮？→ 检查是否补充了重复提交
- 有列表页？→ 检查是否补充了空数据、分页
- 涉及权限？→ 检查是否补充了越权访问
- 有状态流转？→ 检查是否补充了状态回退、幽灵状态
- 有弹窗？→ 检查是否补充了弹窗残留

**第二轮：产品专项缺陷模式**

若阶段 1 已加载产品知识文件，读取其中的「测试重点领域」章节：

- 「历史高频缺陷」→ 对照当前功能是否涉及，补充对应测试点
- 「专项测试检查清单」→ 逐项检查是否已覆盖
- 「性能基线」→ 若涉及性能测试，参考基线值设计测试点
- 「状态流转规则」→ 确保状态相关测试点覆盖所有流转路径

### 产品特定测试点提取（阶段 2 必做）

若阶段 1 已加载产品知识文件，**必须**基于产品知识提取产品特定测试点：

1. **业务规则测试点**：
   - 从产品知识的「业务规则与约束」章节提取所有业务规则
   - 每条业务规则至少对应 1 个正向验证测试点（规则满足时的行为）
   - 每条业务规则至少对应 1 个反向验证测试点（规则违反时的错误处理）
   - 标注来源为「产品知识-业务规则」

2. **状态流转测试点**：
   - 从产品知识的「状态流转规则」章节提取所有状态流转路径
   - 每条合法流转路径对应 1 个正向测试点
   - 每条非法流转路径（不允许的状态变更）对应 1 个反向测试点
   - 覆盖所有状态组合，包括并发状态变更场景
   - 标注来源为「产品知识-状态流转」

3. **高频缺陷测试点**：
   - 从产品知识的「高频缺陷模式」章节提取历史缺陷模式
   - 若当前功能涉及相关模块/场景，补充对应的防御性测试点
   - 标注来源为「产品知识-高频缺陷」

4. **专项检查测试点**：
   - 从产品知识的「专项测试检查清单」章节提取检查项
   - 逐项检查当前功能是否涉及，涉及则补充对应测试点
   - 标注来源为「产品知识-专项检查」

5. **数据校验测试点**：
   - 从产品知识的「数据字典与校验规则」章节提取字段校验规则
   - 每个字段的校验规则对应多个测试点（有效值、无效值、边界值）
   - 标注来源为「产品知识-数据校验」

**产品特定测试点标注规则**：
- 在测试点清单的「来源」列标注「产品知识-{章节名}」
- 便于追溯测试点的来源，确保产品知识的完整应用

### 设计方法选择决策树

- 输入域有明确范围？→ 等价类划分 + 边界值分析
- 存在多个条件组合？→ 判定表/因果图
- 系统有明确状态流转？→ 状态迁移法
- 业务流程测试？→ 场景法（基本流 + 备选流 + 异常流）

### 测试点清单输出模板

```markdown
## 测试点清单

| 编号 | 测试点 | 维度 | 设计方法 | 优先级 | 用例数（估） | 需求引用 | 来源 |
|------|--------|------|----------|--------|-------------|----------|------|
| 1 | Endpoint URL 必填校验 | 反向校验 | 等价类+边界值 | P0 | 3 | PRD §3.2 | 需求 |
| 2 | XSS 防御 - URL 字段注入 | 异常与容错 | 安全模式 | P1 | 1 | - | 缺陷模式库 |

**覆盖度自检**：
- [ ] 7 个维度均已扫描
- [ ] 缺陷模式库已对照
- [ ] 每个输入字段至少覆盖：有效值、空值、格式错误、边界值
- [ ] 每个用户角色/权限组合均已覆盖
- [ ] 所有联动关系均已识别

🔴 **CHECKPOINT · 阶段 2 完成**：测试点清单输出后，必须展示给用户确认再进入阶段 3。用户确认前不得开始生成用例。

### 阶段 2 失败模式与 Fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| 测试点遗漏（阶段 4 发现） | 回溯 7 维度逐项核对，补充遗漏维度的测试点 | 标记遗漏维度，用例中标注「待补充」 |
| 缺陷模式库无匹配 | 使用通用缺陷模式（输入校验/状态/权限/数据/UI） | 仅输出 7 维度扫描结果，不强制补充 |
| 设计方法选择困难 | 优先使用场景法（基本流+备选流+异常流） | 多种方法并用，阶段 4 去重 |
```

## 阶段 3：编写测试用例

### AI 生成模式（默认）

**流程**：
1. 读取 [knowledge/prompt-strategy.md](knowledge/prompt-strategy.md) 中的结构化提示词模板
2. 将需求理解 + 测试点清单填入 Prompt 模板
3. **若已加载产品知识**：将产品知识中的「业务规则」「常见测试场景模板」作为上下文注入 Prompt
4. AI 生成 70%-90% 的基础用例
5. **人工审核**：检查业务逻辑准确性、补充领域特有边界条件、验证优先级合理性
6. 人工补充 AI 遗漏的复杂业务逻辑用例

**产品知识注入要点**：
- 使用产品知识中的「数据校验规则」确保用例中的输入值符合业务约束
- 使用产品知识中的「状态流转规则」确保状态相关用例覆盖完整路径
- 使用产品知识中的「测试数据与环境」信息完善用例的前置条件
- 使用产品知识中的「常见测试场景模板」作为 Few-shot 示例

**AI 生成的用例标记** `source: ai-generated`，人工补充的标记 `source: manual`。

**审核重点**：
- 业务逻辑是否准确
- 领域特有边界条件是否覆盖
- 优先级是否合理
- steps 是否可直接执行

🛑 **STOP · 阶段 3 AI 生成后**：AI 生成的用例必须经过人工审核才能输出。未经审核的用例禁止标记 `source: ai-generated`。

### 阶段 3 失败模式与 Fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|----------|------------|
| AI 生成质量不达标（覆盖率<80%） | 调整 Prompt：增加约束、补充 Few-shot 示例 | 分批生成（单次 10-15 条），人工补充剩余 |
| AI 生成格式不合规 | 要求 AI 按 YAML 模板重新输出 | 人工手动修正格式，标记 `source: manual` |
| 适配器转换失败 | 检查 stderr 错误信息，修复草稿后重试 | 放弃适配器，直接输出通用格式 |
| steps 与 expected_results 数量不一致 | 检查每步操作是否都有可观察结果，补充或合并 | 重新梳理步骤，确保一步一验 |

### 适配器检查（输出前必做）

在构思和输出 YAML 用例之前，先检查 `adapters/` 目录：
- **无适配器** → 用通用格式直接输出，不校验
- **有适配器** → 构思用通用格式，转换用适配器规则，输出后校验

### 默认输出功能测试用例

除非用户明确要求或需求内容明显涉及以下类型，否则只输出功能测试用例（`type: functional`）：

| 额外类型 | 触发条件 |
|----------|----------|
| performance | 需求提到响应时间、并发量、吞吐量要求 |
| security | 涉及认证、授权、敏感数据、支付 |
| compatibility | 需求提到多浏览器、多设备、多系统版本 |

当需要额外类型时，在功能测试用例之后独立成组追加。

### 通用 YAML 格式

```yaml
- id: TC_MODULE_FUNC_001
  title: 被测对象 - 测试意图（具体行为）
  priority: P0
  type: functional
  req_ref: Story-42
  trace: TP-03
  source: ai-generated
  description: |
    补充说明测试场景的业务背景
  preconditions:
    - 已登录为 A 类用户
    - 已打开 "Add webhook" 弹窗
  steps:
    - 操作步骤 1（祈使句，一步一动作）
    - 操作步骤 2
  expected_results:
    - 预期结果 1（步骤1完成后立即可观察的结果）
    - 预期结果 2（步骤2完成后立即可观察的结果）
  tags: [场景标签]
  auto: false
```

### 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | `TC_{模块}_{功能}_{三位序号}` |
| `title` | string | 是 | 不超过 40 字符，动宾结构 |
| `priority` | P0/P1/P2/P3 | 是 | 定义与比例见 [knowledge/test-standards.md](knowledge/test-standards.md) |
| `type` | string | 是 | functional / ui / security / performance / compatibility / usability / accessibility / observability |
| `req_ref` | string | 否 | 需求追溯（Story ID） |
| `trace` | string | 否 | 测试点追溯（TP 编号） |
| `source` | string | 否 | 用例来源：ai-generated / manual / ai-reviewed |
| `description` | string | 否 | 业务背景补充 |
| `preconditions` | string[] | 否 | 前置条件 |
| `steps` | string[] | 是 | 操作步骤 |
| `expected_results` | string[] | 是 | 预期结果 |
| `tags` | string[] | 否 | 标签 |
| `auto` | boolean | 否 | 默认 false |

### 编写与审核铁律

**title**：`{被测对象} - {具体行为}`，不超过 40 字符，动宾结构

**steps**：
- 祈使句，每步一个操作，含具体输入值
- **不超过 7 步**，超过 7 步必须拆分为多条用例
- 一个用例只覆盖单一测试逻辑

**expected_results**：
- 可直接判定 pass/fail，引用实际文案
- **禁用模糊词**（完整清单见 [knowledge/test-standards.md](knowledge/test-standards.md)）
- 数量与 steps 一致，一一对应（一步一验）

**steps ↔ expected_results 一一对应（强制）**：
- `steps[N]` 和 `expected_results[N]` 数量必须一致
- 每步操作后立即写出对应的预期结果
- 预备动作上移到 preconditions

**测试点→用例映射规则**：
- 同一流程不同测试数据 → 合并为 1 条用例，数据参数化
- 不同前置条件/预期结果/测试逻辑 → 拆分为独立用例
- 同一字段多类校验 → 按类拆分，每类 1 条

### 适配器转换与校验（若有适配器）

当 `adapters/` 目录下存在项目适配器时，优先使用程序化转换：

```bash
# 1. Agent 把通用格式 YAML 写入中间文件
# 2. 一键转换 + 校验
.venv-tools/bin/python $SKILL_ROOT/scripts/transform_yaml.py \
    .tmp/draft.yaml \
    -o testing-tm-cases/releases/<release>/<feature>.yaml \
    --validate \
    --module "<模块名>" --feature "<功能名>"
```

校验失败 → 阅读 stderr 中的错误信息 → 修复草稿 → 重新跑脚本

## 阶段 4：自检与补全

**行为约束**：发现遗漏直接补充到用例，不另外说明。

### 用例自检清单

```markdown
## 用例自检

**可追溯性**：
- [ ] 主要用例与测试点编号可对应
- [ ] 追溯信息（req_ref/trace 或 description 首行）完整

**覆盖度检查**：
- [ ] 所有测试点均已有对应用例
- [ ] 正向 Happy Path 全覆盖
- [ ] 每个输入字段的反向校验全覆盖
- [ ] 角色/权限矩阵全覆盖
- [ ] 联动关系全覆盖
- [ ] 边界条件全覆盖

**优先级比例检查**：
- [ ] P0 占比 10%~15%
- [ ] P1 占比 30%~40%
- [ ] P2 占比 30%~40%
- [ ] P3 占比 10%~15%

**质量检查**：
- [ ] 步骤无歧义，按流程顺序编排
- [ ] 包含具体输入数据和预期输出
- [ ] expected_results 无模糊描述
- [ ] steps 具体到可操作
- [ ] steps 与 expected_results 数量一致
- [ ] ID 无重复

**遗漏扫描**：
- [ ] 是否遗漏了某个字段/按钮/链接？
- [ ] 是否遗漏了某个用户角色？
- [ ] 是否遗漏了取消/返回/刷新等辅助操作？
- [ ] 涉及安全时：是否对照了安全测试检查清单？

**产品知识覆盖检查**（若有产品知识文件）：
- [ ] 产品知识中的「历史高频缺陷」是否已覆盖
- [ ] 产品知识中的「专项测试检查清单」是否逐项检查
- [ ] 产品知识中的「状态流转规则」是否已完整覆盖
- [ ] 产品知识中的「业务规则」是否已在用例中体现

**AI 生成质量检查**（仅 AI 生成模式）：
- [ ] AI 生成用例是否覆盖了所有测试点中的标准场景
- [ ] 人工补充的用例是否覆盖了 AI 容易遗漏的领域特有业务逻辑
- [ ] AI 生成用例的 source 标记是否准确

🔴 **CHECKPOINT · 阶段 4 完成**：自检通过后，必须展示最终用例给用户确认。用户确认前不得视为完成。
```

### 效率度量输出（AI 生成模式必做）

阶段 4 自检完成后，输出效率报告：

```markdown
## 效率度量报告

| 指标 | 本次值 |
|------|--------|
| 用例总数 | X 条 |
| AI 生成 | Y 条 (Z%) |
| 人工补充 | W 条 |
| 预估传统工时 | N 人天 |
| 实际工时 | M 人天 |
| 工时节省 | (N-M)/N * 100% |
| 效率提升 | N/M 倍 |

**质量观察**：
- AI 生成用例中需修改的比例：约 A%
- AI 补充的异常/安全场景数量：B 条
```

**传统工时估算基准**：
- 低（单表单/单页面，3-5 字段）：0.5 人天
- 中（多表单/多页面，含联动/权限）：1-2 人天
- 高（复杂业务流程，多角色多状态）：3-4 人天
- 极高（跨系统集成，核心业务链路）：5+ 人天

- **默认**：不允许跳过阶段 1/2 直接写完整用例
- **快速**：压缩阶段 1/4，但必须输出测试点清单，文首声明未做完整理解与自检
- **探索式**：额外输出探索章程，与测试点清单并列

## 探索章程（Exploration Charter）

**关注点**：[风险区域、接口联动、竞态、边界]
**须记录证据**：[截图、日志片段、trace id、请求/响应摘录]

## 缺陷分析与定位

五步定位法：复现 → 隔离 → 定位 → 验证 → 报告。

**报告模板**：
```markdown
**现象** / **环境** / **版本/构建号** / **复现率** / **附件/日志**
**根因**：[直接原因] / [深层原因]
**影响范围** / **修复建议**（紧急 + 长期） / **回归测试**
```

**鱼骨图分类维度**（根因分析时按此归类）：
- **人**：操作失误、权限配置、沟通遗漏
- **机**：服务器、网络、中间件、环境配置
- **料**：数据质量、依赖服务、第三方接口
- **法**：流程缺陷、规范缺失、代码逻辑
- **环**：环境差异、版本差异、部署时序

**5 Whys 根因分析**：从问题表象逐层追问"为什么"，直到找到可落地的根因。

## 反例与黑名单（不要做什么）

> **设计依据**：基于 SkillLens 论文（arXiv 2605.23899）实证——只写"应该做 X"没有"不要做 Y"会导致 LLM judge 准确率下降。

### 工作流反模式

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|----------|
| 1 | **跳过阶段 1 直接写用例** | 需求理解不完整导致用例偏离业务意图 | 必须完成阶段 1 需求理解输出，哪怕快速模式也要压缩而非跳过 |
| 2 | **跳过阶段 2 直接写用例** | 测试点遗漏导致用例覆盖不全 | 必须输出测试点清单，阶段 3 基于清单生成用例 |
| 3 | **AI 生成后不审核直接输出** | AI 可能产生业务逻辑错误或格式不合规 | 人工审核后才标记 `source: ai-generated` |
| 4 | **用例中使用模糊词** | 无法直接判定 pass/fail，用例形同虚设 | 替换为具体数值或明确描述（见模糊词禁用清单） |
| 5 | **steps 与 expected_results 数量不一致** | 无法一一对应验证，测试执行时混乱 | 重新梳理步骤，确保一步一验 |
| 6 | **单条用例覆盖多个测试逻辑** | 失败时无法定位具体问题点 | 拆分为独立用例，每条只验证一个逻辑 |
| 7 | **用位置描述代替元素名称** | UI 变更后用例失效 | 使用具体的元素名称（按钮名、输入框名） |
| 8 | **expected_results 使用"正确""正常"** | 无法客观判定，依赖主观判断 | 引用实际 UI 文案或具体数值 |

### 缺陷分析反模式

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|----------|
| 1 | **只复现不隔离** | 无法确定根因范围，修复可能引入新问题 | 先隔离变量，再定位根因 |
| 2 | **跳过 5 Whys 直接下结论** | 可能只找到表象原因，未触及深层根因 | 逐层追问，直到找到可落地的根因 |
| 3 | **忽略鱼骨图分类** | 根因分析遗漏重要维度 | 按人/机/料/法/环五维度逐一排查 |
| 4 | **修复建议只有紧急方案** | 临时方案可能累积技术债务 | 同时给出紧急修复 + 长期改进方案 |

### 输出格式反模式

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|----------|
| 1 | **YAML 缺少必填字段** | 适配器转换失败或校验不通过 | 对照字段定义表，确保 id/title/priority/type/steps/expected_results 齐全 |
| 2 | **title 超过 40 字符** | 在测试管理工具中显示截断 | 精简为 `{被测对象} - {具体行为}`，动宾结构 |
| 3 | **steps 超过 7 步** | 执行时容易遗漏或出错 | 拆分为多条用例，每条控制在 7 步内 |
| 4 | **ID 重复或格式不规范** | 无法唯一标识，追溯困难 | 使用 `TC_{模块}_{功能}_{三位序号}` 格式 |

### 触发检查时机

- **阶段 1 完成时**：对照工作流反模式 1-2
- **阶段 3 输出前**：对照工作流反模式 3-8 + 输出格式反模式
- **阶段 4 自检时**：对照所有反模式
- **缺陷分析时**：对照缺陷分析反模式
