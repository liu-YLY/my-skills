# 如何编写高质量的测试用例

编写测试用例是测试工程师的核心工作之一。很多人觉得写测试用例很简单，不就是把操作步骤列出来吗？但实际上，真正高质量的测试用例需要经过深入的需求分析和系统的测试点提取。这篇文章就和大家分享一下我的方法。

## 理解需求是第一步

拿到需求之后，不要急着写用例。先问自己几个问题：这个功能的核心价值是什么？有哪些用户角色会用到？涉及哪些数据流转？

举个例子，假设我们要测试一个"用户注册"功能。表面上看很简单——填表、提交、注册成功。但如果深入分析，你会发现需要考虑的场景非常多。

## 提取测试点的方法

我常用的方法是"7 维度扫描法"：

1. 正向功能：正常路径的 Happy Path
2. 反向校验：每个输入字段的异常输入
3. 权限/角色：不同用户看到的内容差异
4. 状态联动：选项之间的联动关系
5. 边界与限制：数量上限、长度限制
6. UI 交互：按钮状态、提示文案
7. 异常与容错：网络异常、服务端错误

每个维度都至少输出 2-3 个测试点，这样能保证覆盖面。

## 编写用例的规范

写用例也有讲究。title 要简洁有力，动宾结构。steps 用祈使句，每步一个操作。最重要的是 expected_results 要和 steps 一一对应，一步一验。

下面是一个标准的用例格式：

```
id: TC_USER_REGISTER_001
title: 正确填写注册信息并提交
priority: P0
type: functional
steps:
  - 打开注册页面
  - 输入有效手机号
  - 输入符合规则的密码
  - 点击注册按钮
expected_results:
  - 页面显示注册表单
  - 手机号输入框显示已输入内容
  - 密码以掩码形式显示
  - 提示注册成功并跳转至登录页
```

## Python 自动化示例

对于可自动化的用例，可以用 pytest 来编写：

```python
def test_register_success():
    # 准备测试数据
    phone = "13800138000"
    password = "Test@123456"
    
    # 调用注册接口
    response = register_api(phone, password)
    
    # 验证结果
    assert response.status_code == 200
    assert response.json()["code"] == 0
    assert "token" in response.json()["data"]
```

## 常见问题

很多新手容易犯的错误：把所有的校验逻辑写在一个用例里。比如既验证必填为空又验证格式错误又验证边界值，这样一旦失败很难定位。正确做法是一条用例只验证一个测试逻辑。

还有一点，preconditions 和 steps 容易混淆。打开页面、登录这些预备动作应该放在 preconditions，只有在特定步骤有验证价值时才放在 steps。

## 总结

写好测试用例没有捷径，就是多分析需求、多提取测试点、多练习编写。掌握了正确的方法论之后，效率会越来越高。
