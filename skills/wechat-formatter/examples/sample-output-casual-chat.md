# 写了五年测试用例，我才发现这件事

你有没有遇到过这种情况——

辛辛苦苦写了两百条测试用例，上线后 Bug 一个没少，还被开发怼了一句「你测的啥？」

说实话，这种感觉真的很崩溃 😅

我刚入行那两年，几乎每个月都要经历一次。每次都在想：我明明写了很多用例啊，为什么线上的 Bug 跟我的用例像是两条平行线？

后来慢慢琢磨明白了。核心问题就一个。

—

## 🤔 问题出在哪

先说一个反直觉的结论：**测试用例不是越多越好。**

100 条重复逻辑的用例，不如 10 条精准覆盖关键路径的用例。

我刚入行时最大的误区就是：把同一个逻辑用不同的测试数据重复写了 N 条。比如测手机号输入框，13800138000 写一条，13900138001 写一条，13700138002 又写一条……

看着用例数挺多，评审的时候很有成就感。但实际上这三条本质上是同一条——它们都只覆盖了「有效手机号」这一个等价类。

真正应该测的，是那些你没想过要测的地方。

—

平时我们很容易只关注「用户正常操作会发生什么」和「用户输错了会怎么样」。

但其实还有几个维度，是我在经历了好几次线上事故之后才学会重视的。

—

## 🎯 那些容易漏掉的测试点

**第一件事：权限。**

有一次上线后，一个普通用户发现他能看到管理员的报表。

排查了半天，原因是后端改了接口逻辑但没同步更新权限校验。测试用例里压根没覆盖这个场景——我们只测了「管理员能看到报表」和「未登录用户看不了」，跳过了「已登录的普通用户」这个中间状态。

从那以后，我每次写用例都会列一个矩阵：所有角色 × 所有功能点，一个个交叉问——他能不能看到？能不能操作？

**第二件事：状态联动。**

这个比权限更容易漏。举个例子——你选择「中国」，区号自动变成 +86，这个谁都会测。但如果用户在 +86 之后又切回「请选择」，区号会清空吗？还是会保留 +86？

这类场景特别容易被忽略，但往往是 Bug 的高发区。

**第三件事：异常与容错。**

说实话，我刚入行时几乎不测异常场景——觉得服务挂了就是后端的事，跟我有什么关系？

直到有一次，短信服务超时，整个注册流程直接挂了——用户看到的不是「短信验证码发送失败，请稍后重试」，而是一个白屏。

产品的投诉电话打了一下午。

从那以后我才意识到：**测试不只是验证功能对不对，更是验证功能「挂得优不优雅」。**

—

## 💡 我的方法：七维度扫描法

后来我摸索出一个比较系统的方法——给每个功能按七个维度扫一遍。

七个维度听起来挺多，但其实就是回答七个问题：

1. **正常走通了吗？**（正向功能）
2. **用户瞎输入会怎样？**（反向校验）
3. **不同角色的表现一样吗？**（权限）
4. **改了一个选项，关联的东西变了吗？**（联动）
5. **到了上限还能继续吗？**（边界）
6. **按钮、文案、跳转都对了没？**（UI 交互）
7. **服务挂了的时候，用户看到什么？**（异常容错）

你可能会觉得——七个维度，每个写 3 条，一个功能就要 21 条，一个系统几百个功能……

其实不用每个功能都七维度全写。

我现在一般是 P0 功能七个维度全覆盖，P1 功能覆盖前四个维度，P2/P3 功能走一遍正向就行。重点不是数量，是**不要每次都从同一个习惯的视角看问题**。

—

## 🔥 写用例的几个铁律

说几个我这些年踩坑踩出来的经验。

**铁律一：一条用例只验证一个测试逻辑。**

这个看上去简单，但做起来特别容易犯规。

我见过太多人（包括以前的我）把必填校验、格式校验、边界值校验全塞进一条用例。写的时候觉得省事，等这条用例失败了你就傻眼了——到底是哪个校验出的问题？

不同逻辑必须拆开，同逻辑不同数据可以用参数化。

**铁律二：preconditions 和 steps 不要搞混。**

preconditions 放的是「为了让测试能开始而必须做的事」——打开页面、登录、准备数据。

steps 放的是「测试过程中有验证价值的操作」。

打开页面不是测试的核心行为，把它扔到 preconditions 里，让你的 steps 干净一点。

**铁律三：预期结果要精确到可以写断言。**

"系统正常运行"不是一个预期结果。你得写清楚——页面提示了什么文案？跳转到哪个页面？数据以什么格式返回？

这些细节在你手工测试的时候可能觉得无所谓（反正眼睛能看到），但一旦要做自动化，模糊的预期结果就是灾难。

—

下面是一段 pytest 代码，展示一下精确断言长啥样：

```python
def test_register_success():
    phone = "13800138000"
    password = "Test@123456"

    response = register_api(phone, password)

    # 注意：断言的不是"成功"，而是具体的字段值
    assert response.status_code == 200
    assert response.json()["code"] == 0
    assert "token" in response.json()["data"]
```

看到区别了吗？

不是 `assert response is success`，而是精确到 `status_code == 200`、`code == 0`、返回体里有 `token`。

这三个断言，随便哪一个失败了你都能立刻知道问题出在哪 🎯

—

## 写在最后

说到底，写测试用例这件事跟写代码一样——是需要「品味」的。

品味不是你写的有多快、用了多少高大上的工具。而是你知道哪些值得测、哪些可以放一放；你知道每条用例必须独立、必须可验证；你知道覆盖率的百分比不重要，覆盖了哪些场景才重要。

这些东西，书本上不太会教你，只能靠踩坑和复盘慢慢攒。

但好在这个过程并不漫长——多经历几次线上事故，你很快就明白了 😂
