## 前言

**测试用例**（Test Case）是测试工程师的核心产出物。很多人认为编写测试用例不过是把操作步骤罗列出来，但真正高质量的测试用例，背后是深入的需求分析和系统的测试点提取。

本文分享一套经过实践验证的方法论，帮助你写出覆盖全面、结构清晰、可执行的测试用例。

## 理解需求：不要急着写用例

拿到需求之后，第一步不是打开用例模板，而是回答三个问题：

- 这个功能的**核心价值**是什么？
- 有哪些**用户角色**会用到？
- 涉及哪些**数据流转**？

以"用户注册"功能为例。表面上看很简单——填表、提交、注册成功。但如果深入分析，你会发现需要考虑：

- 手机号是否已注册？
- 密码强度规则是什么？
- 注册后是否需要邮箱验证？
- 不同渠道（App / H5 / 小程序）的注册流程是否一致？

这些都是在需求理解阶段必须明确的。

> **注意**：需求不明确的地方不要做假设，列出问题清单向产品确认。

## 提取测试点：七维度扫描法

我常用的方法是**7 维度扫描法**，对每个功能按以下维度逐一扫描：

### 维度一：正向功能

验证核心业务流程的正常路径。每个用户角色至少 1 条 Happy Path。

### 维度二：反向校验

每个输入字段覆盖四类输入：有效值、空值、格式错误、边界值。

### 维度三：权限与角色

不同用户类型看到的内容和可用操作可能不同，需要按角色 × 功能点建立矩阵。

### 维度四：状态联动

当某个选项变化时，检查关联选项是否联动变化。

### 维度五：边界与限制

覆盖数量上限、长度限制、重复校验等场景。

### 维度六：UI 交互

按钮状态、提示文案、页面跳转、加载状态等前端表现。

### 维度七：异常与容错

网络断开、服务端 5xx、并发冲突等异常场景。

每个维度至少输出 2-3 个测试点，保证覆盖面。

## 编写用例的规范

### 基本格式

用例 `title` 要简洁有力，动宾结构。`steps` 用祈使句，每步一个操作。最重要的是 **expected_results 和 steps 必须一一对应**，实现一步一验。

以下是一个标准的用例结构：

```
id: TC_USER_REGISTER_001
title: 正确填写注册信息并提交
priority: P0
type: functional
steps:
  - 打开注册页面
  - 输入有效手机号
  - 输入符合规则的密码
  - 点击注册按钮
expected_results:
  - 页面显示注册表单
  - 手机号输入框显示已输入内容
  - 密码以掩码形式显示
  - 提示注册成功并跳转至登录页
```

### 一条用例只验证一个测试逻辑

这是新手最容易犯的错误：把必填校验、格式校验、边界值校验全塞进一条用例。这样的用例一旦失败，你很难定位是哪个校验出了问题。

**正确做法**：每个测试逻辑独立成一条用例。同一逻辑不同数据可合并（参数化）；不同逻辑必须拆分。

### preconditions 与 steps 的区分

`preconditions` 放预备动作（打开页面、登录等），`steps` 放有验证价值的具体操作。不要在 steps 里塞入纯预备动作。

## Python 自动化示例

对于可自动化的用例，推荐使用 `pytest` 框架编写：

```python
def test_register_success():
    # 准备测试数据
    phone = "13800138000"
    password = "Test@123456"

    # 调用注册接口
    response = register_api(phone, password)

    # 验证结果
    assert response.status_code == 200
    assert response.json()["code"] == 0
    assert "token" in response.json()["data"]
```

测试数据的构造和断言是自动化用例的两个关键点。**数据要独立于环境**（避免硬编码依赖特定数据库记录），**断言要精确**（验证具体字段值而非仅判断成功/失败）。

## 总结

- **先理解需求，再提取测试点，最后写用例**——这个顺序不能乱
- **7 维度扫描法**是系统化提取测试点的有效工具
- **一条用例一个测试逻辑**是保障用例质量的铁律
- **steps 与 expected_results 一一对应**，一步一验
- 自动化用例关注**数据独立性**和**断言精确性**
