## 目标

完成本教程后，你将掌握：

- 用 7 维度扫描法系统化提取测试点
- 编写符合规范的 YAML 格式测试用例
- 用 pytest 编写可复用的自动化测试脚本

## 前置条件

- 了解软件测试的基本概念
- 熟悉一种编程语言（推荐 Python）
- 已安装 Python 3.8+ 和 pytest

## 环境准备

### Step 1: 安装 pytest

执行以下命令安装 pytest：

```bash
pip install pytest
```

预期输出：

```
Successfully installed pytest-7.4.0 ...
```

> **✅ CHECK**：执行 `pytest --version` 确认安装成功。

## 核心步骤

### Step 2: 理解需求

**目标**：明确功能的核心价值、用户角色和数据流转。

**操作**：

1. 阅读需求文档，标注关键信息
2. 识别所有涉及的**用户角色**
3. 梳理完整的数据链路（前端 → API → 后端 → 数据库）

> **💡 NOTE**：以"用户注册"为例，表面看很简单，但实际上要考虑手机号是否已注册、密码强度规则、是否需要邮箱验证、不同渠道（App / H5 / 小程序）的注册流程是否一致等问题。

### Step 3: 提取测试点

**目标**：用 7 维度扫描法系统化提取所有可测试点。

**操作**：

1. 按以下 7 个维度逐一扫描目标功能
2. 每个维度至少输出 2-3 个测试点
3. 对照常见缺陷模式补充防御性测试点

7 个维度如下：

- 正向功能：正常路径的 Happy Path
- 反向校验：每个输入字段的有效值、空值、格式错误、边界值
- 权限/角色：不同用户看到的内容差异
- 状态联动：选项之间的联动关系
- 边界与限制：数量上限、长度限制、重复校验
- UI 交互：按钮状态、提示文案、加载状态
- 异常与容错：网络断开、服务端 5xx、并发冲突

> **📌 TIP**：用 Excel 或 Notion 建一个「字段级测试矩阵」，以字段为行、校验类型为列，逐格填入测试值和预期结果，确保无遗漏。

### Step 4: 编写测试用例

**目标**：将测试点转化为结构化的 YAML 用例。

**操作**：

1. 确定用例 ID：`TC_{模块}_{功能}_{三位序号}`
2. 编写 title：动宾结构，不超过 40 字符
3. 列出 steps：祈使句，每步一个操作
4. 编写 expected_results：与 steps 一一对应，一步一验

标准用例格式如下：

```
id: TC_USER_REGISTER_001
title: 正确填写注册信息并提交
priority: P0
type: functional
steps:
  - 打开注册页面
  - 输入有效手机号
  - 输入符合规则的密码
  - 点击注册按钮
expected_results:
  - 页面显示注册表单
  - 手机号输入框显示已输入内容
  - 密码以掩码形式显示
  - 提示注册成功并跳转至登录页
```

> **⚠️ WARNING**：最常见的错误是把多条校验逻辑塞进一个用例。比如既验证必填为空又验证格式错误——用例失败后难以定位根因。记住：**一条用例只验证一个测试逻辑**。

> **⚠️ WARNING**：`preconditions` 和 `steps` 容易混淆。打开页面、登录等预备动作应放在 `preconditions`，不要放在 `steps` 里。

### Step 5: 编写自动化脚本

**目标**：用 pytest 将测试用例转化为可重复执行的自动化脚本。

**操作**：

1. 创建测试文件 `test_register.py`
2. 编写测试函数，构造测试数据
3. 调用接口并断言结果

示例代码：

```python
def test_register_success():
    # 准备测试数据
    phone = "13800138000"
    password = "Test@123456"

    # 调用注册接口
    response = register_api(phone, password)

    # 验证结果
    assert response.status_code == 200
    assert response.json()["code"] == 0
    assert "token" in response.json()["data"]
```

> **📌 TIP**：测试数据应独立于环境——使用专门的测试账号或动态生成数据，避免依赖特定数据库记录导致用例不稳定。

> **📌 TIP**：断言要精确——验证具体的字段值（如 `response.json()["code"] == 0`），而不是仅判断是否成功。精确的断言能帮你更快定位问题。

## 验证与测试

> **✅ CHECK**：运行 `pytest test_register.py -v` 确认所有用例通过。

## 常见问题 (FAQ)

- **Q: 测试点太多，怎么写得完？**
  A: 按优先级分批。P0 先写（核心路径），P1 补充（重要校验），P2/P3 用探索式测试覆盖。

- **Q: 每条用例必须对应一条自动化脚本吗？**
  A: 不必。优先自动化高频回归的 P0/P1 用例，UI 交互类用例手工执行效率更高。
