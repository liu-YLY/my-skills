# 测试用例编写实战指南

> **🎯 完成本教程后，你将能够：**
>
> - **系统化提取测试点**：使用 7 维度扫描法对任意功能模块进行全面的测试点分析
> - **编写规范的测试用例**：按照标准化格式编写结构清晰、可执行、可验证的用例
> - **实现自动化测试**：使用 pytest 将核心用例转化为可复用的自动化脚本
>
> ⏱️ **预计耗时**：20 分钟  |  🎚️ **难度**：入门

---

## 前置条件

开始之前，请确保你已具备：

- 软件测试的基本概念（什么是测试用例、正向/反向测试）
- 基础的 Python 语法知识（变量、函数、列表）
- 已安装 Python 3.8 及以上版本

> **💡 提示**：如果尚未安装 pytest，本教程 Step 1 会带你完成安装。

---

## 阶段一：环境准备

### Step 1 — 安装 pytest

**🎯 这一步的目标**：在你的开发环境中安装 pytest 测试框架。

**操作步骤**：

1. 打开终端（Mac/Linux）或命令提示符（Windows）
2. 执行以下命令安装 pytest：

```bash
pip install pytest
```

3. 等待安装完成

**✅ 预期结果**：终端输出类似以下内容，提示安装成功。

```
Successfully installed pytest-7.4.0 iniconfig-2.0.0 pluggy-1.2.0
```

> **✅ CHECK ｜ 阶段一验证**
>
> 执行 `pytest --version`，确认输出 `pytest 7.4.0`（或更高版本）。

---

## 阶段二：理解需求与分析测试点

### Step 2 — 理解需求

**🎯 这一步的目标**：在动手写用例之前，先明确功能的边界和隐含的业务规则。

**操作步骤**：

1. 拿到需求文档后，用荧光笔标注所有**业务规则**（如"手机号不能重复"）
2. 列出所有涉及的**用户角色**（如普通用户、管理员、未登录用户）
3. 画出数据流转路径：前端 → API 网关 → 业务服务 → 数据库 → 外部服务

> **💡 NOTE ｜ 补充说明**
>
> 以「用户注册」功能为例，表面看需求只有一句话——填写手机号和密码完成注册。但深入分析后你会发现至少要考虑：手机号唯一性、密码强度规则、是否需短信验证、不同渠道（App/H5/小程序）的流程一致性。这些「隐藏需求」恰恰是 Bug 的高发区。

> **⚠️ WARNING ｜ 常见陷阱**
>
> 需求不明确的地方千万不要做假设。比如「密码长度应该是 6-20 位」如果文档里没写，就列到问题清单里向产品确认。我见过不止一次因为需求理解偏差导致整批用例返工的情况。

---

### Step 3 — 用 7 维度扫描法提取测试点

**🎯 这一步的目标**：系统化地提取测试点，确保覆盖全面，不遗漏关键场景。

**操作步骤**：

1. 创建一个表格或 Notion 页面，列出以下 7 个维度
2. 对每个维度逐一思考「这个维度下有哪些可测试的场景」
3. 每个维度至少输出 2-3 个测试点
4. 对照常见缺陷模式（空值、并发、超时）补充防御性测试点

7 个维度的详细说明：

| 维度 | 关注点 | 示例（注册功能） |
|------|--------|----------------|
| 1. 正向功能 | Happy Path 完整链路 | 正确填写信息 → 注册成功 |
| 2. 反向校验 | 空值、格式错误、边界值 | 手机号少 1 位、密码为空 |
| 3. 权限/角色 | 不同角色看到什么 | 已登录用户访问注册页 |
| 4. 状态联动 | 选项变化触发关联变化 | 选择国家 → 区号自动切换 |
| 5. 边界/限制 | 上限、下限、重复提交 | 连续点击注册按钮 3 次 |
| 6. UI 交互 | 按钮状态、文案、跳转 | 注册按钮防重复点击 |
| 7. 异常/容错 | 网络断开、服务端错误 | 短信服务超时时的表现 |

> **📌 TIP ｜ 效率技巧**
>
> 用 Excel 或 Notion 建立「字段级测试矩阵」：以字段为行（手机号、密码、验证码...），以校验类型为列（有效值、空值、格式错误、边界值...），逐格填入测试值和预期结果。这个方法虽然看起来笨，但能保证你永不遗漏字段级校验。

## 阶段三：编写测试用例

### Step 4 — 编写标准化用例

**🎯 这一步的目标**：将测试点转化为结构清晰、可独立执行的测试用例。

**操作步骤**：

1. 确定用例 ID：推荐格式 `TC_{模块}_{功能}_{三位序号}`，如 `TC_REGISTER_POSITIVE_001`
2. 编写 title：动宾结构，不超过 40 字符，如「正确填写注册信息并提交」
3. 列出 preconditions：打开页面、准备数据等前置操作
4. 编写 steps：祈使句，每步一个操作，不多不少
5. 编写 expected_results：与 steps **一一对应**，一步一验证

完整的标准用例示例：

```
id: TC_USER_REGISTER_001
title: 正确填写注册信息并提交
priority: P0
type: functional
preconditions:
  - 手机号 13800138000 未注册
  - 短信验证码服务可用
steps:
  - 打开注册页面
  - 输入有效手机号 13800138000
  - 输入符合规则的密码 Test@123456
  - 点击「注册」按钮
expected_results:
  - 注册页面正常加载，各输入框可交互
  - 手机号输入框正确显示已输入内容
  - 密码字段以掩码形式（●）显示
  - 提示「注册成功」并自动跳转至登录页面
```

> **⚠️ WARNING ｜ 常见陷阱**
>
> 最常见的错误：把多条校验逻辑塞进一条用例。比如在一条用例里同时验证「手机号为空」「密码格式错误」「验证码过期」——用例失败后你根本无法定位是哪个校验触发的。**记住铁律：一条用例只验证一个测试逻辑。**

> **📌 TIP ｜ 效率技巧**
>
> 写用例时优先覆盖 P0（核心路径）和 P1（重要校验），这两层覆盖了 80% 以上的线上缺陷。P2 和 P3 级别的用例可以利用探索式测试来补充，不必追求 100% 用例覆盖率。

## 阶段四：自动化实现

### Step 5 — 用 pytest 编写自动化脚本

**🎯 这一步的目标**：将核心用例转化为可重复执行的 pytest 自动化脚本。

**操作步骤**：

1. 在项目目录下创建 `test_user_register.py`
2. 编写测试类 `TestUserRegister`，按测试逻辑组织测试方法
3. 每个测试方法对应一条手工用例
4. 使用 `@pytest.mark.parametrize` 对同一逻辑的不同数据进行参数化

完整代码示例：

```python
import pytest


class TestUserRegister:
    """用户注册接口 — 自动化测试"""

    def test_register_success(self, api_client):
        """正向用例：正确填写注册信息并提交"""
        phone = "13800138000"
        password = "Test@123456"

        response = api_client.register(phone, password)

        assert response.status_code == 200
        assert response.json()["code"] == 0
        assert "token" in response.json()["data"]

    @pytest.mark.parametrize("phone,password,expected_msg", [
        ("", "Test@123456", "手机号不能为空"),
        ("13800138000", "", "密码不能为空"),
        ("1380013800", "Test@123456", "手机号格式错误"),
        ("13800138000", "12345", "密码长度不能少于8位"),
    ])
    def test_register_field_validation(
        self, api_client, phone, password, expected_msg
    ):
        """反向校验：空值、格式错误、边界值"""
        response = api_client.register(phone, password)

        assert response.json()["code"] != 0
        assert expected_msg in response.json()["message"]
```

> **📌 TIP ｜ 效率技巧**
>
> 测试数据应当独立于环境。不要硬编码依赖特定数据库记录的数据——使用专门的测试账号、动态生成唯一手机号，或用 fixture 在每次测试前插入所需数据。这样你的用例在任何环境（本地/CI/预发）都能稳定运行。

## 常见问题 (FAQ)

**Q1：测试点写得太多，根本执行不完怎么办？**

A：按优先级分批执行。P0 先写先测（核心路径，每次发版必跑），P1 补充（重要校验，回归时跑），P2/P3 用探索式测试覆盖。另外，同逻辑不同数据的用例用参数化合并，可以显著减少用例条数。

**Q2：每条手工用例都必须对应一条自动化吗？**

A：不必。优先自动化**高频回归**的 P0/P1 用例。UI 交互类（如按钮颜色、动画效果）和一次性的边界探索用手工执行效率更高。

**Q3：preconditions 和 steps 到底怎么区分？**

A：preconditions 是「为了让测试能开始而必须做的事」，steps 是「测试过程中有验证价值的操作」。打开页面是 precondition，在页面上输入数据并验证是 step。
