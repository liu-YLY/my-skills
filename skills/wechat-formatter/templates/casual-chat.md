# 轻松聊天风格排版规则

> **何时阅读**：用户选择 `casual-chat` 风格时必读。
> **覆盖范围**：口语化表达、Emoji 规范、对话感营造、整体结构模板。
> **可跳过条件**：未选择此风格时跳过。

---

## 风格定位

亲和、有趣、易读。像技术朋友在咖啡馆聊天，轻松但不轻浮，有趣但不油腻。

---

## 排版规则

### 1. 标题规范

- 标题尽量口语化、有趣味性，但表意清晰
- 使用 `##` 和 `###`，不使用 `#`
- 允许标题中使用 Emoji，如 `## 🎯 我是怎么写测试用例的`

```markdown
✅ 好的标题：
## 🤔 为什么你的代码总是 Bug 不断？
## 一个测试工程师的日常
## 那些年我踩过的坑

❌ 不好的标题（太正式）：
## 测试用例编写方法论
## 软件开发缺陷分析报告
```

### 2. 段落与节奏

- **短段落**：每段 1-3 句话，大量留白
- **对话感**：多用"你"、"我"、"咱们"，营造聊天氛围
- **语气词**：适当使用"说实话"、"你懂的"、"对吧"、"其实"
- **分段逻辑**：一个观点 = 一段，说完就换行
- 重要观点可以单独成段，用 `**粗体**` 标出

### 3. Emoji 使用规范

**允许且鼓励使用 Emoji**，但遵循以下原则：

| 用途 | 推荐 Emoji | 示例 |
|------|-----------|------|
| 章节标题点缀 | 🎯 💡 🔥 ⚡ 🚀 | `## 🚀 开始动手吧` |
| 列表项前缀 | ✅ ❌ ⚠️ 💡 | `✅ 这样做是对的` |
| 情绪表达 | 😂 🤔 😅 💪 🙈 | 段落中穿插 |
| 分隔/停顿 | — | `—` 用于段落间分隔 |
| 关键提示 | ⭐ | `⭐ 重点来了` |

**限制**：
- 每段不超过 2 个 Emoji
- 不在代码块附近使用 Emoji
- 不在严肃结论旁使用搞笑 Emoji

### 4. 口语化转换表

| 正式表达 | 口语化转换 |
|---------|-----------|
| 本文将会介绍... | 今天咱们聊聊... |
| 首先需要配置... | 第一步，先把...装上 |
| 需要注意的是... | 这里有个坑，千万别踩！ |
| 总结一下 | 说到底，其实就是... |
| 推荐使用... | 我个人比较推荐... |
| 根据经验... | 说实话，做了这么多年... |

### 5. 代码块格式

代码块保持专业，但前后说明语口语化：

```markdown
你可能会写出这样的代码（别问我怎么知道的 😅）：

```python
# 这种写法虽然能跑，但...
result = []
for i in range(100):
    result.append(i * 2)
```

其实一行就够了：

```python
result = [i * 2 for i in range(100)]
```

是不是清爽多了？
```

### 6. 整体结构模板

```
# [口语化标题]

## 开场白
[像聊天一样切入话题，1-2 段]

—

## 核心分享
[主要内容，用轻松的风格呈现]

—

## 我的一些想法
[个人观点/经验/建议]

—

## 随便聊聊
[轻松的收尾，邀请互动]

**你在工作中遇到过类似的情况吗？欢迎在评论区聊聊 👇**
```

**结构说明**：
- 用 `—` 作为章节分隔符（比 `---` 更轻量）
- 不强制要求「总结」章节，用「想法」替代
- 结尾必须有互动引导（评论、点赞、在看）

- [ ] Emoji 数量合理（全篇 8-15 个），分布均匀
- [ ] 代码块讲解口语化
- [ ] 用 `—` 分隔章节
- [ ] 结尾有互动引导
- [ ] 全文语气一致，不生硬
